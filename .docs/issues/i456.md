# I456 — In-App Markdown Reader for Entity Documents

**Status:** Open
**Priority:** P2
**Version:** 1.1.0
**Area:** Frontend / UX

## Summary

Currently, entity documents (meeting notes, playbooks, case studies, Drive imports, etc.) stored in `Accounts/{id}/Documents/`, `Projects/{id}/Documents/`, and `People/{id}/Documents/` are inaccessible within the app. Clicking a document reference in the file list or timeline opens Finder, breaking the user's flow. A read-only in-app markdown renderer makes entity context immediately visible without leaving DailyOS.

**Scope for v1.1.0 (MVP):** Read-only markdown rendering with basic syntax highlighting. Files viewed in a modal or sidebar drawer. Simple navigation by entity.

**Future (v1.2.0+):** Inline editing, markdown preview mode, search/jump to headings, export to PDF, inline comments.

## Acceptance Criteria

### File Access Points

1. **File list in entity detail page** — Account/project/person detail pages show a Documents section. Clicking a document opens the markdown reader modal. Verify: Click document in account detail → modal opens with file content.

2. **Timeline references** — When a summary, action, or outcome references a source document (via `source_label` or frontmatter), clicking the reference opens that document in the reader. Verify: Click "See Acme Case Study" in a meeting prep → reader opens to that file.

3. **Drive import notifications** — When a Drive document is imported and available in the entity's Documents/, user can click "View" button in the toast or in-app notification to open the reader.

### Markdown Rendering

4. **Basic markdown support** — H1–H6 headings, bold/italic, lists (ordered/unordered), blockquotes, code blocks, links, tables. Verify: Import a markdown document with mixed formatting; all elements render correctly.

5. **Syntax highlighting** — Code blocks use language-aware syntax highlighting (JavaScript, Python, SQL, etc. detected from code fence). Dark/light theme support via design tokens. Verify: Code block renders with appropriate colors for the active theme.

6. **Images and embeds** — Embedded images (e.g., from Drive PDF conversion or user-uploaded images) render inline. Verify: Document with images renders images in-place.

### UI & Navigation

7. **Modal/drawer layout** — Reader is a full-height modal or right-slide drawer (consistent with existing detail modals in DailyOS). Close button top-right. Verify: Modal closes on close button click or Escape key.

8. **File metadata header** — Top of reader shows: file name, source (e.g., "Google Drive" or "Manually added"), entity breadcrumb, last modified date. Verify: Header displays correct metadata.

9. **Scrolling & TOC (optional for MVP)** — For documents > 2000 characters, offer an optional sidebar table-of-contents extracted from headings. Jump to heading on click. Verify: Click heading in TOC → scroll jumps to that heading.

10. **Read-only enforcement** — No inline editing UI. Content is strictly read-only. Verify: No cursor changes, no edit affordances.

### Content Loading

11. **Lazy loading** — Large files (> 10MB) load content on first open. Don't fetch during list render. Verify: Open a large document; content loads within 1 second.

12. **Error handling** — If file is missing or unreadable, show a friendly message: "Unable to load document. File may have been moved or deleted." Verify: Delete a file from Documents/, try to open it in reader → error message shown.

### Integration Points

13. **Entity context preservation** — Reader knows which entity owns the document (passed via props or route param). Links within the document to other entities work intelligently (e.g., "@Acme" links to Acme entity page if reader is open from an Acme doc). Verify: Open a doc from an account, click an internal entity link → navigates to that entity.

14. **`cargo test` passes** — No regressions on existing components. Reader component has unit tests for markdown parsing edge cases.

## Implementation Details

### Component Structure

**New component:** `src/components/document-reader/DocumentReaderModal.tsx`

- Props: `documentPath: string`, `entityId: string`, `entityType: EntityType`, `onClose: () => void`
- Internal state: document content (fetched via Tauri command), parsed markdown AST
- Rendering: Use existing markdown parser (`remark`/`unified` stack if available, or `marked` as fallback)

**Backend command:** `read_entity_document(entity_id, entity_type, filename) → { content: string, metadata: {...} }`

- Validates path stays within the entity's Documents/ directory (prevent directory traversal)
- Reads file from workspace
- Returns raw content (parsing happens in React)

### Markdown Parsing

**Library:** `remark` + `rehype` for AST-to-React conversion. Falls back to `marked` if not available.

```tsx
import { unified } from "unified";
import remarkParse from "remark-parse";
import remarkRehype from "remark-rehype";
import rehypeReact from "rehype-react";

const processor = unified()
  .use(remarkParse)
  .use(remarkRehype)
  .use(rehypeReact, { components: { /* custom components */ } });

const ast = processor.parse(content);
const rendered = processor.runSync(ast);
```

### Styling

- Use design tokens for typography, spacing, colors
- Code blocks: Use `highlight.js` or `shiki` for syntax highlighting
- Tables: Render as styled divs (not HTML tables for better control)
- Links: Use entity-aware link component (checks if link target is a DailyOS entity)

### Tauri Integration

**Backend command in commands.rs:**

```rust
#[command]
pub async fn read_entity_document(
    entity_id: String,
    entity_type: String,
    filename: String,
    state: State<'_, AppState>,
) -> Result<DocumentContent, String> {
    let workspace = state.get_workspace_path()?;
    let doc_path = workspace
        .join(format!("{}s", entity_type))  // "Accounts", "Projects", "People"
        .join(&entity_id)
        .join("Documents")
        .join(&filename);

    // Validate path (prevent directory traversal)
    if !doc_path.starts_with(&workspace) {
        return Err("Invalid path".to_string());
    }

    let content = std::fs::read_to_string(&doc_path)
        .map_err(|e| format!("Failed to read document: {}", e))?;

    Ok(DocumentContent {
        content,
        filename,
        modified_at: /* file mtime */,
    })
}
```

### Design Consistency

- Reader uses existing editorial typography (Newsreader for headings, DM Sans for body)
- Spacing follows design-tokens.css grid
- Colors use semantic tokens (text-primary, text-secondary, etc.)
- Consistent with magazine aesthetic (margins, rules, no over-decoration)

## Why Post-1.0.0?

1. **Non-blocking for MVP.** Intelligence and entity context work without reader. Files exist; users can open in external editors if needed.
2. **Enhances Drive connector value (v0.14.3).** Users won't see Drive imports if they can't view them in-app. Reader makes imports truly useful.
3. **Design & testing debt.** Markdown parsing, code highlighting, and edge cases (large files, broken links, image loading) need careful testing.
4. **Clear scope.** MVP is read-only. Avoids conflicts with other v1.0.0 priorities (search, offline, export).

## Coordination

- **I436 (Workspace File Deprecation)** removes the workspace filesystem as a storage layer. I456's `read_entity_document` reads from `{workspace}/Accounts/{id}/Documents/`. If I436 lands first, documents may be archived to `~/.dailyos/workspace-archive/` and the reader's backend command needs to adapt its read path accordingly. If I456 ships first (more likely given version ordering), the reader works against the existing workspace. Revisit after I436 lands.

## Notes

**Why 1.1.0 instead of 1.0.0?** Documents are supplementary context. Intelligence, briefings, and action tracking work without them. Post-GA, this reader turns Drive imports (v0.14.3) from "documents stored somewhere" into "documents visible and useful."

**Read-only in v1.1.0 is intentional.** Editing adds complexity (conflict detection, undo, version control). Ship reader first, prove the UX, add editing in v1.2.0.

**Entity breadcrumb matters.** Opens from different contexts: file list, timeline, search results, Drive notification. Breadcrumb keeps the user oriented about which entity owns the doc.

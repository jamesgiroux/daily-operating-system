# I651 — Product Classification from Glean (Bundled into Financial Dimension)

**Status:** Spec Ready (Phase 6)

**Thesis:** DailyOS can identify product subscription types (CMS, Analytics) and tier levels from Glean Salesforce data via the Financial dimension `chat` call. This classification enriches meeting prep and intelligence context so CSMs walk in knowing exactly what the customer has bought, without building product inventory UI.

Different from I624: I624 discovers product *names* from transcripts and Glean. I651 classifies *subscription type and tier* from explicit Salesforce fields. Both feed the Intelligence Loop but serve different purposes.

**Architecture:** Extends the existing Financial dimension Glean MCP `chat` call to request product classification fields. Single response includes both financial intelligence and product data. Parse response to extract product JSON and upsert to `account_products` table.

---

## Problem

A CSM sits down for a meeting with Globex Holdings. The dossier shows ARR and renewal status. But there's no place to see: "This customer has **CMS Premier at $60k/year** and **Analytics Standard at $36k/year**." The CSM has to guess or dig into Salesforce themselves.

Glean already returns these fields (`Estimated CMS ARR`, `Parsely Customer`, `Parsely Premier` flags). We have the data. We just need a place to store the classification, and inject it into meeting prep context so the prompt can say: "Expand their analytics tier, which is currently underutilized."

---

## Solution

1. **Extend Financial dimension Glean MCP `chat` call to request product classification.**
   - Modify the prompt in `dimension_prompts.rs` (Financial) to include request for:
     - CMS subscription status (`Customer Account Subscription Status`)
     - CMS tier (`Support Package`)
     - CMS ARR (`Estimated ARR` or `Estimated CMS ARR`)
     - Analytics subscription status (`Parsely Customer` flag)
     - Analytics tier (`Parsely Premier` flag)
     - Billing terms (`Billing Terms`)
   - Single `chat` call returns both financial intelligence AND product classification JSON
   - Runs as part of existing parallel dimension processing (no new async task)

2. **Parse Financial dimension response to extract product classification.**
   - Response includes structured section: `"product_classification": { "products": [...] }`
   - Extract and deserialize to `Vec<AccountProduct>`
   - Upsert to `account_products` table immediately after dimension merge
   - Emit signal (best-effort; if parse fails, log and continue)

3. **Store in `account_products` table (extend I624's schema).**
   - Add `product_type: enum [cms, analytics]`
   - Add `tier: enum [premier, standard, basic, unknown]` (nullable)
   - Add `billing_terms: text` (Annual, Monthly, Multi-Year)
   - Add `arr: real` for annual recurring revenue
   - Upsert on each Glean refresh (one row per product type per account)

4. **Inject into meeting prep and intelligence context.**
   - `gather_account_context()` includes: `products: [{type: "cms", tier: "enhanced", arr: 185400, billing_terms: "annual"}, ...]`
   - `build_intelligence_context()` includes product data for downstream prompts
   - Meeting prep prompt can now say: "Customer has CMS Enhanced at $185.4k/year (annual). No current Analytics contract."

5. **Extraction rules (Salesforce-only, via Glean `chat`).**

| Glean Field | Product Type | Tier Field | ARR Field | Notes |
|---|---|---|---|---|
| `Customer Account Subscription Status` contains "CMS" | CMS | `Support Package` (Enhanced, Signature, Standard, Premier) | `Estimated ARR` or `Estimated CMS ARR` | Current subscription. Renewal/active opportunities confirm currency. |
| `Parsely Customer` = true | ANALYTICS | `Parsely Premier` (true=Premier, false=Standard) | Account-level field or opportunity `analyticsACV` if current contract exists | Current subscription. Open opportunities are expansion (separate enrichment). |
| Neither condition met | — | — | — | No row created for that product type. |

   - **Current vs. Expansion:** This spec extracts *current subscriptions* only. Open opportunities (status "Solution Development", "Proposal") are expansion signals, tracked in separate enrichment (Outlook chapter, expansion intelligence). I651 filters by current subscription status.
   - **Upsert on refresh:** Each Glean refresh upserts product records. If CMS tier changes from Enhanced to Signature and ARR drops, existing row updates. `last_verified_at` tracks when Salesforce was last checked.
   - **Partial data:** If only CMS is current and Analytics is not (`Parsely Customer=false`), only CMS row is stored. No "UNKNOWN" rows.
   - **Example (Globex Holdings):** `Customer Account Subscription Status = "CMS Customer"`, `Support Package = "Enhanced"`, `Estimated ARR = 185400`. Result: one row (CMS, Enhanced, 185400). Parse.ly opportunity with $36k `analyticsACV` is open/expansion → not included in I651, handled by expansion enrichment elsewhere.

## Acceptance Criteria

1. **Schema & migration.**
   - Extend `account_products` table (from I624) with columns:
     - `product_type: text` (cms, analytics)
     - `tier: text` (nullable; Enhanced, Signature, Standard, Premier)
     - `billing_terms: text` (nullable; Annual, Monthly, Multi-Year, etc.) — from Salesforce `Billing Terms` field
     - `data_source: text` enum (salesforce; future: ai_inference, user_correction)
   - Add indexes on `(account_id, last_verified_at)` for fresh-product queries
   - `last_verified_at` timestamp tracks when Salesforce data was last seen for this product

2. **Financial dimension `chat` call modification.**
   - Modify the Financial dimension prompt in `intelligence/dimension_prompts.rs` to request product classification fields alongside existing financial intelligence
   - Prompt asks Glean `chat` to return JSON with two sections:
     ```
     {
       "financial_intelligence": { /* existing financial insights */ },
       "product_classification": {
         "products": [
           {"type": "cms", "tier": "enhanced", "arr": 185400, "billing_terms": "annual"},
           {"type": "analytics", "tier": "standard", "arr": null, "billing_terms": null}
         ]
       }
     }
     ```
   - Single `chat` call returns both → no new API calls, no new async tasks

3. **Parse and store product classification from Financial response.**
   - In `glean_provider.rs` after merging Financial dimension result:
     - Extract `response["product_classification"]["products"]`
     - For each product: create `AccountProduct { type, tier, arr, billing_terms, data_source: "salesforce", last_verified_at: NOW() }`
     - Upsert to `account_products` table (one row per product type per account)
     - If parse fails: log error and continue (Financial intelligence still merges successfully)
   - Example (Globex Holdings Glean response): Two products extracted and upserted: (CMS, Enhanced, 185400, Annual) and nothing for Analytics (Parsely Customer=false)

4. **Test fixtures from real Glean output.**
   - Create `tests/fixtures/glean_globex_product_sample.json` with actual Salesforce fields returned by Glean:
     - Account level: `Estimated ARR`, `Customer Account Subscription Status`, `Parsely Customer`, `Parsely Premier`, `Support Package`
     - Opportunities: `opportunityType`, `cmsACV`, `analyticsACV`, `stage`
   - Fixture for Globex Holdings: `Estimated ARR: 185400`, `Customer Account Subscription Status: "CMS Customer"`, `Support Package: "Enhanced"`, `Parsely Customer: false`, `Parsely Premier: false`
   - Unit test: `test_classify_from_globex_fixture()` validates extraction → (CMS, Enhanced, 185400) with no Analytics row
   - Edge case tests:
     - Missing `Support Package` (gracefully handle)
     - `Parsely Customer: true` with `Parsely Premier: true` → create Analytics row with Premier tier
     - `Parsely Customer: true` with `Parsely Premier: false` → create Analytics row with Standard tier
     - Open Parse.ly opportunity → not extracted by I651 (expansion signal handled separately)

5. **Context inclusion.**
   - `gather_account_context()` queries latest products via `SELECT * FROM account_products WHERE account_id = X ORDER BY last_verified_at DESC`
   - Serializes to JSON: `{products: [{type: "cms", tier: "enhanced", arr: 185400, billing_terms: "annual"}, {type: "analytics", tier: "premier", arr: 36000, billing_terms: "annual"}]}`
   - `build_intelligence_context()` includes same product data in enrichment prompts
   - Meeting prep prompt and intelligence prompts can reference: "Customer has CMS Enhanced at $185.4k/year (annual billing) — growth opportunity: evaluate Analytics tier or expansion"

6. **Type safety.**
   - Define Rust enums with serde derives:
     ```rust
     #[derive(Serialize, Deserialize)]
     pub enum ProductType { #[serde(rename = "cms")] Cms, #[serde(rename = "analytics")] Analytics }
     #[derive(Serialize, Deserialize)]
     pub enum Tier { #[serde(rename = "premier")] Premier, #[serde(rename = "standard")] Standard, #[serde(rename = "basic")] Basic, #[serde(rename = "unknown")] Unknown }
     ```
   - Compiler prevents invalid product types or tiers

7. **Verification with real data.**
   - Glean-enriched account (Globex Holdings): open app, trigger Glean refresh, confirm `account_products` table now has 2 rows (CMS + ANALYTICS)
   - Account detail dossier: verify products section is populated (if I624 UI exists; if not, visibility in meeting prep is sufficient)
   - Meeting prep context: run meeting prep query, confirm JSON payload includes `products: [{type: "cms", tier: "standard", acv: 60000}, ...]`
   - Real prep prompt sees product data and can reference in expansion signals

8. **Degradation & edge cases.**
   - **Without Glean:** `account_products` table is empty for that account (expected)
   - **Incomplete Salesforce:** CMS ARR present, Parse.ly flag missing → one CMS row created; if Parse.ly data arrives later, ANALYTICS row added on next refresh
   - **No products:** Neither CMS ARR nor Parse.ly flag → no rows created (not "UNKNOWN" row — just empty)
   - **Financial dimension chat timeout/error:** Product extraction silently skipped, Financial intelligence still merges (best-effort)
   - **Product JSON parse failure in response:** Error logged, Financial intelligence still processed, upsert skipped

9. **Freshness & updates.**
   - On next Glean Financial dimension call, products are upserted (not re-inserted)
   - Example: CMS tier changes from Standard to Premier, ARR increases to $75k → existing row updated, `last_verified_at` refreshed
   - `last_verified_at` is the canonical freshness signal for product data

10. **All tests pass.**
   - `cargo clippy -- -D warnings` — clean
   - `cargo test` (including new product classification tests) — all pass
   - `pnpm tsc --noEmit` — clean (if any TS changes for context serialization)

## Design Notes

1. **Why Salesforce-only?** Salesforce fields are explicit and authoritative (`CMS ARR`, `Parsely Customer` flags). Zendesk and Gong signals are hints, not facts. Stick to facts in Phase 6. Zendesk/Gong triangulation is Phase 7 work if needed.

2. **Why one row per product, not JSON array?** Normalized schema is simpler to upsert (update one tier without touching the other), easier to query, and more DRY-compliant. Single-row JSON would require careful merge logic on update.

3. **Why upsert, not "check before insert"?** Upsert is simpler and handles tier/ARR changes naturally. If Salesforce data changes (upsell, downsell, renewal), next Glean refresh updates the existing row. No need for change detection.

4. **Why `last_verified_at` instead of version history?** Current state is all that matters for context injection. Historical product changes are not part of Phase 6 scope. If needed later, we can add a product change log, but for now, "what does Salesforce say today?" is sufficient.

5. **Why not store as user-editable?** Products are discovered, not managed. If a CSM wants to correct product info, I624's inline edit pattern handles it (emits `user_correction` signal). This issue is pure extraction.

# I611 — Full console.error → toast sweep (remaining ~136 instances)

**Area:** Frontend / UX
**Priority:** P1
**Version:** v1.0.1
**Source:** v1.0.0 GA engineering review (2026-03-16)
**Depends on:** v1.0.0 ships with top-20 user-facing error paths fixed (5A)

---

## Problem

156 bare `console.error()` calls exist across the React frontend without corresponding user-visible feedback (`toast.error()`). v1.0.0 fixes the top 20 user-facing paths (saves, refreshes, connector toggles). The remaining ~136 are in lower-traffic areas but still represent silent failures.

Hotspots:
- Connector internals (`features/settings-ui/connectors/*`): ~20 instances
- Report pages (AccountHealth, EBR/QBR, WeeklyImpact, SWOT, BoB): ~12 instances
- MeetingDetailPage: ~6 instances
- Background data fetching hooks: ~15 instances
- Remaining scattered across pages and components

## Scope

1. Audit all `console.error` calls in `src/` (excluding test files)
2. Categorize each as:
   - **User-facing failure** → replace with `toast.error(userFriendlyMessage)` + keep `console.error` for debugging
   - **Background/expected** → keep `console.error` only, add `// Expected: {reason}` comment
   - **Dead code** → remove
3. Use the sonner toast pattern already established: `import { toast } from "sonner"`
4. Follow ADR-0083 vocabulary in all toast messages (no "entity", "intelligence", "enrichment", "signal", "prep")

## Acceptance Criteria

1. `grep -rn 'console.error' src/ --include="*.tsx" | grep -v 'toast\.\|// Expected\|\.test\.'` returns zero results
2. Every user-initiated action (click, toggle, save, refresh) that can fail shows a toast on failure
3. Toast messages use product vocabulary (ADR-0083) — no system terms
4. No new `console.error` without either a toast or an `// Expected:` comment
5. Zero visual regressions — toasts don't disrupt layout

## Out of Scope

- Changing error handling patterns in Rust backend
- Adding retry logic (separate concern)
- Toast styling customization (sonner defaults are fine)

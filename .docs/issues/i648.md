# I648 - Source coverage matrix + field promotion rules

**Area:** Backend / Intelligence Architecture + Product Planning + Data Governance
**Priority:** P1
**Version:** v1.1.0 Phase 5
**Effort:** Medium
**Depends on:** I644, ADR-0100, I576
**Source:** Follow-up to Blackstone Glean source-inventory review. Glean can access a much broader field surface than DailyOS currently models, but there is no explicit promotion matrix deciding what becomes canonical fact, structured assessment, historical record, meeting-only context, or report-only material.

---

## Problem

We now know Glean can surface a very wide field universe across Salesforce, Zendesk, Gong, Google Docs, P2, Slack references, Rocketlane, and internal tooling links.

What we still do not have is a formal field taxonomy answering:

- which fields should become canonical DB-backed account truth
- which should remain structured assessment only
- which should be current by default vs historical by default
- which belong on account detail vs meeting detail vs reports vs daily briefing
- which are factual enough to reinforce hero vitals
- which should never be promoted because they are too inferred, too noisy, or too stale-prone

Without this matrix, the product risks drifting into arbitrary enrichment:

- some valuable fields never get surfaced
- some low-authority fields get over-promoted
- different surfaces may render contradictory views of the same account

---

## Solution

Create an explicit source coverage and field promotion matrix for account intelligence.

### 1. Enumerate the field families

Initial field families from Glean-accessible evidence:

- commercial facts
- renewal / procurement facts
- stakeholder / org coverage
- support / incident health
- technical footprint
- products / entitlements / add-ons
- adoption / usage telemetry
- services / delivered work
- compliance / security context
- strategic themes / technical exploration themes
- historical pricing / discount behavior
- source coverage / freshness metadata

### 2. Classify every field by destination

Each field gets a destination classification:

- `canonical_account_fact`
- `structured_assessment`
- `current_default`
- `historical_default`
- `meeting_brief_only`
- `report_only`
- `do_not_surface`

And an authority classification:

- `fact`
- `signal`
- `inference`

### 3. Define promotion rules

For each field, specify:

- best source system
- corroborating sources
- minimum confidence / authority required for promotion
- freshness window
- conflict handling rule
- user override behavior
- target surfaces

Example rows:

- `arr`
  - destination: `canonical_account_fact`
  - authority: `fact`
  - best source: Salesforce opportunity/account
  - surfaces: vitals, outlook, reports, meeting brief

- `procurement_complexity`
  - destination: `structured_assessment`
  - authority: `inference` unless backed by explicit contract notes
  - surfaces: outlook, meeting brief

- `recent_tls_incident`
  - destination: `structured_assessment`
  - authority: `signal`
  - current_default if unresolved or recent
  - historical_default after resolution window

### 4. Use the matrix as an implementation gate

No new field family should ship into a surface without first being mapped in the matrix.

This keeps the richness expansion disciplined instead of ad hoc.

---

## Acceptance Criteria

1. DailyOS has a documented field coverage matrix for Glean-accessible account intelligence, covering at least the field families listed above.
2. Every new field proposed for account detail, meeting detail, or reports is classified by:
   - destination
   - authority
   - best source
   - freshness window
   - target surfaces
3. The matrix explicitly identifies which fields are eligible to reinforce hero vitals.
4. The matrix explicitly identifies which fields are report-only or historical-default so current surfaces do not become overloaded.
5. The matrix is referenced by the implementation issues so schema and UI work stay aligned.

---

## Implementation Notes

- This can live as a section in the issue spec set or as a dedicated appendix doc, but it must be version-controlled and implementation-facing.
- Favor small, opinionated classifications over exhaustive theoretical taxonomy.
- This issue is about product and architecture discipline, not about building a user-facing field browser.

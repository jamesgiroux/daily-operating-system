# I576 — Source-Aware Intelligence Reconciliation

**Version:** v1.0.0 Phase 5b
**Depends on:** I574 (Parallel dimension enrichment), I575 (Progressive enrichment)
**Type:** Enhancement — Backend / Intelligence / Architecture
**Scope:** `intelligence/glean_provider.rs`, `intelligence/glean_prompts.rs`, `intelligence/dimension_prompts.rs`, `intel_queue.rs`, `intelligence/io.rs`, `intelligence/health_scoring.rs`

---

## Problem

### The Overwrite Problem

Current intelligence merge is field-level overwrite. When Glean returns a sparse JSON response (it omits fields it has no data for), `merge_glean_into_existing` in `glean_provider.rs` replaces entire arrays:

```rust
macro_rules! merge_vec {
    ($field:ident) => {
        if !glean.$field.is_empty() {
            existing.$field = glean.$field;  // WIPES all existing items
        }
    };
}
```

This means: if PTY enrichment found 3 risks from transcript analysis, and Glean returns 2 risks from Salesforce data, the 3 transcript-derived risks are gone. The user's local knowledge — what they heard in a meeting, what they observed in a document — is overwritten by the org system of record.

### The Attribution Problem

No intelligence item (risk, win, stakeholder insight, commitment) carries source attribution. Once data enters `IntelligenceJson`, its provenance is lost. The user can't distinguish:
- A risk they personally heard a customer articulate in a meeting
- A risk that Glean inferred from a Zendesk ticket trend
- A risk that PTY synthesized from meeting notes
- A risk the user manually entered as a correction

Without provenance, the system can't make intelligent decisions about what to keep during reconciliation, what to weight in health scoring, or what to show the user as high-confidence vs. speculative.

### The Confidence Problem

All intelligence items are treated as equally reliable. A user correction and an AI synthesis have the same weight. ADR-0100 defines confidence tiers, and `signal_weights` implements Bayesian learning at the signal level, but none of this flows into intelligence item-level confidence. Health scoring treats a PTY-guessed stakeholder the same as a CRM-confirmed stakeholder.

### Why This Matters — The DailyOS Thesis

This is the architectural issue that determines whether DailyOS is personal intelligence or just another org dashboard.

Glean knows what Salesforce says. It knows what Zendesk tickets look like. It knows what was said in Gong recordings. This is org knowledge — shared, canonical, system-of-record.

DailyOS knows what *you* heard in a meeting. What *you* wrote in a document. What *your* priorities are. What *you* corrected when the AI got it wrong. This is personal context — private, experiential, high-signal.

The reconciliation engine is where these two knowledge sources meet. If org knowledge always wins, DailyOS is a Glean skin. If personal knowledge always wins, DailyOS ignores the organization. The right answer is: both are preserved with source attribution, weighted by confidence, and the user sees the full picture.

---

## Solution

### Design: Glean Chat as Reconciliation Engine (Option B)

Rather than building a separate reconciliation layer that runs after enrichment, make the AI enrichment call itself the reconciliation engine. Local context (transcripts, workspace files, user corrections) is injected into each dimension prompt with explicit source tags and confidence weights. The AI produces source-attributed output. For non-Glean users, PTY does the same reconciliation with the same prompt pattern.

This means:
- No separate "merge" step after enrichment
- No field-level overwrite logic
- The AI sees all sources simultaneously and produces a reconciled view
- Source attribution is baked into the output, not bolted on after

### 1. Source Hierarchy

Every intelligence item carries a `source` field indicating its provenance and a `confidence` value derived from the source tier and Bayesian signal weight history.

| Source | Confidence | Description |
|--------|-----------|-------------|
| `user_correction` | 1.0 | User manually edited this item. Never overwritten. |
| `glean_crm` | 0.9 | From Salesforce via Glean (system of record). |
| `local_file` | 0.85 | From a workspace file (project doc, playbook, user notes). |
| `glean_zendesk` | 0.85 | From Zendesk via Glean (support data). |
| `transcript` | 0.8 | Extracted from a meeting transcript by AI. |
| `glean_gong` | 0.8 | From Gong call recordings via Glean. |
| `glean_chat` | 0.7 | Glean AI synthesis (cross-source inference). |
| `email` | 0.6 | Extracted from email content. |
| `pty_synthesis` | 0.5 | PTY synthesis from local files (baseline, no external data). |

**Effective confidence** = `base_confidence * bayesian_weight`, where `bayesian_weight` is the entity-specific signal weight for that source from `signal_weights` (learned via Bayesian feedback from user corrections). Default Bayesian weight is 1.0.

### 2. Source Field Format

Every intelligence list item type gains `source` and `confidence` fields:

```rust
// intelligence/io.rs — additions to existing types

/// Source attribution for an intelligence item.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct ItemSource {
    /// Source identifier from the hierarchy.
    pub source: String,       // "user_correction" | "transcript" | "local_file" | "glean_crm" | etc.
    /// Base confidence from source tier (0.0 - 1.0).
    pub confidence: f64,
    /// ISO 8601 timestamp of when this item was last refreshed from this source.
    pub sourced_at: String,
    /// Optional: specific source reference (e.g., meeting date, file name, ticket ID).
    #[serde(skip_serializing_if = "Option::is_none")]
    pub reference: Option<String>,
}
```

Added to these existing types:

| Type | Current Fields | Added |
|------|---------------|-------|
| `IntelRisk` | `text`, `urgency`, `sub_type` | `source: Option<ItemSource>` |
| `IntelWin` | `text`, `impact`, `sub_type` | `source: Option<ItemSource>` |
| `StakeholderInsight` | `name`, `role`, `assessment`, `engagement` | `source: Option<ItemSource>` |
| `ValueItem` | `date`, `statement`, `source_text`, `impact` | `source: Option<ItemSource>` |
| `CompetitiveInsight` | `competitor`, `context`, `threat` | `source: Option<ItemSource>` |
| `OrgChange` | `change`, `when`, `impact` | `source: Option<ItemSource>` |
| `OpenCommitment` | `description`, `owner`, `due_date`, `status` | `source: Option<ItemSource>` |
| `ExpansionSignal` | `signal`, `strength`, `source_text` | `source: Option<ItemSource>` |

The `source` field is `Option<ItemSource>` for backward compatibility — existing data without source attribution continues to work (treated as `pty_synthesis` at baseline confidence).

### 3. Prompt Pattern — Source-Tagged Context Injection

Each dimension prompt (from I574) injects local context with explicit source tags so the AI knows the provenance of each piece of information.

**Current prompt pattern (I535):**

```
### Recent Captures (wins/risks/decisions)
- Risk: "They're evaluating Contentful" — mentioned in call with VP Product
- Win: Reduced reporting time from 5 days to same-day
```

**New prompt pattern with source tags:**

```
### Context with Source Attribution
Each item below has a source tag and confidence level. When producing output,
PRESERVE these source attributions. If you find the same information from
multiple sources, keep both entries with their respective sources.

[transcript, 0.8, meeting 2026-03-10] Risk: "They're evaluating Contentful"
  — VP Product mentioned in Q1 planning review
[local_file, 0.85, playbook.md] Risk: Integration timeline at risk per
  internal project doc
[user_correction, 1.0, 2026-03-12] Win: Reduced reporting from 5 days to
  same-day saving $30K/month (user verified this number)
[glean_crm, 0.9] Stakeholder: Sarah Chen, VP Product, high engagement
[glean_zendesk, 0.85] Risk: 3 P1 tickets open, CSAT trending down

OUTPUT RULES:
1. Every item in your output MUST include a "source" object with source type,
   confidence, and reference.
2. When Glean/CRM data and local data describe the same thing, keep BOTH with
   their respective sources. Do not merge into a single item.
3. User corrections (confidence 1.0) are ALWAYS preserved verbatim. Never
   rephrase, downgrade, or omit a user correction.
4. When sources conflict, flag with "discrepancy": true and preserve both
   versions.
5. If you synthesize a new insight from multiple sources, use source
   "glean_chat" (if via Glean) or "pty_synthesis" (if via PTY) with
   confidence reflecting the lowest-confidence input source.
```

### 4. Reconciliation Rules

These rules govern how the AI (Glean or PTY) reconciles multiple sources:

**Rule 1: User corrections are sacred.** Any item with `source: "user_correction"` is included verbatim in every enrichment output. The prompt injects user corrections with a `[USER CORRECTION — DO NOT MODIFY]` tag. If the AI omits or modifies a user correction, the post-processing step restores it.

**Rule 2: Similar items from different sources are both kept.** If a transcript says "they're evaluating Contentful" and Glean's CRM says "competitive threat: Contentful," both items appear in the output with their respective sources. They are not merged into one. The user sees both and understands where each came from.

**Rule 3: Discrepancy flagging.** When two sources directly contradict (e.g., transcript says "renewal confirmed" but CRM says "renewal at risk"), both are kept and the item gains a `discrepancy: true` flag. Health scoring uses the higher-confidence source for scoring but surfaces the discrepancy in the narrative.

**Rule 4: Source survival across refreshes.** When an enrichment refresh runs, items from sources that were NOT re-queried are preserved. A transcript-derived risk doesn't disappear just because a Glean refresh didn't mention it — the transcript wasn't re-analyzed.

**Implementation of Rule 4:**

```rust
// New in intel_queue.rs — source-aware merge
fn reconcile_enrichment(
    existing: &IntelligenceJson,
    new_output: &IntelligenceJson,
    refreshed_sources: &[&str],  // which sources were queried in this enrichment
) -> IntelligenceJson {
    let mut result = new_output.clone();

    // For each list field: keep existing items whose source was NOT refreshed
    reconcile_vec(&mut result.risks, &existing.risks, refreshed_sources);
    reconcile_vec(&mut result.recent_wins, &existing.recent_wins, refreshed_sources);
    reconcile_vec(&mut result.stakeholder_insights, &existing.stakeholder_insights, refreshed_sources);
    // ... for all source-attributed list types

    // User corrections are always preserved regardless of refresh scope
    preserve_user_corrections(&mut result, existing);

    result
}

fn reconcile_vec<T: HasSource>(
    new_items: &mut Vec<T>,
    existing_items: &[T],
    refreshed_sources: &[&str],
) {
    // Keep existing items whose source is NOT in refreshed_sources
    for item in existing_items {
        if let Some(source) = item.item_source() {
            if !refreshed_sources.contains(&source.source.as_str()) {
                new_items.push(item.clone());
            }
        }
    }
}
```

### 5. User Corrections — Item-Level Extension

The existing `user_edits` mechanism tracks field-level corrections (e.g., "user edited executiveAssessment"). This issue extends it to item-level:

- User edits a specific risk's text: that risk item gains `source: { source: "user_correction", confidence: 1.0 }`
- User adds a new risk manually: the new risk has `source: "user_correction"`
- User deletes a risk: a tombstone record is stored so the same risk isn't re-created by the next enrichment

**Tombstone format:**

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct DismissedItem {
    pub item_type: String,   // "risk" | "win" | "stakeholder" | etc.
    pub item_hash: String,   // content hash for matching
    pub dismissed_at: String,
    pub reason: Option<String>,
}
```

Stored in `IntelligenceJson.dismissed_items: Vec<DismissedItem>`. During reconciliation, any new item whose content hash matches a tombstone is dropped.

### 6. Health Scoring Integration

Health scoring (`intelligence/health_scoring.rs`) currently treats all intelligence items equally. With source attribution, scoring weights items by confidence:

```rust
// In compute_dimension_score:
fn weighted_item_score(items: &[impl HasSource], scorer: impl Fn(&T) -> f64) -> f64 {
    let total_weight: f64 = items.iter()
        .map(|i| i.effective_confidence())
        .sum();
    if total_weight == 0.0 { return 0.0; }

    items.iter()
        .map(|i| scorer(i) * i.effective_confidence())
        .sum::<f64>() / total_weight
}
```

A user-confirmed risk (confidence 1.0) has more weight than a PTY-synthesized risk (confidence 0.5). A CRM-sourced stakeholder (0.9) contributes more to stakeholder coverage scoring than an AI-inferred stakeholder (0.7).

### 7. Glean Dimension Prompt Updates

Each per-dimension Glean prompt (from I574, in `intelligence/glean_prompts.rs`) is updated to:

1. Include source-tagged local context (transcripts, files, user corrections)
2. Request source-attributed output
3. Include the reconciliation rules in the prompt

Example for the Core Assessment dimension:

```
You are analyzing the customer account "Acme Corp" for its core health assessment.

### Local Context (source-tagged — preserve these attributions)
[transcript, 0.8, meeting 2026-03-10] "They're evaluating Contentful for the docs portal"
[user_correction, 1.0, 2026-03-12] "Renewal confirmed by Sarah Chen for 2 years"
[local_file, 0.85, account-plan.md] "Strategic goal: 80% user adoption by Q3"

### Task
Search Salesforce, Zendesk, Gong, and Slack for this account. Produce a core assessment
combining org knowledge with the local context above.

### Output Rules
- Every risk/win item MUST include a "source" object: { "source": "glean_crm|glean_zendesk|...", "confidence": 0.9, "sourcedAt": "ISO date", "reference": "ticket/deal/call ID" }
- Local items (transcript, local_file, user_correction) MUST be included verbatim with their original source tags. Do not rephrase user corrections.
- When org data and local data describe the same thing differently, keep BOTH with "discrepancy": true.
- Return ONLY valid JSON matching this schema:
{
  "executiveAssessment": "...",
  "risks": [{ "text": "...", "urgency": "critical|watch|low", "source": { "source": "...", "confidence": 0.9, "sourcedAt": "...", "reference": "..." }, "discrepancy": false }],
  "recentWins": [{ "text": "...", "impact": "high|medium|low", "subType": "...", "source": { ... } }],
  "currentState": { "working": [...], "notWorking": [...], "unknowns": [...] }
}
```

### 8. PTY Prompt Updates

The same source-tagged context injection and source-attributed output rules apply to PTY prompts (`intelligence/dimension_prompts.rs`). The only difference: PTY can't query Salesforce/Zendesk/Gong, so its sources are limited to `transcript`, `local_file`, `email`, `user_correction`, and `pty_synthesis`.

PTY prompts explicitly acknowledge this: "You have access to local context only. Tag your synthesized insights as `pty_synthesis` with confidence 0.5. Items from specific sources (transcripts, files) retain their original source tags."

### 9. `HasSource` Trait

A trait for intelligence item types that carry source attribution:

```rust
pub trait HasSource {
    fn item_source(&self) -> Option<&ItemSource>;
    fn effective_confidence(&self) -> f64 {
        self.item_source()
            .map(|s| s.confidence)
            .unwrap_or(0.5)  // default: PTY baseline
    }
}
```

Implemented for: `IntelRisk`, `IntelWin`, `StakeholderInsight`, `ValueItem`, `CompetitiveInsight`, `OrgChange`, `OpenCommitment`, `ExpansionSignal`.

### 10. Frontend Source Display

Intelligence items on the account detail page show a provenance tag:

- Transcript-sourced: "from meeting 2026-03-10" (muted text, small)
- CRM-sourced: "from Salesforce" (muted text, small)
- User correction: "you edited this" (subtle indicator, not obnoxious)
- Discrepancy: a small warning icon with tooltip explaining the conflicting sources

This is a minimal UI treatment — a single line of muted text beneath the item content. Not a badge, not a color, not a banner. The goal is transparency without clutter.

## Out of Scope

- Embedding-based deduplication of similar items across sources (mentioned in plan as ">0.8 embedding similarity" — deferred to a future issue; this issue uses content hash matching which is cheaper and sufficient for exact/near-exact duplicates)
- Signal-level source attribution (already exists in `signals/` via `source` field on signals — this issue is about intelligence *item* attribution)
- Frontend filtering by source (e.g., "show me only CRM-sourced risks") — future UX enhancement
- Bayesian weight learning from item-level feedback (existing signal-level Bayesian feedback continues; item-level feedback is a future extension)

## Acceptance Criteria

1. Every intelligence item type (`IntelRisk`, `IntelWin`, `StakeholderInsight`, `ValueItem`, `CompetitiveInsight`, `OrgChange`, `OpenCommitment`, `ExpansionSignal`) has an `Option<ItemSource>` field with `source`, `confidence`, `sourced_at`, and optional `reference`.
2. `ItemSource.source` is one of: `user_correction`, `transcript`, `local_file`, `glean_crm`, `glean_zendesk`, `glean_gong`, `glean_chat`, `email`, `pty_synthesis`. No other values accepted (validated at parse time).
3. Glean enrichment does NOT overwrite transcript-derived items. Verified: enrich an account with PTY (produces transcript-sourced risks), then run Glean enrichment. Transcript-sourced risks survive the Glean refresh.
4. User corrections (confidence 1.0) are never overwritten by any source. Verified: user edits a risk, then run enrichment. The user-corrected risk is preserved verbatim in the output. Its `source.source` is `"user_correction"`.
5. When Glean and local context produce conflicting information about the same topic, both items are preserved with their respective sources and `discrepancy: true`. Verified: inject a transcript risk "renewal confirmed" and a CRM risk "renewal at risk" for the same account. Both appear in the output with discrepancy flag.
6. Items from non-refreshed sources survive enrichment refreshes. Verified: an account has a `local_file`-sourced risk. Run a Glean-only refresh (no transcript reprocessing). The `local_file` risk is still present after the refresh.
7. Dismissed items (tombstones) prevent re-creation. Verified: user dismisses a risk, run enrichment, the same risk is not re-created. The `dismissed_items` array contains a matching tombstone.
8. Health scoring weights items by `effective_confidence()`. Verified: an account with one user-corrected risk (1.0) and one PTY-synthesized risk (0.5) produces a different signal_momentum score than if both were at 0.5. The user-corrected risk has more influence.
9. Per-dimension prompts (both Glean and PTY) include source-tagged local context with `[source, confidence, reference]` tags. Verified by logging the prompt text for a real enrichment run.
10. Per-dimension prompts include the reconciliation output rules (source objects required, user corrections preserved, discrepancies flagged). Verified by prompt inspection.
11. PTY enrichment produces source-attributed output with `pty_synthesis` source at 0.5 confidence for synthesized items and `transcript`/`local_file`/`email` sources at their respective tiers for items derived from specific local sources.
12. Existing intelligence data without `source` fields continues to load and display correctly (backward compatibility). Missing `source` is treated as `pty_synthesis` at 0.5 confidence for scoring purposes.
13. Account detail page shows provenance tags beneath intelligence items. Transcript-sourced items show "from meeting YYYY-MM-DD". CRM-sourced items show "from Salesforce". User corrections show "you edited this". Discrepancies show a warning indicator with tooltip.
14. `HasSource` trait implemented for all 8 intelligence item types. `effective_confidence()` returns the item's confidence (or 0.5 default for items without source attribution).
15. Local workspace file intelligence is tagged as `local_file` source with 0.85 confidence. Verified: process a workspace file, check the resulting intelligence items have `source.source == "local_file"`.
16. Run full enrichment on a real account with both Glean and transcript history. Verify: output contains items from multiple sources, user corrections are preserved, no source data is lost, discrepancies are flagged where sources conflict. Field-by-field comparison shows richer output than pre-I576 monolithic merge.
17. All existing tests pass. Backward-compatible deserialization of `IntelligenceJson` without source fields works without error.

# I639 — Mock data schema validation — CI gate

**Area:** Backend / Testing / CI
**Priority:** P1
**Version:** v1.0.4
**Effort:** Small
**Source:** Engineering retro (2026-03-22). Demo data used `enriched_captures` (renamed to `captures` in migration 070). No validation caught this until runtime.

---

## Problem

Mock data in `demo.rs` and `devtools/mod.rs` contains SQL INSERT statements that reference table and column names. When migrations rename tables or add/remove columns, mock data silently becomes invalid:

- **v1.0.3 bug:** `INSERT INTO enriched_captures` failed silently because the table was renamed to `captures` in migration 070. Demo meetings had no captures data. Took hours to diagnose.
- **v1.0.3 bug:** `meetings` INSERT missing `attendees` column, causing The Room section to be empty in demo mode.
- **Risk:** Every future migration that renames a table or column can break mock data without any test catching it.

## Approach

A Rust integration test that:
1. Creates an in-memory SQLite database with all migrations applied
2. Runs every INSERT statement from `demo.rs` and `devtools/mod.rs` against it
3. Fails with a clear error if any table or column doesn't exist

This is simpler and more reliable than parsing SQL strings — just execute them and catch errors.

## Acceptance Criteria

1. Test in `src-tauri/src/db/mod_tests.rs` (or `src-tauri/src/demo.rs` under `#[cfg(test)]`)
2. Calls `install_demo()` against an in-memory test DB with all migrations applied
3. If any INSERT fails due to table/column mismatch, the test fails with the SQL error
4. Also validates `devtools::install_full_scenario()` if callable from tests (may need a test-only entry point)
5. Test runs as part of `cargo test` — no separate CI step needed
6. Clear error messages: the SQLite error naturally says "no such table: enriched_captures" or "table captures has no column named foo"

## Files

- `src-tauri/src/db/mod_tests.rs` — new test(s)
- No production code changes

# I640 — Meeting Record integration tests

**Area:** Backend / Testing
**Priority:** P2
**Version:** v1.0.4
**Effort:** Medium
**Source:** Engineering retro (2026-03-22). v1.0.3 shipped the prediction scorecard, continuity thread, and meeting record markdown with 11 unit tests for Jaccard similarity but no integration tests for the full pipeline.

---

## Problem

The Meeting Record intelligence pipeline has three new subsystems:

1. **Prediction scorecard** (I635) — compares prep_frozen_json predictions against transcript captures. Tested at the unit level (Jaccard similarity) but not integration-tested with real DB data.
2. **Continuity thread** (I637) — queries previous meetings, completed actions, health deltas, new attendees. No tests.
3. **Meeting record markdown** (I636) — generates structured markdown from pipeline output. No tests.

These subsystems query multiple tables (`captures`, `meeting_entities`, `actions`, `health_score_history`, `meeting_attendees`, `meeting_prep`) and assemble complex output. Unit tests on the similarity function don't catch query-level bugs.

## Acceptance Criteria

### Prediction scorecard tests
1. Seed a meeting with `prep_frozen_json` containing 2 risks and 1 win
2. Seed captures with 1 matching risk and 1 surprise risk
3. Call `compute_scorecard()` — verify: 1 confirmed, 1 not-raised, 1 surprise
4. Verify `has_data` is true
5. Seed a meeting with no prep — verify scorecard returns None
6. Seed a meeting with prep but no captures — verify returns None

### Continuity thread tests
1. Seed two meetings (7 days apart) linked to the same account via `meeting_entities`
2. Seed 1 completed action and 1 pending action between the two dates
3. Seed health_score_history entries for both dates
4. Seed meeting_attendees for both meetings (one new attendee in the second)
5. Call `get_continuity_thread()` — verify:
   - `is_first_meeting` is false
   - `actions_completed` has 1 item
   - `actions_open` has 1 item
   - `health_delta` is populated with correct values
   - `new_attendees` has the new person's name
6. Seed only one meeting — verify `is_first_meeting` is true

### Meeting record markdown tests
1. Call `generate_meeting_record_markdown()` with sample data
2. Verify YAML frontmatter contains: meeting_id, title, date, type, generated_at
3. Verify `## Summary` section present when summary provided
4. Verify `## Key Findings` has wins/risks/decisions subsections
5. Verify sections are absent when data arrays are empty
6. Verify file path computation follows entity routing priority (account > project > person > archive)

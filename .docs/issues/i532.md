# I532 — Intelligence Surfacing Threshold Model

**Priority:** P1
**Area:** Backend / Signals + Frontend / Briefing
**Version:** v1.1.0
**Depends on:** I499 (health scoring — dimensions to evaluate), I508 (intelligence schema — structured signals to surface), I530 (signal taxonomy — feedback loop teaches threshold quality)

## Problem

DailyOS can detect everything — declining health, stale champions, engagement gaps, expansion signals, competitive threats, org changes. The self-healing detectors find gaps. The health scoring engine quantifies risk. The intelligence schema captures structured signals across 6 dimensions. But there is no model for **when** a signal rises to the level of interrupting the user's attention.

Without a surfacing threshold, the system has two failure modes:

1. **Surface everything** — the briefing becomes a wall of alerts. Every declining dimension, every stale enrichment, every gap gets surfaced. The user stops reading. This is Gainsight's failure mode: a dashboard full of red that becomes invisible.

2. **Surface nothing proactively** — the system waits for the user to open each account and read its intelligence. Insights exist in the DB but never reach the user at the right moment. This is DailyOS's current failure mode: intelligence is comprehensive but passive.

The goal is a chief of staff who knows when to tap you on the shoulder and when to stay quiet. Not a dashboard that shows everything. Not a silent database that shows nothing.

### What "surfacing" means in DailyOS

Surfacing is not notifications. DailyOS doesn't send push notifications, doesn't ping, doesn't interrupt. Surfacing means: which intelligence items appear in the **daily briefing** (Attention section), the **Portfolio page** (Exceptions list), and the **meeting prep** (context that wasn't there yesterday). The user opens the app on their schedule. The question is: what's waiting for them when they do?

## Design

### 1. Signal significance scoring

Every intelligence change gets a **significance score** (0.0 to 1.0) that determines whether it surfaces or stays as background context:

```rust
pub struct SignificanceScore {
    pub score: f64,           // 0.0 (background) to 1.0 (critical)
    pub factors: Vec<SignificanceFactor>,
    pub surface_to: Vec<SurfaceTarget>,  // where this should appear
}

pub enum SurfaceTarget {
    DailyBriefing,            // Attention section
    PortfolioExceptions,      // Portfolio page exception list
    MeetingPrep { meeting_id: String },  // Context for upcoming meeting
    AccountDetail,            // Account page intelligence update indicator
}

pub struct SignificanceFactor {
    pub factor: String,       // what contributed to significance
    pub weight: f64,          // how much it contributed
}
```

### 2. Significance computation

```rust
pub fn compute_significance(
    signal: &SignalEvent,
    entity_health: &AccountHealth,
    entity_context: &EntityContext,  // ARR, lifecycle, meeting proximity
) -> SignificanceScore {
    let mut score = 0.0;
    let mut factors = Vec::new();

    // Factor 1: Magnitude of change (0.0-0.3)
    // A health score dropping 15 points is more significant than dropping 2.
    // A new risk appearing is more significant than a risk description update.
    let magnitude = compute_change_magnitude(signal);
    score += magnitude * 0.3;

    // Factor 2: Account importance (0.0-0.2)
    // Higher ARR accounts get amplified significance.
    // Accounts with imminent renewals get amplified.
    // Accounts the user marked as priority get amplified.
    let importance = compute_account_importance(entity_context);
    score += importance * 0.2;

    // Factor 3: Temporal relevance (0.0-0.2)
    // Signal matters more if there's a meeting with this account tomorrow.
    // Signal matters more during renewal window.
    // Signal matters less if it's about an account with no meetings for 3 months.
    let temporal = compute_temporal_relevance(signal, entity_context);
    score += temporal * 0.2;

    // Factor 4: Pattern detection (0.0-0.2)
    // A single declining metric is noise. Three declining metrics is a trend.
    // Champion going quiet + engagement declining + health dropping = convergent risk.
    let pattern = compute_pattern_significance(signal, entity_health);
    score += pattern * 0.2;

    // Factor 5: Novelty (0.0-0.1)
    // First time we detect this type of issue for this account = higher significance.
    // Same issue persisting = lower significance (already surfaced).
    // Issue resolved then recurring = high significance (regression).
    let novelty = compute_novelty(signal, entity_context);
    score += novelty * 0.1;

    // Determine surface targets based on score
    let surface_to = determine_surface_targets(score, signal, entity_context);

    SignificanceScore { score, factors, surface_to }
}
```

### 3. Surfacing thresholds

| Score Range | Behavior | Example |
|---|---|---|
| 0.8-1.0 | **Critical** — surfaces in daily briefing Attention section AND portfolio exceptions. Persists until acknowledged or resolved. | Health dropped 20+ points in a week. Champion departed. Renewal in 30 days with readiness "critical". |
| 0.5-0.79 | **Notable** — surfaces in daily briefing if there's a meeting within 7 days, otherwise in portfolio exceptions only. Surfaces once, then decays. | New risk detected from Glean. Engagement declining for 2+ weeks. Expansion signal identified. |
| 0.3-0.49 | **Context** — surfaces in meeting prep for the next meeting with this account. Does not appear in briefing or portfolio page. | Minor health dimension change. New stakeholder detected. Email volume shift. |
| 0.0-0.29 | **Background** — stored in DB, available on account detail page, but not proactively surfaced. Updates the intelligence record silently. | Routine enrichment update. Minor signal with no trend. |

### 4. Surfacing fatigue prevention

The system must avoid alert fatigue. Rules:

- **Daily budget:** Maximum 5 items in the daily briefing Attention section. If more qualify, show top 5 by significance score + "and N more" link.
- **Per-account cooldown:** An account can surface at most 1 critical item per day in the briefing. Multiple signals from the same account are grouped: "Acme: 3 risk signals detected" with one-click expand.
- **Decay:** A surfaced item's significance decays by 0.1 per day it goes unacknowledged. After 5 days, even a critical item drops to Notable. After 10 days, it drops to Context. This prevents stale alerts from clogging the briefing.
- **Acknowledgment:** User viewing an account detail page counts as implicit acknowledgment for that account's surfaced items. Thumbs up/down on a surfaced item (I529) is explicit acknowledgment.

### 5. Feedback loop (connects to I529/I530)

The surfacing model learns from user behavior:

- **User engages with surfaced item** (clicks through, reads account, takes action) → significance model was correct. Source weights unchanged.
- **User thumbs-down a surfaced item** (I529) → this was surfaced when it shouldn't have been. Adjust the significance factors that contributed: reduce magnitude weight for this signal type, or reduce account importance multiplier.
- **User thumbs-up a surfaced item** → this was correctly surfaced. Reinforce the factors.
- **User ignores a surfaced item for 5+ days** → ambiguous. Decay handles this. No penalty. The system doesn't assume the user disagrees — they might just be busy.

Over time, the surfacing model learns what the user considers worth their attention vs what's noise. This is where the Bayesian scoring system (I530's signal weights) connects to the surfacing layer: source reliability affects confidence, which affects significance magnitude.

### 6. Integration with existing surfaces

**Daily briefing (Attention section):**
- Currently shows meeting prep quality gaps and email items
- Add: intelligence surfacing items scored >= 0.5 with meeting within 7 days, or >= 0.8 regardless
- Editorial treatment: "Since yesterday" framing. One-line per item: account name + what changed + why it matters

**Portfolio page (I492, Exceptions list):**
- Currently shows declining health, at-risk renewals, meeting gaps
- Add: any intelligence surfacing item scored >= 0.5 across the book
- Grouped by account, sorted by max significance within each account

**Meeting prep:**
- Currently shows entity intelligence + prep topics
- Add: "New since your last meeting" section showing intelligence changes scored >= 0.3 that are temporally relevant to this meeting
- This is where the chief of staff says "before you walk in, you should know..."

## Acceptance Criteria

1. Health score dropping 20+ points in 7 days surfaces in daily briefing Attention section with account name, magnitude, and one-sentence rationale
2. Champion departure (champion_health dimension goes null) surfaces as critical (>= 0.8) in briefing and portfolio exceptions
3. New expansion signal detected during enrichment surfaces as notable (>= 0.5) in meeting prep for the next meeting with that account
4. Routine enrichment update that changes no dimension by more than 5 points does NOT surface — stays as background update
5. Daily budget: with 10 accounts all changing simultaneously, briefing shows top 5 + "and 5 more" link. Not all 10.
6. Per-account cooldown: account "Acme" with 3 simultaneous signals shows as one grouped item in the briefing, not 3 separate items
7. Decay: a surfaced item unacknowledged for 5 days drops from Critical to Notable. After 10 days, to Context.
8. Feedback: thumbs-down on a surfaced item reduces future significance for that signal type. Thumbs-up reinforces it.
9. Meeting prep shows "New since your last meeting" section with intelligence changes scored >= 0.3 — only changes that are temporally relevant to this specific meeting
10. Account importance: a $500K ARR account with declining health surfaces before a $50K account with the same decline

## Out of Scope

- Push notifications (Slack, email, mobile) — DailyOS is pull-only by design
- User-configurable threshold levels ("show me more/fewer alerts") — learn from behavior instead
- Cross-account pattern detection ("3 accounts in FinServ all declining") — portfolio-level pattern recognition is a future enhancement
- Real-time surfacing — computed during enrichment and daily pipeline, not on every signal event
- Surfacing for reports — reports are on-demand, not proactively surfaced

## Design Philosophy

The surfacing model is the difference between a dashboard and a chief of staff. A dashboard shows everything and trusts the user to scan. A chief of staff filters and prioritizes, earning trust by being right about what matters and quiet about what doesn't. The Bayesian feedback loop (I529/I530) is what makes this learnable — not a static threshold but a system that improves with every interaction.

The initial thresholds (0.8/0.5/0.3) are hypotheses. They will be wrong. The feedback loop is the mechanism that makes them right over time. Ship with reasonable defaults, then learn.

# I596 — Email Mock Data Foundation

**Version:** v1.0.1
**Area:** Backend / Dev Tools
**Priority:** P0 (prerequisite for all v1.0.1 email features)

## Problem

The `emails` table has zero rows in mock data. Only 2 `email_signals` rows exist. Every v1.0.1 feature (reply debt, triage, cadence, meeting linkage) needs mock email data to develop and verify against. Without this, developers build blind and verification is impossible.

## Design

### Email Table Seeding

Add 15-20 mock emails to `seed_database()` in `devtools/mod.rs`, distributed across accounts and scenarios:

**Acme Corp (6 emails)**:
- `mock-email-acme-1`: Sarah Chen (VP Eng), "Re: Platform Migration Timeline" — high priority, enriched, `user_is_last_sender = false`, 3 days old (reply debt + aging signal)
- `mock-email-acme-2`: Sarah Chen, "Re: Q2 Planning Discussion" — high priority, enriched, `user_is_last_sender = false`, 1 day old (reply debt, normal)
- `mock-email-acme-3`: Mike Torres (Eng Lead), "POC Results Summary" — medium priority, enriched, `user_is_last_sender = true`, positive sentiment, 2 days old
- `mock-email-acme-4`: Sarah Chen, "Budget Approval for Phase 2" — high priority, enriched, urgency high, commitment detected ("will confirm budget by Friday"), `pinned_at` set
- `mock-email-acme-5`: noreply@acme.com, "Your Acme Support Ticket #4521" — low priority, automated sender, noise
- `mock-email-acme-6`: Mike Torres, "Quick Question on API Limits" — medium priority, enriched, question detected, `user_is_last_sender = false`, today (repliesNeeded candidate)

### Email Signals Seeding

Expand from 2 to 12+ `email_signals` rows:
- `email_sentiment` signals for enriched emails (positive for Acme-3, negative for Globex-1)
- `email_commitment` signals for commitment-containing emails (Acme-4)
- `email_urgency_high` for urgent emails (Acme-4, Globex-4)
- `pre_meeting_context` signals linking Acme emails to today's Acme meeting (Sarah Chen is attendee)

### Email Cadence Seeding

Seed `entity_email_cadence` table:
- Acme: avg_interval_days = 3, last_email = 3 days ago (normal)
- Globex: avg_interval_days = 5, last_email = 18 days ago (gone quiet — 3.6× normal)
- Initech: only 2 historical emails (excluded from cadence — insufficient baseline)

### Enrichment State Distribution

- 10 emails: `enrichment_state = 'enriched'`
- 3 emails: `enrichment_state = 'pending'` (shows "Building context..." in UI)
- 2 emails: low priority noise, `enrichment_state = 'pending'` (won't be enriched)

### Relevance Scores

Pre-seed `relevance_score` on enriched emails:
- Priority band (> 0.40): Acme-1, Acme-4, Globex-1 (3 emails)
- Monitoring band (0.15-0.40): Acme-2, Acme-3, Acme-6, Globex-2, Globex-5 (5 emails)
- Other band (< 0.15): all noise/automated emails

### Commitments and Questions

Pre-seed on enriched emails (these fields are populated by AI enrichment, but mock data pre-populates):
- Acme-4: `commitments = ["Will confirm Phase 2 budget by Friday"]`
- Globex-1: `commitments = ["Will send updated contract terms by EOW"]`
- Acme-6: `questions = ["What are the current API rate limits for the enterprise tier?"]`
- Globex-5: `questions = ["Can we move the Thursday review to next week?"]`

### Purge

All mock email IDs use `mock-email-*` prefix. Add `DELETE FROM emails WHERE id LIKE 'mock-email-%'` to `purge_mock_data()`. Same pattern for email_signals with mock email IDs.

## Acceptance Criteria

1. `full` scenario seeds 15-20 emails across 3 accounts + 2 unlinked
2. Emails distributed across priority tiers: ~4 high, ~7 medium, ~5 low
3. Enrichment states distributed: ~10 enriched, ~3 pending, ~2 noise-pending
4. At least 3 emails have `user_is_last_sender = false` with age > 24h (reply debt scenarios)
5. At least 2 emails have `pinned_at` set
6. At least 2 emails have commitments, 2 have questions
7. 12+ email_signals rows seeded (sentiment, commitment, urgency, pre_meeting_context types)
8. `entity_email_cadence` seeded for 3 accounts with varied scenarios (normal, gone quiet, insufficient)
9. Relevance scores distributed across all 3 bands
10. `purge_mock_data()` cleans all `mock-email-*` rows
11. The Correspondent page renders all sections (YOUR MOVE, COMMITMENTS, UPDATES, INBOX with score bands) from mock data without errors

# I561 — Onboarding: Three Connectors

**Version:** v1.0.0 Phase 5
**Depends on:** I535 (Glean intelligence provider — account discovery), I560 (connector optimization — automatic management)
**Type:** Enhancement — onboarding wizard restructure
**Scope:** Frontend (wizard flow, connector chapters) + Backend (discovery trigger, background enrichment orchestration)

---

## Problem

The current onboarding flow (I56/I57, v0.16.0) treats connector setup as a single step (Google OAuth) and treats account creation as manual-only:

```
Connect Google → Set up profile → Add accounts manually → Prime DailyOS → Done
```

This is wrong for two reasons:

1. **Claude Code is invisible.** It's required for transcript processing, local intelligence enrichment, and PTY fallback — but the wizard doesn't surface it. Users discover they need it only when features silently fail.

2. **Glean is absent.** Glean-connected users can auto-discover accounts from Salesforce, Gong, and Zendesk — skipping the tedious manual add step and getting full intelligence in minutes instead of days.

### What this spec corrects

The onboarding wizard should present **three connectors** as explicit, independent setup steps. Each can be skipped, but the wizard makes clear what each enables.

---

## Design

### Three Connectors Model

```
Step 1: Connect Google Calendar + Gmail
        ↓ (required for schedule, emails, briefings)

Step 2: Connect Claude Code
        ↓ (required for transcripts, local intelligence)

Step 3: Connect Glean
        ↓ (optional — enables account discovery + enterprise intelligence)

Step 4: Set up profile (pre-filled from Glean if connected)

Step 5: Add accounts
        ├── Glean connected → "We found N accounts" (auto-discovery)
        └── No Glean → Add manually (existing flow)

Step 6: Prime DailyOS → Done
```

**Key principle:** No connector replaces another. Google provides data (calendar, email). Claude Code provides compute (transcript extraction, PTY enrichment). Glean provides enterprise context (CRM, Zendesk, Gong, org chart). All three are additive.

### Step 1: Connect Google (unchanged)

Existing Google OAuth flow. Required for calendar events, email fetch, and daily briefing.

### Step 2: Connect Claude Code (NEW)

```
Power Your Briefings

DailyOS uses Claude to analyze your meetings, extract action items
from transcripts, and build intelligence about your accounts.

Requirements:
  • Claude Pro or Max subscription
  • Claude Code CLI installed

  [Verify Connection]     [Skip for now]
```

**Behavior:**
- "Verify Connection" runs a lightweight PTY health check (spawn Claude, echo test, confirm version)
- On success: green checkmark, "Claude Code connected" — proceed
- On failure: show specific error (not installed, not authenticated, wrong version)
- "Skip for now" proceeds — transcript processing and local enrichment are disabled, user sees a setup banner in Settings

### Step 3: Connect Glean (NEW)

```
Connect Your Company Knowledge

Glean connects DailyOS to your company's Salesforce, Zendesk, Gong,
and Slack — so your briefings include the full picture, not just
what's in your calendar.

  [Connect Glean]     [Skip — I'll set up later]
```

**Behavior:**
- "Connect Glean" triggers existing `start_glean_auth` OAuth flow
- On success: immediately trigger account discovery in background
- If user already has Glean token in Keychain, auto-detect and skip to discovery
- "Skip" proceeds to manual profile + account setup

### Step 4: Set Up Profile

Existing profile step (name, title, company, priorities). If Glean is connected, pre-fill from org directory:

```rust
let user_profile = glean_provider.chat(
    "Find the profile for {email}. Return JSON: { name, title, department, company }"
).await;
```

User confirms or edits. If Glean lookup fails, form is blank (existing behavior).

### Step 5: Add Accounts

**If Glean connected** → Show discovery results:

```
We Found Your Accounts

Based on your activity in Salesforce, Gong, and Zendesk:

  ☑ Nexus Analytics          TAM (co-owner)       via Gong calls
  ☑ Globex Holdings       Involved             via Zendesk tickets
  ☑ Juniper Software    Owner                via Salesforce
  ☑ CloudCo           Owner                via Salesforce
  ☐ Keystone Finance     TAM                  via Zendesk tickets
  ... and 4 more

  [Add Selected (4)]     [Add All (8)]     [Skip — I'll add later]
```

- All accounts checked by default
- "Add Selected" creates entities and starts background Glean enrichment
- Enrichment runs in parallel while onboarding continues — user doesn't wait
- Accounts already in DailyOS show "Already added" badge

**If Glean not connected** → Existing manual account add flow (unchanged).

### Step 6: Prime DailyOS

Existing priming step. If Glean enrichment is still running, show progress:

```
Getting Ready

  Nexus Analytics          ████████████████████  Complete ✓
  Globex Holdings       ████████████░░░░░░░░  Analyzing...
  Juniper Software    ░░░░░░░░░░░░░░░░░░░░  Queued

  [Continue to DailyOS]
```

"Continue to DailyOS" is always available — enrichment continues in background.

## Backend

### New Commands

```rust
#[tauri::command]
pub async fn onboarding_check_claude_code(
    state: State<'_, Arc<AppState>>,
) -> Result<ClaudeCodeStatus, String>
// PTY health check: spawn, echo, confirm version

#[tauri::command]
pub async fn onboarding_discover_accounts(
    state: State<'_, Arc<AppState>>,
) -> Result<Vec<DiscoveredAccount>, String>
// Calls discover_accounts_from_glean (I535) with the user's email

#[tauri::command]
pub async fn onboarding_import_accounts(
    state: State<'_, Arc<AppState>>,
    account_names: Vec<String>,
) -> Result<OnboardingImportResult, String>
// Creates entities + enqueues Glean enrichment for each

#[tauri::command]
pub async fn onboarding_enrichment_status(
    state: State<'_, Arc<AppState>>,
) -> Result<Vec<EnrichmentProgress>, String>
// Returns per-account enrichment progress for the onboarding UI

#[tauri::command]
pub async fn onboarding_prefill_profile(
    state: State<'_, Arc<AppState>>,
) -> Result<Option<UserProfileSuggestion>, String>
// Queries Glean for the user's profile data
```

### Enrichment Orchestration

During onboarding, enrichment runs at `Priority::Onboarding` (higher than `ProactiveHygiene`, lower than `UserTriggered`). Max 3 concurrent Glean `chat` calls. Total timeout: 5 minutes — remaining accounts enqueued at normal priority.

## Acceptance Criteria

### Connector Flow
1. Wizard presents three connector steps in order: Google, Claude Code, Glean.
2. Each connector can be skipped. Skipping shows what features are unavailable without it.
3. Claude Code verification runs a PTY health check. Success shows version; failure shows specific remediation (not installed / not authenticated).
4. Glean OAuth uses existing `start_glean_auth`. On success, account discovery begins immediately in background.
5. If user already has Glean token in Keychain, Step 3 is auto-skipped and discovery starts.

### Account Discovery (Glean connected)
6. After Glean auth, wizard shows "We Found N Accounts" with checkboxes, roles, and evidence sources.
7. "Add Selected" creates entities and starts background Glean enrichment.
8. Enrichment runs in parallel while onboarding continues — user is never blocked.
9. "Continue to DailyOS" is always available. Enrichment continues in background.

### Profile Pre-fill
10. If Glean connected, profile step pre-fills name, title, department, company from org directory. User confirms or edits.
11. If Glean lookup fails, form is blank (existing behavior) — no error shown.

### Backend
12. `onboarding_check_claude_code` spawns a PTY test and returns status within 5 seconds.
13. `onboarding_import_accounts` creates entities and enqueues Glean enrichment at `Priority::Onboarding`.
14. Max 3 concurrent Glean `chat` calls during onboarding. Total timeout: 5 minutes.
15. Accounts not enriched within timeout are enqueued at normal priority.
16. `onboarding_enrichment_status` returns real-time progress per account (polled every 3 seconds).

### Intelligence Loop Integration
17. Each account created via `onboarding_import_accounts` emits `entity_created` signal via `emit_signal_and_propagate()`.
18. Profile pre-fill data committed to `user_entity` BEFORE any enrichment begins. `build_intelligence_context()` includes the user's role/title.
19. After enrichment completes, `entity_assessment.health_json` is non-null with all 6 dimensions scored.
20. `Priority::UserTriggered` always preempts `Priority::Onboarding`.

### Editorial Standards
21. All onboarding screens use editorial treatment: Newsreader headings, DM Sans body, design tokens.
22. Zero ADR-0083 vocabulary violations in user-facing copy.
23. Loading states use EditorialLoading patterns.

### Mock Data
24. Mock data (`full` scenario) seeds mock Glean discovery response with 8 accounts (3 matching existing, 5 new).
25. Mock data seeds mock Claude Code health check response.
26. DevTools scenario `onboarding_glean` resets onboarding state for testing the full flow.

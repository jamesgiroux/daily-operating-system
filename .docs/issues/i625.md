# I625 — Provenance inline markers: editorial attribution scoped to Glean sub-sources

**Area:** Frontend / Components + Backend / Data
**Priority:** P1
**Version:** v1.1.0
**Effort:** Medium
**Depends on:** ADR-0098 DataSource enum (existing), Glean MCP integration (existing)
**Source:** CEO plan review (2026-03-21). Scoped to Glean sub-sources per user feedback — "if we can scope to things within Glean, such as Salesforce, Gong, etc."

---

## Problem

AI-sourced and CSM-entered data on the account detail page are visually indistinguishable. When the dossier shows ARR as "$137,000," the CSM can't tell whether that came from Salesforce (via Glean), AI inference, or their own input. When the intelligence narrative says "strong renewal signal," they don't know if that's based on CRM data, email analysis, or Gong call patterns.

This matters for trust. CSMs need to know the provenance of data to evaluate its reliability. "Via Salesforce" carries different weight than "AI synthesis."

## Design Principle

**Editorial attribution, not badges.** Provenance uses subtle inline markers that match the magazine aesthetic (ADR-0073/0076). Like attribution lines in a newspaper article — small, muted, informative. Not color-coded badges, not icons, not a separate "sources" panel.

## Solution

### Provenance Labels

Subtle text line below key fields showing source + timestamp:

```
ARR: $137,000
┊ you updated · Mar 10

Health: 78 (Green)
┊ 6 dimensions · last scored 2h ago

Renewal Outlook:
"Strong renewal signal — order form signed."
┊ AI synthesis · Mar 15
```

### Source Labels (scoped to Glean sub-sources)

| Source | Label | Confidence Tier |
|--------|-------|-----------------|
| Salesforce (via Glean) | "via Salesforce" | 0.9 |
| Zendesk (via Glean) | "via Zendesk" | 0.85 |
| Gong (via Glean) | "via Gong" | 0.8 |
| Glean AI synthesis | "via Glean" | 0.7 |
| CSM input | "you noted" or "you updated" | 1.0 |
| AI enrichment (PTY) | "AI synthesis" | varies |
| System | "system" (not shown to user) | — |

These map to existing signal confidence tiers from ADR-0100.

### Conflict Indicators

When Glean returns a value that differs from the CSM-entered value for the same field:

```
ARR: $137,000
┊ you updated · Mar 10
┊ Salesforce shows $142,000  [Accept] [Dismiss]
```

- **Accept:** Updates field value, changes source to Glean sub-source, emits signal
- **Dismiss:** Hides the suggestion for this field, emits negative feedback signal (penalizes the Glean source for this field type via Bayesian weights)

### Data Model

Per-field provenance tracking. Two approaches (implementation decides):

**Option A: Dedicated provenance columns.** Add `_source TEXT` and `_updated_at TEXT` columns for each provenance-tracked field on the `accounts` table (e.g., `arr_source`, `arr_updated_at`, `lifecycle_source`, `lifecycle_updated_at`). Simple but adds columns.

**Option B: Provenance JSON column.** Add `provenance_json TEXT` column to `accounts` with structure: `{"arr": {"source": "glean", "sub_source": "salesforce", "updated_at": "...", "glean_value": 142000}, ...}`. Single column but requires JSON parsing.

Recommendation: **Option A** for v1.1.0 (simpler queries, no JSON parsing overhead). Can migrate to Option B later if the number of tracked fields grows beyond 5-6.

### Fields with Provenance

For v1.1.0, track provenance on these account fields:
- ARR (`arr`)
- Lifecycle (`lifecycle`)
- Health score (`health` — already has scoring metadata, extend with source)
- Renewal date (`contract_end`)
- NPS (`nps`)

Product provenance is handled by I624's `source` column on `account_products`.

### Frontend Component

```tsx
<ProvenanceLabel
  source="salesforce"       // glean sub-source or "user" or "ai"
  timestamp="2026-03-15"    // when this value was set
  conflict={{               // optional: when Glean disagrees
    source: "salesforce",
    value: "$142,000",
    onAccept: () => ...,
    onDismiss: () => ...,
  }}
/>
```

**Design tokens:**
- Font: `var(--font-mono)` at `var(--font-xs)` size (JetBrains Mono, 11px)
- Color: `var(--color-text-muted)` for attribution text
- Left border: 1px `var(--color-paper-linen)` for visual grouping (the ┊ character)
- Conflict line: `var(--color-spice-turmeric)` for the suggested value
- Accept/Dismiss: small text buttons, not prominent

## Acceptance Criteria

1. ARR, lifecycle, renewal date, and NPS on account detail show provenance attribution: source label + relative timestamp.
2. CSM-entered values show "you updated · {date}." Glean-sourced values show "via {sub-source} · {date}."
3. When Glean returns a different value than CSM entered for the same field, conflict indicator appears: "{sub-source} shows {value} [Accept] [Dismiss]."
4. Accept updates the field and source. Dismiss hides the suggestion.
5. Accept emits a signal. Dismiss emits negative feedback signal.
6. Provenance labels use `--font-mono` at `--font-xs`, `--color-text-muted`. No badges, no icons.
7. Provenance doesn't clutter the hero section — labels are subtle and scannable.
8. Fields without provenance data (source not tracked yet) show no label — don't show "unknown source."
9. Health score provenance shows "6 dimensions · last scored {time}" (existing metadata, not a new source field).
10. `ProvenanceLabel` component reusable — can be applied to any field, not hardcoded to account detail.

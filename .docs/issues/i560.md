# I560 — Glean Mode Connector Optimization + Token Lifecycle + Data Governance

**Version:** v1.0.0 Phase 5
**Depends on:** I535 (Glean intelligence provider)
**Type:** Enhancement — connector management, auth UX, data governance
**Scope:** Backend (connector control, token monitoring, purge fixes) + Frontend (status indicators, re-auth flow, notifications)

---

## Problem

Three interconnected issues with the Glean-connected experience:

### 1. Additive vs Governed is a confusing user choice

The current Settings UI presents a radio button: "Additive" (Glean + local connectors active) vs "Governed" (Glean replaces local connectors). This distinction was designed for ADR-0095's "Glean as supplementary context" model. In the Glean-first world (ADR-0100), the question is no longer "which context feeds the prompt" — the `chat` tool does its own synthesis. The real decision should be made automatically based on what's optimal, not exposed as a confusing toggle.

### 2. Token expiry makes Glean mode unreliable

The Glean DCR token expires after ~7 days. When it expires:
- Enrichment silently falls back to PTY (user doesn't know)
- The refresh token also fails (DCR client registration may expire)
- Re-auth requires full OAuth consent flow (browser SSO)
- Current architecture requires app restart to reinitialize the context provider (`Arc<dyn ContextProvider>` is set at `AppState::new()` time)

This makes Glean-first mode feel broken after a week. No warning, no graceful handling, no in-app recovery.

### 3. PII and company data governance gaps

The I559 spike + PII audit revealed:
- Glean-derived intelligence (`entity_assessment.*`) persists after Glean disconnection — `purge_source()` explicitly skips it
- `org_health_json` (Salesforce CRM data via Glean) persists after disconnection
- `emails` table not cleared on Google OAuth revocation
- `glean_document_cache` has no scheduled TTL enforcement — stale entries accumulate
- AI-synthesized text containing company data (from Zendesk tickets, Salesforce, Gong) is stored permanently in `entity_assessment`

For enterprise deployment, the question is: when a user disconnects from Glean (or leaves the company), what company data remains on their local machine?

## Solution

### 1. Remove Additive/Governed — Automatic Connector Optimization

Replace the Additive/Governed radio buttons with automatic, intelligent connector management:

```
Glean connected:
  ├── Calendar: ALWAYS ACTIVE (scheduling backbone, prep timing)
  ├── Transcript processing: ALWAYS ACTIVE (local text > Gong summaries)
  ├── Gmail: ACTIVE, reduced frequency (15 min → 30 min)
  │   └── Email signals still extracted — personal context Glean doesn't have
  │   └── If Glean indexes Gmail, email_signals from both sources are deduped
  ├── Google Drive: DISABLED (Glean indexes it more completely)
  ├── Clay: ACTIVE as secondary enrichment
  │   └── Clay detects LinkedIn changes faster than Glean
  │   └── Clay data augments Glean contacts, doesn't replace
  ├── Gravatar: ACTIVE for avatars only
  │   └── Primary function is photo_url — Glean doesn't provide avatars
  ├── Enrichment: Glean chat primary, PTY fallback
  └── Linear: ACTIVE (not in Glean's index)

Glean NOT connected:
  ├── Everything active as before
  └── PTY enrichment is the only path
```

**Settings UI change:** Remove the Additive/Governed radio buttons. Show an informational connector status panel:

```
Connectors
  Google Calendar    ● Active
  Gmail              ● Active (every 30 min)
  Google Drive       ○ Managed by Glean
  Clay               ● Active (secondary)
  Gravatar           ● Active (avatars)
  Glean              ● Connected — enrichment primary
```

No user choice needed. The system makes the right decision.

**Backend change:** Remove `GleanStrategy` enum. The connector control logic moves from a user toggle to a function:

```rust
pub fn should_run_connector(connector: Connector, glean_connected: bool) -> ConnectorBehavior {
    match (connector, glean_connected) {
        (Connector::Calendar, _) => ConnectorBehavior::Active,
        (Connector::Transcripts, _) => ConnectorBehavior::Active,
        (Connector::Gmail, true) => ConnectorBehavior::ReducedFrequency(Duration::from_secs(1800)),
        (Connector::Gmail, false) => ConnectorBehavior::Active,
        (Connector::GoogleDrive, true) => ConnectorBehavior::Disabled("Managed by Glean"),
        (Connector::GoogleDrive, false) => ConnectorBehavior::Active,
        (Connector::Clay, _) => ConnectorBehavior::Active,
        (Connector::Gravatar, _) => ConnectorBehavior::Active,
        (Connector::Linear, _) => ConnectorBehavior::Active,
    }
}
```

### 2. Token Lifecycle Management

#### 2a. Proactive token health monitoring

Add a background task that checks token expiry on every enrichment cycle (and independently every 6 hours):

```rust
pub async fn check_glean_token_health() -> GleanTokenHealth {
    match token_store::load_token() {
        Ok(token) => {
            let expiry = token.expiry_datetime();
            let now = Utc::now();
            let remaining = expiry - now;

            if remaining < Duration::zero() {
                GleanTokenHealth::Expired
            } else if remaining < Duration::hours(24) {
                GleanTokenHealth::ExpiringsSoon { hours_remaining: remaining.num_hours() }
            } else {
                GleanTokenHealth::Healthy { expires_at: expiry }
            }
        }
        Err(_) => GleanTokenHealth::NotConfigured,
    }
}
```

#### 2b. Pre-expiry notification

When token health is `ExpiringSoon` (< 24h remaining):

- Emit a frontend event `glean-token-expiring` with hours remaining
- Show a non-intrusive banner in the app: "Glean connection expires in {N} hours. [Reconnect now]"
- The banner uses the same editorial treatment as the offline mode indicator
- Clicking "Reconnect now" triggers the OAuth flow without navigating away

When token health is `Expired`:

- Emit `glean-token-expired`
- Show a status indicator: "Running in local mode — Glean disconnected. [Reconnect]"
- Enrichment automatically falls back to PTY (no blank screens, no errors)
- All Glean-sourced surfaces show a "Last updated via Glean: {date}" staleness indicator

#### 2c. In-app re-auth without restart

The current architecture requires app restart because `Arc<dyn ContextProvider>` is set once at `AppState::new()`. Fix:

```rust
// In AppState:
pub glean_intelligence_provider: RwLock<Option<GleanIntelligenceProvider>>,

// On re-auth:
pub async fn reconnect_glean(&self, endpoint: &str) -> Result<()> {
    // Run OAuth flow
    let result = glean::oauth::run_glean_consent_flow(endpoint).await?;
    // Create new provider with fresh token
    let provider = GleanIntelligenceProvider::new(endpoint);
    // Swap in place — no restart needed
    *self.glean_intelligence_provider.write() = Some(provider);
    // Re-enable Glean-first enrichment
    // Emit event to frontend
    Ok(())
}
```

The key change: `GleanIntelligenceProvider` is behind a `RwLock`, not an `Arc<dyn>`. The enrichment pipeline reads it with a read lock; re-auth swaps it with a write lock. No restart.

**Auto-refresh attempt:** Before showing the "Reconnect" banner, attempt a silent token refresh. If the refresh token is still valid (< 7 days since last consent), the user never sees the banner. Only surface the re-auth prompt when silent refresh fails.

### 3. Data Governance Fixes

#### 3a. Purge gaps — fix `purge_source(DataSource::Glean)`

Add to `purge_source()`:

```sql
-- Clear Glean-derived CRM data
UPDATE entity_assessment SET org_health_json = NULL WHERE org_health_json IS NOT NULL;

-- Clear AI assessments that were Glean-synthesized
-- (tagged via source_manifest_json containing "glean" entries)
UPDATE entity_assessment
SET executive_assessment = NULL,
    risks_json = NULL,
    stakeholder_insights_json = NULL,
    current_state_json = NULL,
    company_context_json = NULL,
    dimensions_json = NULL
WHERE source_manifest_json LIKE '%glean%';
```

After purge, enqueue all affected entities for re-enrichment via PTY (local-only), so intelligence regenerates from local context.

#### 3b. Purge gaps — fix Google OAuth revocation

Add to existing Google disconnect handler:

```sql
DELETE FROM emails WHERE 1=1;
DELETE FROM email_signals WHERE 1=1;
DELETE FROM email_threads WHERE 1=1;
UPDATE meetings SET description = NULL;
```

#### 3c. Gravatar cache purge

```sql
DELETE FROM gravatar_cache WHERE 1=1;
```

Add to `purge_source(DataSource::Gravatar)`.

#### 3d. Glean document cache TTL enforcement

Add a scheduled job (runs daily) that cleans expired cache entries:

```rust
pub fn purge_expired_glean_cache(db: &ActionDb) -> Result<usize> {
    db.execute(
        "DELETE FROM glean_document_cache
         WHERE (kind = 'document' AND cached_at < datetime('now', '-1 hour'))
            OR (kind = 'person_profile' AND cached_at < datetime('now', '-1 day'))
            OR (kind = 'org_graph' AND cached_at < datetime('now', '-4 hours'))",
    )
}
```

#### 3e. Data residency awareness

For enterprise deployments, add a Settings → Data section that clearly states:

```
Data Storage

All data is stored locally on this device at:
  ~/Library/Application Support/com.dailyos.desktop/data.db

What's stored:
  • Calendar events (title, time, attendees)
  • Email metadata (subject, sender, AI summary — NOT full email bodies)
  • Contact profiles (name, email, title, organization)
  • AI-generated intelligence (assessments, risks, recommendations)
  • Meeting transcripts (full text, on filesystem)

What's NOT stored:
  • Full email bodies (only Gmail snippets)
  • Glean search results (cached temporarily, purged on disconnect)
  • Passwords or authentication tokens (stored in macOS Keychain)

When you disconnect a service, all data from that service is purged.
```

## Out of Scope

- End-to-end encryption of local SQLite (post-1.0, I527)
- Remote wipe capability (enterprise MDM handles this)
- GDPR data export of stored PII (I429 data export covers this)
- Extending DCR token lifetime (Glean admin controls this)

## Acceptance Criteria

### Connector Optimization
1. Settings → Context Source no longer shows Additive/Governed radio buttons when Glean is connected.
2. Settings shows connector status panel: Calendar (active), Gmail (active, 30 min), Google Drive (managed by Glean), Clay (active), Gravatar (active), Glean (connected).
3. With Glean connected, Google Drive poller is automatically disabled. Gmail poller runs at 30-minute intervals instead of 5-minute.
4. With Glean disconnected, all connectors revert to normal behavior.
5. Clay and Gravatar remain active in Glean mode — Clay for LinkedIn title changes, Gravatar for avatars.
6. `GleanStrategy` enum removed from codebase. `ContextMode::Glean` no longer has a `strategy` field.

### Token Lifecycle
7. Token health checked on every enrichment cycle and independently every 6 hours.
8. When token expires in < 24 hours, a non-intrusive banner appears: "Glean connection expires soon. [Reconnect]"
9. When token is expired, banner shows: "Running in local mode — Glean disconnected. [Reconnect]"
10. Clicking "Reconnect" runs OAuth consent flow. On success, Glean enrichment resumes **without app restart**.
11. Silent token refresh attempted before showing the banner. If refresh succeeds, user never sees the notification.
12. During token expiry, enrichment falls back to PTY transparently. No blank screens, no errors, no data loss.

### Data Governance
13. `purge_source(DataSource::Glean)` clears `org_health_json` and Glean-tagged fields in `entity_assessment`. After purge, affected entities enqueued for PTY re-enrichment.
14. Google OAuth revocation clears `emails`, `email_signals`, `email_threads` tables and nulls `meetings.description`.
15. Gravatar disconnect clears `gravatar_cache` table.
16. `purge_expired_glean_cache()` runs daily via scheduler. No stale cache entries older than TTL.
17. Settings → Data shows clear, human-readable description of what's stored locally and what's not.

### Integration
18. `ContextMode::Glean` serialization is backward-compatible — old `mode_json` with `strategy` field still deserializes (defaults to connector auto-optimization).
19. Migration updates existing `context_mode_config` rows to remove `strategy` field.
### Intelligence Loop Integration
20. After `purge_source(DataSource::Glean)` runs and entities are re-enqueued for PTY re-enrichment, existing `user_edits_json` is preserved. Re-enrichment reads user edits and does not overwrite them.
21. Connector status change (Glean connected → disconnected) marks all health scores as stale. AccountsPage shows staleness indicator until PTY re-enrichment completes.
22. During Glean token expiry, Glean-sourced callout signals older than 7 days are suppressed from the morning briefing (or marked "Last updated via Glean: {date}"). Verify: after `glean_expired` mock scenario, briefing does not show stale Glean callouts as current.

### Mock Data
21. Mock data (`full` scenario) seeds `context_mode_config` with Glean mode active (endpoint: mock URL). Connector status panel renders correctly showing automatic optimization.
22. Mock data seeds a mock `GleanTokenHealth` state cycle: one account with `Healthy` (expires in 5 days), accessible via `get_glean_token_health` mock response.
23. DevTools scenario includes a `glean_expiring` variant that simulates `ExpiringSoon` (2 hours remaining) — triggers the pre-expiry banner in the UI for visual development.
24. DevTools scenario includes a `glean_expired` variant that simulates `Expired` — triggers the "Running in local mode" banner and PTY fallback indicator.
25. After applying `full` mock scenario, Settings → Context Source shows the connector status panel (no radio buttons). Settings → Data shows the data residency description. No console errors.

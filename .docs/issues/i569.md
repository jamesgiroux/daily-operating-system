# I569 — Glean context provider async bridging

**Priority:** P1
**Area:** Backend / Performance
**Version:** v1.0.0 (Phase 6)
**Depends on:** None
**Type:** Enhancement

## Problem Statement

The `GleanContextProvider` in `context_provider/glean.rs` uses `tokio::task::block_in_place(|| handle.block_on(...))` in two locations to run async network calls from the synchronous `ContextProvider` trait:

1. **Line 113** — `get_token()`: `block_in_place(|| handle.block_on(crate::glean::get_valid_access_token()))` to refresh an expired Glean OAuth token.
2. **Lines 1078-1079** — `gather_entity_context()` impl: `block_in_place(|| handle.block_on(self.gather_glean_context(db, entity_id, entity_type, &gap_q)))` for the full Glean context-gathering pipeline (search, chat, document fetch).

`block_in_place` tells Tokio "I'm about to block this thread — move other tasks away." But with the default `multi_thread` runtime (typically 4-8 worker threads), if multiple entities are being enriched simultaneously, this saturates the thread pool. Glean `chat` calls take 10-30 seconds (`GLEAN_CHAT_TIMEOUT = 60s`), so even 2-3 concurrent enrichments can starve the entire Tokio runtime — no other async work (IPC handlers, email sync, timer ticks) can proceed.

### The ContextProvider trait constraint

The `ContextProvider` trait (`context_provider/mod.rs:91-108`) is synchronous:

```rust
pub trait ContextProvider: Send + Sync {
    fn gather_entity_context(
        &self,
        db: &ActionDb,
        entity_id: &str,
        entity_type: &str,
        prior: Option<&IntelligenceJson>,
    ) -> Result<IntelligenceContext, ContextError>;
}
```

It returns `Result<IntelligenceContext, ContextError>`, not a `Future`. This trait is called from `intel_queue.rs` (the background enrichment processor). The local provider (`context_provider/local.rs`) is pure DB reads — legitimately sync. Glean is the outlier: it makes network calls that are inherently async.

## Solution

Replace `block_in_place` + `block_on` with `tokio::task::spawn_blocking` at the call site in `intel_queue.rs`, or restructure the trait to support async gathering.

### Option A: spawn_blocking wrapper in intel_queue (minimal change)

The `intel_queue.rs` processor already runs on a Tokio async task. Before calling `provider.gather_entity_context()`, wrap the call in `spawn_blocking`:

```rust
let ctx = tokio::task::spawn_blocking(move || {
    provider.gather_entity_context(&db, &entity_id, &entity_type, prior.as_ref())
}).await.map_err(|e| format!("spawn_blocking join error: {e}"))??;
```

This moves the entire `gather_entity_context` call (including the internal `block_on` for Glean) off the Tokio worker threads entirely. The `block_on` inside `GleanContextProvider` then runs on a dedicated OS thread where blocking is safe.

### Option B: Async trait (larger refactor)

Add an `async fn gather_entity_context_async()` method with a default implementation that calls the sync version. Glean overrides it with a native async implementation. `intel_queue.rs` calls the async version.

**Recommendation: Option A.** It's a 5-line change in `intel_queue.rs`, eliminates the thread starvation risk, and doesn't require trait changes. The `block_in_place` calls in `glean.rs` become harmless (they're on a `spawn_blocking` thread, not a Tokio worker). Option B is cleaner long-term but can be done as a follow-up when `IntelligenceProvider` (ADR-0091) lands.

### Token refresh

The `get_token()` method (line 111-122) also uses `block_in_place`. With Option A, this is automatically handled — the entire provider call runs on a blocking thread. No separate fix needed.

## Acceptance Criteria

1. No `block_in_place` calls execute on Tokio worker threads during Glean context gathering. Either: (a) the call is wrapped in `spawn_blocking` at the caller, or (b) the trait method is natively async for remote providers.
2. Concurrent enrichment of 3+ entities with Glean provider does not block IPC command handlers. Test: trigger 3 entity enrichments, then call a fast IPC command (e.g., `get_config`) — it responds in <100ms, not queued behind enrichment.
3. Glean context gathering still works end-to-end: entity intelligence is produced with Glean search + chat data when the provider is active.
4. Local provider behavior is unchanged — no `spawn_blocking` overhead for pure DB reads.
5. `cargo test` — all pass.
6. `cargo clippy -- -D warnings` — clean.

## What This Does NOT Do

- Does not convert `ContextProvider` to an async trait — that's a larger refactor (Option B) for post-1.0.
- Does not change the Glean MCP HTTP transport or timeout configuration.
- Does not address the `std::thread::sleep` in Keychain retry loops — that's I570.

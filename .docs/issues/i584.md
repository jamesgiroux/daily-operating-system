# I584 — Action Completion Feeds Health Scoring

**Version:** v1.0.2
**Area:** Backend / Intelligence / Actions
**Priority:** P0

## Problem

Action mutations emit signals (`action_completed`, `action_accepted`, etc.) but no propagation rules fire from them and they don't feed any health scoring dimension. "3 actions completed for Acme this week" is a strong engagement signal. "0 actions completed in 30 days" is equally telling. The data exists — it just doesn't flow into the Intelligence Loop.

## Design

### Signal Propagation Rules

Add new rules in `signals/rules.rs`:

1. **`rule_action_velocity`** — Triggered by `action_completed` signals. For the entity, count completed actions in the last 30 days. If ≥ 3 completions, emit `positive_action_momentum` (confidence 0.7). If 0 completions in 30 days and entity has ≥ 5 pending actions, emit `action_stagnation` (confidence 0.6).

2. **`rule_overdue_action_escalation`** — Enhance existing `rule_overdue_actions()`. Currently triggers at ≥3 overdue. Add: emit `overdue_action_warning` at ≥1 overdue for P1 actions (these are user-marked critical).

### Health Scoring Dimension Integration

In `intelligence/health_scoring.rs`, feed action data into **two dimensions**:

- **Engagement dimension**: Action completion rate (completed / total pending over 30 days). High completion rate → positive engagement signal. Stagnation → negative.
- **Momentum dimension**: Trend of action velocity (are completions accelerating, steady, or declining over 60-day window). Uses the same rolling-window approach as email cadence.

New query: `get_action_velocity_for_entity(entity_id, days)` — returns `{ completed_count, pending_count, overdue_count, avg_days_to_complete }`.

### Intelligence Context

In `intelligence/prompts.rs`, add action velocity to `build_intelligence_context()`:
- "X actions completed in last 30 days, Y pending, Z overdue"
- "Average time to complete: N days"
- This gives the LLM context for generating assessments that reference action health

### Briefing Callouts

Add `action_stagnation` and `positive_action_momentum` to `CALLOUT_SIGNAL_TYPES` so they surface in the daily briefing when relevant.

## Mock Data

Requires I597 (action mock data expansion). Uses:
- Acme: 5 completions in 30 days → should trigger `positive_action_momentum`
- Globex: 2 completions in 30 days + 5 pending → near stagnation threshold
- Mock signal_events include `positive_action_momentum` (Acme) and `action_stagnation` (Globex) for briefing callout verification

## Acceptance Criteria

1. `action_completed` signal triggers `rule_action_velocity` propagation rule
2. ≥3 completions in 30 days emits `positive_action_momentum` signal
3. 0 completions + ≥5 pending in 30 days emits `action_stagnation` signal
4. P1 overdue actions emit warning at count ≥1 (not ≥3)
5. Action velocity data feeds engagement and momentum health dimensions
6. `build_intelligence_context()` includes action velocity stats
7. `action_stagnation` appears as briefing callout when triggered
8. Entity health score recalculation reflects action completion changes (verified by completing an action and checking score delta)
9. **Mock data**: Acme triggers `positive_action_momentum` (5 completions). Globex shows stagnation signal. Velocity stats render in intelligence context.

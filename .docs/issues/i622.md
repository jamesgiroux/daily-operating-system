# I622 — Fix past-meeting opacity: outcomes should be full brightness

**Area:** Frontend / Meeting Detail
**Priority:** P1
**Version:** v1.0.2
**Source:** v1.0.0 GA review (2026-03-17)

---

## Problem

On past meetings with transcript outcomes, the `pastMeetingOpacity` CSS class applies reduced opacity to everything below the outcomes divider — including the meeting outcomes themselves. This is wrong. The outcomes (wins, risks, decisions, actions, engagement dynamics, champion health) are the most valuable content on a past meeting page. The pre-meeting briefing is historical context that correctly belongs at lower opacity.

### Current structure (simplified)

```
MeetingDetailPage
├── PostMeetingIntelligence or OutcomesSection  ← full opacity ✓
├── outcomesDivider
├── "Pre-Meeting Context" label
└── <div className={pastMeetingOpacity}>        ← reduced opacity
    ├── ACT I: "Ground Me" (headline, risks)
    ├── ACT II: "The Room" (attendees)
    ├── ACT III: "Your Plan" (agenda, open items)
    └── FinisMarker
```

The outcomes render ABOVE the opacity div, so they're actually fine. But looking at the user's report, the issue might be more nuanced — the entire page container after `hasAnyContent || outcomes || postIntel` gets the opacity class:

```tsx
<div className={isPastMeeting && outcomes ? styles.pastMeetingOpacity : undefined}>
```

This wraps everything — including the pre-meeting briefing sections (correct) but the structural placement means it might also affect layout or visual weight of elements that appear before the div in the scroll flow.

### Root cause

Line 1057 in MeetingDetailPage.tsx:
```tsx
{(hasAnyContent || outcomes || postIntel) && (
  <div className={isPastMeeting && outcomes ? styles.pastMeetingOpacity : undefined}>
```

The `pastMeetingOpacity` class is applied to the container that holds the pre-meeting briefing content. The outcomes/post-intel render in a sibling div above this. However, the visual experience reads as "the whole bottom section is faded" because:
1. The outcomes divider creates a visual break
2. Everything below it (including the label "Pre-Meeting Context") is faded
3. The faded section is usually the larger portion of the page
4. Users perceive the fade as applying to "everything after the transcript" rather than "just the pre-meeting prep"

## Scope

1. Verify the opacity class only applies to pre-meeting briefing content (Acts I-III), not outcomes
2. If outcomes are currently outside the opacity wrapper (they should be), confirm this visually with `/browse`
3. If the issue is perceptual (large faded section makes outcomes feel faded by context), consider:
   - Adding a stronger visual separator between outcomes and pre-meeting content
   - Reducing the opacity difference (current value vs. a subtler fade)
   - Adding a "Pre-Meeting Context" header that clearly labels the faded section as historical

### Files

- `src/pages/MeetingDetailPage.tsx` — line 1057, `pastMeetingOpacity` wrapper
- `src/pages/meeting-detail.module.css` — `pastMeetingOpacity` class definition

## Acceptance Criteria

1. Meeting outcomes (PostMeetingIntelligence or OutcomesSection) render at full opacity on past meetings — verified visually.
2. Pre-meeting briefing content (Acts I-III: headline, risks, the room, your plan) renders at reduced opacity — verified visually.
3. Clear visual separation between outcomes (full opacity) and pre-meeting context (faded): a labeled divider or distinct section break.
4. The transition from full to faded doesn't create a jarring visual cliff — the fade feels intentional and editorial, not broken.
5. No opacity changes on future/current meetings — only past meetings with outcomes get the treatment.
6. Screenshot comparison before/after with `/browse`.

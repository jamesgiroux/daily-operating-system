# I572 — Audit log coverage for OAuth token lifecycle

**Priority:** P2
**Area:** Backend / Security
**Version:** v1.0.0 (Phase 6)
**Depends on:** None
**Type:** Enhancement

## Problem Statement

The audit log (ADR-0094, `audit_log.rs`) covers high-level OAuth connect/disconnect events (e.g., `start_glean_auth` in `commands/integrations.rs:2530` emits `"oauth_connected"`), but individual token operations are invisible:

- **Token save**: When a refreshed token is written to Keychain, no audit record is created.
- **Token refresh**: When an expired token is refreshed via the token endpoint, no audit record is created.
- **Token delete**: When a user disconnects and the Keychain entry is removed, no audit record is created (the command-level "disconnected" event exists, but the actual credential removal is not logged).
- **Token expiry detection**: When a token is found expired and refresh is attempted, there's no record of the expiry event itself.
- **Migration**: When `load_token_macos()` migrates a legacy `token.json` to Keychain (`google_api/token_store.rs:264-268`), no audit record is created.

### Why this matters

For enterprise observability (ADR-0094), the audit log should provide a complete picture of credential lifecycle. If a token is compromised, the audit trail should show when it was last refreshed, from which flow, and when it was rotated. Currently, only the initial connect is visible — everything after is a black hole.

### Current audit log API

`AuditLogger::append(category, event, detail)` in `audit_log.rs:52`:
- `category`: `"security"`, `"data_access"`, `"ai"`, `"anomaly"`, `"config"`, `"system"`
- `event`: snake_case identifier
- `detail`: `serde_json::Value` (structured data, never PII)

The `AuditLogger` is stored in `AppState` as `Arc<Mutex<AuditLogger>>` (`state.rs:209`).

### Token store functions that need audit coverage

**`google_api/token_store.rs`:**
- `save_token()` (line 46) — called after token refresh and initial auth
- `delete_token()` (line 68) — called during disconnect
- `load_token_macos()` (line 257) — migration path from `token.json` to Keychain

**`glean/token_store.rs`:**
- `save_token()` (line 98) — called after token refresh and initial auth
- `delete_token()` (line 119) — called during disconnect

**`glean/mod.rs` (or wherever `get_valid_access_token` lives):**
- Token refresh flow — the point where an expired token is detected and refreshed

## Solution

### Approach: Audit at the call site, not inside token_store

The token store functions (`save_token`, `delete_token`, `load_token`) are low-level Keychain operations. They don't have access to `AppState` or the `AuditLogger`. Rather than threading the logger through every token store function, emit audit records at the call sites where token operations occur — these are the command handlers and auth flows that already have `State<AppState>`.

### Events to add

| Event | Category | When | Detail |
|-------|----------|------|--------|
| `token_saved` | `security` | After `save_token()` succeeds | `{"provider": "google"\|"glean", "trigger": "auth_flow"\|"refresh"\|"migration"}` |
| `token_refreshed` | `security` | After a successful token refresh | `{"provider": "google"\|"glean", "expiry_detected": true\|false}` |
| `token_deleted` | `security` | After `delete_token()` succeeds | `{"provider": "google"\|"glean", "trigger": "disconnect"\|"revoke"}` |
| `token_refresh_failed` | `security` | When token refresh fails | `{"provider": "google"\|"glean", "error_class": "network"\|"invalid_grant"\|"other"}` |
| `token_migrated` | `security` | After legacy token.json migrated to Keychain | `{"provider": "google"}` |

**Detail rules (ADR-0094):** Never include the token value, refresh token, or any credential material. Only log the provider name, trigger context, and outcome.

### Implementation locations

1. **Google token refresh** — wherever `google_api::refresh_access_token()` or equivalent is called. Emit `token_refreshed` on success, `token_refresh_failed` on failure.
2. **Glean token refresh** — `glean::get_valid_access_token()`. Emit `token_refreshed` on success.
3. **Google disconnect** — `commands/integrations.rs` disconnect handler. Emit `token_deleted` after `delete_token()`.
4. **Glean disconnect** — `commands/integrations.rs` Glean disconnect handler. Emit `token_deleted` after `delete_token()`.
5. **Keychain migration** — `load_token_macos()` is called from `load_token()` which doesn't have AppState access. The migration audit can be emitted at startup when the migration is detected (check if `token.json` was present before and is now gone).

### Threading the logger

For token refresh flows that happen in background tasks (e.g., during `refresh_emails`), the `AppState` is available as an `Arc`. Pass `Arc<Mutex<AuditLogger>>` (cloned from `state.audit_log`) to the refresh function, or emit the audit record in the caller after the refresh returns.

## Acceptance Criteria

1. Google token refresh emits a `token_refreshed` audit record with `{"provider": "google"}`.
2. Glean token refresh emits a `token_refreshed` audit record with `{"provider": "glean"}`.
3. Google token delete (disconnect) emits a `token_deleted` audit record.
4. Glean token delete (disconnect) emits a `token_deleted` audit record.
5. Failed token refresh emits a `token_refresh_failed` audit record with the error class (not the error message, which could contain sensitive info).
6. No credential material (access_token, refresh_token, client_secret) appears in any audit record detail field.
7. Audit records use category `"security"` and snake_case event names.
8. `verify_audit_log()` passes after a session that includes token refresh and disconnect operations — the hash chain is intact.
9. `cargo test` — all pass.
10. `cargo clippy -- -D warnings` — clean.

## What This Does NOT Do

- Does not add audit records for token *load* operations (reads) — that would be extremely noisy and low value.
- Does not change the audit log format, rotation policy, or hash chain mechanism.
- Does not add audit coverage for Clay/Smithery API key operations — Clay is optional and lower priority.
- Does not move audit logging inside the token store modules — the logger threading cost isn't justified for these low-level functions.

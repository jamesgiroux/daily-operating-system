# I495 — Ephemeral Account Query

**Priority:** P1
**Area:** Backend / Connectors + Frontend
**Version:** 1.1.2
**Depends on:** I494 (Glean account discovery — shares query infrastructure)

## Problem

A VP in a meeting hears about an account their colleague manages. They want to know what the org knows about it — ARR, health, recent activity, key contacts — without creating a persistent entity. Today, the only way to learn about an account is to create it first, wait for enrichment, then read the intelligence. That's a 5-minute commitment for what should be a 10-second lookup.

"Tell me about X" is the most natural question a chief of staff should answer. The answer should be instant, transient, and offer a path to persistence if the user wants to track the account long-term.

## Design

### Backend — `query_ephemeral_account` Command

New Tauri command that queries Glean for a given account name and synthesizes a transient briefing.

```rust
#[tauri::command]
async fn query_ephemeral_account(
    name: String,
    state: tauri::State<'_, AppState>,
) -> Result<EphemeralBriefing, String>
```

**Query flow:**
1. Check if `name` matches an existing DailyOS account (case-insensitive). If yes, return a redirect hint: `{ redirect_to: account_id }` so the frontend can navigate to the existing account detail page instead.
2. Send Glean queries shaped by active role preset vocabulary:
   - `"{name}"` — broad match
   - `"{name} account"` — CRM context
   - `"{name} renewal"` / `"{name} health"` / `"{name} ARR"` — preset-specific (reuse I482 query patterns)
3. Assemble results into a structured briefing.
4. Return the briefing. Do NOT persist to DB. Do NOT create an entity.

**Response shape:**
```rust
struct EphemeralBriefing {
    account_name: String,
    summary: String,              // 2-3 sentence narrative synthesis
    sections: Vec<BriefingSection>,  // Structured sections from Glean data
    source_count: usize,          // Number of Glean documents referenced
    source_urls: Vec<String>,     // Attribution links
    can_add: bool,                // true if not already in DailyOS
    existing_account_id: Option<String>,  // set if already exists
}

struct BriefingSection {
    heading: String,    // e.g., "Key Metrics", "Recent Activity", "Key People"
    content: String,    // Markdown-ish text content
    source: String,     // e.g., "Salesforce", "Gong", "Zendesk"
}
```

**Section assembly:** Group Glean results by source type where identifiable (CRM data, meeting notes, support tickets, documents). Each becomes a `BriefingSection`. If source type is unclear, use "Organization Knowledge" as the heading. The summary is a synthesized narrative across all sections.

**No caching:** Ephemeral queries are not cached. Each query hits Glean fresh. This is intentional — the user is asking "what do we know right now?" not "what did we know 4 hours ago."

### Frontend — EphemeralBriefing Component

New component `src/components/editorial/EphemeralBriefing.tsx` that renders the transient briefing.

**Entry points:**
1. **Accounts page:** New "Look up an account" text input in the discovery panel (I494). User types a name, hits Enter, sees ephemeral briefing inline.
2. **Cmd+K search (future, when I427 ships):** Search for an account name not in local DB. Results include "Look up in Glean" option that renders the ephemeral briefing.

For v1.1.2, the primary entry point is the Accounts page.

**UI on Accounts page:**

Add a search-style input below the discovery button area:

```
┌──────────────────────────────────────────────────────────┐
│  Tell me about...                                        │  ← search-input, placeholder
│  [Acme Corp                                        ⏎]   │  ← user types, Enter to query

```

**Briefing rendering (inline, below the input):**

```

│                                                          │
│  KEY METRICS                            from Salesforce  │  ← Margin grid: mono label + source
│  ARR: $1.2M · Health: Green · Renewal: Oct 2026         │  ← DM Sans 15px
│                                                          │

│                                                          │
│  ─────────────────────────────────────────────           │
│  Based on 8 documents from your organization             │  ← Mono 10px, tertiary
│                                                          │
│  [+ Add to my accounts]                                  │  ← Turmeric button
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**Design principles:**
- Editorial treatment: Newsreader headline, italic narrative, margin grid sections, section rules
- No cards — this is inline content with dividers
- Source attribution per section in mono 10px, right-aligned, tertiary color
- Document count at bottom for trust: "Based on 8 documents from your organization"
- Transient feel: no save button, no bookmark. It's a lookup, not a page.

**Loading state:** Use `GeneratingProgress` component (editorial generating indicator, not a spinner). Message: "Looking up Acme Corp..."

**Empty/error states:**
- Glean not connected: "Connect Glean in Settings to look up accounts across your organization" — DM Sans 15px, tertiary, with link to Settings
- No results from Glean: "No information found for 'Acme Corp' in your organization's knowledge base" — EditorialEmpty style
- Already exists in DailyOS: Instead of briefing, show "Acme Corp is already in your accounts" with link to account detail page

**"Add to my accounts" flow:**
1. User clicks "+ Add to my accounts" button
2. Creates account entity via `create_account` with name pre-filled
3. Button changes to "Added — View account" with link to the new account detail page
4. Enrichment begins immediately via `intel_queue`
5. Bridges to I494's import flow — the created account gets the same enrichment treatment

### Keyboard Interaction

- Enter in the input field triggers the query
- Escape clears the briefing and input
- If the input is empty and user presses Enter, no action

## Acceptance Criteria

1. Enter account name not in local DB. System queries Glean. Returns briefing with available org knowledge (whatever Glean provides — could be CRM fields, support tickets, meeting mentions, docs)
2. Briefing is transient — not persisted to DB, no entity created
3. "Add to my accounts" follow-up creates entity and begins persistent enrichment (bridges to I494 flow)
4. Without Glean connected: returns "No external context available — connect Glean in Settings to discover accounts"
5. Briefing renders in editorial style — Newsreader headline, section rules, narrative synthesis
6. Loading state shows editorial generating indicator, not a spinner
7. Accessible via Accounts page (Cmd+K integration deferred to I427)

### Intelligence Loop Integration
8. After "Add to my accounts" creates an entity, health scores are computed within one enrichment cycle. `entity_assessment.health_json` is non-null. The account appears on AccountsPage with health indicators.
9. After an ephemeral briefing with risk indicators is followed by "Add to my accounts", the next enrichment produces risks in `risks_json` from Glean data. Risks meeting severity threshold surface as briefing callouts.

### Inline Editing + Feedback (I529/I530)
10. Each section in `EphemeralBriefing` renders with thumbs up/down feedback buttons on hover (I529 pattern). Feedback recorded to `intelligence_feedback` with `entity_id = NULL` (ephemeral) and `source = 'glean_chat'`.
11. Migration: `intelligence_feedback.entity_id` altered to allow NULL values (or sentinel `'ephemeral'`). Required for ephemeral feedback — I529 defined it as NOT NULL.

### Mock Data
12. Mock data (`full` scenario) seeds a mock ephemeral briefing response for a known test account name (e.g., "Aperture Science") that returns a pre-built JSON briefing matching the editorial render structure.
9. After applying `full` mock scenario, entering "Aperture Science" in the ephemeral query UI renders the mock briefing in editorial style with all sections. "Add to my accounts" button creates the entity. No console errors.

## Out of Scope

- Persisting ephemeral briefings to a history/cache — each query is fresh
- Cmd+K search integration — deferred to I427
- Briefing for entities other than accounts (people, projects) — future issue
- AI synthesis beyond Glean result assembly — the briefing uses Glean content directly, not an LLM pass
- Structured data extraction from Glean results into vitals fields — that's I482 follow-on territory
- Offline fallback — Glean requires network connectivity

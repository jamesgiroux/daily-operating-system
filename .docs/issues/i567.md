# I567 — Narrow refresh_emails semaphore scope

**Version:** v1.0.0 Phase 6
**Depends on:** I565 (typed resource permits)
**Type:** Enhancement — Backend / Performance
**Scope:** Backend: `prepare/orchestrate.rs`. No frontend changes.

---

## Problem

`refresh_emails` in `prepare/orchestrate.rs` (line 1321) acquires the heavy-work semaphore (after I565: `permits.pipeline`) at line 1324 and holds it for the entire function — a pipeline that includes:

1. **Config reads** (lines 1328–1377) — pure memory reads, ~0ms
2. **Gmail API fetch** (line 1379, `fetch_and_classify_emails`) — network I/O, 2–10s
3. **DB persistence** (lines 1392–1444) — SQLite writes, ~50ms
4. **Sent thread position update** (lines 1447–1462) — Gmail API + DB, 1–3s
5. **Email enrichment via PTY** (lines 1466–1478, `enrich_pending_emails_two_phase`) — PTY subprocess per email, 10–60s total
6. **Signal emission** (lines 1480–1489) — DB writes, ~10ms
7. **Email scoring** (lines 1493–1530) — ONNX inference, 2–10s
8. **Directive write** (remaining lines) — file I/O, ~10ms

Only step 5 (PTY enrichment) needs the pipeline/PTY permit. Steps 1–4 are network + DB work that doesn't contend with any other heavy resource. Steps 6–8 are fast DB/inference operations.

Holding the permit for the entire 30–300 second pipeline means:
- Manual refresh is blocked for the full duration (even after I565, if `permits.pipeline` is held)
- Other scheduled pipelines (`prepare_day`, `prepare_week`) queue behind email refresh
- The email fetch itself (2–10s of network I/O) unnecessarily holds the permit

---

## Solution

### 1. Acquire permit only around PTY enrichment phase

Move the permit acquisition from the function entry (line 1324) to just before the PTY enrichment call (line 1466):

```rust
// Lines 1321-1377: Config reads — no permit needed
// Lines 1379-1444: Gmail fetch + DB persist — no permit needed
// Lines 1447-1462: Sent thread update — no permit needed

// Line 1466: PTY enrichment — needs permit
let _pty_permit = state.permits.pty.acquire().await.map_err(|_| {
    ExecutionError::ConfigurationError("PTY permit closed".to_string())
})?;
let enriched = super::email_enrich::enrich_pending_emails_two_phase(
    state, workspace, &ai_config, 20
);
drop(_pty_permit); // Release immediately after PTY work

// Lines 1480-1530: Signal emission + scoring — no permit needed
```

### 2. Use PTY permit, not pipeline permit

After I565, email enrichment's PTY call should use `permits.pty` (not `permits.pipeline`), since the resource being serialized is the PTY subprocess. The pipeline permit should only be used if the entire pipeline must be serialized against other pipelines — which it doesn't need to be. Email fetch, DB writes, and scoring can safely overlap with other pipelines.

If pipeline-level serialization is needed to prevent duplicate email fetches (two `refresh_emails` running concurrently would fetch the same emails), use a lighter-weight mechanism: an `AtomicBool` flag (`email_refresh_in_progress`) checked at function entry, not a semaphore that blocks other resource types.

### 3. Scoring phase: use embedding permit if applicable

Step 7 (`score_emails`) uses ONNX inference via the embedding model. If I565 assigns embedding work to `permits.embedding`, the scoring phase should acquire `permits.embedding` briefly:

```rust
let _embed_permit = state.permits.embedding.acquire().await.map_err(|_| {
    ExecutionError::ConfigurationError("Embedding permit closed".to_string())
})?;
let scores = crate::signals::email_scoring::score_emails(...);
drop(_embed_permit);
```

This is a secondary optimization — the primary fix is narrowing the PTY permit scope.

## Acceptance Criteria

1. `refresh_emails` does NOT hold any semaphore/permit during the Gmail API fetch phase (`fetch_and_classify_emails`). Verified by log: permit acquisition timestamp is AFTER email fetch completion.
2. `refresh_emails` does NOT hold any permit during DB email persistence (lines 1392–1444).
3. `refresh_emails` acquires the PTY permit (`permits.pty`) only for the `enrich_pending_emails_two_phase` call and releases it immediately after.
4. During email fetch (2–10s), user-initiated manual refresh can acquire its own permit and start enrichment concurrently.
5. Two concurrent `refresh_emails` calls do not produce duplicate email records — either a dedup guard prevents the second call, or `upsert_email` handles the collision idempotently (it already does via `ON CONFLICT`).
6. The email scoring phase does not block PTY enrichment — scoring and PTY use different permits (or scoring runs without any permit).
7. Total wall-clock time of `refresh_emails` is unchanged — narrowing the permit scope doesn't add latency.
8. `cargo test` passes. `cargo clippy -- -D warnings` clean.

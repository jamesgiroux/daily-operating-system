# I494 — Glean Account Discovery Flow

**Priority:** P1
**Area:** Backend / Connectors + Frontend
**Version:** 1.1.2
**Depends on:** Glean connector (v0.15.2), I486 (Glean person writeback, v1.1.0)

## Problem

The cold-start problem is the biggest barrier to VP adoption. "Go set up all your accounts" is a week of data entry. A VP with 20 accounts in Salesforce should not have to manually create each one in DailyOS. If Glean already indexes the org's CRM, we can present a list of discoverable accounts and let the user import them with pre-populated context in one click.

Without this, every new user faces the same friction: manually creating accounts, typing names, leaving vitals empty, waiting for enrichment to fill in what the CRM already knows.

## Design

### Backend — `discover_accounts` Method

Add `discover_accounts()` to `GleanContextProvider` in `src-tauri/src/context_provider/glean.rs`.

**Query strategy:** Send targeted Glean search queries shaped by the active role preset:
- CS preset: `"my accounts"`, `"account list"`, `"customer accounts"` — queries that surface CRM account objects
- Fallback: `"accounts"`, `"customers"` if no preset is active

**Response processing:**
1. Parse Glean search results for account-like entities (company names, CRM records)
2. Extract available metadata: name, domain, industry, ARR (whatever Glean provides)
3. Match against existing DailyOS accounts by name (case-insensitive) and domain
4. Return only accounts NOT already in DailyOS

**Caching:** Results cached in memory with 4-hour TTL. Cache key: `"discover_accounts:{preset_id}"`. Cache invalidated when Glean connection status changes.

**Result shape:**
```rust
struct DiscoveredAccount {
    name: String,
    domain: Option<String>,
    industry: Option<String>,
    arr: Option<f64>,
    context_preview: String,    // 1-2 sentence summary from Glean results
    glean_source_urls: Vec<String>,  // Source documents for attribution
    already_in_dailyos: bool,   // true if matched to existing account
}
```

### Backend — New Tauri Command

```rust
#[tauri::command]
async fn discover_accounts_from_glean(
    state: tauri::State<'_, AppState>,
) -> Result<Vec<DiscoveredAccount>, String>
```

Returns discovered accounts. Errors if Glean not connected (frontend handles gracefully).

### Backend — Import with Pre-populated Context

When user clicks "Add", use existing `create_account` flow but pre-populate:
- `name` from discovery
- `domain` from discovery (if available)
- Immediately enqueue for intelligence enrichment via `intel_queue`
- Store Glean source URLs as initial context entries

### Frontend — Discovery UI on Accounts Page

**Trigger:** "Discover from Glean" button on AccountsPage, positioned in the folio actions area next to "+ New". Styled as mono 11px uppercase, turmeric border, matching existing button style.

**Button visibility:** Only rendered when Glean is connected. Check via existing `get_connector_status` or equivalent. No error state — button simply absent when Glean is unavailable.

**Discovery panel:** When clicked, opens an inline section (not a modal) below the page header.

**Row structure:** Thin divider-separated rows (not cards). Each row:
- Account name: Newsreader 19px, primary color
- Metadata line: industry + domain in DM Sans 13px, secondary
- Context preview: 1-2 sentences from Glean in DM Sans 13px italic, tertiary
- Action: "+ Add" button (mono 11px, turmeric) OR "Already added" sage pill

**Duplicate detection:** Accounts matched to existing DailyOS accounts show "Already added" badge (sage pill). Not a second add button.

**Large result sets:** Show first 50 results. Filter input narrows the list client-side. If >50, show "Showing first 50 — search to find more" in mono 11px.

**Add flow:**
1. User clicks "+ Add"
2. Button changes to "Adding..." (disabled)
3. `create_account` called with pre-populated fields
4. On success: button changes to "Added" sage pill. Account appears in the main list on next refresh.
5. Enrichment begins immediately — intelligence populated within one `intel_queue` cycle

**Close/dismiss:** "Done" button or click outside closes the discovery panel. Discovery results persist in memory (4h cache) so re-opening is instant.

## Acceptance Criteria

1. With Glean connected, "Discover from Glean" button visible on Accounts page
2. Discovery returns accounts from Glean-indexed CRM not already in DailyOS (matched by name + domain)
3. Each discovered account shows available context preview: name, domain, industry (whatever Glean provides)
4. "Add" creates entity with available context pre-populated (no empty shell)
5. Enrichment begins immediately on add — intelligence populated within one intel_queue cycle
6. Button hidden when Glean not connected — no error state, just absent
7. Results cached with 4-hour TTL to avoid repeated Glean queries
8. Handles large result sets gracefully — pagination or "showing first 50" with search
9. Duplicate detection: account already in DailyOS shows "Already added" badge, not a second add button

### Intelligence Loop Integration
10. Importing a discovered account emits `entity_created` signal via `emit_signal_and_propagate()`. Verify: `signal_events` has `entity_created` row for the imported account. Propagation rules fire — the account is enqueued for enrichment and meeting prep scheduling.
11. Glean context preview from discovery is stored as an initial context entry on the new entity (not discarded after import). `build_intelligence_context()` includes this seeded context on first enrichment. Verify: first enrichment produces intelligence referencing the pre-populated context.
12. Accounts imported with renewal risk data from Glean trigger the `renewal_data_updated` callout path (per I535) on the next briefing cycle after enrichment completes.

### `DiscoveredAccount` Type (canonical definition)
13. `DiscoveredAccount` type defined in I535 is extended with fields needed by I494: `domain`, `industry`, `contextPreview`, `alreadyInDailyos`. I494 uses the same type, not a separate struct. Both I535 and I494 reference the single canonical definition.

### Mock Data
14. Mock data (`full` scenario) seeds a mock Glean discovery response with 6 entries: 3 matching existing mock accounts (Acme, Globex, Initech — for duplicate detection badges), 3 new accounts (for import flow). Each entry has `name`, `myRole`, `evidence`, `source`.
11. After applying `full` mock scenario, the "Discover from Glean" UI renders all 6 entries: 3 with "Already added" badges, 3 with "Add" buttons. Clicking "Add" on a new account creates the entity. No console errors.

## Out of Scope

- Auto-importing all accounts without user confirmation — user must explicitly choose
- Team-level account population (governance design needed)
- Writing back to Glean or CRM
- Discovery for non-Glean users — button is simply hidden
- Structured extraction from Glean results (mapping Salesforce fields to vitals) — I482 follow-on
- Ephemeral queries for accounts not imported — that's I495

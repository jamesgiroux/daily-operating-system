# I630 — Reconnection audit: sweep for built-but-not-wired components

**Area:** Code Quality / All surfaces
**Priority:** P2
**Version:** v1.0.2
**Effort:** Medium (audit: 1-2 hours, fixes: varies)
**Source:** CEO plan review assessment (2026-03-21). The I629 briefing expansion panel regression revealed a pattern — rapid build velocity (798 commits in February) likely left other components built but not connected.

---

## Problem

DailyOS was built at extreme velocity. Features were added when possible, not always fully integrated. I629 (PrepGrid + MeetingActionChecklist exported but not rendered) is one confirmed example. There are likely others:

- Components exported from one file but never imported elsewhere
- Sections that render conditionally on data that's never populated
- Backend services that compute results never consumed by the frontend
- Signal types defined in propagation rules but never emitted
- Frontend types describing richer data than the backend delivers

The Feb 2026 audit found similar patterns: "intelligence lifecycle faking 'enriched', 3/6 propagation rules listening for signals never emitted, prep invalidation implemented but never called."

## Solution

Systematic sweep of the codebase to identify disconnections. This is an audit + fix issue — the audit produces a list, quick fixes are shipped immediately, larger gaps become new issue specs.

## Audit Methodology

### 1. Exported-but-unused components (Frontend)
```bash
# Find all exported components/functions in src/
# Cross-reference with imports across the codebase
# Flag any export with zero external imports
```

### 2. Conditional renders on never-populated data (Frontend)
Review each major page for sections that render conditionally (e.g., `{data.foo && <FooSection />}`). Check whether `foo` is actually populated by the backend. Known risk areas:
- Account detail page chapters that depend on intelligence fields
- Meeting detail sections gated on prep data fields
- Daily briefing sections gated on dashboard data fields

### 3. Dead signal types (Backend)
```bash
# Grep for signal type strings in propagation rules
# Cross-reference with emit_signal/emit_signal_and_propagate calls
# Flag signal types that are listened for but never emitted
```

### 4. Backend computes, frontend ignores (Backend → Frontend gap)
Review Tauri IPC command return types. Are all fields in the returned structs consumed by the frontend? Known risk: the `DashboardData` struct may have fields the briefing doesn't render.

### 5. Frontend types richer than backend delivers
Compare TypeScript interfaces (`src/types/index.ts`) against Rust serialization structs. Flag any TS field that doesn't have a corresponding Rust field.

## Acceptance Criteria

1. **Audit document produced.** A markdown file (`.docs/audits/reconnection-audit-v1.0.2.md`) listing every disconnection found, categorized by type (exported-unused, conditional-dead, signal-dead, backend-unused, type-mismatch).
2. **Quick fixes shipped.** Any disconnection fixable in < 30 minutes is fixed in this version. Each fix is a separate commit with the disconnection described.
3. **Remaining gaps tracked.** Disconnections requiring > 30 minutes of work are filed as new issue specs with version assignment (likely v1.1.0+ or backlog).
4. **Signal type audit complete.** Every propagation rule in `signals/propagation.rs` has at least one corresponding `emit_signal` call. Dead rules removed or marked with `// TODO: emit from {source}`.
5. **No new disconnections introduced.** The audit methodology is documented so future reviews can repeat it.

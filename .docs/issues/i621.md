# I621 — Meeting transcript output redesign

**Area:** Frontend / Meeting Detail
**Priority:** P1
**Version:** v1.0.2
**Source:** v1.0.0 GA review (2026-03-17)

---

## Problem

The post-meeting intelligence section on meeting detail pages renders transcript extraction output but lacks structure and editorial quality. Two separate rendering paths exist:

1. **`PostMeetingIntelligence` component** — used when enriched captures exist (I558). Renders engagement dynamics, champion health, categorized captures (wins/risks/decisions/commitments), and role changes. Currently 405 lines with ChapterHeading sections but flat item rendering.

2. **`OutcomesSection` inline component** — fallback when no post-intel data exists. Simple flat list of wins/risks/decisions/actions with no sub-type or urgency rendering.

### Current issues

**Rendering gaps.** Several extracted fields aren't surfaced or are poorly presented:
- Win sub-types (ADOPTION, EXPANSION, VALUE_REALIZED, RELATIONSHIP, COMMERCIAL, ADVOCACY) are extracted but not visually distinguished — they render as flat bullet points
- Risk urgency tiers (RED, YELLOW, GREEN_WATCH) are sorted correctly but the urgency level isn't visually communicated (no color coding, no badges)
- Evidence quotes (`evidenceQuote` field) are present on enriched captures but rendered as plain text with no visual distinction from the content
- Speaker attribution (`speaker` field) is extracted but not shown on most items
- Impact assessment (`impact` field) is extracted but not rendered
- The `OutcomesSection` fallback doesn't render actions at all when the dedup guard blocks them (separate issue I620)

**No report structure.** The sections render in order (Engagement → Champion → Wins → Risks → Decisions → Commitments → Role Changes) but feel like a content dump. No executive summary, no visual grouping of related sections, no hierarchy that communicates "here's what matters most."

**Design quality.** The component uses basic div layouts without the editorial treatment (margin labels, section rules, typography hierarchy) applied to other pages like Account Detail.

## Scope

### 1. Rendering completeness

- **Win sub-types**: render as categorized groups (e.g., "Adoption (2)" header followed by items) or as colored tags on each item
- **Risk urgency**: RED items get terracotta accent/badge, YELLOW gets turmeric, GREEN_WATCH gets sage. Urgency badge visible at a glance
- **Evidence quotes**: render in a distinct pull-quote style (italic, left border, smaller text) — the editorial pattern used elsewhere
- **Speaker attribution**: show speaker name as a subtle tag on items that have it
- **Impact field**: render as a one-line context note below the content when present

### 2. Report-like structure

Reorganize the post-meeting section into a clear hierarchy:

```
Meeting Outcomes
├── Summary (executive summary from Phase 1)
├── Engagement Dynamics (talk balance, sentiment, signals)
│   ├── Talk Balance Bar
│   ├── Speaker Sentiment
│   ├── Competitor Mentions (if any)
│   └── Escalation Language (if any)
├── Key Findings
│   ├── Wins (grouped by sub-type, with evidence quotes)
│   ├── Risks (sorted by urgency, color-coded)
│   └── Decisions (with tags)
├── Commitments (with owner and timeline)
├── Actions (proposed items from extraction)
├── Champion Health (status + evidence)
└── Role Changes (if any)
```

Each section uses ChapterHeading. "Key Findings" groups related outcome types under one heading rather than three separate top-level sections. Champion Health moves lower since it's analytical context, not a primary finding.

### 3. Editorial treatment

- Use the design token system for all colors and spacing
- Section rules between major groups
- Consistent typography: Newsreader for headings, DM Sans for body, JetBrains Mono for metadata (urgency badges, speaker names, dates)
- Evidence quotes use the pull-quote pattern from Account Detail
- Ensure the component follows COMPONENT-INVENTORY.md patterns

## Acceptance Criteria

1. Every field in `MeetingPostIntelligence` + `EnrichedCapture` types is rendered in at least one component path. No extracted data is silently dropped.
2. Win sub-types visually grouped or tagged — a reader can identify "3 adoption wins, 1 advocacy win" without reading every line.
3. Risk urgency communicated visually: RED items immediately distinguishable from YELLOW and WATCH. Color coding uses design tokens (terracotta, turmeric, sage).
4. Evidence quotes rendered in editorial pull-quote style (italic, left border accent) — distinct from content text.
5. Speaker names shown on items that have attribution. Subtle, doesn't dominate the content.
6. Sections follow the structured hierarchy: Summary → Engagement → Key Findings → Commitments → Actions → Champion → Role Changes.
7. No inline styles — all CSS in `PostMeetingIntelligence.module.css` using design tokens.
8. The `OutcomesSection` fallback (when PostMeetingIntelligence data isn't available) renders a consistent subset of this design — not a completely different treatment.
9. Visual diff reviewed with `/browse` — no regressions to existing meeting detail pages.

## Out of Scope

- Editing post-meeting intelligence fields (inline editing is a separate feature)
- PDF export of meeting outcomes
- Changing what the AI extracts (prompt changes) — this is purely a rendering issue

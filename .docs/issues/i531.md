# I531 — Glean-Powered Proactive Self-Healing

**Priority:** P1
**Area:** Backend / Self-Healing / Connectors
**Version:** v1.1.0
**Depends on:** I508 (intelligence schema — dimension-aware gap queries), I513 (DB as sole source — filesystem dependencies removed), I487 (Glean signal emission — Glean signals flow into the bus)

## Problem

The self-healing system (`hygiene.rs`, `self_healing/`) detects intelligence gaps proactively — missing champions, stale enrichments, empty risk assessments, phantom entities — but can only fix them by re-running the enrichment pipeline against locally available context. When context is thin (few meetings, no emails), re-enrichment produces the same thin results.

With Glean connected, the system has access to the entire org knowledge graph — CRM records, internal docs, Slack threads, org charts — but never searches it proactively. Glean is only queried during user-initiated enrichment (via `GleanContextProvider::gather()`). The self-healing detectors identify exactly what's missing, but have no way to go find it.

### Current behavior

```
Hygiene scanner detects: account "Acme" has empty risks, stale champion, no exec sponsor
  → Re-enrich with same local context
  → Same thin results (no new information available)
  → Gap persists until user manually triggers enrichment or new meetings occur
```

### Desired behavior

```
Hygiene scanner detects: account "Acme" has empty risks, stale champion, no exec sponsor
  → Generate targeted Glean queries for each gap (via semantic_gap_query())
  → Search Glean for risk-related content, org chart changes, executive mentions
  → Feed results into enrichment context
  → Gaps filled with real org knowledge
```

Additionally, several self-healing mechanisms have signal emission gaps that prevent downstream propagation (prep invalidation, cross-entity rules). These need to be fixed for the feedback loop to work correctly.

## Design

### 1. Hygiene → Glean query pipeline

When `run_hygiene()` detects gaps during Phase 3 (AI-budgeted gap filling), and Glean is connected:

```rust
// In hygiene.rs Phase 3, after gap detection
if glean_connected {
    let gap_queries = semantic_gap_queries(&entity_intel, &entity_type);
    for query in gap_queries {
        let glean_results = glean_provider.search(&query.query, &query.dimension).await;
        // Merge Glean results into enrichment context
        enrichment_context.glean_gap_results.push(glean_results);
    }
    // Re-enrich with Glean-augmented context
    enqueue_enrichment(entity_id, enrichment_context);
}
```

This reuses `semantic_gap_queries()` from I508 — the same function that generates dimension-aware search terms during enrichment. The difference: during enrichment, gap queries supplement the main Glean context gather. Here, gap queries are the *primary* reason for the Glean search — triggered by the hygiene scanner, not by a user action.

### 2. Proactive detector → Glean integration

Three detectors in `self_healing/detector.rs` should trigger targeted Glean searches:

| Detector | Current behavior | With Glean |
|---|---|---|
| `stale_champion` | Flags accounts with champion set > 90 days ago, no recent contact | Search Glean for org chart changes, role changes for the champion's company |
| `missing_exec_sponsor` | Flags accounts with no executive relationship | Search Glean for executive contacts at the account's company |
| `engagement_decay` | Flags accounts with declining meeting frequency | Search Glean for recent internal mentions, Slack discussions about the account |

Each detector emits a `GapQuery` that feeds into the Glean search pipeline from §1.

### 3. Signal emission fixes

Fix the two bugs and seven gaps documented in `.docs/architecture/SELF-HEALING.md` §8:

**Bugs (use `emit_and_propagate()` instead of `emit()`):**
- `fix_auto_merge_duplicates()` — person merge should invalidate prep for affected meetings
- `fix_co_attendance_links()` — new relationship link should propagate to entity intelligence

**Missing signal emissions (add `emit_signal_and_propagate()`):**
- `resolve_names_from_emails()` — emit `person_profile_updated` when name resolved
- `detect_domain_aliases()` — emit `entity_updated` when domain alias linked
- `archive_phantom_entities()` — emit `entity_archived` when phantom removed
- `fix_entity_type_mismatches()` — emit `entity_updated` when type corrected
- `fix_orphaned_meetings()` — emit `meeting_updated` when meeting relinked
- `fix_missing_entity_links()` — emit `entity_linked` when link created
- `detect_stale_enrichments()` — emit `enrichment_stale` as informational signal

### 4. Filesystem → DB migration for hygiene

Post-I513, `resolve_names_from_emails()` currently reads `emails.json` from the workspace filesystem. After I513 eliminates workspace files, this function must read from `email_metadata` in SQLite instead. The function signature stays the same — only the data source changes.

Similarly, the overnight maintenance scan writes `maintenance.json` that nothing reads. Remove this dead write entirely.

### 5. Budget and rate limiting

Glean searches from hygiene are budget-gated (same as Phase 3 AI enrichment):

- **Daily Glean query budget:** 50 queries per hygiene run (configurable)
- **Per-entity limit:** Max 3 Glean gap queries per entity per hygiene run
- **Cooldown:** Entity won't be Glean-queried again for 24h after a gap-fill attempt
- **Priority ordering:** Entities sorted by `trigger_score` (imminence × 0.35 + staleness × 0.25 + quality_deficit × 0.20 + importance × 0.10 + signal_delta × 0.10)

This prevents runaway Glean API usage while ensuring the highest-priority gaps get filled first.

## Acceptance Criteria

1. With Glean connected, hygiene run on an account with empty risks produces Glean gap queries targeting risk-related content. After enrichment, risks populated if Glean returned relevant results.
2. `stale_champion` detector triggers Glean search for org chart changes. If Glean finds a role change, enrichment context includes it.
3. Signal emissions: `fix_auto_merge_duplicates()` uses `emit_and_propagate()` — person merge invalidates prep for affected meetings.
4. Signal emissions: `fix_co_attendance_links()` uses `emit_and_propagate()` — new relationship propagates to entity intelligence.
5. All 7 previously-silent hygiene mechanisms emit signals (verifiable in `signal_events` table).
6. `resolve_names_from_emails()` reads from `email_metadata` table, not `emails.json`.
7. `maintenance.json` no longer written by overnight scan.
8. Glean query budget respected: no more than 50 gap queries per hygiene run, 3 per entity, 24h cooldown per entity.
9. Without Glean connected, hygiene behavior unchanged — no errors, no degradation.

### Intelligence Loop Integration
10. Each of the 7 newly-emitting hygiene mechanisms uses `emit_signal_and_propagate()` (not `emit_signal()`) where the signal should cause downstream prep invalidation or re-enrichment. Specifically: `person_profile_updated`, `entity_updated`, `entity_linked`, `meeting_updated` all use `emit_and_propagate()`. Verify: after `fix_missing_entity_links()` runs, `prep_frozen_json` is cleared for upcoming meetings with the newly linked entity.
11. After `stale_champion` detector triggers a Glean gap-fill that returns a champion departure, `compute_champion_health()` produces a lower score for that account. Verify: `championHealth` dimension score changes in `health_json` after hygiene + Glean-augmented enrichment completes.
12. Glean gap-fill results from hygiene are written to `entity_assessment` via the same write path as direct enrichment. Verify: after hygiene + Glean gap-fill, `build_intelligence_context()` for that entity returns non-null data in the previously-gapped dimension.
13. None of the 7 hygiene-emitted signal types appear in `signal_events` with `source = 'user_correction'` or `source = 'user_curation'`. Hygiene mechanisms use `source = 'hygiene'` or `source = 'system'` — never user-attributed sources.

### CRM Divergence Detection (Glean-connected)
14. New consistency check: if `entity_assessment.executive_assessment` sentiment implies "healthy" but `entity_assessment.org_health_json.health_band` is "red" (or vice versa), flag as `GLEAN_CRM_DIVERGENCE`. This is a deterministic check (no AI call) comparing local AI narrative against Glean-sourced CRM data.
15. `GLEAN_CRM_DIVERGENCE` findings stored in `entity_assessment.consistency_findings_json` and surfaced via `consistency_status = 'flagged'` on the account detail page.

### Mock Data
14. Mock data (`full` scenario) seeds at least 2 accounts with intelligence gaps detectable by hygiene (e.g., empty `risks_json`, stale `enriched_at` > 14 days, missing champion in `account_stakeholders`).
11. Mock data seeds `signal_events` rows for all 9 previously-silent hygiene mechanisms (verifiable: `SELECT DISTINCT signal_type FROM signal_events WHERE source LIKE 'hygiene%'` returns 9+ types).
12. After applying `full` mock scenario and running hygiene, `signal_events` contains new hygiene-emitted signals. No console errors from gap detection or signal emission.

## Out of Scope

- Glean webhooks / push notifications (Glean is pull-only for now)
- Autonomous remediation beyond enrichment (e.g., auto-creating relationships, auto-archiving)
- User notification of self-healing actions ("I found new information about Acme")
- Self-healing for reports (detecting stale report content)
- Hygiene analytics dashboard

# I599 — Welcome Screen: Eliminate Blank Window on Startup

**Version:** v1.0.0 (Phase 6 — GA Readiness)
**Area:** Frontend + Backend / Startup UX
**Priority:** P0

## Problem

When DailyOS launches, the user sees a blank cream window for 400-1100ms before any content appears. The startup sequence is:

1. Tauri `setup()` runs (~200ms) — window not yet visible
2. Window appears with empty `<div id="root">` — **blank screen**
3. React hydrates, config check runs (~100-500ms) — **still blank** (startup gate returns `"checking"`)
4. Dashboard skeleton appears
5. Dashboard data loads (~500-2000ms) — skeleton visible
6. First meaningful paint — briefing renders

The blank window is the worst moment. It looks broken. A 3-5 second startup is fine if it's *intentional* — like opening a magazine to the cover before flipping to the articles.

## Design

### Layer 1: Static HTML Welcome Screen (zero-JS, instant)

Embed a branded welcome screen directly in `index.html` inside the `<div id="root">`. This renders at the exact moment the Tauri window becomes visible — before React, before JS, before fonts.

```html
<div id="root">
  <!-- Static welcome — replaced when React hydrates -->
  <div id="static-welcome" style="
    position: fixed; inset: 0;
    display: flex; flex-direction: column;
    align-items: center; justify-content: center;
    background: #f5f2ef;
    font-family: 'Newsreader', Georgia, 'Times New Roman', serif;
  ">
    <div style="font-size: 64px; line-height: 1; color: #c4a882; margin-bottom: 16px;">✳</div>
    <div style="font-size: 28px; font-weight: 400; color: #3d3a36; letter-spacing: -0.01em;">
      DailyOS
    </div>
  </div>
</div>
```

Key constraints:
- Inline CSS only (no external stylesheet dependency)
- System-serif fallback (`Georgia, 'Times New Roman', serif`) — Newsreader loads later
- Cream background matches `--color-background` exactly (`#f5f2ef`)
- Turmeric/gold asterisk mark matches brand identity (ADR-0076)
- No spinner, no progress bar — the mark and name are the entire screen
- Minimal DOM — React's `createRoot` replaces the contents of `#root`, removing this automatically

### Layer 2: React Welcome Component (takes over seamlessly)

Replace the blank `"checking"` gate state with a `<WelcomeScreen />` component that visually matches the static HTML but adds:

- **Proper fonts**: Newsreader for the mark/name (now loaded via CSS)
- **Date line**: Today's date in DM Sans, below the name: "Saturday, March 15"
- **Personality greeting**: After ~500ms delay, fade in a line like "Preparing your briefing…" (personality-aware via `getPersonalityCopy()`)
- **Subtle animation**: The asterisk mark does a gentle opacity pulse (0.7 → 1.0, 2s cycle) — editorial-appropriate, not tech-loading-spinner

The transition from static HTML → React welcome should be imperceptible. Same layout, same colors, same position. React just adds the date and greeting as enhancements.

### Layer 3: Minimum Display Time

The welcome screen should show for a **minimum of 1.5 seconds** even if the dashboard data loads faster. This prevents a jarring flash where the welcome appears and vanishes in 200ms. The experience should feel like:

1. App opens → welcome screen (instant)
2. Welcome screen holds for ~1.5-3s (greeting fades in)
3. Smooth crossfade to dashboard content

Implementation:
- Track `welcomeShownAt` timestamp when React welcome mounts
- When dashboard data is ready AND `Date.now() - welcomeShownAt >= 1500ms`, transition to dashboard
- If data is ready before 1.5s, hold the welcome screen (don't flash)
- Transition: opacity crossfade (welcome fades out, dashboard fades in, ~300ms)

### Startup Gate Changes

In `routerStartupGate.ts`, the `"checking"` state currently renders a blank div. Change it to render `<WelcomeScreen />`. All other gates (encryption-recovery, database-recovery, lock, onboarding) continue to render their own screens — they take priority when they fire.

The welcome screen only appears during normal startup (no recovery needed, no lock, no onboarding). It's the happy path experience.

### Font Loading Strategy

Currently all fonts use `font-display: block` which can cause invisible text for up to 3s. For the welcome screen:
- Static HTML uses system-serif fallback — visible immediately
- When Newsreader loads, the text updates seamlessly (same layout, slightly different font)
- Consider changing `font-display: swap` for Newsreader only (so text is always visible with fallback, then swaps to Newsreader when ready)

### Window Background Color

Add `background-color: #f5f2ef` to the `<body>` tag in `index.html` so even before any CSS loads, the window background matches the brand color instead of default white.

## Vocabulary (ADR-0083)

- Greeting text: "Preparing your briefing…" / "Getting your day ready…" / "Good morning." (personality-dependent)
- Never: "Loading…", "Initializing…", "Please wait…"

## Acceptance Criteria

1. Window appears with branded welcome screen immediately on launch — no blank white or cream frame
2. Welcome screen shows DailyOS asterisk mark and name, centered, on cream background
3. Static HTML version renders before any JS executes (visible at window-show time)
4. React welcome component takes over seamlessly — no visual jump or flash
5. Date line and personality greeting fade in after ~500ms
6. Welcome screen holds for minimum 1.5 seconds even if data loads faster
7. Smooth crossfade transition (300ms) from welcome to dashboard content
8. All startup gates (encryption, DB recovery, lock, onboarding) still work correctly — welcome only shows on happy path
9. No "Loading…" or technical language in any user-visible startup text
10. `<body>` background color set to `#f5f2ef` in `index.html` (eliminates white flash before CSS loads)
11. Works correctly on cold start (first launch, no cache) and warm start (re-open from tray)

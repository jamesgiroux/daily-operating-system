# I662 — Archive recovery: re-route stranded transcripts and records

**Area:** Backend / Transcripts
**Priority:** P1
**Version:** v1.1.2
**Effort:** Medium (2-3 hours)
**Source:** Investigation 2026-04-03. 57+ files stranded in ~/Documents/DailyOS/_archive/ since mid-February.

---

## Problem

Between mid-February and April 2026, all transcript and meeting record routing has fallen through to `_archive/{date}/` because entity resolution couldn't match meetings to accounts (I660). This has accumulated:

- **57+ markdown files** across 20 date-based directories
- Mix of transcripts (`*-transcript.md`) and meeting records (`*-record.md`)
- Customer meetings (Heroku, Cox, Jefferies, etc.) mixed with internal meetings
- Files have YAML frontmatter with `meeting_id` and `meeting_type` but no `account` field

These files are invisible to:
- MCP `search_content` (searches entity directories, not _archive)
- `build_intelligence_context()` (entity-scoped)
- Meeting prep context gathering
- Account-level transcript history

## Solution

A one-time recovery process that runs after I660 (domain population) and I661 (routing fallback) are in place.

### Phase 1: Classify and route

For each `.md` file in `_archive/{date}/`:

1. **Parse frontmatter** — extract `meeting_id`, `meeting_title`, `meeting_type`
2. **Look up meeting in DB** — use `meeting_id` to find the CalendarEvent and any existing entity links in `meeting_entities`
3. **Determine correct destination** using the same priority routing as I631+I661:
   - Account-linked → `Accounts/{name}/Call-Transcripts/` or `Meeting-Records/`
   - Project-linked → `Projects/{name}/Call-Transcripts/` or `Meeting-Records/`
   - Person-linked (1:1) → `People/{name}/Call-Transcripts/` or `Meeting-Records/`
   - If meeting_id not found in DB, try attendee domain matching (I661 fallback)
   - If still unresolvable → leave in `_archive` (genuinely unroutable)
4. **Move file** to correct destination
5. **Update frontmatter** — add `account:` field if missing, add `recovered_from: "_archive/{date}"` for audit trail
6. **Update DB** — if `meeting_transcripts.transcript_path` exists, update to new path

### Phase 2: Classify internal vs customer

Files that remain in `_archive` after Phase 1 should be categorized:
- `meeting_type: "internal"`, `"teamsync"`, `"allhands"` → expected to be in `_archive` (no account)
- `meeting_type: "customer"`, `"qbr"` → flag as unresolved (needs manual account assignment or better entity hints)
- `meeting_type: "oneonone"` → check if person is linked, route if possible

### Implementation

This should be a Tauri command (not a migration) so it can be re-run if needed:

```
#[tauri::command]
async fn recover_archived_transcripts(state: State<'_, AppState>) -> Result<RecoveryReport, String>
```

Returns a `RecoveryReport` with:
- `files_recovered: usize` — moved to entity directories
- `files_skipped: usize` — internal/unresolvable, left in _archive
- `files_failed: Vec<(String, String)>` — file path + error
- `details: Vec<RecoveryDetail>` — per-file: source, destination, routing_method

Expose via devtools or a one-time startup migration flag.

## Acceptance Criteria

1. Customer meeting transcripts (Heroku, Cox, etc.) are moved to their correct `Accounts/{name}/Call-Transcripts/` directory.
2. Meeting records are moved to `Accounts/{name}/Meeting-Records/` directory.
3. Internal meetings (team syncs, all-hands) remain in `_archive` (correct behavior).
4. Non-transcript files (Book-of-Business-Review, HTML files) remain in `_archive` (manually placed).
5. `meeting_transcripts.transcript_path` in DB is updated to reflect new locations.
6. Recovery report shows how many files were moved, skipped, and failed.
7. MCP `search_content` for "Heroku" finds the recovered transcripts.
8. Running the command twice is idempotent (already-moved files are skipped).
9. Files that were correctly in `_archive` (genuinely unlinked) are not touched.

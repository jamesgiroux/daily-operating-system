# I661 — Defensive fallback in transcript routing

**Area:** Backend / Transcripts
**Priority:** P0
**Version:** v1.1.2
**Effort:** Low (1-2 hours)
**Source:** Investigation 2026-04-03. Transcript processor ignores YAML frontmatter from source files and has no fallback when `meeting.account` is None.

---

## Problem

The transcript processor (`processor/transcript.rs:127-182`) routes files using a single data source: `meeting.account` from the `CalendarEvent` struct, populated by meeting classification.

When classification fails to resolve an account (the common case today, see I660), the processor:
1. Skips the account routing branch entirely
2. Falls through project routing (usually no linked project)
3. Falls through person routing (only for 1:1s with linked people)
4. Lands in `_archive/{date}/` as the catch-all fallback

The processor **generates** new YAML frontmatter from `meeting.account` rather than **parsing** existing frontmatter from the source file. Even if the transcription service (Otter, Fireflies, etc.) included account metadata in the file, it would be overwritten.

Meanwhile, the generic inbox file processor (`processor/router.rs`) has robust frontmatter parsing via `frontmatter_value()` and `infer_entity_tracker_path()`. This capability exists in the codebase but isn't used by the transcript pipeline.

## Solution

Add a fallback routing step in `process_transcript_with_kind()` between the current `meeting.account` check and the archive fallback:

### Step 1: Try `meeting.account` (existing, keep as-is)

### Step 2: Try linked_entities (existing via I631, keep as-is)

### Step 3: NEW — Try attendee domain matching

If `meeting.account` is None and linked_entities didn't resolve:
1. Extract external attendee email domains from the CalendarEvent
2. Query `db.lookup_account_candidates_by_domain()` for each domain
3. If exactly one account matches with high confidence, route there
4. Log the fallback: `"Account resolved via attendee domain fallback: {account}"`

### Step 4: NEW — Try frontmatter parsing from source file

If the source file already has YAML frontmatter with an `account:` field:
1. Parse using the existing `frontmatter_value()` from `router.rs`
2. Validate account exists in DB via `get_account_by_name()`
3. If valid, route to that account directory
4. Log: `"Account resolved via source file frontmatter: {account}"`

### Step 5: Archive fallback (existing, keep as-is)

### Routing audit log

Add a `routing_method` field to the log output:
- `"account_classification"` — meeting.account matched
- `"linked_entity"` — project or person entity
- `"attendee_domain_fallback"` — Step 3
- `"source_frontmatter"` — Step 4
- `"archive_fallback"` — no match found

## Acceptance Criteria

1. A transcript from a meeting with `contact@cloudco.com` as attendee, where CloudCo has domains populated (I660), routes to `Accounts/CloudCo/Call-Transcripts/` even when `meeting.account` is None.
2. A transcript file with existing `account: "Jane-Software"` frontmatter routes correctly when classification fails.
3. Archive fallback still works for genuinely unresolvable meetings.
4. Logs show which routing method was used for every transcript.
5. Meeting records (`compute_meeting_record_path()`) get the same fallback chain.
6. No regression: meetings that already route correctly via `meeting.account` continue to do so.

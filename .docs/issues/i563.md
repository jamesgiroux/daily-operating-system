# I563 — shellConfig render loop on AccountDetailEditorial

**Version:** v1.0.0 Phase 6
**Depends on:** none
**Type:** Bug fix — Frontend / Architecture
**Scope:** Frontend: `useMagazineShell.ts`, `AccountDetailEditorial.tsx`. No backend changes.

---

## Problem

`useRegisterMagazineShell` triggers a max update depth warning on `AccountDetailEditorial`. The root cause is a feedback loop between the shell context provider and the page component:

1. `AccountDetailEditorial` builds `shellConfig` in a `useMemo` (line 256) with volatile dependencies: `acct.enriching`, `acct.enrichSeconds`, `reportsOpen`, `toolsOpen`, `saveStatus`, and multiple `acct.*` callbacks. These values change on nearly every render.

2. When any dependency changes, `useMemo` produces a new object reference for `shellConfig`.

3. `useRegisterMagazineShell` (line 361) passes `shellConfig` to a `useEffect` whose dependency is the config object itself (line 79 in `useMagazineShell.ts`). A new object reference triggers the effect.

4. The effect calls `ctx.register(config)` (line 75), which calls `setConfig(c)` in the shell provider (line 50), re-rendering `MagazinePageLayout`.

5. `MagazinePageLayout` re-renders its children, including `AccountDetailEditorial`, which re-evaluates `useMemo`, producing another new `shellConfig` reference — and the loop restarts.

React caps the update depth so it doesn't crash, but the console warning fires and there's unnecessary render churn degrading performance on the most-visited detail page.

The dependency list on line 359 includes 17 items, many of which are callback references from `useAccountDetail` that are recreated every render (not memoized with `useCallback`).

---

## Solution

Split the shell config into stable structural props and volatile render content:

### 1. Separate stable config from volatile folioActions

The `useRegisterMagazineShell` effect should only re-fire when the *structural* shell properties change — `folioLabel`, `atmosphereColor`, `activePage`, `backLink`, `chapters`. These change only when the account or route changes.

The `folioActions` JSX (refresh button, Reports dropdown, Tools dropdown) and `folioStatusText` are the volatile parts — they depend on `enriching`, `enrichSeconds`, `reportsOpen`, `toolsOpen`, `saveStatus`. These should be delivered via a separate mechanism that doesn't trigger re-registration.

**Option A — Ref-based volatile slot**: Store `folioActions` and `folioStatusText` in a `useRef` that `MagazinePageLayout` reads on each render without causing re-registration. `useRegisterMagazineShell` accepts an optional `volatileRef` alongside the stable config.

**Option B — Split context**: Add a second context value for volatile folio content (`folioActions`, `folioStatusText`) with its own setter that doesn't trigger the re-registration effect. The shell provider reads both: stable config from the effect-driven path, volatile content from a direct state update.

### 2. Stabilize callback references

In `AccountDetailEditorial`, wrap `saveMetadata`, `saveAccountField`, and the `backLink.onClick` handler in `useCallback` so they don't change identity on every render. The `acct.*` callbacks from `useAccountDetail` should also be memoized at the hook level if they aren't already.

### 3. Deep-equality guard in useRegisterMagazineShell

Add a shallow-equality check in the `useEffect` before calling `ctx.register(config)`:

```typescript
useEffect(() => {
  // Only re-register if the structural config actually changed
  ctx.register(config);
  return () => ctx.unregister();
}, [ctx.register, ctx.unregister, config]);
```

Replace the raw `config` dependency with a stable serialization key (e.g., `JSON.stringify` of the non-JSX fields) or use `usePrevious` + shallow compare to gate the `register` call.

## Acceptance Criteria

1. Opening `AccountDetailEditorial` produces zero "Maximum update depth exceeded" warnings in the browser console.
2. The folio bar still shows live "Refreshing... Ns" countdown text when enrichment is running.
3. Reports dropdown opens/closes without triggering a shell re-registration (verify via `console.log` in `useRegisterMagazineShell` effect — should not fire on dropdown toggle).
4. Tools dropdown opens/closes without triggering a shell re-registration.
5. Save status transitions (`idle` → `saving` → `saved` → `idle`) display correctly in the folio bar without re-registration.
6. Navigating between two different account detail pages re-registers the shell exactly once per navigation (new account = new config).
7. `React.Profiler` on `AccountDetailEditorial` shows render count reduced by > 50% compared to pre-fix baseline during a 10-second idle period on the page.
8. No regressions on other magazine shell pages (`MeetingDetailPage`, `PersonDetailEditorial`, `ProjectDetailEditorial`) — verify shell still registers correctly.
9. `pnpm build` succeeds. `pnpm test` passes. No TypeScript errors.

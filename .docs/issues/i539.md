# I539: Database recovery UX for migration/DB failure

## Problem

Users currently have no in-app recovery path when database startup fails due to migration or schema integrity issues. Recovery requires manual/devtools intervention, which is not acceptable for production.

## Design

1. Startup blocker recovery screen (same precedence level as encryption recovery) that prevents normal app usage while recovery is required.
2. Settings > Data recovery controls available during normal operation.
3. Backend commands to list backups and restore a selected backup.

## Backend API additions

- `get_database_recovery_status()`
- `list_database_backups()`
- `restore_database_from_backup(backupPath)`

## Frontend surfaces

- `DatabaseRecovery` startup blocker route/screen.
- `Settings > Data` recovery card with backup list, create backup, and restore actions.

## Public API / Types

- `DatabaseRecoveryStatus { required, reason, detail }`
- `BackupInfo { path, createdAt, sizeBytes, kind }`
- Frontend recovery-status hook for startup and Settings surfaces.
- Router startup gate for database recovery-required state.

## Acceptance Criteria

1. If DB is recovery-required, app blocks normal UI and shows restore options.
2. Restore succeeds from startup blocker and from Settings.
3. Invalid/corrupt backup restore shows a clear error.
4. Encryption-key-missing flow remains unchanged (no regression).

## Scope

- Includes backend command/state plumbing and frontend recovery UX.
- Out of scope for I511; tracked independently to keep I511 backend-only.

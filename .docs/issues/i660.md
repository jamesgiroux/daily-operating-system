# I660 — Populate account_domains table from meeting attendees

**Area:** Backend / Entity Resolution
**Priority:** P0
**Version:** v1.1.2
**Effort:** Medium (2-3 hours)
**Source:** Investigation 2026-04-03. All customer accounts have 0 domains in account_domains, making domain-based entity resolution impossible. This is the root cause of transcript mis-routing to _archive. See also: entity resolution phase 2 investigation (2026-03-29).

---

## Problem

The `account_domains` table exists but is never populated. Domains are extracted from meeting attendees during `prepare_today()` orchestration (`extract_customer_domains()`), used for email classification, then discarded. They're never persisted to `account_domains`.

Without domains in the table:
- `classify_meeting_multi()` can't match attendee email domains to accounts via entity hints
- `CalendarEvent.account` is `None` for most meetings
- Transcript routing falls through to `_archive/{date}/` instead of `Accounts/{name}/Call-Transcripts/`
- Meeting records never reach account directories
- Entity resolution domain signals are dead code

This has been silently broken since mid-February 2026. 57+ transcripts and records are stranded in `_archive`.

## Solution

Three paths, implemented in order:

### Path 1: Extract from entity linking (immediate)

When `event_trigger.rs` links a meeting to an account via `link_meeting_entity_if_absent()`, extract attendee email domains and store in `account_domains`.

**Where:** `src-tauri/src/signals/event_trigger.rs`, after successful entity linking (~line 116)

```
After linking:
1. Get meeting attendees from the event
2. Extract email domains (split on @, lowercase)
3. Filter out user's own domains and personal email domains (gmail, outlook, etc.)
4. Call db.set_account_domains(account_id, &discovered_domains)
```

### Path 2: Backfill from existing meeting-account links

One-time migration or command to:
1. Query all `meeting_entities` links where `entity_type = 'account'`
2. For each link, get meeting attendees
3. Extract external domains
4. Merge into `account_domains` (deduplicated)

**Where:** New function in `services/accounts.rs` or a migration script

### Path 3: Extract from enrichment (future)

When Glean or Clay returns firmographic data with company domains, store them. This is lower priority since Path 1+2 cover the immediate need.

## Acceptance Criteria

1. After a meeting is linked to an account, the account's domains appear in `account_domains` table.
2. `SELECT account_id, COUNT(*) FROM account_domains GROUP BY account_id HAVING COUNT(*) > 0` returns results for accounts with meeting links.
3. Backfill populates domains for all existing account-meeting links.
4. Personal email domains (gmail.com, outlook.com, etc.) are excluded.
5. User's own company domains are excluded.
6. EntityHint structs loaded for classification include the populated domains.
7. `classify_meeting_multi()` resolves a meeting with `contact@heroku.com` to the Heroku account.

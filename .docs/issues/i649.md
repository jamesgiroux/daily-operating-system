# I649 - Technical footprint, adoption, and service-delivery intelligence

**Area:** Backend / Intelligence Schema + Frontend / Dossier + Meeting Prep
**Priority:** P1
**Version:** v1.1.0 Phase 5
**Effort:** Large
**Depends on:** I644, I645, I646, I647
**Source:** Globex Holdings source inventory and prompt outputs showed a large class of high-value account truth that is not purely commercial or relationship-based: apps, environments, org IDs, domains, add-ons, support SLAs, traffic footprint, usage telemetry, and professional services deliverables.

---

## Problem

DailyOS currently treats "technical/account reality" too narrowly.

But for RM / CSM / TAM work, these fields are often what make the account feel real:

- what apps and environments actually exist
- which domains and org IDs matter
- what add-ons and entitlements are purchased
- what support package and SLA the customer expects
- what usage / engagement telemetry suggests about adoption
- what services work was delivered and whether it created value
- what technical themes are active right now vs historical

These are not edge details. They materially shape:

- renewal authority
- expansion authority
- meeting readiness
- operational empathy
- how trustworthy the dossier feels

Without them, the account page is commercially aware but not operationally grounded.

---

## Solution

Add a dedicated technical/adoption/services intelligence slice to the DB-backed assessment and surface it across the dossier and meeting brief.

### 1. Technical footprint

Model structured technical footprint including:

- applications / site IDs
- environments (`prod`, `staging`, `preprod`, `sandbox`)
- domains
- org IDs
- key integrations / external systems
- hosting / CDN / analytics stack notes
- technical maturity

### 2. Products, add-ons, and entitlements

Extend product intelligence with richer operational context:

- primary product
- add-ons purchased
- non-prod entitlements
- support package and tier
- SLA profile

### 3. Adoption / usage telemetry

Model usage and engagement signals such as:

- Pendo last visit
- visitor count
- time on site
- engagement minutes windows
- adoption trend

These should influence both dossier rendering and later prep/report context.

### 4. Services and delivered work

Capture professional services and technical delivery evidence:

- active or recent services engagements
- project objectives
- focus URLs
- delivered reports / executive summaries
- delivered outcomes / recommendations
- follow-on opportunities

### 5. Surface mapping

Recommended placement:

- `State of Play`
  - technical footprint
  - entitlements
  - adoption telemetry
  - current technical themes
- `Outlook`
  - upsell / expansion paths tied to add-ons, analytics, services, or architecture
- `The Record`
  - completed services work
  - historical footprint changes
  - older technical initiatives
- meeting briefing
  - likely technical concerns
  - current environments / apps in scope
  - recent services outcomes worth reinforcing

---

## Acceptance Criteria

1. The DB-backed assessment can represent technical footprint, add-ons/entitlements, adoption telemetry, and services/delivery evidence as structured account data.
2. Account detail can surface current technical footprint and entitlements in `State of Play` without treating them as ad hoc notes.
3. Account detail can surface recent or completed services work in `The Record` and connect it to value delivered where appropriate.
4. Meeting briefing can include current technical themes, relevant environments/apps, and recent services or support context likely to arise in the meeting.
5. Adoption telemetry and support tier / SLA expectations can be carried into downstream prompt context and report generation.
6. Timeliness rules distinguish active technical themes from stale historical technical context.

---

## Implementation Notes

- This issue is not a request for inventory CRUD. The rendering should remain dossier-style and source-aware.
- Do not overload the hero with technical detail. The goal is richness and authority deeper in the dossier.
- Source references matter here too: app IDs, org IDs, service records, and report links should be traceable.

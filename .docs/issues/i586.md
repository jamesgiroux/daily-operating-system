# I586 — Action Completion → Milestone Advancement

**Version:** v1.0.2
**Area:** Backend / Actions + Success Plans
**Priority:** P1

## Problem

Completing an action linked to an objective does NOT advance its milestone. The user creates an objective "Close Acme Renewal," adds a milestone "Send proposal," links an action "Draft and send renewal proposal" to the objective, completes the action — and the milestone stays pending. The UI shows them together but the systems don't talk. This makes The Work section feel decorative rather than functional.

## Design

### Explicit Action-Milestone Linking

Extend `action_objective_links` table (or create new `action_milestone_links`):
- Add optional `milestone_id` to the link — an action can be linked to a specific milestone within an objective
- UI: When linking an action to an objective, show milestones as sub-options ("Link to: Objective / Specific milestone")
- If linked to milestone: completing the action auto-completes the milestone

### Auto-Advance Logic

In `services/actions.rs` `complete_action()`, after marking the action complete:
1. Query `action_milestone_links` for the completed action
2. For each linked milestone that is still pending:
   - Check if ALL actions linked to that milestone are now completed
   - If yes: auto-complete the milestone via `complete_milestone()`
   - This cascades: `complete_milestone()` already checks if all milestones in the objective are done → auto-completes objective
3. Emit `milestone_auto_completed` signal with `source: "action_completion"`, confidence 0.85

### Fuzzy Auto-Link on Action Creation

When an action is created (from any source) and an account has active objectives:
- Compare action title against milestone titles for that account (fuzzy match, >0.7 similarity)
- If match found: auto-link the action to the milestone
- If no match but objective match: auto-link to objective without milestone specificity
- Auto-links are suggested, not forced — emit `action_auto_linked` signal and surface in UI for user confirmation

### Signal Emission

- `milestone_auto_completed` (confidence 0.85) — milestone completed because all linked actions were completed
- `action_auto_linked` (confidence 0.6) — action was auto-linked to a milestone based on fuzzy match
- Both feed into entity health scoring (momentum dimension)

## Mock Data

Requires I597 (action mock data expansion). Milestone cascade scenarios:
- 2 completed actions linked to the same Acme milestone — verifies auto-complete when all linked actions done
- 1 waiting action linked to a Globex milestone — verifies guard (don't auto-complete when waiting action exists)
- 1 auto-linked action (fuzzy match) with "Auto-linked" badge

## Acceptance Criteria

1. Actions can be linked to specific milestones (not just objectives)
2. Completing all actions linked to a milestone auto-completes the milestone
3. Milestone auto-completion cascades to objective auto-completion (existing logic)
4. Auto-completion emits `milestone_auto_completed` signal
5. New actions are fuzzy-matched against existing milestones and auto-linked when confidence > 0.7
6. Auto-linked actions show a confirmation prompt in The Work section
7. User can unlink auto-linked actions (correction feeds Bayesian learning)
8. The Work section visually shows action→milestone linkage (not just action→objective)
9. **Mock data**: Acme milestone with 2 linked completed actions (cascade test). Globex milestone with 1 waiting action (guard test). The Work section renders linkage from mock data.

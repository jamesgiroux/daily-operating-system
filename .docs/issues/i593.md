# I593 — Captured Commitments → Milestone Candidates

**Version:** v1.0.2
**Area:** Backend / Intelligence + Success Plans
**Priority:** P2

## Problem

Transcript commitments are extracted into the `captured_commitments` table and surface as objective suggestion candidates via `get_objective_suggestions()`. But they're never matched against existing milestones. When a customer says "We'll go live by end of March" in a call, and the account has an active objective with a "Go-live" milestone — the commitment should auto-associate with that milestone, not just float as an unconnected suggestion.

## Design

### Commitment → Milestone Matching

After transcript processing extracts commitments to `captured_commitments`:

1. For the account, get active objectives with pending milestones
2. For each new commitment, compare against milestone titles (same fuzzy matching as I592)
3. If match confidence > 0.7:
   - Update the milestone's `target_date` if the commitment includes a date and milestone has no date (or commitment date is more recent)
   - Link the commitment to the milestone (new `milestone_id` FK on `captured_commitments`)
   - Emit `commitment_milestone_linked` signal
4. If no match but commitment is clearly a deliverable (has a date and owner):
   - Suggest as a new milestone under the most relevant objective
   - Surface in The Work section: "New milestone candidate: [commitment]" under the matched objective

### Schema

Add to `captured_commitments` table:
- `milestone_id TEXT NULL` — FK to `account_milestones`
- `suggested_objective_id TEXT NULL` — objective this would be a milestone for (if no milestone match)

### Frontend — The Work

Milestones with linked commitments show evidence: "Confirmed in call on March 10" with source meeting link. This makes milestones feel grounded in real conversations, not just user-invented checkpoints.

New milestone suggestions appear inline under the relevant objective with "Add milestone" action.

### Auto-Detect Signal Enhancement

When a commitment matches a milestone that has `auto_detect_signal` set, and the commitment language matches the signal type (e.g., "we'll go live" matches `auto_detect_signal: "go_live"`), boost the confidence: the customer confirmed the milestone in conversation. Don't auto-complete yet — but flag it as "customer confirmed" in the milestone UI.

## Acceptance Criteria

1. New captured commitments are matched against existing milestones (fuzzy, > 0.7)
2. Matched commitments link to the milestone via `milestone_id`
3. Commitment dates update milestone `target_date` when milestone has no date
4. Unmatched commitments with dates/owners suggested as new milestones under relevant objective
5. Milestones show evidence from linked commitments ("Confirmed in call on [date]")
6. Milestone suggestions appear inline in The Work section with "Add milestone" action
7. `commitment_milestone_linked` signal emitted on match
8. Commitments matching `auto_detect_signal` language flag milestone as "customer confirmed"

# I635 — Prep prediction scorecard: what we predicted vs what happened

**Area:** Backend / Intelligence + Frontend / Meeting Detail
**Priority:** P1
**Version:** v1.0.3
**Effort:** Medium
**Depends on:** I634 (temporal transformation — the scorecard lives in Stage 4: Processed)
**Source:** CEO plan review SCOPE EXPANSION (2026-03-21). Builds trust in prep quality and feeds Bayesian learning.

---

## Problem

DailyOS assembles pre-meeting prep with risks, wins, and talking points. After the meeting, it extracts outcomes from the transcript. But it never connects the two — "we predicted X, and Y actually happened." This is a missed opportunity:

1. **Trust building.** If the CSM sees their prep predicted 2 of 3 risks correctly, they trust the system more.
2. **Bayesian learning.** Correct predictions should increase the confidence weight of the signal sources that generated them. Wrong predictions should decrease weights.
3. **The "aha" moment.** Seeing ⚡ surprises (outcomes with no prior signal) tells the CSM what their blind spots are.

## Solution

Compare `prep_frozen_json` (frozen when meeting starts) against transcript outcomes (from Phase 2). Display in the Meeting Record as "What We Predicted vs What Happened."

### Comparison Algorithm

```
For each risk in prep_frozen_json:
  Semantic similarity against transcript risks
  If match >= 0.7 → ✓ Confirmed (predicted and happened)
  If no match → ✗ Not raised (predicted but didn't come up)

For each risk in transcript outcomes:
  If no match against prep_frozen_json risks → ⚡ Surprise (happened but not predicted)

Same logic for wins:
  Predicted win matched against transcript win → ✓ Validated
  Predicted win not in transcript → ✗ Not observed
  Transcript win not in prep → ⚡ Unexpected win
```

**Similarity matching:** Use the existing embedding model (nomic-embed-text-v1.5) for semantic comparison. Generate embeddings for prep items and transcript items, compute cosine similarity. Threshold: 0.7 for match.

**Fallback (no embeddings):** Keyword overlap scoring — extract key terms from both sides, compute Jaccard similarity. Threshold: 0.5 for match.

### Display

```
WHAT WE PREDICTED VS WHAT HAPPENED
───────────────────────────────────

RISKS
  ✓ Support ticket concerns — raised by customer
    ┊ predicted from email signals · confirmed in transcript
  ✗ Competitor evaluation — not discussed
    ┊ predicted from Glean · didn't materialize
  ⚡ API stability concern — new from Sarah Chen
    ┊ no prior signal · first mention

WINS
  ✓ Adoption milestone reached — customer acknowledged
  ⚡ Partner referral offered — unexpected
```

### Bayesian Feedback

After comparison:
- ✓ Confirmed: emit positive feedback for the source that generated the prep risk/win
- ✗ Not raised: no action (absence of evidence ≠ evidence of absence)
- ⚡ Surprise: emit signal noting a gap in intelligence coverage (logged, not penalizing)

## Acceptance Criteria

1. Processed meeting record shows "What We Predicted vs What Happened" section.
2. Three categories: ✓ confirmed, ✗ not raised, ⚡ surprise — each with icon and color.
3. Each prediction result shows provenance: "predicted from {source}" and "confirmed in transcript" or "didn't materialize."
4. Semantic similarity matching with ≥ 0.7 threshold (embedding-based or keyword fallback).
5. Correct predictions emit positive feedback signal for the generating source.
6. Meetings with no prep or no transcript outcomes: section hidden (not empty state).
7. Section appears between "The Thread" and "Key Findings" in the Meeting Record layout.

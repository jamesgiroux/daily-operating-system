# I634 — Meeting page temporal transformation: 4-stage lifecycle

**Area:** Frontend / Meeting Detail + Backend / Transcript Processing
**Priority:** P0
**Version:** v1.0.3
**Effort:** Large
**Depends on:** I621 (editorial transcript output, v1.0.2), I622 (past-meeting opacity, v1.0.2)
**Source:** CEO plan review SCOPE EXPANSION (2026-03-21). The meeting page is the crown jewel — it should tell a temporal story, not stack two documents on one page.

---

## Problem

The meeting detail page has two modes: future (full prep, editable) and past (outcomes on top, faded prep below). This creates several problems:

1. **Two documents fighting for space.** Outcomes are stacked on top of faded prep. It reads like two separate documents glued together, not a cohesive meeting record.
2. **No "just-ended" state.** The page goes from in-progress to past with no acknowledgment of the processing window. The CSM sees stale prep content while the transcript is being processed.
3. **Pre-meeting prep isn't compared against outcomes.** The prep said "watch for X" — did X happen? The system knows both but never connects them.
4. **No temporal journey.** The page doesn't *evolve* through the meeting lifecycle. It's the same page with content swapped.

## Solution

Transform the meeting page into a 4-stage temporal journey. Each stage is a purpose-built editorial document.

### Stage 1: UPCOMING (existing, minimal changes)

The current pre-meeting briefing. Full prep content, inline editable. The "Brief me before this meeting" experience.

**Changes from current:**
- No structural changes — this stage works well
- Consider: lock prep editing 5 minutes before meeting starts (transitions to Stage 2 feel cleaner)

### Stage 2: IN-PROGRESS (enhanced)

When the meeting is happening (start time reached, end time not passed).

**Changes from current:**
- Prep content becomes read-only (no more inline editing during the meeting)
- Status indicator: "Meeting in progress — recording active" (if Granola/Quill detected)
- Granola auto-check continues (existing 20s polling)
- Folio bar label changes to "In Progress"

### Stage 3: JUST-ENDED (new state)

The window between meeting end and transcript processing completion (0-30 min typically).

**Detection:** `endTime < now < endTime + 30min` AND no processed transcript.

**Content:**
```
Meeting ended 15 minutes ago.
Waiting for transcript...

[Or, if processing has started:]

Assembling meeting intelligence...

● Phase 1: Summary & actions ─── done (7 actions extracted)
● Phase 2: Outcomes & risks ──── done (3 wins, 2 risks, 1 decision)
○ Phase 3: Deep analysis ─────── in progress...
```

- Uses existing `transcript-progress` events (already fire per-phase with counts)
- Pre-meeting prep content hidden (not faded — hidden entirely during this transitional state)
- Folio bar label: "Processing..."

### Stage 4: PROCESSED (transformed — the Meeting Record)

The page transforms into a cohesive post-meeting document. NOT "outcomes on top of prep" — a purpose-built Meeting Record.

**Layout:**

```
THE MEETING RECORD
═══════════════════════════════════════════

[Headline — AI-generated one-liner about this meeting]
"Renewal secured but migration concerns emerged."

[Meeting metadata: date, time, duration, type, entity]
[Intelligence quality badge]

───────────────────────────────────────────

THE THREAD (I637)
Since your last meeting with Acme Corp on Mar 7...
• 2 actions completed
• Support ticket volume decreased
• Mike Torres promoted to VP

───────────────────────────────────────────

WHAT WE PREDICTED VS WHAT HAPPENED (I635)
✓ Support concerns raised (predicted risk)
✗ Expansion discussion (predicted but not raised)
⚡ API stability concern from Sarah Chen (surprise)

───────────────────────────────────────────

KEY FINDINGS
[I621 editorial treatment — structured outcomes]

  WINS (3)
  ───────
  [grouped by sub-type, evidence quotes]

  RISKS (2)
  ────────
  [urgency-coded, evidence quotes]

  DECISIONS (1)
  ─────────────
  [with tags and context]

───────────────────────────────────────────

ENGAGEMENT
[Interaction dynamics, champion health]

───────────────────────────────────────────

COMMITMENTS & ACTIONS
[Commitments with owner, actions with due dates]

───────────────────────────────────────────

THE ROOM
[Attendee summary — who was there, RSVP status]

═══════════════════════════════════════════

▼ Pre-Meeting Context (collapsed)
  [Full prep content from before — collapsed by default]
  [Expandable for reference]

═══════════════════════════════════════════
FINIS
```

**Key design decisions:**
- Pre-meeting content is a **collapsed appendix**, not faded content below. Clicking expands it.
- The Meeting Record is the primary content — it's what you read.
- Headline is generated from the transcript summary (Phase 1 output), not the prep context.
- Folio bar label: "Meeting Record"
- Folio bar actions: Share (copy formatted record), Reprocess, View Transcript

## Implementation Notes

**State detection logic (extend existing):**
```typescript
type MeetingStage = 'upcoming' | 'in-progress' | 'just-ended' | 'processed';

function getMeetingStage(meeting, now, hasProcessedTranscript): MeetingStage {
  if (hasProcessedTranscript) return 'processed';
  if (endTime && now > endTime && !hasProcessedTranscript) return 'just-ended';
  if (startTime && now >= startTime && (!endTime || now <= endTime)) return 'in-progress';
  return 'upcoming';
}
```

**The page renders different content per stage, not different components.** Same MeetingDetailPage component, conditional sections based on stage. This avoids the over-engineering risk of Approach C (separate components per state).

**prep_frozen_json is key.** The prep is frozen when the meeting starts. The prediction scorecard (I635) compares `prep_frozen_json` against transcript outcomes. This data is already stored in the `meeting_prep` table.

## Acceptance Criteria

1. Upcoming meeting shows full prep briefing, inline editable. No changes from current.
2. In-progress meeting shows read-only prep with "Meeting in progress" indicator.
3. Just-ended meeting (0-30 min after end, no transcript) shows "Meeting ended X minutes ago. Waiting for transcript..." No stale prep visible.
4. When transcript processing starts, phase progress shows with counts.
5. Processed meeting transforms into Meeting Record layout — headline, outcomes, prep collapsed as appendix.
6. Pre-meeting content is accessible via "Pre-Meeting Context" collapsed section at bottom.
7. Folio bar label updates per stage: Briefing → In Progress → Processing... → Meeting Record.
8. Transition between stages is smooth — no full page reload, no layout shift.
9. Direct URL to a processed meeting (`/meeting/{id}`) renders the Meeting Record, not the prep.
10. Back navigation from Meeting Record returns to the correct surface (daily briefing or account dossier).

# I535 — Glean MCP Chat Intelligence Provider

**Version:** v1.0.0 Phase 5
**Depends on:** I559 (validation spike — COMPLETE, go confirmed), I508a (intelligence schema types), I487 (Glean signal emission), I554 (transcript signal fidelity — defines the enriched prompt schema that Glean prompts must also use), I555 (captures metadata — defines the storage schema for Glean-produced captures)
**Supersedes:** Original I535 spec (Agents REST API design — API not available, `chat` MCP tool is superior)
**Type:** Feature — new intelligence provider
**Scope:** Backend: new provider, enrichment pipeline integration, onboarding bootstrap, health scoring augmentation. Frontend: Settings updates, onboarding flow.

---

## Context

The I559 validation spike (2026-03-14) confirmed that Glean's MCP `chat` tool:

1. Returns structured JSON matching our I508 schema when prompted
2. Synthesizes across Salesforce, Zendesk, Gong, Slack, P2, and org directories in a single call
3. Can discover a user's accounts from their email identity alone
4. Responds in 10-30 seconds (vs 60-180s for PTY calls)
5. Is included in the enterprise Glean subscription (no per-token cost)

The Agents REST API is not available on our Glean instance and is not needed — the `chat` MCP tool IS the agent. This simplifies the design: single transport (existing `GleanMcpClient`), existing auth, no new HTTP client.

See: ADR-0100 (Glean-first intelligence architecture), `.docs/research/2026-03-14-glean-agent-validation-spike.md`

## Design

### 1. GleanIntelligenceProvider

New struct implementing a provider interface that parallels `LocalIntelligenceProvider`:

```rust
pub struct GleanIntelligenceProvider {
    mcp_client: GleanMcpClient,  // existing, reused
    cache: GleanContextCache,     // existing, reused
}

impl GleanIntelligenceProvider {
    /// Full entity intelligence enrichment via Glean chat.
    /// Returns IntelligenceJson-compatible output.
    pub async fn enrich_entity(
        &self,
        entity_id: &str,
        entity_type: &str,
        entity_name: &str,
        local_context: &IntelligenceContext,  // meetings, actions, user edits — always local
    ) -> Result<GleanEnrichmentResult, ContextError>

    /// Discover accounts for a user by email.
    /// Used during onboarding bootstrap.
    pub async fn discover_accounts(
        &self,
        user_email: &str,
        user_name: &str,
    ) -> Result<Vec<DiscoveredAccount>, ContextError>

    /// People/stakeholder enrichment for an account.
    pub async fn enrich_stakeholders(
        &self,
        account_name: &str,
    ) -> Result<Vec<GleanStakeholder>, ContextError>

    /// Targeted gap-fill query for a specific intelligence dimension.
    pub async fn fill_dimension_gap(
        &self,
        entity_name: &str,
        dimension: &str,
        gap_description: &str,
    ) -> Result<serde_json::Value, ContextError>
}
```

### 2. Structured Prompts Per Intelligence Role

Each enrichment call sends a structured prompt to the `chat` MCP tool requesting JSON output matching our I508 types. The prompts are built in a new `intelligence/glean_prompts.rs` module.

**Entity enrichment prompt pattern:**

```
You are a customer success intelligence system. Analyze the customer account {name}
using all available data (support tickets, Salesforce, Gong calls, internal documents, Slack).

Current local context (from calendar and transcripts):
- Recent meetings: {meeting_history}
- Open actions: {actions}
- User notes: {context_entries}

Return ONLY a JSON object matching this schema:
{Full I508 + I554 enriched schema — see below}
```

The local context is injected so Glean's synthesis includes what DailyOS already knows — preventing contradictions and enabling Glean to add what's missing rather than duplicate what's known.

### Phase 4b Schema Alignment (CRITICAL)

Glean prompts MUST request the same enriched schema that PTY calls produce after I554 lands. Both intelligence paths produce identical output types — same fields, same sub-types, same urgency tiers. The only difference is the data source, not the output shape.

**Glean prompts must include:**

| I554 Addition | How Glean Prompt Requests It |
|---|---|
| **Win sub-types** | `"recentWins": [{ "text": "...", "impact": "high\|medium\|low", "subType": "adoption\|expansion\|value_realized\|relationship\|commercial\|advocacy" }]` |
| **Risk urgency tiers** | `"risks": [{ "text": "...", "urgency": "critical\|watch\|low", "subType": "champion_departure\|competitor_eval\|usage_decline\|..." }]` |
| **Value delivered quantification** | `"valueDelivered": [{ "statement": "quantified outcome — must include a number", "impact": "revenue\|cost\|risk\|speed", "date": "...", "source": "..." }]` — with guidance: "only extract when measurable, not vague usage" |
| **Champion health** | `"championHealth": { "name": "...", "status": "strong\|weak\|lost\|none", "evidence": "...", "risk": "..." }` — with MEDDPICC criteria (Power, Vested Interest, Active Advocacy) |
| **COMMITMENTS** | `"commitments": [{ "content": "...", "ownedBy": "us\|them\|joint", "successCriteria": "...", "targetDate": "..." }]` |
| **successPlanSignals** | `"successPlanSignals": { "statedObjectives": [...], "mutualSuccessCriteria": [...], "milestoneCandidates": [...] }` |
| **Sentiment** | `"sentiment": { "overall": "...", "ownershipLanguage": "customer\|vendor\|mixed", "forwardLooking": true, "internalAdvocacyVisible": true }` |
| **Role changes** | `"roleChanges": [{ "personName": "...", "oldStatus": "...", "newStatus": "...", "evidence": "..." }]` |

**Glean-specific additions** (data Glean has that PTY doesn't):

| Field | Why Glean Can Produce It |
|---|---|
| `supportHealth` | Glean reads Zendesk directly — ticket counts, severity, SLA, trends |
| `salesforceContext` | Glean reads Salesforce — opportunity stage, pipeline value, renewal probability |
| `gongCallSummaries` | Glean reads Gong — recent call topics, sentiment, speakers |
| `orgChartChanges` | Glean reads HRIS/directory — role changes, departures, reorgs |

These Glean-specific fields are written to `entity_assessment.dimensions_json` (same column as the I508a dimension types) and flow through to all surfaces via `get_entity_intelligence()`.

**Storage of Glean-produced captures:**

When Glean's response includes categorized wins, risks, decisions, and commitments, these are written to both:
1. `entity_assessment` (intelligence JSON) — for narrative rendering on detail pages
2. `captures` table with I555 metadata columns (`sub_type`, `urgency`, `impact`, `evidence_quote`, `speaker`) and `source = "glean_chat"` — for report content pipeline (I556) and timeline rendering
3. `captured_commitments` table — for I551 objective suggestion pipeline

This dual-write ensures Glean-produced intelligence flows through the same Intelligence Loop paths as PTY-produced intelligence.

**Account discovery prompt pattern:**

```
I am {name} ({email}), a {role}. Find all customer accounts I own, manage,
or am actively involved with. Search Salesforce account ownership, Gong call
participation, Zendesk ticket assignments, and internal documents.

Return ONLY a JSON object: { "accounts": [{ "name", "myRole", "evidence", "source" }] }
```

**Dimension gap-fill prompt pattern:**

```
For the customer account {name}, I need specific intelligence about {dimension}.
Current assessment shows a gap: {gap_description}.
Search all available sources and return ONLY a JSON object: {dimension-specific schema}
```

### 3. Integration Into Enrichment Pipeline

In `intel_queue.rs`, the enrichment flow changes:

```rust
// Existing: gather local context (meetings, actions, captures, user context)
let local_context = build_intelligence_context(db, entity, ...);

// NEW: check if Glean is connected
if let Some(glean_provider) = &state.glean_intelligence_provider {
    // Glean-first path
    match glean_provider.enrich_entity(
        &entity.id, &entity.entity_type, &entity.name, &local_context
    ).await {
        Ok(glean_result) => {
            // Merge Glean output with local data
            let merged = merge_glean_with_local(glean_result, &local_context);
            // Write to entity_assessment
            write_intelligence(db, &entity.id, &merged)?;
            // Emit signals with source="glean" at tiered confidence
            emit_glean_signals(db, &engine, &entity, &glean_result)?;
            return Ok(());
        }
        Err(e) => {
            log::warn!("Glean enrichment failed for {}: {}, falling back to PTY", entity.name, e);
            // Fall through to PTY path
        }
    }
}

// Existing PTY path (fallback or non-Glean users)
let pty_result = run_enrichment_pty(state, entity, &local_context).await?;
write_intelligence(db, &entity.id, &pty_result)?;
```

### 4. Signal Emission from Glean Data

Glean responses emit signals at tiered confidence per ADR-0100:

```rust
fn emit_glean_signals(
    db: &ActionDb,
    engine: &PropagationEngine,
    entity: &Entity,
    result: &GleanEnrichmentResult,
) -> Result<()> {
    // CRM data at 0.9
    if let Some(renewal) = &result.renewal_data {
        emit_signal_and_propagate(db, engine, &entity.entity_type, &entity.id,
            "renewal_data_updated", "glean_crm",
            &serde_json::to_string(renewal)?, 0.9)?;
    }

    // Support health at 0.85
    if let Some(support) = &result.support_health {
        emit_signal_and_propagate(db, engine, &entity.entity_type, &entity.id,
            "support_health_updated", "glean_zendesk",
            &serde_json::to_string(support)?, 0.85)?;
    }

    // AI synthesis at 0.7
    emit_signal(db, &entity.entity_type, &entity.id,
        "intelligence_refreshed", "glean_chat",
        &serde_json::to_string(&result.intelligence)?, 0.7)?;

    // Competitive mentions at 0.7
    for comp in &result.competitive_context {
        emit_signal(db, &entity.entity_type, &entity.id,
            "competitor_mentioned", "glean_chat",
            &serde_json::to_string(comp)?, 0.7)?;
    }

    Ok(())
}
```

### 8. Merge Strategy: Glean + Local

Glean output is merged with local data, not replaced by it:

| Field | Merge Rule |
|---|---|
| `executiveAssessment` | Glean replaces (it has richer cross-source context) |
| `risks` | Union of Glean risks + local-only risks (from transcripts). Deduplicate by text similarity. |
| `recentWins` | Union. Glean adds CRM wins; local adds transcript wins. |
| `stakeholderInsights` | Glean replaces for org-level view. Local attendee data supplements. |
| `health` | Algorithmic dimensions computed from ALL signals (local + Glean). Narrative from Glean. |
| `currentState` | Glean replaces (cross-source view is more complete). |
| `competitiveContext` | Glean replaces (has Gong + Slack + CRM data we can't match locally). |
| `valueDelivered` | Union. Glean adds CRM-sourced outcomes; local adds transcript-captured outcomes. |
| `supportHealth` | Glean replaces (Zendesk is source of truth). |

**User edits always win.** If a user has edited a field via inline editing (tracked in `user_edits_json`), that field is protected from both Glean and PTY overwrites.

### 9. Error Handling + Graceful Degradation

```rust
pub struct GleanCallResult<T> {
    pub data: Option<T>,
    pub source: GleanSource,
    pub latency_ms: u64,
    pub fell_back: bool,
}

pub enum GleanSource {
    Chat,          // Full chat synthesis
    SearchOnly,    // Chat failed, fell back to search
    LocalOnly,     // Glean unavailable, used PTY
}
```

- **Chat timeout (60s):** Fall back to search-based context gathering (existing `GleanContextProvider` behavior) + PTY enrichment
- **Auth failure (401):** Attempt token refresh. If refresh fails, mark Glean as disconnected, fall back to local.
- **Rate limit (429):** Exponential backoff. Budget: max 20 chat calls per enrichment cycle.
- **Parse failure:** If chat returns text that doesn't parse as JSON, fall back to PTY with Glean search context.
- **Partial data:** If JSON parses but is missing expected fields, merge what's available and mark missing dimensions as locally-filled.

## Acceptance Criteria

### Core Intelligence
1. With Glean connected, entity enrichment uses `chat` MCP tool as primary intelligence source. PTY call does NOT fire when Glean succeeds.
2. Glean enrichment output parses into `IntelligenceJson` and writes to `entity_assessment`. All existing surfaces (account detail, meeting prep, reports) render Glean-sourced intelligence identically to PTY-sourced.
3. When Glean call fails (timeout, auth, parse error), enrichment falls back to PTY call transparently. User sees no difference.
4. Without Glean connected, enrichment behavior is identical to pre-Phase 5 (pure PTY).

### Signal Integration (Intelligence Loop)
5. Glean enrichment emits signals with `source = "glean_crm"` (confidence 0.9), `"glean_zendesk"` (0.85), `"glean_chat"` (0.7) — tiered per ADR-0100.
6. `signal_weights` tracks Glean sources independently. User correction on Glean-sourced data penalizes the Glean source, not the PTY source.
7. `renewal_data_updated` and `support_health_updated` added to `CALLOUT_SIGNAL_TYPES`. Salesforce renewal at <50% probability within 90 days surfaces as critical callout in morning briefing.
8. Health scoring dimensions consume Glean-sourced signals: `championHealth` reads Gong engagement, `financialProximity` reads Salesforce renewal data, `signalMomentum` includes Zendesk ticket velocity.

### Account Discovery
9. `discover_accounts_from_glean` returns accounts associated with the user's email, with role attribution and evidence source.
10. Onboarding flow shows "We found N accounts" after Glean auth. User confirms subset. Confirmed accounts created as entities with immediate enrichment.
11. Discovery hidden when Glean not connected.

### Merge + Data Integrity
12. User edits (tracked in `user_edits_json`) are never overwritten by Glean data.
13. Local-only data (transcript wins, meeting-sourced risks, user context entries) is preserved in the merged result, not replaced.
14. `data_source = "glean"` tagged on all Glean-sourced stakeholders and signals. `purge_source(DataSource::Glean)` removes all on disconnect.

### Performance + Error Handling
15. Chat call timeout: 60 seconds. Fallback to PTY within 5 seconds of failure.
16. Rate limit: max 20 chat calls per enrichment cycle. 429 triggers exponential backoff.
17. Enrichment cycle with 10 accounts completes in < 10 minutes via Glean (vs current ~30 minutes via PTY).

### Verification
18. Enrich an account with Glean. Compare output to PTY-only enrichment for the same account. Glean output includes data not available locally (support tickets, Salesforce fields, competitive intel from Gong). PTY output is limited to calendar + email context.
19. Disconnect Glean. Re-enrich. PTY fires. No errors. Intelligence updates with local-only data.
20. User thumbs-down a Glean-sourced risk. `signal_weights` for `"glean_chat"` source gets beta += 1. Next enrichment produces different output.

22. Glean enrichment produces risks with urgency tiers (`critical`, `watch`, `low`) — identical to I554 PTY output. Verify: `entity_assessment.risks_json` for a Glean-enriched account contains `urgency` fields.

# I641 — Folio bar dropdown menus broken on account detail page

**Area:** Frontend / Navigation / MagazineShell
**Priority:** P0
**Version:** v1.0.4
**Effort:** Small
**Source:** User report (2026-03-22). Previously fixed in Mar 15 commits (6bbdf2f, ba44f7e) but the fix didn't hold.

---

## Problem

On the account detail page, clicking Reports or Tools in the folio bar does nothing — dropdown menus don't open, or if they open, the items navigate to the wrong account. This was previously fixed by introducing a `useUpdateFolioVolatile()` hook with a `repaintKey` parameter, but the underlying subscription mechanism is broken.

## Root Cause

`useFolioVolatile()` in `src/hooks/useMagazineShell.ts` (line ~155) reads `folioPaintCount` with a void expression:

```typescript
export function useFolioVolatile(): FolioVolatileState {
  const ctx = useContext(MagazineShellContext);
  void ctx.folioPaintCount;  // ← BUG: does NOT subscribe to state changes
  return ctx.volatileRef.current;
}
```

In React, `void ctx.folioPaintCount;` reads and discards the value without creating a dependency. The consuming component (MagazinePageLayout) never re-renders when `folioPaintCount` changes, so it keeps displaying stale `folioActions` JSX with handlers that capture the old `accountId`.

### The Stale Closure Chain

1. User navigates to Account A → `AccountDetailEditorial` renders with `accountId = "A"`
2. `useUpdateFolioVolatile()` writes new folioActions JSX (containing dropdown buttons with A's handlers) to `volatileRef.current`
3. `requestFolioRepaint()` increments `folioPaintCount` in context state
4. `MagazinePageLayout` calls `useFolioVolatile()` — but `void ctx.folioPaintCount` does NOT subscribe
5. **MagazinePageLayout does not re-render** → renders stale folioActions from initial mount
6. Dropdown buttons fire handlers that reference the wrong accountId or don't exist yet

### Why Other Pages Aren't Affected

ProjectDetailEditorial and PersonDetailEditorial use simpler folio actions (local state toggles, not parameterized navigation). Their handlers work even with stale closures because they don't depend on route params.

## Fix

Replace the void read with an actual subscription. Two options:

**Option A (simplest):** Return `folioPaintCount` as part of the volatile state so the consumer component actually references it:

```typescript
export function useFolioVolatile(): FolioVolatileState & { _paintCount: number } {
  const ctx = useContext(MagazineShellContext);
  return { ...ctx.volatileRef.current, _paintCount: ctx.folioPaintCount };
}
```

The consumer destructures the result, and because `_paintCount` is part of the return value, React sees the context dependency.

**Option B (cleaner):** Use `useSyncExternalStore` or simply read `folioPaintCount` in a way that creates a render dependency:

```typescript
export function useFolioVolatile(): FolioVolatileState {
  const ctx = useContext(MagazineShellContext);
  const _ = ctx.folioPaintCount; // Reference in scope creates dependency
  return ctx.volatileRef.current;
}
```

Actually, reading from context IS a subscription in React — the issue may be that `folioPaintCount` isn't changing the context value object identity. Check if `MagazineShellContext.Provider` creates a new value object when `folioPaintCount` changes.

**Most likely fix:** The context provider value is memoized and doesn't include `folioPaintCount` as a dependency, so changing the count doesn't change the context value identity, and consumers don't re-render.

1. Navigate to Account A → folio bar shows Reports/Tools dropdowns → they open and navigate correctly
2. Navigate from Account A to Account B → folio bar dropdowns update → Reports items navigate to Account B (not A)

4. Project detail page folio actions still work (no regression)
5. Person detail page folio actions still work (no regression)
6. No full page remount — fix must be within the existing volatile ref system

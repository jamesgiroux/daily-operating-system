# I594 — Decision-Requiring Actions

**Version:** v1.0.2
**Area:** Backend / Intelligence + Frontend / Actions
**Priority:** P2

## Problem

The `needs_decision` flag exists on the actions table and `flag_action_as_decision()` / `get_flagged_decisions()` queries exist — but nothing ever sets the flag. Actions that require a decision ("pending legal approval," "waiting on budget sign-off," "needs exec sponsor alignment") look identical to routine tasks. CS users need to distinguish between actions they can just execute and actions that are blocked on a decision.

## Design

### Intelligence-Driven Flagging

During entity enrichment (or as a lighter post-enrichment step), scan pending actions for decision indicators:

**Pattern matching** (no LLM needed for initial pass):
- Keywords: "approval," "decision," "sign-off," "pending review," "blocked on," "needs alignment," "budget," "legal"
- Context patterns: action title or context contains decision language
- Waiting-on + overdue: actions delegated and past due likely need escalation (a decision)

**LLM enrichment** (during intel queue processing):
- For actions matching keyword patterns, include in enrichment prompt: "Is this action blocked on a decision? If yes, who needs to make the decision and what's at stake?"
- Returns `needs_decision: true/false`, `decision_owner: string`, `decision_stakes: string`

### Schema Addition

Extend actions table:
- `decision_owner TEXT NULL` — who needs to make the decision
- `decision_stakes TEXT NULL` — what's at stake (one sentence)
- `needs_decision` already exists (INTEGER, currently always 0)

### Frontend — Decision Surface

**Actions Page**: New filter/view: **"Needs Decision"** — shows all actions where `needs_decision = 1`.

Each item shows:
- Action title
- Decision owner ("Needs: VP Engineering approval")
- Stakes ("Blocks go-live timeline")
- Account name
- Age since flagged
- Action: "Resolved" (clears the flag, emits signal)

**Daily Briefing**: Decision-requiring actions for today's meeting accounts surface as callouts: "Decision needed before Acme call: budget approval for Phase 2."

**Meeting Prep**: Decision actions for meeting attendees highlighted in prep context (I587 handles this structurally; this issue ensures the data is populated).

### Signal Emission

- `decision_flagged` (confidence 0.6 for pattern match, 0.8 for LLM-confirmed)
- `decision_resolved` (confidence 0.8, source: user_action)
- Decision count feeds engagement dimension (many unresolved decisions = stagnation signal)

## Mock Data

Requires I597 (action mock data expansion). Uses 2 pre-flagged actions:
- `mock-act-decision-1`: Acme budget approval, `decision_owner = "VP Engineering"`, `decision_stakes = "Blocks Phase 2 kickoff"`
- `mock-act-decision-2`: Globex renewal terms, `decision_owner = "Legal"`, `decision_stakes = "Renewal at risk"`

## Acceptance Criteria

1. Pending actions with decision-language keywords get `needs_decision = 1` flag set
2. LLM enrichment adds `decision_owner` and `decision_stakes` for flagged actions
3. "Needs Decision" filter on Actions page shows all flagged actions
4. Each item shows decision owner, stakes, account, and age
5. "Resolved" action clears the flag and emits `decision_resolved` signal
6. Daily briefing shows decision callouts for today's meeting accounts
7. Decision count feeds health scoring (unresolved decisions = stagnation)
8. Pattern matching runs without LLM (fast, no PTY needed); LLM enrichment adds detail during intel queue
9. **Mock data**: "Needs Decision" view shows 2 items (Acme budget + Globex renewal) with owner and stakes from mock data.

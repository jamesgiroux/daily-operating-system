# I564 — PTY enrichment blocks async runtime

**Version:** v1.0.0 Phase 6
**Depends on:** none
**Type:** Bug fix — Backend / Performance
**Scope:** Backend: `intel_queue.rs`. No frontend changes.

---

## Problem

The background intelligence processor in `intel_queue.rs` calls `run_enrichment()` (line 398) and `run_batch_enrichment()` (line 413) directly on Tokio async worker threads. Both functions are synchronous — they spawn a PTY subprocess (`PtyManager::spawn_claude`) and block for 30–180 seconds waiting for the process to complete.

Tokio's async runtime has a limited thread pool (typically equal to CPU cores). When `run_enrichment()` blocks an async worker thread for 60+ seconds, that thread cannot service any other futures. With `MAX_BATCH_SIZE = 3` and potentially multiple enrichment cycles, this starves IPC command handlers (`invoke()` calls from the frontend), causing:

- macOS beach balls when clicking UI elements during background enrichment
- Delayed responses to navigation, dropdown clicks, and save operations
- Stalled `refresh_emails`, meeting prep, and other async pipelines waiting for a thread

The correct pattern already exists in `services/intelligence.rs` (lines 125, 136), which wraps PTY calls in `tauri::async_runtime::spawn_blocking()`. The intel queue processor does not follow this pattern.

The same issue affects the PTY fallback paths inside `run_glean_enrichment_with_fallback()` (lines 907, 915, 1001, 1008) — when Glean enrichment fails and falls back to PTY, those calls also block async threads.

---

## Solution

### 1. Wrap all synchronous PTY calls in `spawn_blocking`

In the processor loop (around lines 396–414), replace direct calls with:

```rust
// Single entity
let input_clone = input.clone();
let ai_config_clone = ai_config.clone();
let result = tauri::async_runtime::spawn_blocking(move || {
    run_enrichment(&input_clone, &ai_config_clone)
}).await;

// Batch
let inputs_clone = inputs;
let ai_config_clone = ai_config.clone();
let batch_results = tauri::async_runtime::spawn_blocking(move || {
    run_batch_enrichment(inputs_clone, &ai_config_clone)
}).await;
```

### 2. Fix PTY fallback paths in `run_glean_enrichment_with_fallback()`

The Glean fallback PTY calls at lines 907, 915, 1001, and 1008 also need `spawn_blocking`. Since `run_glean_enrichment_with_fallback` is already `async`, this is straightforward:

```rust
// Line 1001 (single fallback)
let result = tauri::async_runtime::spawn_blocking(move || {
    run_enrichment(&input, ai_config)
}).await;

// Line 1008 (batch fallback)
let batch = tauri::async_runtime::spawn_blocking(move || {
    run_batch_enrichment(pty_fallback_inputs, ai_config)
}).await;
```

### 3. Fix `run_consistency_repair_retry()` caller

The consistency repair retry at line 1080 (`run_consistency_repair_retry`) also spawns a PTY subprocess synchronously. Its caller in the post-enrichment phase (search for `run_consistency_repair_retry` invocations in the processor loop) must also be wrapped in `spawn_blocking`.

### 4. No changes to `run_enrichment()` or `run_batch_enrichment()` signatures

Keep both functions as `pub fn` (synchronous). They are correct as-is — the fix is in the *callers*, not the functions themselves. This matches the pattern in `services/intelligence.rs` where the same `run_enrichment` function is called via `spawn_blocking`.

## Acceptance Criteria

1. `run_enrichment()` and `run_batch_enrichment()` never execute on a Tokio async worker thread — they always run on the blocking thread pool via `spawn_blocking`.
2. During background enrichment of 3 entities (~90s), clicking "Refresh Intelligence" on an account detail page responds within 2 seconds (IPC not starved).
3. During background enrichment, navigating between pages in the app produces no beach ball or visible delay.
4. Glean-mode PTY fallback paths also use `spawn_blocking` — if Glean fails and falls back to PTY for 2 entities, async threads remain free.
5. `JoinError` from `spawn_blocking` (task panic) is caught and logged, not propagated as a panic on the async thread.
6. `cargo test` passes. `cargo clippy -- -D warnings` clean.
7. No change to enrichment output — same intelligence JSON produced, same DB writes, same signal emissions.

# I411 — User Entity — Data Model, Two-Layer Priorities, Context Entries

**Status:** Open
**Priority:** P1
**Version:** 0.14.0
**Area:** Backend / Database

## Summary

The foundational backend for the user entity. This issue covers: (1) the `user_entity` table with declared professional context fields expanded to include the full field set from ADR-0090, (2) the two-layer priority model — annual priorities (year-level, the big bets) and quarterly priorities (current quarter focus, more specific) — both persist until the user changes them, no automatic expiration, (3) the `user_context_entries` table for embedded professional knowledge inputs that are not tied to any specific entity, and (4) the `_user/context.json` workspace file written on every field update. All Tauri commands for CRUD on these tables.

## Acceptance Criteria

### Table: user_entity

1. The `user_entity` table exists with the following columns:

   - `id` (INTEGER PRIMARY KEY) — single row per installation, id = 1
   - `value_proposition` (TEXT, nullable)
   - `success_definition` (TEXT, nullable)
   - `current_priorities` (TEXT, nullable) — simple freeform text for the initial focus phrase; preserved for the daily briefing's focus directive; distinct from the structured priority arrays below
   - `product_context` (TEXT, nullable)
   - `playbooks` (TEXT, nullable)
   - `company_bio` (TEXT, nullable) — one paragraph describing what the company does and who it serves
   - `role_description` (TEXT, nullable) — what the user actually does, what they're responsible for
   - `how_im_measured` (TEXT, nullable) — the KPIs or outcomes the user tracks
   - `pricing_model` (TEXT, nullable) — category-level description only, not exact figures
   - `differentiators` (TEXT, nullable) — JSON array of strings
   - `objections` (TEXT, nullable) — JSON array of Q&A objects: `[{"question": "...", "response": "..."}]`
   - `competitive_context` (TEXT, nullable)
   - `annual_priorities` (TEXT, nullable) — JSON array of objects: `[{"id": "...", "text": "...", "linked_entity_id": "...", "linked_entity_type": "account|project|person", "created_at": "..."}]`; `linked_entity_id` and `linked_entity_type` are optional per item. Year-level bets. No expiration — persists until the user removes or updates items.
   - `quarterly_priorities` (TEXT, nullable) — JSON array of objects: `[{"id": "...", "text": "...", "linked_to_id": "...", "linked_to_type": "action|meeting|person", "created_at": "..."}]`; `linked_to_id` and `linked_to_type` are optional per item. No expiration — persists until the user removes or updates items.
   - `created_at` (DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP)
   - `updated_at` (DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP)

   The table enforces a single row via application logic: insert on first access if no row exists; subsequent writes are UPDATE. Verify: `SELECT COUNT(*) FROM user_entity` — returns 1 after any `get_user_entity` call.

### Table: user_context_entries

2. The `user_context_entries` table exists with the following columns:

   - `id` (TEXT PRIMARY KEY) — UUID, generated by the backend on creation
   - `title` (TEXT NOT NULL)
   - `content` (TEXT NOT NULL)
   - `embedding_id` (TEXT, nullable) — FK to `content_embeddings.id`; null until embedding completes
   - `created_at` (DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP)
   - `updated_at` (DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP)

### Tauri commands: user_entity

3. The following Tauri commands exist and are registered in `commands.rs`:

   - `get_user_entity` — returns the full user entity row (all fields). If no row exists, inserts a default row (all nullable fields null) and returns it.
   - `update_user_entity_field(field: String, value: String)` — updates a single named field on the user entity row. Rejects unknown field names with a typed error. Updates `updated_at`. After writing to the DB, writes `_user/context.json` to the workspace (see criterion 5).

   The field-by-field update pattern (rather than a full-row replace) ensures concurrent edits to different fields do not overwrite each other. All fields that accept JSON (differentiators, objections, strategic_priorities, tactical_priorities) validate that the value is parseable JSON before writing; reject with a descriptive error if not.

### Tauri commands: user_context_entries

4. The following Tauri commands exist and are registered in `commands.rs`:

   - `get_user_context_entries` — returns all entries ordered by `created_at DESC`.
   - `create_user_context_entry(title: String, content: String)` — generates a UUID for `id`, inserts the row, enqueues the entry body for embedding via the background embedding processor (task #9) using the `user_context` collection label. Returns the created entry (with `embedding_id = null` at creation time).
   - `update_user_context_entry(id: String, title: String, content: String)` — updates `title`, `content`, and `updated_at`. Re-enqueues the updated content for embedding (replaces the old embedding). If an existing `embedding_id` exists, the old embedding record is deleted before the new one is created.
   - `delete_user_context_entry(id: String)` — deletes the entry and its associated embedding record (if present). Cascades cleanly.

   Verify (embedding pipeline): create a context entry via `create_user_context_entry`. Wait for the embedding processor (task #9) to run its next cycle — typically under 2 minutes in dev. Query: `SELECT embedding_id FROM user_context_entries WHERE title = '<title>'` — returns a non-null value.

### workspace file

5. `_user/context.json` is written to the workspace root on every `update_user_entity_field` call. The file format mirrors entity context files: a flat JSON object with all non-null user entity fields. Null fields are omitted from the JSON (no null values in the output). Example partial output:

   ```json
   {
     "value_proposition": "We help media companies publish faster...",
     "success_definition": "Customers succeed when time-to-publish drops...",
     "strategic_priorities": [
       {"id": "sp-1", "text": "Expand Cox to two new business units", "linked_entity_id": "acc-cox", "linked_entity_type": "account", "created_at": "2026-02-22T00:00:00Z"}
     ]
   }
   ```

   Verify: call `update_user_entity_field("value_proposition", "We help media companies...")`. Read `<workspace_path>/_user/context.json` — file exists and contains the updated value. The file is human-readable and consistent with the entity context file pattern.

### Priority entity linking

6. Annual priorities support entity linking. Each item in the `annual_priorities` JSON array may include `linked_entity_id` and `linked_entity_type` ("account" | "project" | "person"). Quarterly priorities support linking to actions, meetings, or people via `linked_to_id` and `linked_to_type`. These links are consumed by I414 (signal scoring).

   Verify: call `update_user_entity_field("annual_priorities", '[{"id":"ap-1","text":"Expand Cox to three BUs","linked_entity_id":"<cox-account-id>","linked_entity_type":"account","created_at":"2026-02-22T00:00:00Z"}]')`. Query: `SELECT annual_priorities FROM user_entity` — the JSON includes the `linked_entity_id` with the Cox account ID.

   **No expiration on either priority type.** Both `annual_priorities` and `quarterly_priorities` persist in full until the user explicitly removes or replaces items. `get_user_entity` returns the complete arrays as stored — no filtering, no automatic archival. This is intentional: requiring users to refresh priorities on a schedule creates maintenance burden that violates the zero-guilt design principle. Stale priorities are still useful context. The user updates them when they want to.

### Backward compatibility

7. All five fields used by the existing YouCard (`user_name`, `user_focus`, `user_company`, `user_title`) remain in the `workspace_config` table, unchanged. Their load/save behaviour is unaffected. The `user_entity` table is additive. `cargo test` passes with no regressions.

## Dependencies

Foundational. Blocks I412, I414, I415, I416, I417.

## Notes / Rationale

The expanded field set (company_bio, role_description, how_im_measured, pricing_model, differentiators, objections, competitive_context) represents the full professional context surface defined in ADR-0090. These fields are all nullable and degrade gracefully — users who don't fill them in get the same behaviour as today.

The two-layer priority model (strategic + tactical) replaces the single `current_priorities` text field as the primary priority interface. `current_priorities` is preserved as a simple freeform field for backward compatibility with the daily briefing's focus directive, which reads it directly. The structured priority arrays are new and additive.

The `user_context_entries` table and its embedding pipeline path are the backend foundation for I417. The distinction between context entries and entity notes matters: entity notes document interactions with a specific entity and live on that entity. Context entries document the user's own professional knowledge and judgment — they are retrieved semantically across all enrichment contexts, not associated with any single entity.

Field-level update (`update_user_entity_field`) rather than full-row replacement is intentional: it prevents concurrent UI edits to different fields from overwriting each other, and it mirrors the pattern used in entity metadata updates.

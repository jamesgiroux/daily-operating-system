# I617 — Top 20 console.error → toast.error for user-facing actions

**Area:** Frontend / UX
**Priority:** P1
**Version:** v1.0.1
**Source:** v1.0.0 GA engineering review (2026-03-16), Issue 5A

---

## Problem

156 bare `console.error()` calls exist without user-visible feedback. When user-initiated actions fail (save, refresh, toggle), the user sees a spinner that never resolves or a button that silently does nothing. Sonner toast is already imported in most files — the pattern exists, it's just not applied consistently to the top user-facing paths.

## Pattern

```typescript
// Before
} catch (err) {
  console.error("Failed to save:", err);
}

// After
} catch (err) {
  console.error("Failed to save:", err);
  toast.error("Failed to save");
}
```

Keep `console.error` for debugging. Add `toast.error` for user visibility. All toast messages use product vocabulary (ADR-0083) — no "entity", "intelligence", "enrichment", "signal", or "prep".

## Acceptance Criteria

1. All 20 locations above have a `toast.error()` call
2. Toast messages are user-friendly, no system terms (ADR-0083)
3. `toast` import from "sonner" added where missing
4. `console.error` retained for developer debugging
5. No layout disruption from toasts — sonner renders in fixed position
6. Remaining ~136 instances tracked in I611 (v1.0.1)

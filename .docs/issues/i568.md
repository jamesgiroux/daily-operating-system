# I568 — Migrate remaining `state.db.lock()` calls to async db_service

**Priority:** P1
**Area:** Backend / Performance
**Version:** v1.0.0 (Phase 6)
**Depends on:** None (db_service infrastructure already exists)
**Type:** Enhancement

## Problem Statement

Six direct `state.db.lock()` Mutex lock calls remain in Tauri command handlers. These acquire the same `Mutex<Option<ActionDb>>` that background tasks (intel_queue, email sync, embedding processor) hold during long operations. When a background enrichment holds the DB lock, IPC commands from the frontend block — the user sees a frozen UI until the lock is released.

The async `db_service` path (`state.db_read()` / `state.db_write()`) was built to solve this: it uses a separate reader connection pool and a dedicated writer, so reads never block writes and vice versa. All new command handlers use it, but these six were never migrated.

### Call sites

1. **`commands/projects_data.rs:174-178`** — `restore_database_from_backup()`: `state.db.lock()` to set `*db_guard = None` before file swap.
2. **`commands/projects_data.rs:184`** — same function, error recovery path: `state.db.lock()` to re-insert reopened DB.
3. **`commands/integrations.rs:1096`** — `enrich_account_from_clay()`: `state.db.lock().ok().and_then(...)` to query `account_stakeholders` for a linked person_id.
4. **`commands/integrations.rs:2470`** — `set_context_mode()`: `state.db.lock()` to enumerate entities for re-enrichment after mode switch.
5. **`commands/integrations.rs:2563`** — `start_glean_auth()`: `state.db.lock()` to enumerate entities for re-enrichment after Glean auth.
6. **`commands/app_support.rs:877,891`** — `delete_all_data()`: `state.db.lock()` to get DB path, then to set `*db_guard = None`.

## Solution

Migrate each call site to the async `db_service` path. The correct replacement depends on the operation:

### Read-only queries (call sites 3, 4, 5)

Replace `state.db.lock().ok().and_then(...)` with `state.db_read(|db| { ... }).await`. These are straightforward: the closure receives `&ActionDb` and runs on a dedicated OS thread, never blocking the Tokio runtime or contending with background writes.

**Call site 3** (`enrich_account_from_clay`):
```rust
// Before
let person_id: Option<String> = state.db.lock().ok().and_then(|g| { ... });

// After
let person_id: Option<String> = state.db_read(|db| {
    db.conn_ref()
        .query_row(
            "SELECT person_id FROM account_stakeholders WHERE account_id = ?1 LIMIT 1",
            [&account_id],
            |row| row.get(0),
        )
        .ok()
        .pipe(Ok)
}).await.ok().flatten();
```

**Call sites 4 and 5** (`set_context_mode`, `start_glean_auth`): The entity enumeration block can be extracted into a closure passed to `state.db_read()`. The `intel_queue.enqueue()` calls happen after the DB read returns, so the lock scope is already minimal — but moving to `db_read` eliminates Mutex contention entirely.

### Lifecycle operations (call sites 1, 2, 6)

These are special: they need to *close* or *replace* the DB handle itself. `db_read`/`db_write` can't do that — they operate on the connection, not the container.

**`restore_database_from_backup` (call sites 1-2):** Already drops `db_service` first (`state.db_service.write().await`). The `state.db.lock()` call closes the legacy sync handle. These must remain as direct Mutex locks because they manage the connection lifecycle, not data queries. However, add a comment documenting why: `// Lifecycle: closing legacy sync handle — not a data query, cannot use db_service`.

**`delete_all_data` (call site 6):** Same pattern — needs to read the DB path then close the handle. The path query can be migrated to `db_read`, but the handle close must remain a direct lock. Split into two steps:
1. `state.db_read(|db| db.conn_ref().query_row("PRAGMA database_list", ...).ok()).await` for the path.
2. Direct `state.db.lock()` only for `*guard = None` (handle close).

### Verification

After migration, `grep -rn 'state\.db\.lock()' src-tauri/src/commands/` should return only the lifecycle operations in `restore_database_from_backup` and `delete_all_data` (handle close/reopen), each annotated with a comment explaining why.

## Acceptance Criteria

1. `enrich_account_from_clay()` uses `state.db_read()` for the `account_stakeholders` query. No `state.db.lock()` in that function.
2. `set_context_mode()` uses `state.db_read()` for entity enumeration. No `state.db.lock()` in that function.
3. `start_glean_auth()` uses `state.db_read()` for entity enumeration. No `state.db.lock()` in that function.
4. `delete_all_data()` uses `state.db_read()` for the path query. Only the handle-close step uses direct lock, with a documenting comment.
5. `restore_database_from_backup()` retains direct lock for handle lifecycle only, with a documenting comment. No data queries through `state.db.lock()`.
6. Zero data-query `state.db.lock()` calls in `src-tauri/src/commands/`. Only annotated lifecycle operations remain.
7. `cargo test` — all pass.
8. `cargo clippy -- -D warnings` — clean.

## What This Does NOT Do

- Does not migrate `state.db.lock()` calls outside `commands/` (e.g., in `state.rs` itself, `scheduler.rs`, or background processors) — those are a separate concern.
- Does not change the `db_service` infrastructure or connection pool configuration.
- Does not remove the legacy `db` Mutex from `AppState` — it's still needed for lifecycle operations and the startup fallback path in `db_read`/`db_write`.

# I565 — Split heavy_work_semaphore into typed resource permits

**Version:** v1.0.0 Phase 6
**Depends on:** none
**Type:** Enhancement — Backend / Performance
**Scope:** Backend: `state.rs` + all semaphore consumers. No frontend changes.

---

## Problem

A single `Semaphore::new(1)` in `state.rs` (line 217, initialized at line 489) serializes all heavy work across the entire application:

| Consumer | File | Line | Duration | Trigger |
|----------|------|------|----------|---------|
| Intel queue processor | `intel_queue.rs` | 368 | 30–180s per cycle | Background (every 5s poll) |
| Embedding processor | `processor/embeddings.rs` | 155 | 5–30s per batch | Background |
| Email enrichment (PTY) | `google.rs` | 1185 | 10–60s | Background (email poll) |
| Manual refresh | `services/intelligence.rs` | 61 | 30–180s | User click |
| Project enrichment | `commands/projects_data.rs` | 165 | 30–120s | User click |
| Email refresh pipeline | `prepare/orchestrate.rs` | 1324 | 60–300s | Background (scheduler) |
| Hygiene sweep | `hygiene.rs` | 1633 | Variable | Background (try_acquire) |

When the intel queue is running a 120-second enrichment cycle, a user clicking "Refresh Intelligence" on an account detail page waits up to 120 seconds for the semaphore — with no feedback, no timeout, and no way to cancel. The same applies to email refresh blocking behind embeddings, or manual project enrichment blocking behind email enrichment.

The fundamental issue: these consumers contend for different physical resources. PTY subprocesses, ONNX embedding inference, and network I/O (email fetch) have no reason to serialize against each other. The semaphore was a reasonable first pass to prevent system overload, but it's now the primary source of user-perceived latency.

---

## Solution

### 1. Replace single semaphore with typed permits in `AppState`

Replace `heavy_work_semaphore: Arc<tokio::sync::Semaphore>` with a `ResourcePermits` struct:

```rust
pub struct ResourcePermits {
    /// PTY subprocess permit — serializes Claude Code invocations (only one PTY at a time
    /// to avoid CPU/memory contention from concurrent LLM processes).
    pub pty: Arc<tokio::sync::Semaphore>,
    /// Embedding inference permit — serializes ONNX model inference.
    pub embedding: Arc<tokio::sync::Semaphore>,
    /// Pipeline permit — serializes multi-stage orchestrated pipelines
    /// (refresh_emails, prepare_day) that include their own internal PTY calls.
    pub pipeline: Arc<tokio::sync::Semaphore>,
    /// User-initiated permit — separate lane for manual refresh so it never
    /// queues behind background hygiene. User work takes priority.
    pub user_initiated: Arc<tokio::sync::Semaphore>,
}

impl Default for ResourcePermits {
    fn default() -> Self {
        Self {
            pty: Arc::new(tokio::sync::Semaphore::new(1)),
            embedding: Arc::new(tokio::sync::Semaphore::new(1)),
            pipeline: Arc::new(tokio::sync::Semaphore::new(1)),
            user_initiated: Arc::new(tokio::sync::Semaphore::new(1)),
        }
    }
}
```

### 2. Migrate consumers to typed permits

| Consumer | Old | New | Rationale |
|----------|-----|-----|-----------|
| `intel_queue.rs` processor loop | `heavy_work_semaphore` | `permits.pty` | Serializes PTY calls, doesn't block user actions |
| `processor/embeddings.rs` | `heavy_work_semaphore` | `permits.embedding` | Embedding can run concurrent with PTY |
| `google.rs` email enrichment | `heavy_work_semaphore` | `permits.pty` | PTY call for email enrichment |
| `services/intelligence.rs` manual refresh | `heavy_work_semaphore` | `permits.user_initiated` | User clicks get own lane |
| `commands/projects_data.rs` manual enrich | `heavy_work_semaphore` | `permits.user_initiated` | User clicks get own lane |
| `prepare/orchestrate.rs` refresh_emails | `heavy_work_semaphore` | `permits.pipeline` | Full pipeline serialization (narrowed further in I567) |
| `hygiene.rs` sweep | `heavy_work_semaphore` | `permits.pty` (try_acquire) | Lowest priority, already uses try_acquire |

### 3. Backward-compatible deprecation

Keep `heavy_work_semaphore` as a deprecated alias during the transition:

```rust
#[deprecated(note = "Use state.permits.pty/embedding/pipeline/user_initiated instead")]
pub heavy_work_semaphore: Arc<tokio::sync::Semaphore>,
```

This avoids a big-bang migration — existing code compiles but produces warnings. Remove the alias once all consumers are migrated.

## Acceptance Criteria

1. `AppState` has a `permits: ResourcePermits` field with four typed semaphores: `pty`, `embedding`, `pipeline`, `user_initiated`.
2. `heavy_work_semaphore` is either removed or marked `#[deprecated]` with a compile warning.
3. Background intel queue enrichment (`permits.pty`) runs concurrently with embedding processing (`permits.embedding`) — verified by log timestamps showing overlapping operations.
4. User-initiated manual refresh (`permits.user_initiated`) starts within 2 seconds even when background intel queue holds `permits.pty`.
5. Background hygiene sweep still uses `try_acquire` on `permits.pty` — it does not block, just skips if PTY is busy.
6. `refresh_emails` uses `permits.pipeline` — it does not block manual refresh or embedding processing.
7. No deadlock: acquiring `permits.user_initiated` + `permits.pty` in sequence (manual refresh needs PTY after getting user permit) cannot deadlock against background acquiring `permits.pty` alone. Verify by code inspection that no consumer holds two permits simultaneously, or if they do, the acquisition order is consistent.
8. `cargo test` passes. `cargo clippy -- -D warnings` clean.

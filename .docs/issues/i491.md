# I491 — Portfolio Health Summary report type

**Priority:** P1
**Area:** Backend / Reports
**Version:** 1.0.0
**Depends on:** I499 (health scoring engine), I508 (intelligence schema), I489 (VP Account Review)

## Scope

**v1.1.0: Single-user portfolio.** The report synthesizes across all accounts in the current user's local DailyOS — the accounts they personally manage and have intelligence on. A VP with 20 accounts in their instance gets a portfolio view of those 20.

**v1.2.0+: Team portfolio.** With the hybrid shared intelligence layer (Google Drive + Remote DB), the report can aggregate across IC-managed accounts the VP doesn't personally own. This is an enhancement to the same report, not a separate feature. See `.docs/research/2026-03-01-portfolio-intelligence-architecture.md`.

The VP is also an IC. They have their own accounts, meetings, and intelligence. The single-user version is useful — it's the VP's personal book health. The team version enriches it with IC data flowing up from the shared layer.

## Problem

DailyOS produces per-account reports but has no cross-account synthesis. A VP of Key Accounts needs a single document that answers: "How is my book doing?" — total ARR under management, health distribution, which accounts need attention, renewal pipeline, and coverage gaps. Today this requires opening each account individually and mentally aggregating.

This is a user-scoped report (not entity-scoped) that gathers intelligence from ALL active accounts in the user's local database to produce a portfolio-level view.

## Design

### Report Type

Add `ReportType::PortfolioHealth` variant. DB string: `"portfolio_health"`. Entity scope: **user entity** (not a single account). This is the first report type scoped to the user rather than an individual account.

### Architecture Change: User-Scoped Reports

The existing report infrastructure assumes `entity_id` + `entity_type` = single account/project/person. Portfolio Health is scoped to the user entity. Implementation approach:

- `entity_id` = user entity ID (from `user_entity` table)
- `entity_type` = `"user"`
- The gather function queries ALL accounts (via `db.list_accounts()`) and aggregates their intelligence
- `compute_intel_hash()` for user-scoped reports hashes the combined state of all account intelligences

### Module Structure

New file `src-tauri/src/reports/portfolio_health.rs` following the two-phase pattern, but with a broader data gathering phase that touches all accounts.

### Output Schema (`PortfolioHealthContent`)

```rust
pub struct PortfolioHealthContent {
    pub book_snapshot: BookSnapshot,
    pub bucket_classification: BucketClassification,
    pub exceptions: Vec<ExceptionItem>,
    pub trend_summary: TrendSummary,
    pub renewal_pipeline: RenewalPipeline,
    pub coverage_gaps: Vec<CoverageGap>,
    pub portfolio_narrative: String,                 // 2 paragraphs
}

pub struct BookSnapshot {
    pub total_accounts: i64,
    pub total_arr: Option<String>,                   // formatted currency if available
    pub health_distribution: HealthDistribution,
    pub as_of_date: String,
}

pub struct HealthDistribution {
    pub healthy: i64,                                // score >= 70
    pub needs_attention: i64,                        // score 40-69
    pub at_risk: i64,                                // score < 40
    pub unscored: i64,                               // null health (should be near 0 after I499)
}

/// Strategic bucket classification following the Growth/Risk/Autopilot model.
/// Maps health score + engagement + expansion signals into actionable categories
/// that answer "where should I spend my time?" — not just "how healthy is this account?"
pub struct BucketClassification {
    pub growth_focus: Vec<BucketItem>,               // expanding, high engagement, active expansion signals
    pub at_risk_saveable: Vec<BucketItem>,            // declining but recoverable — champion engaged, relationship intact
    pub at_risk_save_unlikely: Vec<BucketItem>,       // declining and structural — champion gone, no exec sponsor, gone quiet
    pub autopilot: Vec<BucketItem>,                   // stable, healthy, low-touch — don't over-invest
}

pub struct BucketItem {
    pub account_name: String,
    pub bucket_rationale: String,                    // 1 sentence: why this bucket
    pub health_score: Option<i64>,
    pub arr: Option<String>,
}

pub struct ExceptionItem {
    pub account_name: String,
    pub reason: String,                              // 1 sentence: why this needs VP attention
    pub category: String,                            // "declining_health" | "at_risk_renewal" | "engagement_gap" | "blocker"
    pub urgency: String,                             // "immediate" | "this_week" | "this_month"
}

pub struct TrendSummary {
    pub improving: Vec<AccountTrend>,
    pub stable: Vec<AccountTrend>,
    pub declining: Vec<AccountTrend>,
}

pub struct AccountTrend {
    pub account_name: String,
    pub detail: String,                              // 1 sentence on the trend
}

pub struct RenewalPipeline {
    pub next_30_days: Vec<RenewalItem>,
    pub next_60_days: Vec<RenewalItem>,
    pub next_90_days: Vec<RenewalItem>,
}

pub struct RenewalItem {
    pub account_name: String,
    pub renewal_date: String,
    pub arr: Option<String>,
    pub readiness: String,                           // "ready" | "at_risk" | "critical" | "blocked" | "not_assessed"
}

pub struct CoverageGap {
    pub account_name: String,
    pub gap: String,                                 // "no executive sponsor" | "no meeting in 30+ days" | "single-threaded"
}
```

### Prompt Design

- **System role:** "You are a chief of staff preparing a portfolio health summary for a VP of Key Accounts."
- **Data aggregation:** The prompt receives a structured data block with per-account summaries (name, health score, health trend, renewal date, last meeting date, stakeholder count, key risks). This is assembled mechanically in the gather function — the LLM synthesizes narrative and identifies exceptions.
- **Bucket classification:** Explicit prompt instruction: "Classify every account into exactly one bucket: (1) **Growth Focus** — expanding, high engagement, active expansion signals; prioritize investment. (2) **At Risk but Saveable** — declining health but champion is engaged and relationship is recoverable; prioritize outreach. (3) **At Risk, Save Unlikely** — declining with structural problems (champion departed, no exec sponsor, gone quiet for 30+ days); escalate or accept churn. (4) **Autopilot** — stable and healthy, low-touch; don't over-invest. Provide a one-sentence rationale per account explaining the bucket assignment."
- **Exceptions filter:** Explicit prompt instruction: "The exceptions list should ONLY surface accounts with declining health, at-risk renewals (within 90 days with readiness != ready), or meeting gaps (no meeting in 30+ days). Do NOT list all accounts — only those needing VP attention."
- **Narrative specificity:** "The portfolio narrative must reference specific account names. Generic statements like 'most accounts are healthy' are not acceptable."
- **Scale handling:** Must work with 5 accounts and with 25 accounts. For large books, the prompt budget is managed by summarizing per-account data to key fields rather than sending full intelligence.

## Acceptance Criteria

1. Report gathers intelligence from ALL active accounts, produces single document
2. Contains: book snapshot (total accounts, ARR, health distribution), bucket classification (Growth Focus / At Risk but Saveable / At Risk Save Unlikely / Autopilot), exceptions list (accounts needing VP attention with one-line rationale), trend summary (improving/stable/declining with account names), renewal pipeline (30/60/90 day with readiness ratings), coverage gaps (no exec sponsor, no meeting in 30+ days), portfolio narrative (2-paragraph synthesis)
3. Every account is classified into exactly one bucket with a one-sentence rationale. Bucket assignment considers health score, engagement trend, expansion signals, champion status, and stakeholder coverage — not just health score alone
4. Exceptions list only surfaces accounts with declining health, at-risk renewals, or meeting gaps — NOT all accounts
5. Portfolio narrative references specific accounts by name — no generic "most accounts are healthy" statements
6. Works with 5 accounts and with 25 accounts

## Out of Scope

- Dedicated portfolio health page in the frontend (v1.1.1, I492)
- **Team-level portfolio aggregation** — IC intelligence flowing into VP's portfolio view requires the hybrid shared intelligence layer (Google Drive + Remote DB, v1.2.0+). In v1.1.0, this report only sees accounts in the current user's local database.
- Historical portfolio trend tracking (comparing this month's snapshot to last month's)
- Automatic portfolio report scheduling (e.g., weekly generation)
- ARR sourcing from CRM via Glean (uses manually entered vitals only)

## Relationship to Portfolio Architecture

This report has a natural upgrade path. In v1.1.0, it synthesizes the VP's personal book — accounts they have direct intelligence on. When the shared intelligence layer ships (v1.2.0+), the same `build_portfolio_health_prompt()` function widens its data gathering to include IC intelligence from the shared layer, without changing the output schema or prompt structure. The fractal insight holds: same synthesis pattern, wider scope.

# I643 — Background PTY burn reduction + AI control consolidation

**Area:** Backend / PTY + Intelligence Queue + Settings / System
**Priority:** P0
**Version:** v1.1.0
**Effort:** Large
**Depends on:** Existing PTY diagnostics foundation (I642), existing settings system
**Source:** PTY usage investigation (2026-03-29). Diagnostics showed 492 calls, 393,365 tokens, and 11,287.1s of PTY runtime in a single day, with background pollers and automatic enrichment consuming manual session capacity.

---

## Problem

DailyOS is burning PTY tokens in the background at a rate that blocks real work. The spike is not a single bug; it is a chain:

- Google calendar and email pollers ignore configured minute intervals and instead sleep on `adaptive_network_interval()`, which bottoms out around 30 seconds in background mode
- Poller tasks start on app boot even when Google is disabled, and their gating is too weak
- Automatic enrichment uses expensive Sonnet-heavy paths that are appropriate for manual work but too costly for background maintenance
- The intelligence queue can stack background requests and fan out multiple PTY calls per entity
- There is no rolling token or timeout guard that preserves manual capacity when background work becomes noisy
- Diagnostics know a PTY call happened, but not clearly enough which subsystem and operation are actually responsible

The result: background maintenance can consume a 4-hour token allowance before the user starts intentional work.

## Solution

Treat background AI as a constrained system with its own cadence, model tier, and safety rails.

### 1. Pollers obey configured cadence

- Calendar and email pollers compute `next_due_at` from `calendarPollIntervalMinutes` and `emailPollIntervalMinutes`
- `adaptive_network_interval()` may still delay a due poll indirectly via activity deferral, but it must never accelerate polling
- When the app is `Active`, a due poll defers by 60 seconds and re-checks
- When the app is `Idle` or `Background`, a due poll runs immediately
- Pollers stay dormant unless `google.enabled`, auth is valid, and a workspace exists
- Email's `/today` guard remains, but becomes a defer instead of a tight retry loop

### 2. Background AI has its own model tier

- Extend `AiModelConfig` with `background`
- Recommended routing bundle:
  - `synthesis = sonnet`
  - `extraction = sonnet`
  - `background = haiku`
  - `mechanical = haiku`
- Add `aiModelRoutingVersion = 2`
- On first startup after this change, reset legacy routing bundles to the recommended defaults and persist them
- Consolidate these controls into a visible `AI & Background Work` section in System settings

### 3. Automatic work gets a cheaper execution path

- `CalendarChange`, `ContentChange`, and `ProactiveHygiene` refreshes use a single-pass background enrichment path
- Background enrichment uses a 20-second timeout and no six-way Sonnet fan-out
- Automatic email poll enrichment becomes extraction-only on the background tier
- Manual refresh, onboarding, and workflow-triggered higher-quality work keep the richer paths
- The intelligence queue processes one entity per wake so bursts do not stack PTY calls back-to-back

### 4. Background work is budget-aware

- Persist a recent PTY call window in `app_state_kv` with the last 200 call summaries
- Every PTY call records structured metadata:
  - `subsystem`
  - `operation`
  - `trigger`
  - `tier`
  - `model`
  - `background`
  - `status`
  - `duration_ms`
  - `estimated_tokens`
- Pause background PTY work for 30 minutes when either condition is met:
  - rolling 4-hour background usage exceeds 50,000 estimated tokens
  - last 20 background PTY calls have timeout rate >= 25%, or there are 3 consecutive background timeouts
- Native sync continues while paused; only automatic PTY enrichment is skipped
- Manual actions bypass the pause

### 5. Trigger pressure is reduced at the source

- `CalendarChange` gets its own 10-minute per-entity debounce
- Existing debounce for `ContentChange` and `ProactiveHygiene` remains
- Diagnostics surface top operations, per-model counts, rolling 4-hour totals, timeout rate, and current pause reason

## Acceptance Criteria

1. Calendar and email pollers never run more frequently than their configured minute intervals.
2. Google pollers do not run when Google is disabled, auth is missing, or there is no workspace.
3. App activity may delay a due poll by 60 seconds, but never causes polling to happen earlier than configured cadence.
4. Automatic `CalendarChange`, `ContentChange`, and `ProactiveHygiene` refreshes use the background model tier and the cheap single-pass enrichment path.
5. Automatic email poll enrichment uses extraction-only background routing. Manual refresh and workflow-driven enrichment continue to use the richer path.
6. Intelligence queue processing handles one entity per wake and `CalendarChange` is debounced for 10 minutes per entity.
7. Legacy 3-tier model routing is reset once to the recommended 4-tier bundle, persisted to config, and then left alone on subsequent launches.
8. Background PTY work pauses for 30 minutes when rolling 4-hour usage exceeds 50,000 estimated background tokens.
9. Background PTY work also pauses when the last 20 background calls time out at >= 25%, or when there are 3 consecutive background timeouts.
10. PTY diagnostics expose structured per-operation attribution, per-model counts, rolling 4-hour totals, timeout rate, and pause state in Settings.
11. System settings shows a visible `AI & Background Work` section with model routing, calendar poll interval, email poll interval, pre-meeting refresh window, current background AI state, and a `Reset to Recommended` action.
12. Manual AI actions still work while background PTY work is paused.

## Implementation Notes

- This issue is intentionally separate from I642. I642 answers "how much AI are we spending and where?" I643 answers "how do we stop background AI from consuming the session budget?"
- The migration reset is deliberate. Hidden advanced overrides drifted away from sane defaults and are not trustworthy as a starting point.
- Background maintenance should bias toward preserving manual capacity, even if freshness drops.

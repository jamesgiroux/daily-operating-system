# I623 — Lifecycle Intelligence: AI-Native Detection + Briefing Reporting + Product Context

**Version:** v1.1.0
**Priority:** P0 (lifecycle engine), P1 (products, provenance, briefing fix), P2 (milestone auto-complete)
**Area:** Backend / Services + Frontend / Briefing + Account Detail
**Origin:** CEO plan review 2026-03-21, followed by product philosophy challenge. Original spec built lifecycle management UI (CRM thinking). Philosophy challenge reframed to AI-native: the system detects and acts, the briefing reports, the user confirms or corrects. "It should just know."

---

## Problem

A CSM had email signals confirming a Bring a Trailer renewal (signed order form) but no way to tell DailyOS "this renewal is done." The first instinct was lifecycle management UI — pipelines, breadcrumb bars, product registries with CRUD.

The philosophy challenge revealed this violates core DailyOS principles:
- **"AI produces; users curate"** — management UI makes users produce
- **"It should just know"** — proposing and waiting is not knowing
- **"Zero guilt"** — pending suggestions are guilt vectors
- **ADR-0084** — the account detail is a dossier, not a workbench

**The DailyOS answer:** The system should have detected the renewal from email signals, advanced the lifecycle automatically, and told the CSM in the daily briefing: "Bring a Trailer renewed. [Looks good] [Fix something]."

## Solution

Five issues (I623, I624, I625, I628, I629) building AI-native lifecycle intelligence. The system detects, acts, and reports. The user reads, confirms, or corrects.

## Sub-Issues

### I623 — Lifecycle Intelligence Engine (P0, Effort: L)

The core engine. Detects lifecycle transitions from signals and applies them automatically.

**Schema:**
- Add `renewal_stage TEXT` column to `accounts` table (nullable). Values: approaching, negotiating, contract_sent, signed, processed. NULL when lifecycle ≠ "renewing."

**Detection sources:**

| Source | What it detects | Confidence | Example |
|--------|----------------|------------|---------|
| Email signals | Renewal language, order form mentions, churn indicators | 0.7-0.9 | "Order form signed by both parties" |
| Glean CRM | Opportunity stage changes, contract status | 0.9 | Salesforce opp moved to Closed-Won |
| Calendar patterns | Meeting frequency drop, no contact in 30d | 0.5-0.7 | No meetings with account in 30 days |
| Time triggers | Proximity to contract_end | 1.0 | 120 days to renewal |
| Compound signals | Multiple sources agreeing | 0.95+ | Email + Glean both indicate renewal |

**Transition rules (automatic):**

| Detection | Current Lifecycle | Action | Confidence Required |
|-----------|-------------------|--------|-------------------|
| Renewal language + CRM closed-won | renewing | → active (renewed) | ≥ 0.8 (compound) |
| Churn language + no meetings 30d | active/renewing | → churned + auto-archive | ≥ 0.85 (compound) |
| Go-live detected | onboarding | → active | ≥ 0.7 |
| 120 days to contract_end | active | → renewing (stage: approaching) | 1.0 (time trigger) |
| Escalation language | active | emit at_risk signal (NOT auto-transition) | ≥ 0.7 |
| Meeting frequency drop | active | emit concern signal (NOT auto-transition) | ≥ 0.6 |

**Key principle:** High-confidence compound signals auto-transition. Single weak signals surface in the briefing as intelligence but do NOT change lifecycle state. at_risk requires deliberate escalation evidence, not just absence of activity.

**Briefing integration:**

Lifecycle changes appear in the daily briefing Attention section:

```
ATTENTION
─────────

LIFECYCLE
Bring a Trailer renewed — order form signed
(detected from email). Health: 72 → 78.
[✓ Looks good]  [✎ Fix something]

Acme Corp approaching renewal — 47 days to
contract end. Email engagement declining.
[✓ Noted]

ACTIONS
[existing overdue + today actions]

EMAILS
[existing AI-triaged emails]
```

- Max 2-3 lifecycle items in briefing
- [Looks good] = confirm, emit positive feedback signal
- [Fix something] = inline correction (same pattern as intelligence field edits), emit `user_correction` signal (confidence 1.0), Bayesian weights penalize wrong source
- [Noted] = acknowledge informational item (no state change involved)

**Account detail page (dossier):**

Current lifecycle state displayed as a label with provenance in the hero:

```
Bring a Trailer
───────────────────────────
Active  ·  $137K ARR  ·  Health: 78
┊ renewed Mar 15 · via email signal
```

No breadcrumb bar. No pipeline stages. No management UI. Current state + provenance + when it changed.

**ServiceLayer contract:**
```rust
// Automatic transition (called by signal processing pipeline)
fn apply_lifecycle_transition(account_id: &str, new_lifecycle: &str, new_stage: Option<&str>, source: &str, confidence: f64, evidence: &str) -> Result<()>;

// User correction from briefing [Fix something]
fn correct_lifecycle(account_id: &str, corrected_lifecycle: &str, notes: &str) -> Result<()>;

// User confirmation from briefing [Looks good]
fn confirm_lifecycle_change(account_id: &str, change_id: i64) -> Result<()>;
```

**Lifecycle change log:** New `lifecycle_changes` table to track what the system did and whether the user confirmed/corrected:

```sql
CREATE TABLE lifecycle_changes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL,
    previous_lifecycle TEXT,
    new_lifecycle TEXT NOT NULL,
    previous_stage TEXT,
    new_stage TEXT,
    source TEXT NOT NULL,        -- email_signal, glean_crm, time_trigger, compound, user_correction
    confidence REAL NOT NULL,
    evidence TEXT,               -- human-readable explanation
    user_response TEXT,          -- confirmed, corrected, pending
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### I624 — Product Intelligence (P1, Effort: M)

Products are intelligence, not inventory. AI discovers them. Users correct what's wrong.

**Schema:**
```sql
CREATE TABLE account_products (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL REFERENCES accounts(id),
    name TEXT NOT NULL,
    category TEXT,
    status TEXT DEFAULT 'active',  -- active | churned | trial
    arr_portion REAL,
    source TEXT NOT NULL,           -- glean | ai_inference | user_correction
    confidence REAL DEFAULT 0.7,
    notes TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_account_products_account ON account_products(account_id);
```

**Discovery sources:**
- **Glean MCP**: Query Salesforce for account products during enrichment. Source: "glean", confidence: 0.9.
- **AI inference**: Enrichment prompts and transcript processing detect product mentions. Source: "ai_inference", confidence: 0.5-0.7.
- **User correction**: CSM corrects a product inline. Source: "user_correction", confidence: 1.0.

**No CRUD UI.** The dossier shows products as read-only intelligence:

```
Products & Entitlements
───────────────────────
Parse.ly Analytics    Active
┊ via Salesforce

Coral Talk            Active
┊ via Salesforce

Arc XP                Active
┊ via Gong (mentioned in calls)
```

Wrong? Click to correct inline — same `EditableText` pattern as other intelligence fields. Correction emits `user_correction` signal.

**Intelligence Loop:**
- Products included in `build_intelligence_context()` — enrichment knows what products the account has
- Products included in `gather_account_context()` — meeting prep shows product context
- `product_data_updated` signal emitted on discovery/correction

## Architectural Decisions

1. **AI-native, not confirmable.** System detects and acts. Briefing reports. User confirms or corrects. No "propose and wait." (Philosophical challenge: "confirmable" = CRM thinking, violates "it should just know.")
2. **Confidence-gated transitions.** High-confidence compound signals auto-transition (≥ 0.8). Single weak signals surface as intelligence but don't change state. at_risk requires compound evidence. This prevents false positives while staying AI-native.
3. **Products as intelligence, not inventory.** No CRUD UI. Glean + AI inference discover products. User corrects inline. (Challenge: CRUD = inventory management = CRM = guilt vector.)
4. **Dossier stays a dossier.** No lifecycle breadcrumb bar. No pipeline visualization. Current state as a label with provenance. (ADR-0084: "the dossier, not the action queue.")
5. **Briefing as the action surface.** Lifecycle intelligence lives in the Attention section of the daily briefing. The briefing is where you confirm or correct. The dossier is where you read depth.

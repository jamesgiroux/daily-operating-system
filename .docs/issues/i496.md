# I496 — Stakeholder Map report type (absorbs I459)

**Priority:** P1
**Area:** Backend / Reports
**Version:** 1.0.0
**Depends on:** I504 (AI-inferred relationship extraction), I505 (Glean stakeholder writeback)

## Problem

CSMs and their managers need a structured view of stakeholder coverage for an account: which roles are filled, who is engaged, where the gaps are, and whether the account is single-threaded (only one active contact). Today, stakeholder data exists in `entity_people` and `person_relationships` (after I504), but there is no report that synthesizes this into a coverage assessment with actionable gaps.

This absorbs I459 (Stakeholder Map from the original v1.0.1 scope) with richer acceptance criteria because it now leverages AI-inferred relationships from I504.

## Design

### Report Type

Add `ReportType::StakeholderMap` variant. DB string: `"stakeholder_map"`. Entity scope: single account.

### Module Structure

New file `src-tauri/src/reports/stakeholder_map.rs` following the two-phase pattern.

### Output Schema (`StakeholderMapContent`)

```rust
pub struct StakeholderMapContent {
    pub coverage_grid: Vec<CoverageGridRow>,
    pub engagement_assessment: Vec<EngagementAssessment>,
    pub relationship_depth: RelationshipDepth,
    pub coverage_gaps: Vec<CoverageGapItem>,
    pub influence_network: Vec<InferredRelationship>,
    pub recommended_actions: Vec<StakeholderAction>,
}

pub struct CoverageGridRow {
    pub role: String,                                // from active preset stakeholder_roles
    pub person_name: Option<String>,                 // null if role is unassigned
    pub person_title: Option<String>,
    pub status: String,                              // "filled" | "vacant" | "inferred"
}

pub struct EngagementAssessment {
    pub person_name: String,
    pub role: String,
    pub engagement_level: String,                    // "active" | "warm" | "cold"
    pub last_meeting_date: Option<String>,
    pub meeting_count_90d: i64,
    pub email_signal_count: i64,
    pub notes: Option<String>,                       // 1 sentence on engagement quality
}

pub struct RelationshipDepth {
    pub champion_strength: ChampionStrength,
    pub executive_access: ExecutiveAccess,
    pub single_threaded_risk: bool,
    pub active_contact_count: i64,
    pub total_contact_count: i64,
    pub depth_summary: String,                       // 2 sentences
}

pub struct ChampionStrength {
    pub champion_name: Option<String>,
    pub strength: String,                            // "strong" | "moderate" | "weak" | "none"
    pub rationale: String,                           // 1 sentence
}

pub struct ExecutiveAccess {
    pub exec_name: Option<String>,
    pub access_level: String,                        // "direct" | "indirect" | "none"
    pub last_exec_meeting: Option<String>,
}

pub struct CoverageGapItem {
    pub role: String,                                // the missing role label
    pub impact: String,                              // "critical" | "important" | "nice_to_have"
    pub recommendation: String,                      // 1 sentence action
}

pub struct InferredRelationship {
    pub person_a: String,
    pub person_b: String,
    pub relationship_type: String,                   // "reports_to" | "collaborates_with" | "influences"
    pub confidence: f64,                             // 0.0-1.0
    pub source: String,                              // "ai_enrichment" | "user_confirmed"
}

pub struct StakeholderAction {
    pub action: String,                              // verb phrase
    pub target_person: Option<String>,               // who to engage
    pub target_role: Option<String>,                 // or what role to fill
    pub priority: String,                            // "high" | "medium"
}
```

### Prompt Design

- **System role:** "You are a relationship strategist preparing a stakeholder coverage analysis."
- **Coverage grid construction:** The grid is built mechanically in the gather function by cross-referencing the active preset's `stakeholder_roles` against `entity_people` roles. The LLM adds engagement assessment and depth analysis.
- **Preset-aware roles:** The coverage grid uses the active preset's `stakeholder_roles` labels. CS preset shows customer-side roles (Champion, Executive Sponsor, Technical Lead, Day-to-Day Contact). Sales preset shows prospect-side roles (Economic Buyer, Technical Evaluator, Coach). Agency preset shows client-side roles (Client Lead, Brand Manager, Approver).
- **Engagement thresholds:** Explicit prompt instruction for engagement levels:
  - `active`: meeting within last 14 days
  - `warm`: meeting within last 30 days
  - `cold`: no meeting in 30+ days or never met
- **Inferred relationships:** Pull from `person_relationships` where `source = 'ai_enrichment'` (from I504). Display with "suggested" confidence label when confidence < 0.8. User-confirmed relationships (confidence >= 0.8) display without qualifier.

## Acceptance Criteria

1. Coverage grid: stakeholder roles from active preset x actual people identified from entity_people
2. Engagement assessment per stakeholder: active (meeting in 14 days) / warm (meeting in 30 days) / cold (30+ days or never)
3. Relationship depth: champion strength, executive access, single-threaded risk
4. Coverage gaps visible: "No Executive Sponsor identified" when role is unassigned
5. Inferred relationships (from I504) appear in influence section with "suggested" label
6. Preset-aware: CS shows customer-side roles; Sales shows prospect-side roles; Agency shows client-side roles
7. No internal DailyOS terminology in output

## Out of Scope

- Cross-account stakeholder overlap detection (person appears in multiple accounts)
- Automatic stakeholder discovery from Glean org chart data (I486 writes person data, but discovery of new stakeholders is separate)
- Stakeholder map diffing (showing changes over time)

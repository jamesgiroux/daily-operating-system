# I609 — Retire sync DB path from AppState

**Area:** Code Quality / Backend
**Priority:** P2
**Version:** v1.0.1
**Source:** v1.0.0 GA engineering review (2026-03-16)
**Related:** I404 (AppState decomposition Phase 1), I568 (legacy db.lock() migration)

---

## Problem

`state.rs` carries two DB access mechanisms:

```rust
pub db: Mutex<Option<crate::db::ActionDb>>,              // legacy sync
pub db_service: tokio::sync::RwLock<Option<DbService>>,  // async (current)
```

I568 migrated all command handlers to `db_service`. Zero `state.db.lock()` calls remain in `commands/`. The sync `Mutex<Option<ActionDb>>` is dead weight — it adds a lock primitive that could contend with async paths and increases cognitive load when reasoning about DB access patterns.

## Scope

1. Audit all remaining `state.db` references outside `commands/` (~6 sites in background tasks: `intel_queue.rs`, `hygiene.rs`, `meeting_prep_queue.rs`, `scheduler.rs`, etc.)
2. Migrate each site to `db_service` or `ActionDb::open()` (the pattern used in `services/intelligence.rs:882`)
3. Remove `db: Mutex<Option<ActionDb>>` from `AppState`
4. Remove any `with_db()` or `db_read()` helpers that wrap the old sync path
5. Verify `cargo test` + `cargo clippy -- -D warnings` pass

## Acceptance Criteria

1. `grep -rn "state.db" src-tauri/src/ | grep -v "db_service"` returns zero results outside of test files
2. `AppState` struct has no `Mutex<Option<ActionDb>>` field
3. All 1,189+ tests pass
4. No behavioral change — all existing functionality works identically

## Out of Scope

- Full AppState decomposition (I404) — this is one targeted removal, not a struct redesign
- Changing the `ActionDb::open()` pattern in services — that pattern is correct and stays

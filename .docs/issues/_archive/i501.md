# I501 — Transcript Sentiment Extraction

**Priority:** P1
**Area:** Backend / Processor
**Version:** 1.1.0
**Depends on:** I503 (TranscriptSentiment type)
**ADR:** 0097

## Problem

The transcript processing prompt (`build_transcript_prompt()` in `src-tauri/src/processor/transcript.rs`) extracts SUMMARY, DISCUSSION, ANALYSIS, ACTIONS, WINS, RISKS, and DECISIONS from meeting transcripts. It does not extract any structured sentiment data.

Email enrichment already extracts structured sentiment (`positive | neutral | negative | mixed`) via `src-tauri/src/prepare/email_enrich.rs`. Meeting sentiment is arguably more valuable — a tense meeting is a stronger signal than a terse email — but currently the only sentiment signal from meetings is indirect: wins count as positive, risks count as negative, via the `transcript_outcomes` signal emitted on line 245 of `transcript.rs`.

ADR-0097 defines structured transcript sentiment fields that feed directly into the health scoring dimensions: customer sentiment into email engagement + signal momentum, engagement level into meeting cadence quality, forward-looking language into positive momentum, competitor mentions into risk signals, and champion presence/engagement into the champion health dimension.

## Design

### Prompt Extension

Add a `SENTIMENT` section to the transcript prompt in `build_transcript_prompt()`. This section is requested between DECISIONS and the end of the prompt:

```
DECISIONS:
- <explicit decision made, who decided, any conditions>
END_DECISIONS

SENTIMENT:
- overall: positive|neutral|negative|mixed
- customer: positive|neutral|negative|mixed
- engagement: high|moderate|low|disengaged
- forward_looking: yes|no
- competitor_mentions: [comma-separated list or "none"]
- champion_present: yes|no|unknown
- champion_engaged: yes|no|n/a
END_SENTIMENT
```

Add corresponding rules to the prompt:

```
Rules for sentiment:
- overall: The general tone of the entire meeting. "mixed" when positive and negative signals coexist.
- customer: The customer's tone specifically — ignore internal team sentiment. If no customer was present, use "n/a".
- engagement: How actively the customer participated. "high" = asking questions, making commitments. "disengaged" = short answers, camera off, multitasking cues.
- forward_looking: Did the conversation include future plans, next steps, roadmap discussion? "yes" if any substantive forward-looking language was used.
- competitor_mentions: List any competitor products or vendors mentioned by name. "none" if no competitors discussed.
- champion_present: Was the identified champion (primary internal advocate) in the meeting? "unknown" if champion identity is unclear.
- champion_engaged: If champion was present, were they actively contributing? "n/a" if champion not present or unknown.
```

### Parser Extension

In `src-tauri/src/processor/enrich.rs`, the `parse_enrichment_response()` function parses section markers. Add parsing for the `SENTIMENT:` / `END_SENTIMENT` block.

The parser extracts the key-value pairs between the markers:

```rust
fn parse_sentiment_block(text: &str) -> Option<TranscriptSentiment> {
    // Find SENTIMENT: ... END_SENTIMENT block
    // Parse each "- key: value" line
    // Validate enum values (positive|neutral|negative|mixed for sentiment fields)
    // Return None if block is missing or unparseable
}
```

Validation rules:
- `overall` and `customer`: must be one of `positive`, `neutral`, `negative`, `mixed`, or `n/a`
- `engagement`: must be one of `high`, `moderate`, `low`, `disengaged`
- `forward_looking`: must be `yes` or `no`
- `competitor_mentions`: split by comma, trim whitespace. `"none"` → empty vec.
- `champion_present`: must be `yes`, `no`, or `unknown`
- `champion_engaged`: must be `yes`, `no`, or `n/a`

### Storage

The parsed `TranscriptSentiment` is stored in two places:

1. **`captures` table** — A new capture type `"sentiment"` with the serialized JSON as `content`. This preserves the per-meeting sentiment alongside existing wins/risks/decisions captures. Stored via the existing `insert_capture()` function.

2. **Signal emission** — Emit a `transcript_sentiment` signal via `emit_signal()` with the sentiment data as the signal value JSON. This makes sentiment available to the health scoring engine (I499) through the same signal bus that other meeting signals use.

```rust
// After parsing sentiment
if let Some(sentiment) = &parsed_sentiment {
    // Store as capture
    if let Some(db) = db {
        let json = serde_json::to_string(sentiment).unwrap_or_default();
        let _ = db.insert_capture(
            &meeting.id,
            &meeting.title,
            meeting.account.as_deref(),
            "sentiment",
            &json,
        );
    }

    // Emit signal for health scoring
    if let Some(db) = db {
        let _ = crate::signals::bus::emit_signal(
            db,
            entity_type,
            entity_id,
            "transcript_sentiment",
            "transcript",
            Some(&serde_json::to_string(sentiment).unwrap_or_default()),
            0.8,  // higher confidence than generic transcript_outcomes
        );
    }
}
```

### TranscriptResult Extension

Add `sentiment: Option<TranscriptSentiment>` to the `TranscriptResult` struct in `src-tauri/src/types.rs` (or wherever it is defined — currently in `processor/transcript.rs` via the Default impl). This allows the frontend to access sentiment data for recently processed transcripts.

### Token Budget

The SENTIMENT section adds approximately 100 tokens to the prompt instructions and 50-80 tokens to the response. The current `TRANSCRIPT_MAX_CHARS` limit of 60,000 characters is not affected — the sentiment extraction happens from the same transcript content already being analyzed.

## Acceptance Criteria

1. `build_transcript_prompt()` output includes the `SENTIMENT:` section with rules for each field
2. Given a transcript AI response with a valid SENTIMENT block, `parse_sentiment_block()` returns a populated `TranscriptSentiment` with all 7 fields
3. Given a response missing the SENTIMENT block entirely, `parse_sentiment_block()` returns `None` (graceful degradation for transcripts processed before this change)
4. Invalid enum values are rejected: `"angry"` for overall sentiment returns `None` for that field, not a crash
5. A capture of type `"sentiment"` is stored in the DB for each transcript with valid sentiment data
6. A `transcript_sentiment` signal is emitted via the signal bus with confidence 0.8
7. `competitor_mentions: "Salesforce, HubSpot"` parses to `vec!["Salesforce", "HubSpot"]`; `"none"` parses to empty vec
8. Existing transcript tests in `transcript.rs` still pass (the new section is additive)
9. New unit tests cover: valid sentiment parsing, missing block, partial block, invalid enum values, competitor list parsing

## Out of Scope

- Using transcript sentiment in health dimension scoring (I499 consumes signals, but the mapping of `transcript_sentiment` signals to dimension scores is I499's responsibility)
- Meeting-level analytics (talk ratios, question density) — future enhancement per ADR-0097
- Retroactive sentiment extraction on already-processed transcripts
- Frontend rendering of per-meeting sentiment (I502 may surface this in meeting briefing, but the rendering is I502's scope)

## Original Spec (preserved for reference)

## Problem

`GleanContextProvider` fetches `GleanPersonResult` structs with structured fields: `name`, `email`, `title`, `department`, `manager`, `location`, `start_date`. These are gathered via `search_people()` in `glean.rs` (line 187) and stored in `GleanEntityData.people` (line 494).

Currently, this data is only used to build text blocks for the intelligence prompt context. The structured fields are formatted as strings and appended to `ctx.stakeholders` or `ctx.file_contents`. The data reaches the LLM as unstructured text but is never written to the `people` table.

The `people` table has columns for `role`, `organization`, `title_history`, and more. The `update_person_profile()` function (line 1007 in `db/people.rs`) handles source-priority-aware writes with provenance tracking. The `ProfileUpdate` struct (line 1123) has fields for `role`, `organization`, `bio`, `company_industry`, `company_size`, `company_hq`, and `title_history` — all of which Glean can populate.

The result: Glean's org graph data flows into AI prompts but never persists. A person's title and department from Glean are visible during enrichment but lost afterward. Subsequent operations that read from the `people` table see null fields.

## Design

### 1. Add "glean" as a source in `source_priority()`

In `db/people.rs`, the `source_priority()` function (line 1152) maps source names to priority levels:
- User (4) > Clay (3) > Gravatar (2) > AI (1)

Add `"glean" => 2` — same priority as Gravatar. Glean org graph data is authoritative for organizational fields (title, department, manager) but should not override user corrections or Clay's richer profile data.

```rust
pub fn source_priority(source: &str) -> u8 {
    match source {
        "user" => 4,
        "clay" => 3,
        "glean" | "gravatar" => 2,
        "ai" => 1,
        _ => 0,
    }
}
```

### 2. Write person data after Glean people search

In `glean.rs`, after the people search results are gathered in `gather_glean_context()` (line 446–477), the `GleanPersonResult` structs are available. Add a writeback step before returning `GleanEntityData`.

For each `GleanPersonResult` that has a non-empty email:

1. Look up the person by email: `db.get_person_by_email_or_alias(&email)`
2. If found, build a `ProfileUpdate` from the Glean fields
3. Call `db.update_person_profile(person_id, &profile_update, "glean")`

Map Glean fields to `ProfileUpdate` fields:
- `GleanPersonResult.title` → `ProfileUpdate.role` (Glean calls it "title" but it maps to the `role` column in `people`)
- `GleanPersonResult.department` → `ProfileUpdate.organization` (department is the closest structural match)
- `GleanPersonResult.manager` → Store in `ProfileUpdate.bio` as "Reports to: {manager}" if bio is empty (manager is a text field, not a FK — no person_relationships wiring here, that's I487's signal territory)
- `GleanPersonResult.location` → `ProfileUpdate.company_hq` (person's location, closest available field)
- `GleanPersonResult.start_date` → `ProfileUpdate.title_history` as JSON: `[{"title": "{title}", "start": "{start_date}", "source": "glean"}]` — only if `title_history` is currently null

```rust
for person_result in &people_results {
    if let Some(ref email) = person_result.email {
        if let Ok(Some(existing)) = db.get_person_by_email_or_alias(email) {
            let update = ProfileUpdate {
                role: person_result.title.clone(),
                organization: person_result.department.clone(),
                company_hq: person_result.location.clone(),
                ..Default::default()
            };
            let _ = db.update_person_profile(&existing.id, &update, "glean");
        }
    }
}
```

### 3. Emit signals per person update

After each successful `update_person_profile()` call that writes at least one field (`result.fields_updated` is non-empty), emit a signal:

```rust
if !result.fields_updated.is_empty() {
    let _ = crate::signals::bus::emit_signal(
        db,
        "person",
        &existing.id,
        "profile_enriched",
        "glean",
        Some(&result.fields_updated.join(",")),
        0.7,  // matches glean source weight in bus.rs
    );
}
```

Note: Use `emit_signal()` (not `emit_signal_and_propagate()`) here because the `gather_entity_context` call is already inside the enrichment pipeline. Propagation will run when the enrichment result is applied. Using `emit_signal_and_propagate()` inside `gather_entity_context()` could cause recursive enrichment triggers.

### 4. Thread DB access through `gather_glean_context()`

`gather_glean_context()` already takes `db: &ActionDb` as a parameter (line 365). No signature changes needed — `db` is available for both person lookups and profile writes.

## Acceptance Criteria

1. With Glean connected, enrich account. Linked people show role/department from Glean where previously null
2. `SELECT * FROM enrichment_sources WHERE source = 'glean'` has rows (via the `enrichment_sources` JSON column provenance tracking in `update_person_profile`)
3. Source priority: Glean = 2 (same as Gravatar). User (4) and Clay (3) overrides preserved
4. User-set role is NOT overwritten by Glean (source priority enforcement)
5. Signal emitted per person update via `emit_signal()` with source `"glean"`

## Out of Scope

- Writing Glean manager data to `person_relationships` — that requires resolving manager names to person IDs, which is I487's signal territory
- Adding new columns to `people` table for Glean-specific fields (department, location as distinct columns) — use existing columns via the mapping described above
- Writing back to Glean — DailyOS is a consumer, never a writer
- Glean person discovery (finding new people not yet in the `people` table) — that's a future issue

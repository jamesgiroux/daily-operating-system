# I090 — Product Telemetry + Analytics Infrastructure

**Status:** Open (Parking Lot)
**Priority:** P2
**Version:** Unscheduled
**Area:** Infrastructure

## Summary

DailyOS currently has no telemetry. There's no way to understand how users interact with the app, which features they use, which surfaces they ignore, or whether the intelligence outputs are actually being acted on. This issue covers building the analytics infrastructure — likely a lightweight local-first telemetry layer that respects the privacy-first architecture (data stays on the user's machine or is opt-in aggregated anonymously).

## Acceptance Criteria

Not yet specified. Scope includes: defining what events to track, choosing an analytics approach consistent with local-first architecture, instrumenting key user paths, and creating a summary view for the developer.

## Dependencies

- None blocking. Independent infrastructure work.
- Should be implemented before beta (I056, I057, I280) to collect meaningful usage data from early users.

## Notes / Rationale

Parked due to competing priorities. The privacy constraint (all data local by default) makes standard analytics platforms (Mixpanel, Amplitude) inappropriate without explicit opt-in. The telemetry design needs to fit the local-first model — potentially writing events to a local SQLite table and aggregating on request.

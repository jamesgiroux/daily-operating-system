## Original Spec (preserved for reference)

## Problem

`health_score` and `health_trend` are null for ~80% of accounts because the enrichment prompt explicitly instructs the LLM to return null for sparse accounts. Two prompt fragments in `intelligence/prompts.rs` enforce this gate:

1. Line 1567 — schema instruction: `"healthScore": "number 0-100. Overall account health score. Only populate when the account has 3+ signals and 2+ meetings. Return null for sparse accounts."`

2. Lines 1596–1597 — behavioral instruction: `"For healthScore, healthTrend, successMetrics, openCommitments, and relationshipDepth: return null/empty when the account has fewer than 2 signals. Do NOT hallucinate values for sparse accounts."`

The `HealthTrend` struct in `intelligence/io.rs` has only `direction` and `rationale` — no confidence indicator. This means consumers (reports, account detail) cannot distinguish a high-confidence health assessment from a best-guess on sparse data.

The result: reports that depend on health scores (VP Account Review, Renewal Readiness, Portfolio Health) blank out for most accounts, making them useless for portfolio-level surfaces.

## Design

### 1. Modify the prompt schema instruction

Replace the healthScore schema instruction at line 1567 to always require a score:

```
"healthScore": "number 0-100. Overall account health score. ALWAYS provide a score, even for sparse accounts. When data is limited, use conservative scoring (40-60 range) and explain confidence in healthTrend.rationale."
```

### 2. Modify the behavioral instruction

Replace lines 1596–1597 to remove the null gate and add confidence guidance:

```
"For healthScore and healthTrend: ALWAYS provide values. When the account has fewer than 2 signals, score conservatively (40-60), set healthTrend.confidence to a low value (0.1-0.3), and state the data limitation explicitly in healthTrend.rationale. Do NOT hallucinate specific events — instead describe what is unknown. For successMetrics, openCommitments, and relationshipDepth: return null/empty when insufficient data exists. valueDelivered should include completed commitments and wins."
```

### 3. Add `confidence` field to `HealthTrend`

In `intelligence/io.rs`, add a `confidence` field to the `HealthTrend` struct:

```rust
pub struct HealthTrend {
    pub direction: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub rationale: Option<String>,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub confidence: Option<f64>,  // 0.0–1.0, NEW
}
```

Use `Option<f64>` with `#[serde(default)]` for backward compatibility — existing intelligence.json files that lack this field will deserialize as `None` rather than failing.

### 4. Update healthTrend schema instruction

Add confidence to the healthTrend schema in the prompt (~line 1568):

```
"healthTrend": {"direction": "improving|stable|declining|volatile", "rationale": "1 sentence explaining trend direction", "confidence": "number 0.0-1.0. How confident you are in this assessment given available data."}
```

### 5. No migration needed

The `entity_intelligence` table stores intelligence as a JSON blob (`intelligence_json` column). Adding `confidence` to `HealthTrend` with `#[serde(default)]` is additive and backward-compatible. Existing rows deserialize cleanly — `confidence` will be `None` until re-enrichment.

## Acceptance Criteria

1. After re-enrichment, 95%+ of accounts have a non-null `health_score` in `entity_intelligence`
2. A sparse account (1 meeting, no email signals) has a health score with confidence qualifier in rationale — not null
3. `HealthTrend` struct has a `confidence` field (0.0–1.0) alongside existing `direction` and `rationale`
4. Account Health Review report renders without blanking out the health section for sparse accounts
5. Prompt change: "Always provide health_score (0-100) and health_trend. For sparse accounts, state confidence explicitly in the rationale."

## Out of Scope

- Changing how `successMetrics`, `openCommitments`, or `relationshipDepth` handle sparsity — those remain null when data is insufficient
- Frontend display of confidence badges or indicators — that belongs in a report or account detail UI issue
- Retroactive re-enrichment of all accounts — existing accounts get updated on their next enrichment cycle
- Changing the `health_score` column type in `entity_intelligence` — it already stores `f64`

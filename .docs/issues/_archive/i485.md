## Original Spec (preserved for reference)

## Problem

`extract_inferred_relationships()` in `intelligence/prompts.rs` (line 1929) parses AI-inferred person-to-person relationships from enrichment responses. It returns `Vec<InferredRelationship>` with `from_person_id`, `to_person_id`, and `relationship_type`. The function works — it validates relationship types against the `RelationshipType` enum, filters empty IDs, and handles malformed JSON gracefully.

But it is never called. In `intel_queue.rs` at line 804, there is an explicit deferral comment:

```rust
// NOTE: I391 inferred edge extraction deferred — requires name→ID resolution
// to produce valid person_relationships rows (FK constraint on to_person_id).
// The AI prompt still requests inferredRelationships for future use once
// co-attendee person rosters are included in the prompt context.
```

The enrichment prompt includes `canonical_contacts` (line 43 of `IntelligenceContext`) which provides person IDs to the LLM. The AI response contains `inferredRelationships` with valid person IDs from the canonical contacts list. The name-to-ID resolution problem cited in the deferral comment is already solved by the canonical contacts context — the LLM receives person IDs and returns them.

The result: the AI produces relationship intelligence on every enrichment cycle and it is silently discarded. The `person_relationships` table exists, `upsert_person_relationship()` works, but AI-inferred edges never get written.

## Design

### 1. Call `extract_inferred_relationships()` after enrichment

In `intel_queue.rs`, after the `parse_intelligence_response()` call at line 809 (single enrichment) and after the batch response parsing, call `extract_inferred_relationships()` on the raw PTY output to extract relationship edges.

The call site is in `run_single_enrichment()` — replace the deferral comment at line 804 with actual extraction and persistence:

```rust
// Extract and persist inferred relationships (I485)
let inferred = crate::intelligence::prompts::extract_inferred_relationships(&output.stdout);
```

### 2. Write relationships to `person_relationships` via `upsert_person_relationship()`

For each `InferredRelationship`, build an `UpsertRelationship` and call `db.upsert_person_relationship()`. The `UpsertRelationship` struct (in `db/person_relationships.rs` line 113) requires:

```rust
UpsertRelationship {
    id: &format!("pr-ai-{}-{}", rel.from_person_id, rel.to_person_id),
    from_person_id: &rel.from_person_id,
    to_person_id: &rel.to_person_id,
    relationship_type: &rel.relationship_type,
    direction: "bidirectional",
    confidence: 0.6,
    context_entity_id: Some(&input.entity_id),
    context_entity_type: Some(&input.entity_type),
    source: "ai_enrichment",
}
```

Key decisions:
- **ID format**: `pr-ai-{from}-{to}` — deterministic so re-enrichment upserts (reinforces) rather than duplicates
- **Confidence**: 0.6 — above the 0.5 "noise" threshold but below user-confirmed (0.8+)
- **Source**: `"ai_enrichment"` — distinct from `"user_confirmed"`, `"co_attendance"`, etc.
- **Direction**: `"bidirectional"` — peer/collaborator relationships are symmetric; manager edges the AI may specify directionally but the `direction` field handles that

### 3. Guard against overwriting user-confirmed relationships

The `upsert_person_relationship()` function uses `ON CONFLICT(id) DO UPDATE` which will overwrite. Since the ID is deterministic (`pr-ai-{from}-{to}`), a user-confirmed edge would have a different ID (e.g., `pr-{uuid}`). However, add an explicit check: before upserting, query for an existing relationship between the same two people with `source = 'user_confirmed'` and `confidence >= 0.8`. If found, skip the AI-inferred edge.

```rust
// Don't overwrite user-confirmed relationships
let existing = db.get_relationships_between(&rel.from_person_id, &rel.to_person_id)?;
if existing.iter().any(|r| r.source == "user_confirmed" && r.confidence >= 0.8) {
    continue;
}
```

This requires adding `get_relationships_between(from_id, to_id)` to `db/person_relationships.rs` — a simple SELECT query.

### 4. Emit signals for new relationships

After persisting each relationship, emit a signal via `emit_signal_and_propagate()`:

```rust
let _ = crate::signals::bus::emit_signal_and_propagate(
    &db,
    &propagation_engine,
    &input.entity_type,
    &input.entity_id,
    "relationship_inferred",
    "ai_enrichment",
    Some(&format!("{} -> {}: {}", rel.from_person_id, rel.to_person_id, rel.relationship_type)),
    0.6,
);
```

The signal type `"relationship_inferred"` is new. It should trigger prep invalidation for meetings involving these people.

### 5. Handle batch enrichment

The batch enrichment path (`run_batch_enrichment()`) calls `parse_intelligence_response()` per entity section. The raw response per entity is available in the `parse_batch_response()` output. Add `extract_inferred_relationships()` in the post-processing loop in `apply_enrichment_results()` (after line 1055 where `write_intelligence_json` is called).

Since `apply_enrichment_results()` already opens its own DB connection (line 1060), relationship writes can use the same connection.

## Acceptance Criteria

1. Enrich account with 3+ stakeholders. `SELECT * FROM person_relationships WHERE source = 'ai_enrichment'` has rows
2. Inferred relationships written via `db.upsert_person_relationship()` with source `"ai_enrichment"`, confidence 0.6
3. Signal emitted via `emit_signal_and_propagate()` per new relationship
4. Person detail shows inferred relationships in Network chapter at 0.6 confidence ("suggested")
5. User-confirmed relationships (confidence >= 0.8) are NOT overwritten by AI-inferred ones

## Out of Scope

- Improving the AI prompt to produce better relationships — the prompt already requests `inferredRelationships` and the extraction function handles the response format
- UI for confirming/rejecting inferred relationships — that is a person detail page feature
- Relationship type expansion (adding new types beyond the existing enum) — use current `RelationshipType` variants
- Propagation rules that derive new signals from relationship changes — those can be added later

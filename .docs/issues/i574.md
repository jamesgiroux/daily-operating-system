# I574 — Parallel Dimension Enrichment: Decompose ALL Monolithic AI Calls

**Version:** v1.0.0 Phase 5b
**Depends on:** I535 (Glean-first intelligence provider)
**Type:** Enhancement — Backend / Performance / Intelligence
**Scope:** `intel_queue.rs`, `services/intelligence.rs`, `reports/generator.rs`, `risk_briefing.rs`, `processor/transcript.rs`, `intelligence/glean_provider.rs`, `intelligence/glean_prompts.rs`, `prepare/email_enrich.rs`, `processor/email_actions.rs`, `processor/enrich.rs`, `executor.rs`, `google.rs`

---

## Problem

Every large AI call in DailyOS sends one massive prompt and waits for one massive response. This is slow, fragile, and produces worse output.

**Entity enrichment** (`intel_queue.rs::run_enrichment`) sends a single prompt requesting the entire `IntelligenceJson` schema — executive assessment, risks, wins, stakeholders, competitive context, champion health, success plan signals, and 15+ more fields — in one call. Timeout is 180s. Typical wall-clock: 30-180s depending on context size. If it times out, the entire enrichment is lost. If the LLM hallucinates one field, the entire response may fail JSON parsing. Batch mode (`run_batch_enrichment`) scales this linearly — 3 entities at 180s each = 540s timeout.

**Report generation** (`reports/generator.rs::run_report_generation`) uses a 300s timeout for a single PTY call. Account Health Reports, EBR/QBR, SWOT, Weekly Impact, and Monthly Wrapped all go through this path. A single timeout kills the entire report.

**Transcript processing** (`processor/transcript.rs::process_transcript`) sends the full transcript with all extraction targets (summary, wins, risks, decisions, actions, sentiment, champion health, role changes, commitments, interaction dynamics) in one 180s call. Summary and actions — the highest-value outputs — are held hostage until sentiment analysis and dynamics extraction complete.

**Risk briefing** (`risk_briefing.rs::run_risk_enrichment`) uses a 300s timeout for a single synthesis call.

**Email enrichment** (`prepare/email_enrich.rs`) processes emails sequentially with 60s timeouts per email. A batch of 20 emails takes 20+ minutes wall-clock.

**Glean enrichment** (`intelligence/glean_provider.rs::enrich_entity`) sends the full schema request in one Glean `chat` call. Glean is faster (10-30s), but still returns one massive JSON blob. Sparse responses wipe PTY-populated data (a problem I576 addresses, but parallelism is a prerequisite).

The monolithic pattern also produces worse output. LLMs perform better on focused, specific tasks than on "analyze everything about this account and return a 30-field JSON object." Decomposition improves both latency and quality.

## Solution

### Design Rule

No single AI call (PTY or Glean) exceeds 30s timeout. Every decomposed call writes its result independently. If one call fails, the others still succeed. Partial enrichment is always better than no enrichment.

### 1. Entity Enrichment — 6 Parallel Dimension Calls

Replace the monolithic `run_enrichment` call with 6 parallel, focused calls. Each call targets one dimension group and returns a subset of `IntelligenceJson` fields.

**Dimension groups:**

| Group | Fields | Prompt Focus | Timeout |
|-------|--------|-------------|---------|
| **Core Assessment** | `executiveAssessment`, `currentState`, `risks`, `recentWins` | "What is the current state of this account? What's working, what's not, what are the risks and wins?" | 30s |
| **Stakeholder & Champion** | `stakeholderInsights`, `championHealth`, `coverageAssessment`, `roleChanges`, `organizationalChanges`, `internalTeam` | "Who are the key people? Champion status? Coverage gaps? Org changes?" | 30s |
| **Commercial & Financial** | `health` (dimensions), `contractContext`, `renewalOutlook`, `expansionSignals`, `blockers` | "What is the financial health? Renewal risk? Expansion signals? Commercial blockers?" | 30s |
| **Strategic Context** | `companyContext`, `competitiveContext`, `strategicPriorities` | "What is the company context? Competitive landscape? Strategic alignment?" | 30s |
| **Value & Success** | `valueDelivered`, `successMetrics`, `successPlanSignals`, `openCommitments`, `commitments` | "What value has been delivered? What are the success criteria? Open commitments?" | 30s |
| **Engagement Signals** | `meetingCadence`, `emailResponsiveness`, `productAdoption`, `npsCsat`, `supportHealth`, `gongCallSummaries` | "What are the engagement patterns? Product adoption? Support health?" | 30s |

**Implementation:**

```rust
// New in intel_queue.rs
pub fn run_parallel_enrichment(
    input: &EnrichmentInput,
    ai_config: &AiModelConfig,
) -> Result<EnrichmentParseResult, String> {
    use std::thread;

    let dimensions = [
        ("core_assessment", build_core_prompt(input)),
        ("stakeholder", build_stakeholder_prompt(input)),
        ("commercial", build_commercial_prompt(input)),
        ("strategic", build_strategic_prompt(input)),
        ("value_success", build_value_prompt(input)),
        ("engagement", build_engagement_prompt(input)),
    ];

    let handles: Vec<_> = dimensions
        .into_iter()
        .map(|(name, prompt)| {
            let ws = input.workspace.clone();
            let ai = ai_config.clone();
            thread::spawn(move || {
                let pty = PtyManager::for_tier(ModelTier::Extraction, &ai)
                    .with_timeout(30)
                    .with_nice_priority(10);
                (name, pty.spawn_claude(&ws, &prompt))
            })
        })
        .collect();

    let mut merged = IntelligenceJson::default();
    let mut succeeded = 0u32;
    let mut failed_dimensions: Vec<&str> = Vec::new();

    for handle in handles {
        let (name, result) = handle.join().map_err(|_| "thread panic")?;
        match result {
            Ok(output) => {
                merge_dimension_into(&mut merged, name, &output.stdout)?;
                succeeded += 1;
            }
            Err(e) => {
                log::warn!("Dimension {} failed: {}", name, e);
                failed_dimensions.push(name);
            }
        }
    }

    if succeeded == 0 {
        return Err("All dimension calls failed".to_string());
    }

    // Log partial success
    if !failed_dimensions.is_empty() {
        log::warn!(
            "Partial enrichment for {}: {}/{} dimensions succeeded (failed: {:?})",
            input.entity_id, succeeded, 6, failed_dimensions
        );
    }

    Ok(EnrichmentParseResult { intel: merged, inferred_relationships: vec![] })
}
```

Each dimension prompt includes the same local context (meeting history, captures, email signals) but requests only its specific output fields. The prompt is ~40% shorter per call, improving LLM focus.

**Merge strategy:** Each dimension call returns a partial JSON object. `merge_dimension_into` parses the partial response and sets only the fields belonging to that dimension group. Fields from successful calls are never overwritten by failed calls.

**Batch enrichment removal:** `run_batch_enrichment` (call site #3) is removed. The parallel-per-dimension approach is inherently faster than batching multiple entities into one massive prompt. Multiple entities are processed by queuing each entity's 6 parallel dimension calls.

### 2. Glean Enrichment — 6 Parallel Chat Calls

Mirror the PTY decomposition for Glean. `GleanIntelligenceProvider::enrich_entity` splits into 6 parallel `GleanMcpClient::chat` calls, one per dimension group.

```rust
// New in intelligence/glean_provider.rs
pub async fn enrich_entity_parallel(
    &self,
    entity_id: &str,
    entity_type: &str,
    entity_name: &str,
    ctx: &IntelligenceContext,
    relationship: Option<&str>,
) -> Result<IntelligenceJson, String> {
    let prompts = build_dimension_prompts(entity_name, entity_type, relationship, ctx);

    let futures: Vec<_> = prompts
        .into_iter()
        .map(|(name, prompt)| {
            let client = GleanMcpClient::new(&self.endpoint);
            async move {
                let result = tokio::time::timeout(
                    std::time::Duration::from_secs(30),
                    client.chat(&prompt, None),
                ).await;
                (name, result)
            }
        })
        .collect();

    let results = futures::future::join_all(futures).await;
    // ... merge same as PTY path
}
```

New prompt builders in `intelligence/glean_prompts.rs`:
- `build_glean_core_prompt` — core assessment dimension
- `build_glean_stakeholder_prompt` — stakeholder & champion dimension
- `build_glean_commercial_prompt` — commercial & financial dimension
- `build_glean_strategic_prompt` — strategic context dimension
- `build_glean_value_prompt` — value & success dimension
- `build_glean_engagement_prompt` — engagement signals dimension

Each returns a focused prompt that requests only its dimension fields and includes the "ONLY valid JSON, no markdown" anchor.

### 3. Transcript Processing — Phased Extraction

Replace single 180s call with 3 sequential phases (each under 30s):

| Phase | Outputs | Timeout | Why Sequential |
|-------|---------|---------|----------------|
| **Phase 1: Core** | summary, actions, decisions | 30s | Highest-value, land first |
| **Phase 2: Signals** | wins, risks, sentiment, champion_health | 30s | Needs summary context from Phase 1 for coherence |
| **Phase 3: Dynamics** | interaction_dynamics, role_changes, commitments, engagement_signals, competitor_mentions, escalation_signals | 30s | Lowest urgency, can fail without losing core value |

Phase 1 writes to DB immediately. Meeting detail page shows summary and actions within 15s. Phases 2 and 3 update incrementally. If Phase 3 fails, the user still has summary + actions + risks — 80% of the value.

**Implementation:** Three calls to the same transcript content, each with a focused extraction prompt. The transcript text is included in all three prompts (it's the same source material), but each prompt requests different output sections.

### 4. Report Generation — Section-by-Section

Replace single 300s call with per-section calls. Each report type defines its sections:

| Report Type | Sections | Calls |
|-------------|----------|-------|
| Account Health | Executive Summary, Dimension Analysis, Risk Assessment, Recommendations | 4 |
| EBR/QBR | Period Summary, Health Trends, Value Delivered, Action Items, Priorities | 5 |
| SWOT | Strengths, Weaknesses, Opportunities, Threats, Strategic Recommendations | 5 |
| Weekly Impact | Week Summary, Key Activities, Outcomes, Next Week Focus | 4 |
| Monthly Wrapped | Month Overview, Top Wins, Risks Addressed, Metrics, Forward Look | 5 |

Each section call gets the full input context but requests only its section output. Sections write to DB as they complete (I575 handles the frontend streaming).

**Implementation:** Add a `sections` field to `ReportGeneratorInput`. `run_report_generation` becomes `run_report_section` with a 30s timeout. The caller orchestrates parallel section calls and assembles the final report.

### 5. Risk Briefing — Category Decomposition

Replace single 300s call with parallel category calls:

| Call | Focus | Timeout |
|------|-------|---------|
| Risk Assessment | Current risks with urgency tiers | 30s |
| Opportunity Analysis | Growth signals, expansion paths | 30s |
| Action Recommendations | Concrete next steps with owners | 30s |

### 6. Email Enrichment — Parallel Batch

Replace sequential per-email processing in `prepare/email_enrich.rs` with parallel batches:

- Process up to 5 emails concurrently (5 threads, each with 30s timeout)
- Each email still gets its own PTY call (Extraction tier)
- Total wall-clock for 20 emails: ~120s (4 batches of 5) instead of ~1200s sequential

### 7. Prompt Builder Module

New module: `intelligence/dimension_prompts.rs`

Contains:
- `build_core_prompt(input: &EnrichmentInput) -> String`
- `build_stakeholder_prompt(input: &EnrichmentInput) -> String`
- `build_commercial_prompt(input: &EnrichmentInput) -> String`
- `build_strategic_prompt(input: &EnrichmentInput) -> String`
- `build_value_prompt(input: &EnrichmentInput) -> String`
- `build_engagement_prompt(input: &EnrichmentInput) -> String`
- `merge_dimension_into(target: &mut IntelligenceJson, dimension: &str, response: &str) -> Result<(), String>`

Each prompt builder includes the shared local context block (meeting history, captures, email signals, stakeholders, user context) but requests only its dimension's output fields. The JSON schema in each prompt is ~100 lines instead of ~300.

### 8. Failure Handling

- If 1-5 dimensions succeed: write the partial result. Mark failed dimensions with `"could not be determined"` in their narrative fields.
- If 0 dimensions succeed: fall back to the monolithic call (legacy path preserved as `run_enrichment_legacy`). This is the safety net for the first release.
- Circuit breaker: if an entity fails enrichment 3 times in a row, exponential backoff (existing `MAX_VALIDATION_RETRIES` mechanism extended).
- Each dimension failure emits a `dimension_enrichment_failed` signal for observability.

## Acceptance Criteria

1. No single AI call (PTY or Glean) has a timeout exceeding 30s. All 7 decomposed call sites use 30s timeouts.
2. Entity enrichment decomposes into 6 parallel dimension calls. All 6 run concurrently (verified via log timestamps showing overlapping execution windows).
3. Total entity enrichment wall-clock is < 45s for PTY, < 20s for Glean (down from 60-180s and 10-30s respectively). Measured on a real account with full context.
4. If 1 dimension call fails, the other 5 still write their results. The failed dimension shows "could not be determined" in its narrative fields. No data loss from successful dimensions.
5. If all 6 dimension calls fail, the system falls back to the legacy monolithic call (180s timeout). Fallback produces the same output as pre-I574 behavior.
6. Transcript processing Phase 1 (summary + actions) writes to DB within 15s of transcript submission. Verified by checking `captures` table timestamps. Phase 2 (wins + risks + sentiment + champion health) follows. Phase 3 (dynamics + commitments) follows. Each phase writes independently.
7. If transcript Phase 3 fails, Phase 1 and Phase 2 data are still present in the database. Meeting detail page shows summary, actions, risks, and wins.
8. Report generation decomposes by section. Each section writes to the report record as it completes. A 5-section report takes < 60s wall-clock (down from 300s).
9. Email enrichment processes up to 5 emails concurrently. A batch of 20 emails completes in < 120s wall-clock (down from 1200s sequential).
10. `run_batch_enrichment` (multi-entity single-prompt batching) is removed. Multiple entities are processed as separate parallel-dimension jobs via the queue.
11. Glean enrichment uses 6 parallel `chat` calls matching the PTY dimension groups. Each Glean call has a 30s timeout. Total Glean enrichment wall-clock < 20s.
12. Each dimension prompt includes the full local context block (meeting history, captures, email signals) but requests only its dimension's output fields. Prompt length per dimension is < 50% of the monolithic prompt length.
13. Dimension merge logic is deterministic: fields from dimension A never overwrite fields from dimension B. Each dimension owns a disjoint set of `IntelligenceJson` fields.
14. Risk briefing decomposes into 3 parallel calls (risk assessment, opportunity analysis, action recommendations). Total wall-clock < 45s (down from 300s).
15. Run enrichment on 3 real accounts (1 with Glean, 2 with PTY). Verify: parallel execution, partial failure resilience, output quality equivalent to monolithic path. Compare field-by-field against monolithic output for the same accounts.
16. All existing tests pass. No regression in enrichment output schema or downstream consumers (health scoring, meeting prep, callouts).

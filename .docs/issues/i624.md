# I624 — Product intelligence: AI-discovered products from Glean + transcript/email inference

**Area:** Backend / DB + Intelligence + Frontend / Account Detail
**Priority:** P1
**Version:** v1.1.0
**Effort:** Medium
**Depends on:** Glean MCP integration (existing), enrichment pipeline (existing)
**Source:** CEO plan review + philosophy challenge (2026-03-21). Original spec was product registry with CRUD UI. Reframed to AI-only discovery after challenge against "AI produces; users curate."

---

## Problem

CSMs can't see what products an account has. There's no product data anywhere in DailyOS — not from CRM, not from AI inference, not from manual entry. When preparing for a meeting or reviewing an account, the CSM has to check Salesforce separately for product context.

## Design Principle

**Products are intelligence, not inventory.** No add/edit/remove UI. The system discovers products from Glean (Salesforce) and infers them from transcripts/emails. The CSM corrects what's wrong inline — same as any other intelligence field. This avoids creating a maintenance obligation (violates zero-guilt) and keeps the account detail page as a dossier (ADR-0084), not a workbench.

## Solution

### Schema

New `account_products` table:

```sql
CREATE TABLE account_products (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    account_id TEXT NOT NULL REFERENCES accounts(id),
    name TEXT NOT NULL,
    category TEXT,
    status TEXT DEFAULT 'active',       -- active | churned | trial
    arr_portion REAL,
    source TEXT NOT NULL,                -- glean | ai_inference | user_correction
    confidence REAL DEFAULT 0.7,
    notes TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_account_products_account ON account_products(account_id);
```

### Discovery Sources

1. **Glean MCP (primary):** During enrichment, if Glean is connected, query: "What products or SKUs does {account_name} have in Salesforce? Return as a structured list with product name, status, and contract value if available." Parse structured response. Upsert with `source = 'glean'`, `confidence = 0.9`.

2. **AI inference (secondary):** During enrichment and transcript processing, detect product mentions in context. Example: "they asked about upgrading their analytics tier" → infer product "Analytics" with `source = 'ai_inference'`, `confidence = 0.5-0.7`. Lower confidence means it appears on the dossier but with a softer provenance label.

3. **User correction:** CSM clicks on a product name to correct it (same `EditableText` inline pattern used elsewhere). Correction sets `source = 'user_correction'`, `confidence = 1.0`, emits `user_correction` signal.

### Merge Logic

- Glean can ADD products and UPDATE Glean-sourced products
- Glean CANNOT delete or overwrite `user_correction`-sourced products
- AI inference can ADD products but at lower confidence
- Name-based matching (case-insensitive) determines duplicates
- On conflict: highest source priority wins (User > Glean > AI per ADR-0098)

### Intelligence Loop Integration

- Products included in `build_intelligence_context()` — enrichment prompts know what products the account has
- Products included in `gather_account_context()` — meeting prep shows product context
- `product_data_updated` signal emitted on discovery/correction
- Product context appears in health scoring narrative (not scoring dimensions yet — future version)

### Frontend

"Products & Entitlements" section on account detail dossier. Read-only list:

```
Products & Entitlements
───────────────────────
Parse.ly Analytics    Active
┊ via Salesforce

Coral Talk            Active
┊ via Salesforce

Arc XP                Active
┊ via Gong (mentioned in calls)
```

- No add button. No edit button. No remove button.
- Product name is inline-editable (click to correct) — same pattern as intelligence field corrections
- Provenance label uses I625 provenance markers
- If no products discovered: section shows "No products detected yet" (not hidden)

### ServiceLayer Contract

```rust
// In services/accounts.rs
fn upsert_account_product(account_id: &str, name: &str, category: Option<&str>, status: &str, arr_portion: Option<f64>, source: &str, confidence: f64) -> Result<()>;
fn correct_account_product(product_id: i64, corrected_name: &str, corrected_status: Option<&str>, notes: Option<&str>) -> Result<()>;
fn get_account_products(account_id: &str) -> Result<Vec<AccountProduct>>;
fn remove_account_product(product_id: i64) -> Result<()>; // Only for user_correction source — CSM can dismiss wrong AI inferences
```

## Acceptance Criteria

1. New `account_products` table created via migration.
2. During enrichment with Glean connected, products queried from Salesforce via MCP chat. Products appear on the account dossier with "via Salesforce" provenance.
3. During enrichment/transcript processing, AI-inferred product mentions appear with "mentioned in calls" or "inferred from email" provenance at lower confidence.
4. No add/edit/remove buttons visible on the product section. Only inline correction (click product name to fix).
5. CSM corrects a product name → source changes to "user_correction", `user_correction` signal emitted, Bayesian weights adjusted.
6. CSM can dismiss a wrong AI inference (small × to remove) — but cannot dismiss Glean-sourced products (CRM is authoritative for existence, CSM corrects details).
7. Product data included in `build_intelligence_context()` and `gather_account_context()`.
8. `product_data_updated` signal emitted on all product changes.
9. Without Glean: product section shows "No products detected yet." If AI infers products from transcripts, those appear even without Glean.
10. TypeScript type `AccountProduct` added to `src/types/index.ts`.

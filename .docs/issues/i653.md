# I653 — Meeting→Account Entity Linking Pipeline — Centralized Service + Gap Fixes

**Area:** Backend / Entity Resolution / Meeting Prep
**Priority:** P0
**Version:** TBD
**Effort:** Medium
**Status:** Spec finalized 2026-03-31
**Source:** Production observation (2026-03-31). CloudCo meeting with account name in title and known attendees linked to the CloudCo account does not generate a briefing. Classification correctly identifies the account (slug match at 0.50 confidence), but the entity link never gets persisted to `meeting_entities`, so the prep queue (`INNER JOIN meeting_entities`) skips it entirely.

---

## Problem

Meeting→account entity linking has three independent gaps that compound into silent briefing failures:

1. **`prepare_today` doesn't persist entity links.** `prepare_week` calls `db.link_meeting_entity()` for each resolved entity (orchestrate.rs:1139-1154). `prepare_today` upserts the meeting to DB but skips entity linking entirely (orchestrate.rs:639-726). Today's meetings never get `meeting_entities` rows from the daily pipeline.

2. **Calendar poller doesn't persist classification entities.** `populate_people_from_events` (google.rs:637+) calls `ensure_meeting_in_history` and links people→accounts via `link_person_to_entity`, but never writes to the `meeting_entities` junction table. Classification results are converted to `CalendarEvent` which only stores `account: Option<String>` (a name, not an entity ID) — entity IDs are lost in the `to_calendar_event()` conversion.

3. **Mechanical prep path has no attendee data.** `generate_mechanical_prep_with_inputs` (meeting_prep_queue.rs:496-502) builds classified JSON without attendees, so the entity resolver's `signal_attendee_inference` and `signal_domain_match` signals never fire during prep generation.

### Secondary issues

- **Name-vs-ID comparison bug:** `populate_people_from_events` (google.rs:728) compares `event.account` (a name string like "CloudCo") against entity IDs from `get_meeting_entities` (UUIDs). Type mismatch means reclassification detection never triggers.

- **No person→account chaining at classification time:** Person entity hints carry emails but not linked account IDs. When a known stakeholder attends a meeting, their account doesn't get a confidence boost during `classify_meeting_multi`. This signal only exists in the entity resolver (which requires DB access and runs later), not at the earlier classification step that determines `resolved_entities`.

### Impact

Any meeting that relies on classification-time entity resolution (title matching, domain matching, attendee inference) for its `meeting_entities` rows will fail to generate a briefing. This affects:
- Meetings where the account name appears in the title
- Meetings with attendees from known account domains
- Meetings with attendees who are linked stakeholders

Only meetings that already have manual entity links (user-assigned via UI) or links from `prepare_week` (which only runs on Monday) generate briefings reliably.

---

## Design Direction

### Centralized service function

Create a single `persist_classification_entities` function in the service layer that all paths call. Takes Rust types directly — no JSON intermediary.

```rust
// services/meetings.rs
pub fn persist_classification_entities(
    db: &ActionDb,
    meeting_id: &str,
    entities: &[(String, String)],  // (entity_id, entity_type)
) -> Result<usize, String>
```

- Calls `db.link_meeting_entity()` (INSERT OR IGNORE) for each entity
- Additive: does not remove existing links (preserves manual user links)
- Returns count of newly linked entities
- No signal emission — classification is detection, not user mutation

### Carry entity IDs through CalendarEvent

Add `classified_entities: Vec<(String, String)>` to `CalendarEvent` (types.rs), populated during `to_calendar_event()` from `resolved_entities`. This eliminates the account-name-only bottleneck and enables the calendar poller to persist entity links.

### Pass attendees through mechanical prep

Add `"attendees"` field to the classified JSON in `generate_mechanical_prep_with_inputs` from the DB meeting record. Enables entity resolver's attendee inference during prep generation.

## Acceptance Criteria

### AC1: `prepare_today` persists entity links
After `prepare_today` runs, meetings classified with resolved entities have corresponding rows in `meeting_entities`. Verify: query `meeting_entities` for a meeting whose title matches a known account — row exists.

### AC2: Calendar poller persists entity links
After calendar poll sync, new meetings classified with resolved entities have corresponding rows in `meeting_entities`. Verify: add a new meeting with an account name in the title, wait for poll, query `meeting_entities` — row exists.

### AC3: Mechanical prep includes attendees
`generate_mechanical_prep_with_inputs` passes attendee data to `gather_meeting_context_single`, enabling `signal_attendee_inference` and `signal_domain_match`. Verify: meeting with known attendees (but no title match) resolves to the correct account during prep generation.

### AC4: Single service function for all paths
Both `prepare_today`, `prepare_week`, and `populate_people_from_events` call `persist_classification_entities`. No inline `meeting_entities` INSERT statements outside the DB layer and test code.

### AC5: Name-vs-ID comparison fixed
Reclassification detection in `populate_people_from_events` compares entity IDs (not name strings) from classification against existing `meeting_entities` rows.

### AC6: Person→account chaining at classification time
When a known stakeholder (person linked to account via `account_stakeholders`) attends a meeting, their linked account gets a confidence boost in `resolve_entities` during classification. Verify: meeting with 3 attendees who are CloudCo stakeholders resolves to CloudCo even without title match.

### AC7: CloudCo meeting generates briefing
End-to-end: the specific CloudCo meeting (title contains "CloudCo", attendees linked to CloudCo account) has a `meeting_entities` row, prep is generated, and briefing renders on the frontend.

## Intelligence Loop Check

1. **Signal emission?** Entity resolution already emits `entity_resolution` signals via the entity resolver. The new persistence function is additive (INSERT OR IGNORE), not a user mutation — no new signals needed.
2. **Health scoring?** No new dimensions. Existing meeting→account links feed health scoring correctly once persisted.
3. **Intel context?** Yes — `gather_account_context()` already reads `meeting_entities` to find related meetings. Fixing persistence means accounts get proper meeting history.
4. **Briefing callouts?** Yes — `CALLOUT_SIGNAL_TYPES` includes entity resolution signals already emitted.
5. **Bayesian feedback?** No — entity resolution confidence is already tracked. User corrections to entity links already flow through `link_meeting_entity_with_prep_queue`.

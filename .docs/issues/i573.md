# I573 — Mutex poisoning recovery for critical state

**Priority:** P2
**Area:** Backend / Stability
**Version:** v1.0.0 (Phase 6)
**Depends on:** None
**Type:** Enhancement

## Problem Statement

When a thread panics while holding a `std::sync::Mutex`, the Mutex becomes "poisoned" — all subsequent `lock()` calls return `Err(PoisonError)`. In DailyOS, this means a single thread panic permanently disables core resources for the remainder of the app session. The user must force-quit and relaunch.

### Critical Mutexes in AppState (`state.rs`)

| Field | Type | Line | Current poison handling |
|-------|------|------|------------------------|
| `db` | `Mutex<Option<ActionDb>>` | ~470 | `map_err(\|_\| "DB lock poisoned")` — returns error string |
| `audit_log` | `Arc<Mutex<AuditLogger>>` | 209 | `if let Ok(mut audit) = state.audit_log.lock()` — silently skips on poison |
| `database_recovery_status` | `Mutex<DatabaseRecoveryStatus>` | 207 | `if let Ok(mut guard) = ...` — silently skips |
| `last_activity` | `Mutex<Instant>` | 198 | `if let Ok(mut last) = ...` — silently skips |
| `last_failed_unlock` | `Mutex<Option<Instant>>` | 200 | `if let Ok(mut guard) = ...` — silently skips |

Additionally, two `RwLock` fields use `expect()` which will **panic** on poison, cascading the failure:

| Field | Type | Line | Current poison handling |
|-------|------|------|------------------------|
| `context_provider` | `RwLock<Arc<dyn ContextProvider>>` | 221, 496, 501 | `.expect("context_provider lock poisoned")` — **panics** |

### Failure scenarios

1. **DB Mutex poison**: A thread panics during a DB operation (e.g., corrupted query result triggers an unwrap). Every subsequent IPC command that uses `with_db_read`, `with_db_write`, or `db_read`/`db_write` (fallback path at lines 750, 800) returns "DB lock poisoned". The app is completely non-functional.

2. **Audit log Mutex poison**: All subsequent audit writes silently fail. The hash chain breaks (missing records). Not immediately user-visible but violates the tamper-evident guarantee (ADR-0094).

3. **Context provider RwLock poison**: `context_provider()` at line 496 calls `.expect()` — a poisoned RwLock causes a **second panic**, potentially poisoning more Mutexes on the way down. This is the worst case: a cascade of panics.

## Solution

### Strategy: Recover through the poison, don't just report it

`PoisonError` wraps the `MutexGuard` that the panicking thread was holding. You can recover the inner data via `into_inner()`. The data may be in an inconsistent state, but for most of these resources, re-creation is safe:

### DB Mutex recovery

Replace error-returning poison handling with recovery:

```rust
pub fn with_db_read<T, F>(&self, f: F) -> Result<T, String>
where
    F: FnOnce(&crate::db::ActionDb) -> Result<T, String>,
{
    let guard = self.db.lock().unwrap_or_else(|poisoned| {
        log::error!("DB Mutex was poisoned — recovering inner value");
        poisoned.into_inner()
    });
    let db = guard.as_ref().ok_or_else(|| "Database unavailable".to_string())?;
    f(db)
}
```

The `ActionDb` wraps a `rusqlite::Connection`. If the connection itself is in a bad state (e.g., mid-transaction), the next operation will return a SQLite error — which is a normal, handleable error, not a permanent lockout.

For `db_read`/`db_write` fallback paths (lines 750, 800), apply the same pattern.

### Audit log Mutex recovery

The audit logger's `AuditLogger` struct contains a `PathBuf` and a `last_hash: Option<String>`. If poisoned, recover the inner value. If the inner state is suspect, re-create from the file:

```rust
fn get_audit_logger(&self) -> std::sync::MutexGuard<'_, AuditLogger> {
    self.audit_log.lock().unwrap_or_else(|poisoned| {
        log::error!("Audit log Mutex poisoned — recovering");
        let mut guard = poisoned.into_inner();
        // Re-read last hash from file to ensure chain continuity
        *guard = AuditLogger::new(guard.path().to_path_buf());
        guard
    })
}
```

### Context provider RwLock recovery

Replace `expect()` with `unwrap_or_else` that recovers:

```rust
pub fn context_provider(&self) -> Arc<dyn ContextProvider> {
    match self.context_provider.read() {
        Ok(guard) => Arc::clone(&guard),
        Err(poisoned) => {
            log::error!("Context provider RwLock poisoned — recovering");
            Arc::clone(poisoned.get_ref())
        }
    }
}
```

The `Arc<dyn ContextProvider>` inside is immutable — it's safe to clone it out of a poisoned lock. Same for `swap_context_provider` with `write()`.

### Other Mutexes

- `last_activity`, `last_failed_unlock`, `database_recovery_status`: These already use `if let Ok(guard) = ...` which silently skips on poison. Convert to `unwrap_or_else(|p| p.into_inner())` to recover instead of skip. These hold simple values (`Instant`, `Option<Instant>`, `DatabaseRecoveryStatus`) that are always safe to recover.

### Logging

Every poison recovery MUST log at `error!` level with the field name. This ensures the panic that caused the poison is visible in logs even if the app continues operating.

## Acceptance Criteria

1. After a simulated DB Mutex poison (test helper that panics while holding the lock), subsequent `with_db_read()` and `with_db_write()` calls succeed — they recover the inner value and proceed.
2. After a simulated audit log Mutex poison, subsequent `audit_log.lock()` calls succeed and the audit logger is re-initialized from the file (hash chain continues correctly).
3. After a simulated context provider RwLock poison, `context_provider()` returns a valid provider — no panic, no cascade.
4. Every poison recovery emits an `error!`-level log message identifying which Mutex was poisoned.
5. No `expect("...poisoned")` calls remain on any Mutex or RwLock in `state.rs`. All poison cases are handled with recovery.
6. The app does not enter a permanent error state after a single thread panic. IPC commands continue to function after recovery.
7. Unit tests demonstrate poison-then-recover for at least: DB Mutex, audit log Mutex, context provider RwLock.
8. `cargo test` — all pass.
9. `cargo clippy -- -D warnings` — clean.

## What This Does NOT Do

- Does not prevent the original panics that cause poisoning — that's a broader code quality concern.
- Does not add crash reporting or panic hooks — separate infrastructure.
- Does not address Mutex poisoning in fields outside `AppState` (e.g., `DEV_TOKEN` in `google_api/token_store.rs:14`, `HygieneBudget.last_reset`).
- Does not switch from `std::sync::Mutex` to `parking_lot::Mutex` (which doesn't poison). That's a viable post-1.0 migration but a larger dependency change.

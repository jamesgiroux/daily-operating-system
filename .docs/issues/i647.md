# I647 - Source-aware prompt propagation + downstream PTY context refresh

**Area:** Backend / Intelligence Prompts + Context Assembly + Meeting Prep + Reports
**Priority:** P1
**Version:** v1.1.0 Phase 5
**Effort:** Large
**Depends on:** I644, I645, existing Glean provider and prompt system
**Source:** Architecture review follow-up. Updating the DB schema alone will not improve downstream PTY calls unless the new account truth is wired into context assembly and prompt shapes.

---

## Problem

Most of DailyOS's later AI work depends on DB-backed context assembly.

Today `build_intelligence_context()` still assembles a narrow account facts block:

- health
- lifecycle
- ARR
- renewal date
- NPS
- products
- account team

That means richer Glean-backed account truth will not automatically improve:

- future enrichment refreshes
- meeting prep
- reports
- daily briefing synthesis
- follow-on PTY or fallback calls

The prompt schemas are also too shallow for what Glean can actually return. Commercial, support, relationship, source, freshness, and feedback state need to be first-class in prompt IO.

---

## Solution

Update the prompt and context pipeline so the new DB-backed account truth becomes the input layer for all later surfaces.

### 1. Expand account context assembly

`build_intelligence_context()` and equivalent account context builders should include:

- renewal likelihood
- renewal model
- renewal pricing method
- renewal opportunity stage
- ARR range and conflict notes
- support package / tier
- active subscription count
- growth potential score
- ICP fit score
- current themes
- recent current incidents
- relationship map summary
- relationship risks / opportunities
- feedback suppressions and user overrides
- source coverage summary

### 2. Expand Glean / PTY schemas

Prompt schemas should explicitly support:

- `sourceKind`
- `sourceSystem`
- `observedAt`
- `staleAfter`
- `isCurrent`
- exact source references
- factual vs inferred values
- suppression-aware output rules

### 3. Preserve the fact / signal / inference boundary

Prompt instructions must tell the model:

- which inputs are canonical facts
- which are signals
- which are prior inferences
- which are user overrides that must be preserved

The model should not freely rewrite user corrections or stale dismissals.

### 4. Downstream surface propagation

The improved DB-backed account truth should flow into:

- future account enrichment
- meeting prep assembly
- daily briefing attention synthesis
- account reports and report cache assembly

The principle is simple: once a field is promoted into canonical DB truth or structured DB assessment, later PTY work should consume it mechanically rather than re-invent it from raw events.

### 5. Verification-ready prompt outputs

When prompts produce output intended for account detail or meeting detail, they should return enough source metadata to back the UI verification affordance.

Example requirements:

- stakeholders include source references
- support incidents include exact ticket references
- renewal outlook includes exact opportunity references
- call themes include Gong call references

---

## Acceptance Criteria

1. `build_intelligence_context()` includes the new commercial, support, relationship, and source-coverage facts from the DB-backed model instead of only the old minimal account facts block.
2. Glean dimension prompts can request and parse richer source-aware fields including freshness and exact references.
3. PTY fallback prompts use the same fact / signal / inference conventions as Glean-backed prompts.
4. Meeting prep and daily briefing synthesis consume the expanded DB-backed account truth without reading `intelligence.json` as a runtime source.
5. User overrides, dismissals, and suppressions are injected into prompt context so the next refresh reflects prior feedback.
6. Report assembly and cached report content can rely on the richer DB-backed assessment fields rather than re-deriving the same context from raw tables.
7. Tests prove that a dismissed stale issue is not re-surfaced on the next enrichment unless newer or stronger evidence exists.

---

## Implementation Notes

- This issue is where the schema work becomes product leverage. Without it, the new DB fields will exist but downstream AI work will still act under-informed.
- Prefer shared context builders so enrichment, prep, and report pipelines do not each invent their own partial view of account truth.
- Keep prompt examples and parsing strict. This is a schema-expansion issue, not a "let the model improvise" issue.

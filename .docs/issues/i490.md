# I490 — Renewal Readiness report type (absorbs I458)

**Priority:** P1
**Area:** Backend / Reports + Frontend (vitals)
**Version:** 1.0.0
**Depends on:** I499 (health scoring engine), I508 (intelligence schema — Commercial Context dimension), I511 (schema decomposition — migration must target post-I511 schema)

## Scope

**Single-user, per-account.** The user generates this for accounts they personally manage. Works with whatever data is available — the report adapts its depth and explicitly surfaces what's missing.

This is not a financial forecasting tool. It's a **relationship-informed readiness assessment** — the kind of document a CSM brings to a renewal QBR that goes beyond pipeline stage and dollar amount to answer: "Do we have the relationships, the delivered value, and the operational health to renew this account confidently? And if not, what specifically needs to happen before the renewal conversation?"

## Problem

### What exists today

The Account Health Review provides a general health snapshot. But renewal preparation is a distinct activity with specific questions:

- Is the champion still engaged, or have they gone cold?
- Does the economic buyer know the value we've delivered?
- Are there unresolved commitments that could become objections?
- Is there competitive pressure that could displace us?
- Is this a routine renewal or a contentious one?
- How much runway do we have, and when should we start the conversation?

None of these are answered by a health score alone. The health score is an input to renewal readiness, not the answer.

### What the manual process looks like

Today, renewal preparation is manual and fragmented:
1. Check CRM for renewal date and ARR
2. Review recent meeting notes for risk signals
3. Check who the champion is and when they last engaged
4. Look for open commitments or unresolved issues
5. Ask the RM if there's competitive pressure
6. Synthesize all of this mentally or in a doc
7. Present at a renewal QBR with incomplete context

DailyOS has most of these inputs already — it just doesn't assemble them into a renewal-specific view.

### The data gap problem

The current renewal data model is thin:
- `accounts.contract_end` — a single date field (the entire "contract" model)
- `accounts.arr` — optional, user-entered
- `account_events` — renewal/expansion/churn/downgrade with optional arr_impact
- No contract term type (annual vs multi-year vs auto-renew)
- No previous renewal outcome history
- No customer fiscal year / budget cycle
- `contract_end` and `account_events` renewal rows aren't synced

I508's Commercial Context dimension (Dimension 5) adds `ContractContext`, `ExpansionSignal`, and `RenewalOutlook` to intelligence.json. These provide the AI-synthesized renewal assessment. But the underlying factual data (contract type, ARR, term) needs to be capturable by the user or sourced from Glean.

### The user as a data source

The user doesn't know what they don't know. DailyOS should:
- **Conditionally surface the report** — only show "Generate Renewal Readiness" when a renewal date is set
- **Prompt the user to fill gaps** — if ARR is missing, show "Adding ARR unlocks renewal value analysis" in the vitals strip
- **Show completeness** — "This report has 4 of 7 sections. Add [contract type] and [competitive context] to unlock the full assessment"
- **Never block on missing data** — generate what it can with what it has, and make the gaps visible

This is the "work harder or ask the user" model: the system extracts what it can from meetings/emails/Glean, then transparently tells the user what's missing and how to fill it.

## Design

### 1. Vitals strip enhancements (prerequisite data capture)

The CS preset's vitals strip needs additional fields to support renewal intelligence. These are optional — the report works without them but is richer with them.

**New vitals fields for CS preset:**

| key | label | fieldType | maps to | Why |
|-----|-------|-----------|---------|-----|
| `contract_start` | Contract Start | date | `accounts.contract_start` | Tenure calculation — first renewal vs year 5 renewal is a different playbook |
| `contract_type` | Contract Type | select: annual/multi-year/month-to-month/auto-renew | `accounts.metadata` | Auto-renew with 30 days left ≠ manual renewal with 30 days left |

These map to existing columns (`contract_start` already exists on `accounts` but isn't in the CS preset, `contract_type` goes into the `metadata` JSON blob via the existing preset metadata mechanism).

**Vitals strip gap prompts:**

When the user views an account within 120 days of renewal, the vitals strip shows gentle prompts for missing fields that unlock report depth:

- No ARR → "Adding ARR helps assess renewal risk relative to book"
- No contract type → "Auto-renew or manual? This shapes the renewal timeline"
- No contract start → "Contract tenure helps assess renewal complexity"

These are tooltips or subtle inline hints, not modal interrupts. The user is helping the system help them.

### 2. Account events model enrichment

Extend the `account_events` CHECK constraint to include completion events and add an outcome field:

```sql
-- Migration: extend account_events
ALTER TABLE account_events ADD COLUMN outcome TEXT;
-- outcome values: expanded | flat | contracted | contentious | smooth | null
-- Only meaningful for completed events (renewal_completed, expansion_closed)

-- Extend event_type CHECK to include positive completions
-- renewal_completed, expansion_closed join the existing renewal/expansion/churn/downgrade
```

**Sync behavior:** When a user sets `contract_end` via VitalsStrip and no `account_events` row exists for that date, auto-create a `renewal` event. When a renewal event is created, update `contract_end` if it's empty. This eliminates the dual-source confusion.

### 3. Report availability and conditional surfacing

The report is conditionally available based on renewal date:

- **No renewal date set:** Report button shows as "Set a renewal date to enable Renewal Readiness" (not hidden — the user learns the report exists and what unlocks it)
- **Renewal date > 180 days out:** Available but de-emphasized. Tooltip: "Renewal is [N] months away — consider generating closer to renewal"
- **Renewal date 30-180 days out:** Prominently available. This is the sweet spot.
- **Renewal date < 30 days:** Urgent styling. The report opens with a runway warning.
- **Renewal date passed:** Still available for post-mortem analysis. Framing shifts from "are we ready?" to "how did it go?"

### 4. Report completeness model

The report adapts to available data rather than failing on missing data. Each section has a completeness assessment:

```
Section completeness:
■ Renewal Profile (4/4 fields) — fully populated
■ Relationship Readiness (3/3 signals) — fully populated
□ Competitive Position (0/1 signals) — no competitive intelligence available
  → "Mention competitors in meeting notes or connect Glean to surface competitive context"
■ Value Narrative (2/2 sources) — meeting wins + success metrics
□ Commercial Context (1/3 fields) — ARR present, contract type and fiscal year unknown
  → "Add contract type in account details to refine the renewal timeline recommendation"
■ Readiness Rating — generated (confidence: moderate — missing competitive + commercial data)
```

This is not a progress bar. It's an intelligence completeness signal that motivates the user to fill gaps because they can see exactly what improves with each addition.

### 5. Output schema

```rust
pub struct RenewalReadinessContent {
    /// Section completeness: which sections are fully populated, partially populated, or empty
    pub completeness: ReportCompleteness,

    /// The renewal context: when, how much, what kind of contract, how much runway
    pub renewal_profile: RenewalProfile,

    /// Relationship readiness: are the right people engaged and aligned?
    pub relationship_readiness: RelationshipReadiness,

    /// Competitive position: is there displacement risk?
    pub competitive_position: Option<CompetitivePosition>,

    /// Value narrative: what have we delivered that justifies renewal?
    pub value_narrative: ValueNarrative,

    /// Commercial context: contract mechanics, expansion potential, procurement
    pub commercial_context: CommercialContext,

    /// Blockers: specific impediments to a smooth renewal
    pub blockers: Vec<RenewalBlocker>,

    /// The readiness verdict with confidence and rationale
    pub readiness_rating: ReadinessRating,

    /// Recommended renewal playbook: what to do, in what order, by when
    pub renewal_playbook: Vec<PlaybookAction>,
}

pub struct ReportCompleteness {
    pub sections_complete: i32,
    pub sections_total: i32,
    pub gaps: Vec<CompletenessGap>,  // what's missing + how to fill it
}

pub struct CompletenessGap {
    pub section: String,
    pub missing: String,           // what data is absent
    pub unlock_hint: String,       // "Add [X] to unlock [Y]" — user-facing
}

pub struct RenewalProfile {
    pub renewal_date: String,
    pub days_until_renewal: i64,
    pub runway_assessment: String,  // "ample" | "adequate" | "tight" | "critical" | "overdue"
    pub arr: Option<f64>,
    pub contract_type: Option<String>,
    pub contract_tenure_years: Option<f64>,  // computed from contract_start
    pub is_first_renewal: Option<bool>,      // first renewal = higher risk, different playbook
    pub previous_renewal_outcome: Option<String>, // from account_events history
    pub recommended_conversation_start: Option<String>, // "Start now" / "Begin by [date]"
}

pub struct RelationshipReadiness {
    /// Is there an active champion who can advocate internally?
    pub champion_status: StakeholderReadiness,
    /// Does the economic buyer / decision-maker know us?
    pub decision_maker_status: StakeholderReadiness,
    /// Are we single-threaded (one contact = high risk)?
    pub single_threaded_risk: bool,
    /// How many active contacts vs total known contacts
    pub active_contact_ratio: String,  // "3 of 7 contacts active in last 60 days"
    /// Relationship trajectory: are relationships strengthening or weakening into renewal?
    pub trajectory: String,  // "strengthening" | "stable" | "weakening"
    pub trajectory_evidence: String,   // 1-2 sentences citing specific signals
}

pub struct StakeholderReadiness {
    pub name: Option<String>,
    pub role: Option<String>,
    pub status: String,        // "engaged" | "warm" | "cold" | "unknown" | "departed" | "none_identified"
    pub last_interaction: Option<String>,  // date
    pub risk_note: Option<String>,  // "Champion hasn't been in a meeting since Jan 15"
}

pub struct CompetitivePosition {
    /// Named competitors mentioned in intelligence
    pub competitors_mentioned: Vec<CompetitorThreat>,
    /// Overall competitive pressure assessment
    pub pressure_level: String,  // "none_detected" | "monitoring" | "active_evaluation" | "displacement_risk"
    /// What differentiates us in this account
    pub differentiation: Option<String>,
}

pub struct CompetitorThreat {
    pub name: String,
    pub context: String,        // what was said, when, by whom
    pub threat_level: String,   // "mention" | "evaluation" | "active_threat"
}

pub struct ValueNarrative {
    /// What have we concretely delivered? (cited from intelligence)
    pub delivered_value: Vec<String>,
    /// Success metrics with current status
    pub success_metrics: Vec<SuccessMetricStatus>,
    /// The "renewal justification" — 2-3 sentences a champion could use internally
    pub renewal_justification: String,
}

pub struct SuccessMetricStatus {
    pub metric: String,
    pub target: Option<String>,
    pub current: Option<String>,
    pub on_track: Option<bool>,
}

pub struct CommercialContext {
    /// Is there expansion potential tied to renewal?
    pub expansion_opportunity: Option<String>,
    /// Known procurement requirements that affect timeline
    pub procurement_considerations: Option<String>,
    /// Customer budget cycle alignment
    pub budget_cycle_note: Option<String>,
}

pub struct RenewalBlocker {
    pub blocker: String,
    pub severity: String,       // "critical" | "high" | "medium"
    pub owner: Option<String>,  // who can resolve it
    pub resolution_path: Option<String>,
}

pub struct ReadinessRating {
    /// ready | at_risk | critical | blocked
    pub rating: String,
    /// Confidence in the rating based on data completeness
    pub confidence: String,     // "high" | "moderate" | "low"
    /// Why this rating — grounded in specific evidence
    pub rationale: String,
    /// What would change the rating — specific, actionable
    pub what_would_improve_rating: Vec<String>,
}

pub struct PlaybookAction {
    /// Specific action in verb form
    pub action: String,
    /// Who should do it
    pub owner: String,
    /// When it should happen relative to renewal
    pub timeline: String,
    /// Why this action matters for the renewal
    pub rationale: String,
    /// What this action depends on
    pub prerequisite: Option<String>,
}
```

### 6. Prompt design

**System role:** "You are a renewal strategist preparing a readiness assessment. Your job is to honestly assess whether this account is ready to renew, identify what's blocking or at risk, and recommend a specific playbook. Be direct about gaps — if data is missing, say so. Never inflate confidence."

**Readiness rating derivation (explicit in prompt):**
- `ready`: Health ≥ 70, champion active (meeting in last 30 days), no critical blockers, no active competitive threat
- `at_risk`: Health 40-69 OR champion cold (no meeting in 60+ days) OR 1+ high blocker OR competitive evaluation detected
- `critical`: Health < 40 OR champion departed OR decision-maker disengaged OR 2+ critical blockers
- `blocked`: Unresolved critical commitment overdue OR active competitive displacement OR procurement/legal impediment with no resolution path

**Confidence derivation:**
- `high`: Health score + relationship data + competitive context + value narrative all populated
- `moderate`: Health score + relationship data present, but competitive or value context missing
- `low`: Health score only, or significant data gaps in relationship assessment

**Key prompt instructions:**
- "The Renewal Playbook must be sequenced — actions have dependencies. Don't list 5 parallel tasks; show the critical path."
- "For first renewals (contract_tenure < 18 months), emphasize value narrative more heavily — the customer hasn't yet proven they'll stay."
- "For auto-renew contracts, the risk isn't losing the renewal — it's the customer churning AFTER auto-renewal because no one had the conversation. Adjust playbook accordingly."
- "Every PlaybookAction must name a specific person or role as owner. 'The team' is not an owner."

### 7. What the report looks like for different data levels

**Minimal data (renewal date + some meetings):**
> **Readiness Rating: At Risk (low confidence)**
>
> We have a renewal date but limited intelligence on this account. 3 meetings in the last 90 days provide some relationship context, but we don't know the contract value, the contract type, or whether there's competitive pressure.
>
> **What would improve this assessment:**
> - Add ARR to understand renewal value relative to your book
> - Add contract type to determine if this is auto-renew (changes the urgency model)
> - Note any competitive mentions in your next meeting debrief
>
> **Relationship Readiness:** Sarah Chen is the only active contact (last meeting: Feb 12). Single-threaded risk is high. No executive sponsor identified.

**Rich data (vitals + meetings + intelligence + Glean):**
> **Readiness Rating: At Risk (high confidence)**
>
> Acme Corp's $240k renewal on April 15 has 44 days of runway. This is a manual renewal (not auto-renew) with a contentious previous renewal outcome — last year required VP escalation to close. Champion (Sarah Chen, VP Product) is engaged but the economic buyer (CFO James Liu) hasn't been in a meeting since November. Competitive context: Contentful was mentioned in the Jan 28 QBR by their CTO as "something we're looking at for the marketing team."
>
> **Renewal Playbook:**
> 1. **This week:** Schedule executive alignment with James Liu (CFO). He approved the original deal but hasn't seen delivered value. Owner: [your name]
> 2. **By March 10:** Prepare value narrative document with Sarah Chen's input — she can champion internally but needs ammunition. Owner: [your name] + Sarah Chen
> 3. **By March 15:** Address the Contentful evaluation — understand if it's serious or exploratory. Get competitive intel from Sarah. Owner: [your name]
> 4. **By March 25:** Present renewal proposal with expansion pitch (new analytics module interest detected in Feb meeting). Owner: [your name] + Sales

### 8. Who this report serves at each org level

**IC (CSM):** "Am I ready for this renewal? What should I be doing in the next 2 weeks?" The playbook is their action plan. The completeness model tells them what data to gather.

**Territory Lead:** Reviews an IC's Renewal Readiness before the QBR. Asks: "Does this IC have the relationships needed? Do I need to bring in an executive for alignment?" The relationship readiness section is their primary concern.

**VP:** Scans Renewal Readiness reports for accounts with `critical` or `blocked` ratings. Asks: "Which renewals need my involvement?" The readiness rating + competitive position are their primary concerns. The playbook's VP-level actions (if any) are their takeaways.

All three levels use the same report — the lens shifts based on what they're looking for, not what the report contains.

## Acceptance Criteria

1. **Completeness model works:** Generate report with only renewal date + 2 meetings. Report renders with gaps visible: "3 of 7 sections populated. Add ARR and contract type to unlock commercial context."
2. **Rich report works:** Generate for account with ARR, contract type, 10+ meetings, stakeholder data, competitive context from I508. All sections populated, confidence = high.
3. **Conditional surfacing:** Report button shows as "Set a renewal date to enable Renewal Readiness" when no renewal date exists. Prominently available when renewal is 30-180 days out. Urgent styling when < 30 days.
4. **Playbook is sequenced:** Actions have prerequisites and a critical path, not a flat list of parallel tasks.
5. **First renewal vs routine:** Account with contract_tenure < 18 months emphasizes value narrative. Account with 5-year tenure emphasizes relationship continuity.
6. **Auto-renew distinction:** Auto-renew contracts get a different playbook framing (risk = silent churn, not failed renewal).
7. **Readiness rating aligns with evidence:** Rating of `ready` requires active champion + no critical blockers. Rating of `critical` requires specific evidence (departed champion, competitive displacement, etc.).
8. **Previous renewal informs assessment:** Account with `previous_renewal_outcome = "contentious"` flags this in the renewal profile with a note about what happened last time.
9. **Vitals strip prompts:** Missing ARR shows hint text when within 120 days of renewal. Missing contract type shows hint text. These are visible on the account detail page.
10. No internal DailyOS terminology in output (no "signal", "entity", "enrichment").
11. **account_events sync:** Setting `contract_end` via VitalsStrip creates an `account_events` renewal row. Creating a renewal event updates `contract_end` if empty.

## Out of Scope

- Multi-account renewal pipeline view (that's I491 Portfolio Health + the weekly commercial update research)
- Renewal negotiation playbook or pricing guidance (DailyOS doesn't know pricing)
- Automatic renewal date detection from Glean/CRM (future — depends on Glean/Salesforce authentication gate)
- Notification/alert when renewal is approaching (scheduler concern, separate issue)
- Custom weighting of readiness factors (the rating derivation is fixed; user preferences would add complexity without proven value)
- Multi-user intelligence access — report uses current user's data only. IC intelligence flowing into a lead's renewal view requires shared intelligence layer (v1.2.0+)

## Implementation Notes

### Competitive context

The `competitive_position` section depends on competitive intelligence from I508 (Intelligence Schema — Strategic Assessment dimension). Until I508 ships and enrichment runs, this section will render as `None` with the completeness model showing the gap. The prompt handles this via the completeness model: "No competitive intelligence available. Mention competitors in meeting notes or connect Glean to surface competitive context."

### CHECK constraint table rebuild

SQLite does not support `ALTER TABLE ... DROP CONSTRAINT` or modifying CHECK constraints in place. Extending the `account_events.event_type` CHECK constraint to include `renewal_completed` and `expansion_closed` requires the table rebuild pattern:

1. Create new table with updated CHECK constraint
2. Copy data from old table
3. Drop old table
4. Rename new table

This must be handled in the migration file. The migration must run AFTER I511 (schema decomposition) since I511 may also touch `account_events`. Coordinate migration ordering.

# I605 — Account Reports → Display-Only

**Version:** v1.0.3 Phase 2
**Depends on:** I604 (schema, assembly engine)
**Type:** Enhancement
**Scope:** Backend (reports/account_health, reports/swot, reports/ebr_qbr, services/reports)

## Problem

Account Health, SWOT, and EBR/QBR reports each run PTY calls (1 monolithic or 5 parallel) that re-derive intelligence from raw tables. With I604's assembly engine, these can become display-only surfaces that render pre-computed `entity_assessment` data instantly.

Specific waste today:
- **Account Health** re-counts meetings (90d), re-counts email signals, re-queries renewal date — all already in `health_json.dimensions` and `dimensions_json.contract_context`
- **SWOT** runs 5 parallel PTY calls to categorize strengths/weaknesses/opportunities/threats — all derivable from existing `recent_wins_json`, `current_state_json`, `risks_json`, `dimensions_json.expansion_signals`, `dimensions_json.competitive_context`
- **EBR/QBR** re-derives success metrics, value delivered, open commitments — all stored in `entity_assessment`

## Design

### Account Health → Assembly-Only

Replace `gather_account_health_input()` + `run_report_generation()` with `assembly::assemble_account_health()`.

Field mapping:

| Report Field | Source |
|---|---|
| `overall_assessment` | `executive_assessment` |
| `health_score_narrative` | `health_json.narrative` |
| `relationship_summary` | Template from `stakeholder_insights_json` + `relationship_depth` |
| `engagement_cadence` | Template from `health_json.dimensions.meeting_cadence.score` + `.evidence` |
| `customer_quote` | `best_customer_quote` (new column from I604) |
| `what_is_working` | `current_state_json.working` |
| `what_is_struggling` | `current_state_json.not_working` |
| `expansion_signals` | `dimensions_json.expansion_signals[].description` |
| `value_delivered` | `value_delivered` |
| `risks` | `risks_json` with status field |
| `renewal_context` | Template from `dimensions_json.contract_context.renewal_date` + `health_json.band` |
| `recommended_actions` | `health_json.recommended_actions` |

The report command checks if `entity_assessment` exists and is fresh. If so, assembles mechanically. If not (first run, never enriched), falls back to displaying an empty state with a "Refresh intelligence" prompt rather than running a PTY call.

### SWOT → Mechanical Assembly

Replace 5 parallel PTY calls with `assembly::assemble_swot()`.

Field mapping:

| SWOT Category | Source |
|---|---|
| `strengths` | `recent_wins_json` + high-scoring `health_json.dimensions` (evidence strings) + `value_delivered` |
| `weaknesses` | `current_state_json.not_working` + low-scoring `health_json.dimensions` + non-critical `risks_json` items |
| `opportunities` | `dimensions_json.expansion_signals` + `dimensions_json.strategic_priorities` |
| `threats` | Critical `risks_json` items + `dimensions_json.competitive_context` |
| `summary` | First sentence of `executive_assessment` + health band statement |

Each item gets a `source` field populated from the source data's `item_source` or `source` field where available.

### EBR/QBR → Display from `ebr_cache_json`

The EBR/QBR is unique: it's customer-facing and requires audience-aware register transformation. The AI work is moved to enrichment time (not report time) via the new `ebr_cache_json` column on `entity_assessment`.

During entity enrichment, after the main intelligence is computed, an additional focused PTY call generates customer-facing content:
- `executive_summary` — customer-register version of executive_assessment
- `story_bullets` — 3 strategic framing bullets
- `strategic_roadmap` — forward-looking synthesis

This cache is stored in `ebr_cache_json` and refreshed whenever `enriched_at` updates. The EBR/QBR report assembles from `ebr_cache_json` + `value_delivered` + `success_metrics` + `risks_json` (with status) + `recommended_next_steps_json`.

The internal content filter (no internal-only meeting content in customer-facing docs) is applied during cache generation, not at render time.

### Service Layer Changes

`services/reports.rs` dispatch for account_health, swot, and ebr_qbr:

```
1. Check entity_assessment exists and enriched_at is not null
2. If fresh: assemble mechanically, upsert to reports table, return
3. If missing/stale: return empty state with is_stale = true
```

The `generate_report` command becomes a "refresh" action that triggers entity enrichment (which cascades to `ebr_cache_json` regeneration), not a direct PTY call.

### Frontend Changes

- Remove "Generating..." progress animation for Account Health, SWOT, EBR/QBR
- Reports render instantly from stored data
- Show "Intelligence not available — enrich this account first" empty state when no assessment exists
- "Refresh" button triggers enrichment, not report generation
- Animation primitives from I604 integrated: staggered reveal on sections

## Out of Scope

- BoB (I606), Weekly Impact (I607), Monthly Wrapped (I608)
- Enrichment prompt changes beyond EBR cache (I604 handles risk status, next steps, quote)

## Acceptance Criteria

1. Account Health report renders instantly from `entity_assessment` — zero PTY calls, verified via logs
2. SWOT report renders instantly — zero PTY calls
3. EBR/QBR renders from `ebr_cache_json` — zero PTY calls at report time
4. `ebr_cache_json` is generated during entity enrichment and refreshed on each enrichment cycle
5. Reports narrative matches what the account detail page shows (same source data)
6. Empty state shown when entity has never been enriched (no crash, no hang)
7. "Refresh" triggers enrichment, not direct report generation
8. In-place editing preserved — edits stored in reports table, survive assembly
9. SWOT items have source attribution from underlying data

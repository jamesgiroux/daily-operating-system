# I581 — Email Cadence Awareness

**Version:** v1.0.1
**Area:** Backend + Frontend / Email
**Priority:** P1

## Problem

For CS, silence from an account is often more alarming than noise. "Acme used to email weekly — haven't heard from them in 3 weeks." The `entity_email_cadence` table already tracks per-entity email frequency, but this data is never surfaced as a user-facing insight. Cadence drops are invisible until the account is already at risk.

## Design

### Backend

**New function** in `services/emails.rs` — `detect_cadence_anomalies()`:
- For each tracked account with `entity_email_cadence` data, compare days since last email against the historical average interval
- Flag accounts where silence exceeds 2× their normal cadence interval
- Exclude accounts with fewer than 3 historical emails (insufficient baseline for cadence detection)
- Return `Vec<CadenceAlert>`

**New struct** `CadenceAlert`:
- `entity_id`, `entity_name`
- `normal_interval_days` (historical average)
- `days_since_last_email`
- `last_email_date`
- `last_email_sender` (for context)

**New field on `EmailBriefingData`**: `cadence_alerts: Vec<CadenceAlert>`

**Signal emission**: New signal type `email_cadence_drop` emitted when silence threshold is crossed. Confidence: 0.6 (silence is ambiguous — could be good or bad). Only emit once per entity per threshold crossing — track via `email_signals` dedup index. Do not re-emit daily.

**Intelligence Loop**: `email_cadence_drop` feeds engagement dimension. The signal is moderate confidence because silence can mean "everything's fine, no issues" or "they've checked out." The user can dismiss it (feeding Bayesian correction) or investigate.

### Frontend

New section on The Correspondent — **"GONE QUIET"**.

- Appears after UPDATES, before INBOX
- Each item: account name (links to account detail), typical cadence ("usually emails weekly"), days since last email, last email date
- Section hidden when no anomalies detected
- If fewer than 3 items, fold into UPDATES section rather than a standalone section
- Dismiss action per item (user acknowledges, doesn't need to see again for this cycle)

### Vocabulary (ADR-0083)

- Section heading: "GONE QUIET" (not "cadence anomalies")
- Cadence description: "usually weekly" / "usually every few days" (not "average interval: 7 days")

## Mock Data

Requires I596 (email mock data foundation). Cadence scenarios:
- Acme: `entity_email_cadence` shows weekly cadence, last email 3 days ago (normal — should NOT appear)
- Globex: `entity_email_cadence` shows weekly cadence, last email 18 days ago (2.5× normal — SHOULD appear as "gone quiet")
- Initech: fewer than 3 historical emails (excluded from cadence detection)

## Acceptance Criteria

1. Accounts with email silence exceeding 2× their normal cadence appear in a "GONE QUIET" section
2. Each item shows account name, typical cadence ("usually weekly"), and days since last email
3. Clicking an account name navigates to the account detail page
4. `email_cadence_drop` signal emitted once per entity when silence threshold is crossed
5. Signal feeds entity engagement dimension at confidence 0.6
6. Accounts with fewer than 3 historical emails are excluded (insufficient baseline)
7. Section does not appear when there are no cadence anomalies
8. Dismissing an item hides it for this cycle and emits a dismissal signal
9. **Mock data**: `full` scenario seeds `entity_email_cadence` for Acme (normal), Globex (gone quiet), and Initech (insufficient). "GONE QUIET" shows Globex only.

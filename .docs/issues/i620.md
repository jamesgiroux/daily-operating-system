# I620 — Actions pipeline audit & repair

**Area:** Backend / Frontend / Actions
**Priority:** P0
**Version:** v1.0.2
**Source:** v1.0.0 GA engineering review (2026-03-17)

---

## Problem

Actions have been broken since the v1.0.0 DB migration. The AI extracts actions from transcripts (verified via audit trail — 7 well-formatted actions with priorities, accounts, due dates, and context), but they never surface in the UI. The system silently degrades — empty sections just don't render, so nobody notices.

Six root causes identified:

1. **Proposed actions invisible on daily briefing.** `useProposedActions` loads on mount with no event-driven refresh. After transcript processing completes, the briefing doesn't re-fetch proposed actions. The user sees an empty attention section.

2. **Actions page defaults to "Pending" tab.** If proposed actions exist but the user has already loaded the page, they're staring at an empty pending list. The smart default (line 177) only fires on initial load. No badge notification or auto-switch when new proposed actions arrive.

3. **`get_actions_from_db` excludes proposed.** The main actions query (`get_due_actions`) only returns `WHERE status = 'pending'`. Transcript-extracted actions start as `status = 'proposed'`, so they're invisible to the briefing's regular action list and the actions page pending view.

4. **Dedup guard silently rejects on reprocessing.** `upsert_action_if_not_completed_with_status` returns `Ok(false)` when the title dedup guard matches, and the service layer maps this to `Ok(())` — no error, no log. The caller counts it as a success. Combined with WAL visibility gaps between connections, reprocessing appears to work but inserts nothing.

5. **No lifecycle feedback.** When an action is rejected by dedup or fails to insert, there's no log, no toast, no signal. `extract_transcript_actions` reports "Extracted N actions" even when zero were actually written.

6. **Extraction model was haiku.** Too weak for I554 prompt complexity (6 win sub-types, urgency-tiered risks, champion health, commitments). Bumped to sonnet in dev.

## Scope

### Backend fixes

1. **Make dedup guard observable.** `upsert_action_if_not_completed_with_status` should log when it rejects: `log::debug!("Action dedup: skipped '{}' (title exists for source {})", title, source_id)`. The service wrapper should return the bool so callers can count actual writes.

2. **Fix `extract_transcript_actions` count.** Only increment `count` when the write actually happened (check the bool return). Log both attempted and successful counts.

3. **Same-connection clear for reprocessing.** Already fixed in dev — `process_transcript` deletes prior transcript actions on its own connection before inserting. Verify this works end-to-end.

### Frontend fixes

4. **`useProposedActions` must refresh on events.** Add `useTauriEvent("transcript-processed", refresh)` and `useTauriEvent("intelligence-updated", refresh)` so the proposed actions list updates after transcript processing without requiring navigation.

5. **Actions page: auto-switch to Proposed tab.** When `proposedCount` changes from 0 to >0 (new proposed actions arrive while viewing the page), auto-switch `statusFilter` to `"proposed"` and show a brief toast: "N new suggested actions from transcript."

6. **Daily briefing proposed actions refresh.** The `DailyBriefing` component uses `useProposedActions` — same hook fix (#4) makes this work automatically.

7. **Consider including proposed in `get_due_actions`.** Evaluate whether proposed actions should appear in the main actions list (with a visual distinction) rather than requiring a separate tab. This is a product decision — document the tradeoff.

### Verification

8. **End-to-end test script.** Document a manual verification flow: process a transcript → confirm audit file has ACTIONS → confirm log shows "Extracted N actions (N written)" → navigate to Actions page → Proposed tab shows items → accept one → it moves to Pending → check meeting detail outcomes section shows remaining proposed.

## Acceptance Criteria

1. After processing a transcript with actions, `Proposed` tab on Actions page shows the extracted actions within 5 seconds (no navigation required if already on the page).
2. Daily briefing "Suggested" section shows proposed actions from today's meeting transcripts.
3. Meeting detail "Outcomes → Actions" section shows transcript-extracted actions.
4. `extract_transcript_actions` log shows accurate count: "Extracted 7 actions (5 written, 2 skipped by dedup)".
5. Dedup rejections logged at debug level with title and source context.
6. Accepting a proposed action on the Actions page moves it to Pending. Completing it emits an `action_completed` signal.
7. Reprocessing a transcript clears and re-inserts actions correctly — no stale dedup blocks.
8. Zero silent failures — every action insertion attempt either succeeds (logged) or is explicitly skipped (logged with reason).

## Out of Scope

- Action lifecycle redesign (I583 aging, I584 health scoring) — that's Phase A
- Action-to-objective linking (I586, I592) — that's Phase B
- Rejection learning (I595) — that's Phase C

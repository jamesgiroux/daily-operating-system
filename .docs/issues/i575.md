# I575 — Progressive Enrichment Everywhere

**Version:** v1.0.0 Phase 5b
**Depends on:** I574 (Parallel dimension enrichment)
**Type:** Enhancement — Backend + Frontend / UX / Intelligence
**Scope:** `intel_queue.rs`, `services/intelligence.rs`, `reports/`, `processor/transcript.rs`, `prepare/email_enrich.rs`, frontend hooks and pages

---

## Problem

Every AI operation in DailyOS waits for the full response before writing anything to the database or updating the frontend. Users see a spinner for 30-180s with no feedback on what's happening.

**Entity enrichment:** 6 parallel dimension calls (after I574) complete at different times, but the user sees nothing until all 6 finish and the merged result is written. The refresh button shows "Refreshing... 36s" — a meaningless elapsed timer.

**Report generation:** A 5-section report completes sections at different times, but nothing renders until the final section finishes and the full report is assembled.

**Transcript processing:** Summary and actions are available after Phase 1 (~15s), but meeting detail waits for all 3 phases before updating.

**Email enrichment:** 20 emails process in parallel batches (after I574), but the email page doesn't update until the entire batch completes.

The result: users don't trust the system because they can't see it working. They click refresh again, queueing duplicate work. They navigate away, missing results when they arrive. The system does good work — it just hides it behind a wall of silence.

---

## Solution

### Design Principle

Every discrete unit of AI work writes to DB the moment it completes and emits a Tauri event so the frontend can update immediately. The user sees data filling in, not a spinner.

### 1. Entity Enrichment — Dimension-by-Dimension Writes

Each of the 6 parallel dimension calls (from I574) writes its results to `entity_assessment` as soon as it completes. The write is a partial update — only the fields belonging to that dimension group are SET; other fields are untouched.

**Backend changes (`intel_queue.rs`):**

```rust
// After each dimension call completes:
fn write_dimension_result(
    db: &ActionDb,
    entity_id: &str,
    entity_type: &str,
    dimension: &str,
    partial_intel: &IntelligenceJson,
    app_handle: &AppHandle,
) -> Result<(), String> {
    // 1. Read current assessment from DB
    // 2. Merge this dimension's fields into it
    // 3. Write back to DB
    // 4. Emit progress event
    write_intelligence_json(db, entity_id, entity_type, partial_intel)?;

    app_handle.emit("enrichment-progress", EnrichmentProgress {
        entity_id: entity_id.to_string(),
        entity_type: entity_type.to_string(),
        completed: completed_count,
        total: 6,
        last_dimension: dimension.to_string(),
    }).ok();

    Ok(())
}
```

**Tauri event schema — `enrichment-progress`:**

```typescript
interface EnrichmentProgress {
  entity_id: string;
  entity_type: string;
  completed: number;    // 1-6 (dimension calls completed so far)
  total: number;        // 6 (always, for entity enrichment)
  last_dimension: string; // "core_assessment" | "stakeholder" | "commercial" | "strategic" | "value_success" | "engagement"
}
```

**Completion event — `enrichment-complete`:**

```typescript
interface EnrichmentComplete {
  entity_id: string;
  entity_type: string;
  succeeded: number;    // how many dimensions succeeded
  failed: number;       // how many dimensions failed
  failed_dimensions: string[]; // names of failed dimensions
  wall_clock_ms: number; // total wall-clock time
}
```

### 2. Report Generation — Section-by-Section Writes

Each report section (from I574 decomposition) writes to the report record as it completes. The report record stores sections in a structured JSON field rather than a single prose blob.

**New report storage pattern:**

```rust
// reports table gets a new column: sections_json TEXT
// Each section write appends to this JSON object:
{
  "executive_summary": { "content": "...", "completed_at": "..." },
  "dimension_analysis": { "content": "...", "completed_at": "..." },
  // ... more sections as they complete
  "assembled_at": null  // set when all sections are assembled into final markdown
}
```

**Tauri event — `report-progress`:**

```typescript
interface ReportProgress {
  entity_id: string;
  report_type: string;  // "account_health" | "ebr" | "swot" | "weekly_impact" | "monthly_wrapped"
  completed: number;    // sections completed
  total: number;        // total sections for this report type
  last_section: string; // "executive_summary" | "dimension_analysis" | etc.
}
```

When all sections complete, the report assembler combines them into the final markdown and sets `assembled_at`. The frontend can show sections as they arrive or wait for the full assembly — its choice.

### 3. Transcript Processing — Phase-by-Phase Writes

Transcript processing already decomposes into 3 phases (I574). This issue adds the incremental writes:

**Phase 1 completes (~15s):** Write summary, actions, and decisions to DB. Emit `transcript-progress` with `{ phase: 1, completed: 1, total: 3 }`. Meeting detail page shows summary and actions immediately.

**Phase 2 completes (~30s):** Write wins, risks, sentiment, and champion health. Emit `transcript-progress` with `{ phase: 2, completed: 2, total: 3 }`. Meeting detail page shows risk/win badges, sentiment indicators.

**Phase 3 completes (~45s):** Write interaction dynamics, role changes, commitments, engagement signals. Emit `transcript-progress` with `{ phase: 3, completed: 3, total: 3 }`. Full transcript analysis now visible.

**Tauri event — `transcript-progress`:**

```typescript
interface TranscriptProgress {
  meeting_id: string;
  entity_id: string | null;
  phase: number;        // 1 | 2 | 3
  completed: number;    // phases completed
  total: number;        // 3
  phase_label: string;  // "Core (summary + actions)" | "Signals (risks + wins)" | "Dynamics"
}
```

### 4. Email Enrichment — Per-Email Writes

Each email enrichment result writes to DB immediately upon completion (already the case in the existing loop, but after I574 parallelization, multiple emails complete concurrently).

**Tauri event — `email-enrichment-progress`:**

```typescript
interface EmailEnrichmentProgress {
  completed: number;    // emails enriched so far
  total: number;        // total emails in batch
  last_email_id: string;
  last_email_subject: string;
}
```

Emitted after each email's enrichment result is persisted. The email page can listen and refresh its list incrementally.

### 5. Frontend — `useProgressiveData` Hook

A shared React hook that subscribes to Tauri events and provides progressive loading state to any consuming component.

```typescript
interface ProgressiveState<T> {
  data: T | null;           // current data (partial or complete)
  loading: boolean;         // true while any enrichment is in progress
  progress: number;         // 0.0 - 1.0 completion percentage
  completedSteps: number;
  totalSteps: number;
  lastStep: string;         // human-readable label of last completed step
  error: string | null;     // set if enrichment failed
}

function useProgressiveData<T>(
  entityId: string,
  eventName: string,    // "enrichment-progress" | "report-progress" | etc.
  fetcher: () => Promise<T>,  // re-fetch from backend on each progress event
): ProgressiveState<T>
```

**Behavior:**
1. On mount, call `fetcher()` to load current data.
2. Subscribe to the specified Tauri event filtered by `entity_id`.
3. On each progress event, update `progress`/`completedSteps`/`lastStep` and re-call `fetcher()` to get updated data.
4. On completion event, set `loading: false`, final `fetcher()` call.
5. On unmount, unsubscribe from events.

### 6. Refresh Button — Completion Percentage

Replace the elapsed-seconds display with a completion percentage derived from the progress events.

**Current behavior:** "Refreshing... 36s" (meaningless elapsed timer)

**New behavior:** "Refreshing... 67%" (4 of 6 dimensions complete)

When no progress events have arrived yet: "Refreshing..." (no percentage — avoids showing 0% which feels broken).

When complete: "Updated just now" (fade out after 3s, same as current).

**Implementation in account detail page:**

```typescript
const { progress, loading, lastStep } = useProgressiveData(
  accountId,
  "enrichment-progress",
  () => fetchAccountIntelligence(accountId),
);

// In the refresh button:
{loading ? (
  progress > 0
    ? `Refreshing... ${Math.round(progress * 100)}%`
    : "Refreshing..."
) : null}
```

### 7. Meeting Detail — Progressive Transcript Results

Meeting detail page subscribes to `transcript-progress` events. As each phase completes:

- Phase 1: Summary section renders, action items appear in the actions strip
- Phase 2: Risk/win badges appear, sentiment indicator shows
- Phase 3: Dynamics section fills in, commitments appear

Each section uses a subtle fade-in animation (200ms, `opacity 0 -> 1`) as it arrives, so the user sees content materializing rather than popping in.

### 8. Account Detail — Dimension-by-Dimension Rendering

Account detail page subscribes to `enrichment-progress` events. As each dimension completes, the corresponding section updates:

- Core Assessment arrives: executive assessment text, risks strip, wins strip
- Stakeholder arrives: stakeholder grid, champion health badge, coverage assessment
- Commercial arrives: health score, renewal outlook, expansion signals
- Strategic arrives: company context, competitive landscape, strategic priorities
- Value & Success arrives: value delivered timeline, success metrics, commitments
- Engagement arrives: meeting cadence, email responsiveness, product adoption

Sections that haven't arrived yet show a subtle "Analyzing..." placeholder (one line, muted text, no spinner). Once all 6 complete, the placeholders disappear.

### 9. Report Page — Section-by-Section Rendering

Report generation page subscribes to `report-progress` events. Each section renders as it completes. The report page uses a document-style layout where sections appear in order, even if they complete out of order (hold a later section behind an "Analyzing..." placeholder until its predecessors are done, to maintain reading flow).

## Acceptance Criteria

1. Entity enrichment emits `enrichment-progress` Tauri events as each of the 6 dimensions completes. Events arrive incrementally (not all at once). Verified by subscribing to events in the browser console during a real enrichment.
2. Each `enrichment-progress` event contains `{ entity_id, entity_type, completed, total, last_dimension }` matching the schema above. `completed` increments from 1 to 6.
3. Entity assessment data in the DB updates after each dimension completes, not only after all 6 finish. Verified by querying `entity_assessment` during enrichment and seeing partial data grow.
4. The refresh button on account detail shows completion percentage ("Refreshing... 50%") instead of elapsed seconds. Percentage derives from `completed / total` in progress events.
5. When no progress events have arrived yet (enrichment just started), refresh button shows "Refreshing..." without a percentage. No "0%" displayed.
6. Account detail page updates incrementally as dimensions complete. Core assessment section renders first (executive assessment, risks, wins). Stakeholder section renders when its dimension completes. Each section that hasn't loaded yet shows "Analyzing..." placeholder text.
7. Transcript processing emits `transcript-progress` events after each of the 3 phases. Meeting detail page shows summary and actions after Phase 1 completes (~15s). Risks and wins appear after Phase 2. Dynamics after Phase 3.
8. If transcript Phase 2 or 3 fails, Phase 1 data (summary + actions) is already persisted and visible on the meeting detail page. No spinner, no error state — just the sections that succeeded.
9. Report generation emits `report-progress` events per section. Report page shows sections rendering in document order as they complete.
10. Email enrichment emits `email-enrichment-progress` events per email. Email page shows enriched emails updating incrementally during background sync. Enrichment of email #5 is visible before email #20 finishes.
11. `useProgressiveData` hook correctly subscribes/unsubscribes to Tauri events on mount/unmount. No memory leaks from orphaned event listeners. Verified by mounting/unmounting the account detail page during enrichment.
12. `useProgressiveData` re-fetches data from the backend on each progress event. The `data` field always reflects the current DB state, not a stale snapshot.
13. `enrichment-complete` event fires when all dimensions finish (or fail). Contains `succeeded`, `failed`, `failed_dimensions`, and `wall_clock_ms`. Frontend transitions from loading to complete state.
14. Sections that arrive use a subtle fade-in (200ms opacity transition). No jarring pop-in. Verified visually on account detail and meeting detail pages.
15. Run a full enrichment on a real account. Observe: refresh button shows percentage climbing, sections fill in one by one, total UX wait-to-first-content is < 15s (first dimension result). Compare to pre-I575 behavior where wait-to-any-content was 30-180s.
16. All existing frontend tests pass. New `useProgressiveData` hook has unit tests covering: subscribe, progress updates, completion, error, and unmount cleanup.

# I642 — Token usage observability + efficiency foundations

**Area:** Backend / Intelligence / PTY
**Priority:** P1
**Version:** v1.1.0
**Effort:** Medium
**Source:** Usage spike investigation (2026-03-27). Manual codebase audit revealed no visibility into PTY call volume, token consumption, or cost attribution. Sonnet usage spike correlated with Google reconnection triggering cascading enrichment.

---

## Problem

DailyOS has zero observability into AI token usage. There is no way to know:

- How many PTY calls fire per day (or per trigger type)
- How many tokens each call consumes (input + output)
- What the cost breakdown is across call sites (transcript enrichment vs intel queue vs email enrichment)
- Whether background enrichment is running more often than expected

The audit log (ADR-0094) defines a `token_estimate` field in the `entity_enrichment_completed` event but it is never populated. No other call site tracks token usage at all.

### Current Token Budget (estimated from code analysis)

| Call Site | Tokens In | Tokens Out | Typical Daily Calls |
|---|---|---|---|
| Transcript processing (3 phases) | 40-45K | 3K | 3 phases × 1-5 transcripts |
| Intel queue enrichment | 2.5-7K each | 250-1K each | 10-20 |
| Email enrichment | 600-1.1K each | 30-50 each | 20 |
| Pre-meeting auto-refresh | 2.5-7K each | 250-1K each | 1-2 |
| **Daily total** | **~107-171K** | **~10-19K** | **~35-45 calls** |

Estimated cost: $0.45-0.80/day ($10-18/month) per active user at Sonnet pricing.

### Cascade Risk

A single event (Google reconnection, Glean sync, batch transcript arrival) can cascade:
1. Email sync → email enrichment for backlog
2. Calendar changes → signals → intel queue for every affected entity
3. Transcript processing → captures → signal propagation → more enrichment
4. All with only 30s debounce and 2hr TTL on intel queue

No throttle exists at the aggregate level — individual call sites debounce independently but total daily token spend is unbounded.

---

## Solution

Two parts: **observability** (know what's happening) and **efficiency** (reduce waste).

### Part A: Token Observability

**A1. PTY call metering.** After every PTY call returns, estimate tokens from prompt character count and response character count (chars ÷ 4 heuristic). Log to audit trail with call site, entity_id, model tier, and duration.

**A2. Aggregate daily counter.** In-memory accumulator tracks daily totals by call site (transcript, intel_queue, email, report). Persisted to `app_state` table on each update. Frontend can read via existing diagnostics command.

**A3. Diagnostics surface.** Add a "Token Usage" section to the existing Settings → Diagnostics page showing:
- Today's token count (in/out) by call site
- Total PTY calls today
- Last 7 days trend (daily totals from `app_state`)

### Part B: Efficiency Foundations

**B1. Transcript context deduplication.** The same ~60K char transcript is sent 3 times (once per phase). Phase 2 and Phase 3 prompts should reference Phase 1's summary instead of re-sending the full transcript. For Phase 2+3, send only the transcript sections relevant to that phase's extraction target (e.g., decision-heavy segments). Target: 40-50% reduction in transcript token usage.

**B2. Intel queue daily budget.** Add a configurable daily token budget (default: 500K tokens). When the budget is reached, background enrichment (ProactiveHygiene, ContentChange) pauses until midnight. Manual and CalendarChange triggers still fire. Budget tracked via the A2 accumulator. Emit `intel_budget_exhausted` system event when hit.

**B3. Email enrichment batching.** Current: 1 PTY call per email. New: batch up to 5 emails into a single prompt with structured JSON output. Reduces per-email overhead from ~800 tokens to ~300 tokens (shared system prompt amortized).

**B4. Intel queue TTL increase for background triggers.** Current TTL is 2 hours for all triggers. Increase to 6 hours for ProactiveHygiene and ContentChange. Manual and CalendarChange keep 2hr TTL. Halves background enrichment volume.

---

## Acceptance Criteria

1. Every PTY call logs token estimate (input chars, output chars, estimated tokens, call site, model tier, duration_ms) to `~/.dailyos/audit.log` with event `ai_call_completed`.
2. Settings → Diagnostics shows "AI Usage" section: today's total tokens (in/out), call count by site (transcript/intel/email/report), 7-day daily totals.
3. Transcript Phase 2 and Phase 3 prompts use Phase 1 summary + targeted transcript excerpts instead of full transcript. Total per-transcript token usage drops from ~40-45K to ~22-28K (measurable via A1 metering).
4. `intel_queue_daily_budget` setting exists (default 500K). When reached, background-priority enrichments pause. Manual triggers still work. Budget resets at midnight. Diagnostics shows budget remaining.
5. Email enrichment batches up to 5 emails per PTY call. Per-email token cost drops below 400 tokens average (measurable via A1).
6. Intel queue TTL for ProactiveHygiene and ContentChange triggers is 6 hours. Manual and CalendarChange remain 2 hours.
7. No regressions: enrichment quality (intelligence JSON schema validation pass rate) stays ≥ 95%.

## Implementation Notes

- Token estimation heuristic (chars ÷ 4) is sufficient for observability. Exact counts require API response parsing which PTY subprocess doesn't expose.
- Daily budget should be conservative initially (500K) — can be tuned after observability data is available.
- Transcript context dedup (B1) is the highest-ROI change: 3 transcripts/day × 20K tokens saved = 60K/day.
- Email batching (B3) has moderate ROI but reduces PTY call count significantly (20 calls → 4 calls).

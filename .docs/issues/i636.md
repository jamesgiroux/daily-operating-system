# I636 — Consolidated meeting record markdown for MCP consumption

**Area:** Backend / Transcript Processing
**Priority:** P1
**Version:** v1.0.3
**Effort:** Small
**Depends on:** I634 (temporal transformation), I635 (prediction scorecard data), I637 (continuity thread data)
**Source:** CEO plan review (2026-03-21). MCP users need a single document answering "what happened in this meeting?"

---

## Problem

After transcript processing, two disk artifacts exist:
1. **Transcript file** — raw text with YAML frontmatter in `Call-Transcripts/`
2. **Impact log** — bulleted wins appended to `_today/90-impact-log.md`

Neither is a complete meeting record. The structured intelligence (summary, outcomes, predictions, actions) lives only in the DB. When a Claude Code session asks "what happened in the Acme QBR?" via MCP `search_content`, it finds the raw transcript but not the structured intelligence.

## Solution

After Phase 3 processing completes, generate a structured meeting record markdown file:

**Path:** `~/Accounts/{Name}/Meeting-Records/{date}-{slug}-record.md`
(Or `People/{Name}/` for 1:1s, `Projects/{Name}/` for project meetings per I631)

**Template:**

```markdown
---
id: {meeting_id}
title: {meeting_title}
entity: {linked_entity_name}
entity_type: {account|person|project}
date: {YYYY-MM-DD}
time: {start} - {end}
duration: {minutes}m
attendees: {count}
type: {meeting_type}
generated_at: {ISO timestamp}
---

# {Meeting Title}
{date} · {time_range} · {entity_name}

## Summary
{Phase 1 summary}

## The Thread
Since your last meeting with {entity} on {prev_date}:
{continuity items from I637, or "First meeting with {entity}"}

## What We Predicted vs What Happened
{prediction scorecard from I635, formatted as markdown}

## Key Findings

### Wins
{wins with sub-types and evidence quotes}

### Risks
{risks with urgency and evidence}

### Decisions
{decisions with context}

## Engagement
{interaction dynamics: talk balance, sentiment, signals}

## Champion Health
{champion engagement assessment}

## Commitments
{commitments with owner and timeline}

## Actions
{action items with owner, due date, priority}

## Attendees
{attendee list grouped by side}
```

### Generation Timing

Called after Phase 3 completes and all data is persisted, as the last step before returning `TranscriptResult`. This ensures all intelligence layers are included.

### Relationship to Transcript File

| File | Content | Purpose |
|------|---------|---------|
| `Call-Transcripts/{date}-transcript.md` | Raw transcript text + YAML frontmatter | Archive, full text search |
| `Meeting-Records/{date}-record.md` | Structured intelligence output | MCP consumption, sharing, reference |

Both files are generated. Neither replaces the other.

## Acceptance Criteria

1. After transcript processing, a meeting record markdown exists at `{EntityDir}/Meeting-Records/{date}-{slug}-record.md`.
2. The record contains: YAML frontmatter (metadata), summary, continuity thread, prediction scorecard, wins/risks/decisions, engagement dynamics, champion health, commitments, actions, attendees.
3. MCP `search_content` for the entity finds the meeting record.
4. Record is generated AFTER Phase 3 — includes all intelligence layers.
5. 1:1 meetings → record in person directory. Project meetings → project directory. Account meetings → account directory (per I631 routing).
6. If prediction scorecard or continuity data isn't available (first meeting, no prep), those sections are omitted from the record.
7. Record regenerates on transcript reprocessing (overwrites previous).
8. Record path stored in `meeting_transcripts.record_path` column (new).

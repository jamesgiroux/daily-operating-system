# I644 - Canonical account truth + source references

**Area:** Backend / Accounts + Intelligence + Data Model
**Priority:** P0
**Version:** v1.1.0 Phase 5
**Effort:** Large
**Depends on:** ADR-0100, I576, existing account detail + meeting prep read model
**Source:** Glean account research review (Blackstone, 2026-03-29). Glean returned materially richer commercial, support, stakeholder, and source evidence than DailyOS currently promotes into DB-backed account truth.

---

## Problem

DailyOS is not getting the most out of Glean because the DB schema still treats too much account truth as generic intelligence instead of structured account state.

The current gaps are architectural:

- High-confidence facts and AI inference are still too easy to flatten together
- Account detail and meeting detail cannot rely on richer Glean-backed truth unless it is promoted into the DB read model
- Provenance is not precise enough for the user to verify an item outside DailyOS
- `renewal_stage` is carrying two concepts at once:
  - a timeline / proximity heuristic derived from `contract_end`
  - a commercial / CRM motion stage coming from Glean or Salesforce

This makes it hard to answer basic product questions cleanly:

- Is ARR a fact from Salesforce or an AI estimate?
- Is "approaching" a date-driven renewal window or a commercial stage?
- Which exact opportunity, ticket, call, or doc supports the value shown on the page?
- Which values are safe to feed into hero vitals, lifecycle logic, filters, and downstream PTY prompts?

If we do not split these concerns now, v1.1.0 will surface richer prose but still have weak account truth.

### JSON read audit (2026-03-30)

A full audit of all five detail pages (`.docs/audits/json-read-audit-detail-pages-2026-03-30.md`) found:

**Confirmed violations:**
1. `get_account_detail` reads `dashboard.json` from the workspace for narrative fields (overview, programs, notes)
2. `get_project_detail` reads `dashboard.json` — same pattern
3. `update_intelligence_field` and `dismiss_intelligence_item` fall back to reading `intelligence.json` from disk when the DB row is missing
4. Stale comments in `services/accounts.rs`, `services/projects.rs`, and `commands/core.rs` describe filesystem reads that no longer match the implementation

**Confirmed clean:**
- Daily briefing page: fully DB-backed
- Meeting detail page: fully DB-backed (prep_frozen_json + prep_context_json)
- Person detail page: fully DB-backed on initial load

These findings are absorbed into the ACs below.

## Solution

Adopt a 3-layer DB model for account truth.

### 1. Canonical account facts

Promote high-confidence, system-backed values into DB-backed account fields used by hero surfaces, filtering, lifecycle logic, and prompt context.

These fields should be factual, refreshable, and source-attributed:

- `arr`
- `arr_range_low`
- `arr_range_high`
- `contract_end`
- `lifecycle`
- `renewal_likelihood`
- `renewal_model`
- `renewal_pricing_method`
- `renewal_opportunity_stage`
- `support_package`
- `support_tier`
- `active_subscription_count`
- `growth_potential_score`
- `icp_fit_score`
- `primary_product`
- `customer_status`

Do not promote pure narrative or opinion into top-level account columns.

### 2. Structured account assessment

Keep richer but still structured account context in the DB-backed assessment model for rendering and downstream prep:

- expanded `contract_context`
- expanded `renewal_outlook`
- expanded `support_health`
- `relationship_assessment`
- `source_coverage`
- `current_themes`
- `technical_footprint`
- `recent_incidents`
- `compliance_context`

This remains DB-backed runtime state, not `intelligence.json` reads.

### 3. Exact source references

Add a source reference model so any rendered field or item can point back to the exact originating evidence.

Minimum shape:

```sql
CREATE TABLE entity_source_references (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_id TEXT NOT NULL,
    field_key TEXT NOT NULL,
    item_key TEXT,
    source_system TEXT NOT NULL,      -- salesforce | zendesk | gong | google_docs | p2 | slack | glean_chat
    source_kind TEXT NOT NULL,        -- fact | signal | inference
    reference_type TEXT NOT NULL,     -- account | opportunity | ticket | call | doc | post | thread
    reference_id TEXT,
    reference_title TEXT,
    reference_url TEXT,
    glean_locator_json TEXT,          -- enough to let the UI or future tools ask Glean for the same document
    snippet TEXT,
    observed_at TEXT,
    refreshed_at TEXT,
    stale_after TEXT,
    is_current INTEGER NOT NULL DEFAULT 1,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

This table is not just for labels like "via Salesforce." It is the verification layer.

### Facts vs inference

Every source-backed field/item must declare what kind of truth it is:

- `fact`: direct system-backed value, safe to promote into canonical account state
- `signal`: observed evidence that informs scoring and narrative but is not itself the canonical truth
- `inference`: AI synthesis or editorial judgment

This classification must flow into Bayesian weighting and UI rendering.

### Split renewal concepts cleanly

Keep `renewal_stage` as the date / timeline sub-state already used by lifecycle heuristics, or rename it to `renewal_window_stage` during migration if needed for clarity.

Add a separate `renewal_opportunity_stage` for CRM / Glean commercial state.

Examples:

- `renewal_stage = approaching` because contract end is within window
- `renewal_opportunity_stage = R1 - Not Started` because Salesforce says the opportunity is still early

These are related but not the same thing.

## Acceptance Criteria

### DB-first read model (from JSON audit 2026-03-30)

1. **No detail page reads `intelligence.json` under any condition.** This includes initial load, intelligence edits (`update_intelligence_field`), and dismissals (`dismiss_intelligence_item`). The filesystem fallback in `services/intelligence.rs:614-625` and `services/intelligence.rs:877-883` must be removed — if the DB row is missing, that's an error, not a reason to read from disk.
2. **No detail page reads `dashboard.json`.** Account detail and project detail narrative fields (overview, programs, notes, description, milestones) must be promoted to DB-backed columns so `get_account_detail` and `get_project_detail` become fully DB-read-only. Verified by grep: zero `read_account_json` / `read_project_json` calls on any detail page path.
3. **Stale comments cleaned up.** `services/accounts.rs:1209-1211`, `services/projects.rs:115-118`, and `commands/core.rs:17` describe filesystem reads that no longer match the implementation. Update to match reality after runtime changes.

### Canonical account facts (original scope)

4. DailyOS has a DB-backed canonical fact layer for account commercial, support, and fit fields.
5. `ARR`, `contract_end`, `renewal_likelihood`, `support_tier`, `growth_potential_score`, and `icp_fit_score` can be promoted from Glean-backed factual evidence into DB-backed account state.
6. `renewal_stage` and `renewal_opportunity_stage` are distinct fields with distinct semantics. No code path uses one as a proxy for the other.
7. Each promoted field shown on account detail or meeting detail has at least one `entity_source_references` row marking:
   - `source_system`
   - `source_kind`
   - `reference_type`
   - `reference_id` or `reference_url`
   - `observed_at`
8. A Salesforce-backed ARR is stored and rendered as a `fact`, not an inference.
9. A Glean-generated narrative such as procurement complexity or stakeholder sentiment is stored and rendered as an `inference` unless backed by explicit source evidence rows.
10. Account detail and meeting detail APIs can return both the current value and its exact source references for verification.
11. Existing lifecycle heuristics still work after migration, but commercial opportunity stage no longer has to be squeezed into the same field.

---

## Implementation Notes

- Prefer additive migrations. Do not break existing detail pages while the new fields are backfilled.
- Keep existing `data_source` conventions, but do not treat them as sufficient provenance on their own.
- The source reference model should support multiple references per field or item. Many values will be corroborated by more than one system.
- Where account-level rollup fields disagree with opportunity-level fields, keep the promoted current value plus the conflicting references. Do not hide the disagreement.

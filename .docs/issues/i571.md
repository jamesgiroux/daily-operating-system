# I571 — Surface background work status to user

**Priority:** P1
**Area:** Frontend / UX
**Version:** v1.0.0 (Phase 6)
**Depends on:** None
**Type:** Enhancement

## Problem Statement

The `heavy_work_semaphore` (`state.rs:217`) is a `Semaphore::new(1)` that serializes concurrent heavy operations: PTY subprocess calls (intel_queue), embedding inference, email enrichment, daily briefing preparation, and database restore. When a user triggers an action that needs the semaphore (e.g., "Refresh Intelligence" on an account, or "Prepare Briefing") while a background task holds it, the UI shows nothing — no spinner, no message, no indication that work is queued. The user sees a frozen button or an unresponsive click.

### Current semaphore consumers

From grep of `heavy_work_semaphore` usage:

| File | Operation | User-triggered? |
|------|-----------|----------------|
| `intel_queue.rs:368` | Entity intelligence enrichment | Yes (refresh button) |
| `services/intelligence.rs:61` | Manual intelligence refresh | Yes (refresh button) |
| `commands/projects_data.rs:165` | Database restore from backup | Yes (settings action) |
| `google.rs:1185` | Email enrichment during sync | No (background) |
| `processor/embeddings.rs:155` | Embedding batch processing | No (background) |
| `hygiene.rs:1633` | Proactive hygiene AI calls | No (background) |
| `prepare/orchestrate.rs:1324` | Daily briefing preparation | Indirect (auto or manual) |

When a user clicks "Refresh" on an account and the semaphore is held by email enrichment, nothing happens for 10-60 seconds. The user clicks again. Still nothing. They think the app is broken.

## Solution

Use Tauri's event system to emit background work status events that the frontend can display.

### Backend: Emit status events

Add a helper on `AppState` (or as a standalone function) that emits a Tauri event when:
1. A task starts waiting for the semaphore.
2. A task acquires the semaphore and begins work.
3. A task completes and releases the semaphore.

Event payload:
```rust
#[derive(Clone, Serialize)]
struct BackgroundWorkStatus {
    /// What's happening: "waiting", "running", "completed"
    status: String,
    /// Human-readable label: "Updating account intelligence", "Preparing daily briefing"
    label: String,
    /// Unique ID for this work item (for matching start/complete events)
    work_id: String,
}
```

Emit via `app_handle.emit_all("background-work-status", payload)`. The `app_handle` is available in command handlers and can be stored in `AppState` during setup.

### Key emit points

- **`services/intelligence.rs:61`** (user-triggered refresh): Emit "waiting" before `semaphore.acquire()`, "running" after acquire, "completed" after the enrichment finishes.
- **`commands/projects_data.rs:165`** (database restore): Emit "running" after acquire.
- **`prepare/orchestrate.rs:1324`** (briefing prep): Emit "running" after acquire.

Background-only consumers (email enrichment, embeddings, hygiene) should emit "running"/"completed" but not "waiting" — the user didn't initiate them, so a waiting message would be confusing.

### Frontend: Display status

Add a lightweight toast or status indicator that listens to `background-work-status` events:

1. **Waiting state**: Show a toast: "Updating intelligence in the background -- your request will start shortly." Auto-dismiss when the status transitions to "running" or after 30 seconds.
2. **Running state** (user-triggered only): Optional subtle indicator in the folio bar or near the triggering button. Not blocking — the user can navigate away.
3. **Completed state**: Dismiss any active toast. Optionally show a brief "Done" confirmation for user-triggered work.

Use the existing toast/notification infrastructure if one exists, or a minimal floating indicator anchored to the bottom of the viewport.

### Vocabulary

Per ADR-0083: Never use "intelligence", "enrichment", or "semaphore" in user-facing strings. Use:
- "Updating account insights" (for account intelligence refresh)
- "Preparing your briefing" (for daily prep)
- "Processing in the background" (generic fallback)

## Acceptance Criteria

1. When a user clicks "Refresh" on an account while background enrichment is running, a toast appears within 500ms explaining that the request is queued. The toast includes a human-readable description of what's happening.
2. The toast auto-dismisses when the user's request starts executing (semaphore acquired) or after 30 seconds (timeout).
3. Background-only tasks (email enrichment, embedding processing) do NOT show "waiting" toasts — they run silently.
4. The status event payload uses product vocabulary (ADR-0083): no "intelligence", "enrichment", "semaphore", or "entity" in user-facing strings.
5. The indicator does not block user interaction — the user can navigate to other pages while work is queued.
6. `cargo test` — all pass.
7. `cargo clippy -- -D warnings` — clean.
8. `pnpm tsc --noEmit` — clean.

## What This Does NOT Do

- Does not change the semaphore concurrency limit (remains 1) — that's a separate tuning decision.
- Does not add progress bars or percentage indicators — the operations are not incrementally trackable.
- Does not surface status for non-semaphore background work (e.g., Google Calendar sync, which uses its own polling loop).
- Does not implement cancellation of queued work — that's a future enhancement.

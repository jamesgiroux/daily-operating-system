# I433 — Ollama Provider — Local LLM Support

**Status:** Open
**Priority:** P1
**Version:** 1.1.0
**Area:** Backend / Intelligence

## Summary

An `OllamaProvider` that calls a locally-running Ollama instance for AI enrichment. Completes the local-first story: when Ollama is selected, no data leaves the device at any point — calendar events, emails, entity intelligence, and AI inference are all on-machine. Privacy-first users and users on air-gapped or restricted networks can run DailyOS without any external AI dependency.

The provider calls Ollama's OpenAI-compatible chat endpoint (`http://localhost:11434/api/chat`). Because `OllamaProvider` and `OpenAIProvider` (I434) share the same HTTP client underneath, they are built together: same `reqwest` client, same JSON request shape, different base URL and auth. The prompt format used is the completion-API-style template (system prompt + user message + JSON schema), not the Claude Code workspace-aware format.

## Acceptance Criteria

1. `OllamaProvider` implements `IntelligenceProvider` (I432) in `src-tauri/src/intelligence/provider.rs`. It calls `http://localhost:11434/api/chat` using Ollama's OpenAI-compatible chat API. The request body follows the OpenAI chat completions schema (`messages`, `model`, `stream: false`). The response is parsed from the `choices[0].message.content` field.

2. Settings → AI Provider shows "Ollama (local)" as a selectable option. Selecting it reveals:
   - A model selector, populated by querying `GET http://localhost:11434/api/tags` and listing installed models by name.
   - A "Test connection" button that makes a minimal call to the tags endpoint and reports either "Connected — N models available" or a connection error with the raw error message.
   - If no models are installed, the selector shows a disabled state with an inline message pointing to `ollama.ai` for install instructions.

3. Tier-to-model defaults: Synthesis tier uses the largest available model (or `llama3.3:70b` if installed), Extraction and Mechanical tiers use `llama3.2:3b`. The user can override both via the model selector. Configuration is stored in the app's settings DB, not in the workspace config file.

4. If Ollama is not running when a provider call is made, `OllamaProvider::complete()` returns `Err(ProviderError::ProviderUnavailable)`. The fallback chain in `intel_queue.rs` catches this, logs a WARN with the provider name and error, and retries the call via `ClaudeCodeProvider`. No enrichment is silently skipped.

5. End-to-end verification: with Ollama selected and `llama3.3:70b` running, trigger entity enrichment for a real account (via Settings → Entities → "Refresh intelligence" or by emitting a signal that triggers the intel queue). After the enrichment cycle completes, `SELECT executive_assessment FROM entity_intel WHERE entity_id = ?` returns a non-null, non-empty string. Quality may differ from Claude; correctness of the JSON structure is the requirement.

6. Provider isolation: with Ollama selected, `SELECT count(*) FROM signal_events WHERE source = 'claude_code'` does NOT increase after an enrichment cycle completes. Verify by recording the count before triggering enrichment, running enrichment, and confirming the count is unchanged.

7. Prompt templates: `intelligence/prompts.rs` contains a `completion_api_prompt(entity: &EntityContext) -> (String, String)` function returning `(system_prompt, user_message)`. `OllamaProvider::complete()` uses this function, not the Claude Code prompt builder. Verify: `grep -n "completion_api_prompt" src-tauri/src/intelligence/provider.rs` returns a match.

## Dependencies

Blocked by I432. The `IntelligenceProvider` trait must exist before `OllamaProvider` can implement it.

Can be developed in parallel with I434 (OpenAIProvider) once I432 lands — they share the HTTP client infrastructure.

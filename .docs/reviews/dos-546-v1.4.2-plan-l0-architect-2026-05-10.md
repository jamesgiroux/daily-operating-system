# L0 Architecture Review — v1.4.2 (Personal Intelligence Engine: WordPress Foundation)

**Reviewer:** architect-reviewer
**Date:** 2026-05-10
**Scope:** Plan packet — 01-project-description.md, 02-issues.md, 03-wave-plan.md (22 issues across W0–W6).
**Cross-checked against:** ADR-0102 (§7.1, §7.6 + 2026-05-10 amendments), ADR-0111 §8, ADR-0128, ADR-0129, ADR-0130 (with W0-A amendments pending), ADR-0105, ADR-0108, ADR-0125.

---

## Verdict

**APPROVE WITH NAMED CONDITIONS.** The plan is architecturally sound, substrate-coherent, and correctly preserves the load-bearing invariants: substrate-owned authorship, two-level enforcement (allowed_actors + required_scopes + mcp_exposure), audit attribution via SurfaceClient instance identity + wp_user_id, and signal/invalidation propagation through the existing v1.4.1 signal infrastructure. Wave ordering is correct and the integration seams are clean. The conditions below are blocking findings that must resolve at L0 cycle-1 (before W0 kickoff), not deferrals to consuming waves.

## Category 1 — Substrate coherence

### Pass

* **Substrate-owned composition is preserved.** W1-E's CI lint asserting that only ability code constructs `Composition` is the load-bearing structural gate. ADR-0130 §1 + §5 names this explicitly; the plan enforces it at compile/grep time, not at code review. WP plugin is forbidden from constructing Composition or writing claims directly (W5-A CI lint).
* **Two-level enforcement is canonical, not bridge-private.** W1-B promotes `required_scopes` + `mcp_exposure` onto `AbilityPolicy` itself, not just on a SurfaceClientBridge config. This matches the cycle-2 amendment of ADR-0102 (the substrate is the enforcement point; every surface inherits). ADR-0111 §8 "Both must pass" is structurally implementable: actor class check at the policy gate, scope check at the bridge.
* **Audit attribution round-trip is structurally sound.** W6-A's CI lint that every `Actor::SurfaceClient` log site carries both `actor_instance` and `wp_user_id` makes the invariant unforgeable; combined with W1-A's mandatory audit emission on SurfaceClient operations and W2-C's pairing audit events, the forensic chain is complete.
* **Signal/invalidation propagation reuses v1.4.1 W1-B/E.** No new signal routing layer is introduced. Compositions re-render on claim-invalidation signals via the existing policy registry. W4-B's `claim_version` increment emits the existing signal types; W4-C tamper events emit a new signal class but route through the existing infrastructure.
* **Provenance budget discipline.** ADR-0108's 64KB envelope cap is respected: W1-E ships `ProvenanceRef { invocation_id, field_path }` so blocks reference the single envelope on `AbilityOutput<T>` instead of re-embedding. The plan explicitly cites this constraint in W1-E acceptance and in W4-C ("Signature lives outside the ProvenanceEnvelope per ADR-0108 64KB cap").

### Finding A1 (BLOCKING) — `mcp_exposure` type mismatch between ADR-0102 amendment and v1.4.2 plan

ADR-0102's 2026-05-10 amendment line declares `mcp_exposure: bool`. ADR-0111 §8 echoes `mcp_exposure: false` (bool). The v1.4.2 plan (W1-B + Phase 0 artifact 05 reference) ships a tri-state `McpExposure: None | MetadataOnly | Invocable`. The tri-state is the correct design — it captures the "describe-but-don't-invoke" tier the WP MCP server needs — but the ADR text is now stale and W1-B will land code that contradicts the ADR.

**Resolution:** W0-A must extend its scope from "ADR-0130 amendments" to "ADR-0130 amendments + ADR-0102 §7.1 amendment harmonizing `mcp_exposure` to the tri-state enum." Alternatively, lift it into a separate W0 issue. Either way, the ADR text must lead the code, not trail it. This is a Linear-amendment, not scope expansion: the canonical schema is the substrate contract, and the contract is presently inconsistent across two ADRs.

### Finding A2 (BLOCKING) — `mcp_exposure: None` versus "WP MCP server allowlist defaults to Invocable ∩ Read | Transform"

W3-C names two distinct gates against the default WP MCP server exposing substrate abilities: (1) `mcp_exposure: None | MetadataOnly | Invocable` filter; (2) read-mostly allowlist intersection. The plan describes both but doesn't name the precedence rule for the case where they disagree (`mcp_exposure: Invocable` but ability `category: Maintenance` — the inventory says expose, the allowlist says hold). Without a precedence rule, two CI invariants can pass while a Maintenance ability leaks.

**Resolution:** W1-C inventory schema must encode `mcp_exposure` as the necessary-but-not-sufficient gate; W3-C allowlist construction is the sufficient gate. State this precedence in the W3-C acceptance criteria as a single sentence and add a negative fixture that publishes `mcp_exposure: Invocable + category: Maintenance` and asserts the WP MCP server does NOT enumerate it. This is one acceptance line + one fixture; it is not scope expansion.

### Finding A3 (NON-BLOCKING but record) — `SurfaceScope` vocabulary not yet enumerated

ADR-0111 §8 illustrates scopes (`read.account_overview`, `submit.feedback`, `manage.pairing`) but does not enumerate the canonical set. W1-B's `ScopeSet`/`SurfaceScope` enum becomes the canonical enumeration at substrate level; the plan accepts the default of an empty `required_scopes` for v1.4.0/v1.4.1 abilities. This is fine for backward compat. **Track that the per-ability scope population work — when each ability declares its scopes — is parked to consuming-ability issues (v1.4.3+).** The plan correctly does not try to do this in v1.4.2; just make sure the implicit deferral is named in the project description's Non-goals (it is not currently). Add a one-liner under Non-goals: "Per-ability `required_scopes` population beyond v1.4.0/v1.4.1 defaults — lands per-ability in consuming releases."

## Category 2 — Wave ordering

### Pass

* **W0 → W1 → W2 strict-merge ordering is correct.** ADR-0130 amendments (W0-A) gate W1-E (Composition types). W1-A + W1-B form a stage-1 gate before W1-C + W1-E. W2's strict sequential chain (transport → HMAC → pairing → rate-limit) is the only correct ordering — each piece is meaningless without its predecessor.
* **W4 internal staging is correct.** W4-B (concurrency contract) gates W4-A (first block) because the block carries watermarks. W4-C signatures include `composition_version` from W4-B. W4-E nonce binding includes `composition_version`. W4-D fallback rules are independent of watermark (read-side substrate rule).
* **W6 fan-out is correct.** Audit attribution and clean-machine validation are non-overlapping.

### Finding B1 (BLOCKING) — W3-C unsatisfiable as ordered

W3-C ("Custom MCP server + low-cap user") depends on `Issue: Promote AbilityPolicy to canonical schema` (W1-B) AND `Issue: Ability-surface inventory format` (W1-C). Both are listed dependencies. But the wave plan §"W3 internal staging" says W3-C "runs in parallel with W3-B once W3-A merges." W3-A (plugin skeleton) does not produce the inventory or the policy schema; those are W1 outputs already merged. The plan reads correctly, but the W3 merge-gate matrix needs to make this explicit: **W3-C depends on W2 fully merged AND W1-B + W1-C merged**, not just on W3-A.

The current wording "W3-B + W3-C run in parallel after W3-A merges" understates the dependency: W3-B also needs W2 fully merged (already stated); W3-C needs W2 fully merged plus W1 schema/inventory (under-stated). A code agent reading this risks starting W3-C before W2 closes.

**Resolution:** Tighten W3 §"Internal staging" to read: "W3-A gates W3-B and W3-C. W3-B additionally depends on W2 full merge. W3-C additionally depends on W2 full merge + W1-B + W1-C merges (which precede W2 in the default merge-gate-on-prior-wave rule, but state explicitly because the dependency is load-bearing)." One sentence, blocks the misread.

### Finding B2 (BLOCKING) — W2-D rate-limit matrix calibration depends on W4 ability dispatch cost, not the reverse

W2-D's per-ability and per-scope-class budgets (axes 4 + 5 per Phase 0 artifact 09) are quoted as concrete numbers, but those numbers are sensitive to actual ability dispatch cost, which becomes empirical only after W4-A's first block invokes through `SurfaceClientBridge`. W2-D is currently positioned to ship budgets that may need re-tuning after W4 lands its first realistic workload. This is a smell that the rate-limit matrix's L3 wave adversarial should expect a calibration follow-up.

**Resolution:** Add to W2-D acceptance criteria: "Budgets are config-driven; W4 L3 wave adversarial includes a calibration pass that may amend axis-4 and axis-5 thresholds based on real block-dispatch cost. Amendments land via the v1.4.2 maintenance project, not via a W2 re-open." This is the path-α maintenance treatment from the user's memory: don't loop W2; transfer the residual to maintenance with a named ticket. State it now.

### Finding B3 (NON-BLOCKING) — W5-C negative-fixture catalog ownership boundary

W5-C says "Individual fixtures are owned by their producing issue (e.g., HMAC negative fixtures live in the HMAC issue), but the consolidation … live here." This is the right model. But the wave plan lists W5-C as a single agent with no named cross-wave coordinator role. In practice, W5-C closes only after W1–W4 fixtures merge AND the consolidation pass runs. The wall-clock estimate of 1 week for W5 includes this assumption silently.

**Resolution:** Add to W5 merge gate the explicit statement: "W5-C is the last issue to merge in W5; consolidation runs after W5-A and W5-B merge. Bundle 19-23 release-gate integration is the closing act." No new work; explicit ordering only.

### Finding C1 (NON-BLOCKING) — W3-A WP 7.0 client-side `executeAbility()` deferral

## Category 4 — Missing architectural concerns

### Finding D1 (BLOCKING) — Markdown filesystem projection writer is named but not waved

The project description Scope §"Three-view consistency" lists "Markdown filesystem ↔ WordPress DB write path — substrate of record, WP DB as projection, filesystem as durable archive." DoD §3 requires "edit-in-WP, edit-in-markdown, and concurrent-tab scenarios from Phase 0 artifact 02 negative fixtures all pass." W4-C tamper detection covers markdown signing/verification on read. But **no wave actually owns the markdown projection writer**. W4-C verifies markdown projections but does not author them; the project relationship table says "v1.4.6 — Workspace Memory Refactor will own the markdown ↔ WordPress DB reconciliation that this project bootstraps."

This is the "wire existing substrate, don't ship empty-branch claiming producer is future work" trap from the user's memory. If markdown writes ship in v1.4.6, then `edit-in-markdown` cannot be demonstrated as a DoD criterion in v1.4.2. Either:

(a) v1.4.2 ships a minimal markdown writer (substrate emits projection on claim write — already exists in v1.4.1's filesystem archive path? grep needed), OR
(b) DoD §3 drops the `edit-in-markdown` scenario, OR
(c) "Edit-in-markdown" is restricted to "read-only-verification of out-of-band edits" — i.e., the writer is upstream substrate code that already exists, and W4-C is purely consumer-side.

**Resolution:** L0 cycle-1 grep the substrate for an existing markdown projection writer. If it exists, name it in W4-C dependencies and confirm DoD §3 is achievable. If it does not exist, either add a sub-issue to W4 owning the writer, or amend DoD §3 to scope "edit-in-markdown" to "external edits detected and quarantined" (which is what W4-C actually does). Currently the plan reads as if the writer exists; verify before W1 kickoff.

### Finding D2 (BLOCKING) — `dailyos/account-overview` *ability* existence

W4-A acceptance line: "`dailyos/account-overview` ability — assumed to exist (if not, scope a minimal version in this issue OR file a sub-ticket; at L0 plan time grep for the ability and decide)." This is a deferred dependency on the producer side of the producer/renderer split. Per the user's memory ("wire existing substrate, don't ship empty-branch claiming producer is future work"), the W4-A L0 must close this before W4-A starts. If the ability doesn't exist, the producer-renderer split has no producer in v1.4.2 and the foundation cannot prove its load-bearing claim.

**Resolution:** Grep for `dailyos/account-overview` or `get_account_overview` / `prepare_account_overview` in `abilities-runtime/src/` and `src-tauri/src/abilities/` during L0 cycle-1. If absent, add a W4 sub-issue (W4-A.1) to author a minimal Composition-returning account-overview ability. Do not let W4-A start without the producer in scope.

### Finding D3 (NON-BLOCKING) — Sensitivity tier on `ScopeSet`

ADR-0125 sensitivity tiers (`Public | Internal | Confidential | UserOnly`) appear in W4-D fallback projection rules. ADR-0108 sensitivity rendering composes with `mcp_exposure` per W1-B Privacy Rendering note. But `SurfaceScope` (W1-A/B) does not name a sensitivity-tier component. A scope grant like `read.account_overview` does not say at what tier; an Internal-tier account-overview claim could leak through a Public-tier surface client via a scope that doesn't constrain sensitivity.

This is probably handled in the ability body via ADR-0108 rendering rules, but the load-bearing position is fuzzy. Worth confirming in W1-B L0 plan that the per-request sensitivity gate runs at render time (not scope grant time), so the Privacy Rendering composability claim is real and not aspirational.

### Finding D4 (NON-BLOCKING) — Skillify / abilities-as-WP-plugins boundary

The project description explicitly defers Skillify (`DOS-540`) to a later release. The W1-C inventory format is the canonical format Skillify will consume; if Skillify needs additional fields (per-ability code-package binding, version pinning, signing), they must be additive-only on the inventory schema, not breaking. The plan does not assert this evolvability constraint.

**Resolution:** Add a one-liner to W1-C acceptance: "Inventory schema is additive-only; consuming releases (Skillify, v1.4.7 MCP v2) extend without breaking the v1.4.2 contract." This is evolutionary-architecture discipline, not scope.

## Category 5 — L3/L4/L5 invocation points

### Pass

* **L3 on W1 and W4 is correct.** Both waves integrate multiple parallel lanes (5 agents each) on load-bearing substrate. L3 codex-challenge + architect-reviewer + Suite S/P is appropriate. W2 is sequential (no parallel lanes within the wave) but W2's merge gate names L3 too — which is correct because W2 integrates transport+HMAC+pairing+rate-limit into the trust-boundary as one composite surface even though the lanes are serial.
* **L4 on W3, W4, W5, W6 is correct.** W3 plugin admin UX, W4 first block, W5 feedback + theme, W6 clean-machine flow are all user-facing. W3's L4 (`/qa-only` against plugin install + pairing) is a light-touch correct treatment. W5's promotion to `/qa` (not `/qa-only`) on the feedback flow is correct because feedback is a substrate-mutating user gesture.
* **L5 on W6 is correct.** Drift check against v1.4.2 DoD verbatim is the right closeout. The wave plan also names L5 at W1 and W4 merge gates, which is heavier than the standard playbook but justified for a foundation release (the contract surfaces are load-bearing for v1.4.3+).
* **/cso saturation is correctly heavy on W1-A/B/E, W2-all, W3-B/C, W4-C/E, W5-A.** Every trust-boundary diff gets /cso at both L0 plan and L2 diff. This matches the project's positioning as "surface foundation IS a trust-boundary release."

### Finding E1 (NON-BLOCKING) — L3 on W2 is implicit, should be explicit

The wave plan §"Review ladder — which L levels apply" table says "L3 applies to waves that integrate multiple parallel lanes (W1, W4)" — but W2's merge gate text names "L0 → L2 → L3 → L5 cleared." The table and the merge gate disagree. W2 is sequential, but transport+HMAC+pairing+rate-limit composes into one integrated trust surface; treating it with L3 (Suite S full re-run + Suite P transport latency budget) is correct. The L-level matrix table should add W2 to "Applies."

**Resolution:** One-line table fix.

### Finding E2 (NON-BLOCKING) — Pilot wave skip rationale is sound but artifact 11 risk under-named

The wave plan §"Amendments" says "No pilot wave. v1.4.0 W0 + v1.4.1 startup proved the system." This is correct — the protocol is mature. But artifact 11 (`editable-composition-overlay.md`) is still missing as of plan draft, and W4-A relies on artifact 07 fallback covering the gap if 11 doesn't land by W4 start. This is correctly named as L0 open question #1. The risk is named but the trigger for re-evaluation is "if still missing by W4 start" — at which point W4 is about to start and the cost to slip is high. Suggest moving the trigger forward: re-evaluate at W2 close.

## Final summary

The v1.4.2 plan is **structurally correct** and **substrate-coherent**. The architectural invariants are preserved with CI-gated enforcement (not code-review trust). Wave ordering is sound; the integration seams between W1's contract, W2's transport, W3's plugin, and W4's first block are clean. Acceptance criteria are substantive across the spot-checked subset. The /cso saturation and L3/L4/L5 invocation points are appropriate for a foundation release.

**Blocking findings (must close in L0 cycle-1, before W0 kickoff):**

1. **A1** — ADR-0102 `mcp_exposure` type drift (bool vs tri-state); harmonize via W0-A scope extension or a sibling W0 issue.
2. **A2** — `mcp_exposure` vs allowlist precedence rule in W3-C; one acceptance line + one negative fixture.
3. **B1** — W3-C dependency under-statement; tighten wave plan §"W3 internal staging."
4. **B2** — W2-D rate-limit budgets are post-hoc-calibratable; declare the maintenance-project transfer up front.
5. **D1** — Markdown projection writer ownership unclear; grep substrate or scope a sub-issue or amend DoD §3.
6. **D2** — `dailyos/account-overview` *ability* (producer) must be grep-verified in L0 cycle-1; sub-issue if absent.

**Non-blocking findings (record + apply in consuming waves):**

7. **A3** — Per-ability `required_scopes` population is deferred; name the deferral under project Non-goals.
8. **B3** — W5-C closes after W5-A/B merge; state explicit closing order.
9. **C1** — WP 7.0 client-side `executeAbility()` schema fate decided in W3-B L0; commit to a side.
10. **D3** — Sensitivity-tier composability across `SurfaceScope` + ADR-0108 + ADR-0125 — confirm render-time gate in W1-B L0.
11. **D4** — W1-C inventory schema additive-only constraint; one acceptance line.
12. **E1** — L-level matrix table inconsistency with W2 merge gate; one-line fix.
13. **E2** — Artifact 11 re-evaluation trigger should be W2 close, not W4 start.

The plan does not over-engineer, does not strip foundational substrate, and does not defer load-bearing work to "future producer" gestures. The producer/renderer split is real because W1-E ships substrate types AND W4-A consumes them; if D2 confirms the ability exists or scopes its minimal version, the load-bearing claim is empirically verifiable at W6 close.

**Verdict:** APPROVE with the six blocking findings resolved at L0 cycle-1. The seven non-blocking findings are recordable amendments that can be applied as the consuming waves are planned, not blockers.

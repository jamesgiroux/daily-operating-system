## Top Findings

1. **AbilityPolicy is not actually contract-locked.**
   ADR-0102 defines `mcp_exposure: bool`, while the issue asks for `None | MetadataOnly | Invocable`, and Phase 0 adds `client_side_executable` as a separate field. The wave plan then reopens whether to reuse `mcp_exposure` for client-side execution. See [ADR-0102 §7.1](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/decisions/0102-abilities-as-runtime-contract.md:223), [02-issues W1-B](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:308), [phase-0/05](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/phase-0/05-ability-surface-inventory.md:121), [03-wave-plan open Q2](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/03-wave-plan.md:573).
   **Impact:** W1 cannot implement a stable security contract. MCP discovery, WP client execution, and scope filtering can all diverge.

2. **The core demo block depends on an ability that is only "assumed to exist."**
   ADR-0130 says abilities produce compositions and WordPress only renders them, but the `dailyos/account-overview` block issue punts the actual producing ability to "grep and decide." See [ADR-0130 §5](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/decisions/0130-surface-independent-composition-contract.md:110), [02-issues W4-A](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1157).
   **Impact:** The release's load-bearing proof can fail at W4 because no issue owns the composition-producing account overview ability.

3. **Artifact 11 is missing, but downstream artifacts already rely on it.**
   The plan says artifact 11 can be absent and fallback rules will cover the gap, but Phase 0 block and plugin artifacts use artifact 11 for projection snapshots, composition version behavior, claim refs, and edit-to-feedback semantics. See [03-wave-plan](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/03-wave-plan.md:571), [phase-0/14](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/phase-0/14-gutenberg-block-account-overview.md:102), [phase-0/14](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/phase-0/14-gutenberg-block-account-overview.md:311).
   **Impact:** The edit/correction path is under-specified. "Fallback projection" is not a substitute for editable projection semantics.

4. **Three-view consistency is overpromised.**
   The project promises WP DB, substrate, and markdown stay consistent, including edit-in-markdown. But the project also says full markdown ↔ WP reconciliation belongs to v1.4.6, and W4 tamper detection explicitly excludes reconciliation tooling. See [01-project-description User Outcome](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/01-project-description.md:48), [01 relationship](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/01-project-description.md:131), [02-issues W4-C](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1287).
   **Impact:** The DoD can only pass by redefining "consistent" as "tamper detected," not actual edit propagation across three views.

5. **Tamper detection contradicts its own Phase 0 crypto contract.**
   The issue says projection signatures are HMAC-SHA256 with live runtime-key verification. Phase 0 says Ed25519 only, no HMAC, public verification keys may live in WP/markdown, and SurfaceClient verifies without a runtime round trip. See [02-issues W4-C](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1284), [phase-0/03](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/phase-0/03-tamper-detection-contract.md:157), [phase-0/03](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/phase-0/03-tamper-detection-contract.md:171).
   **Impact:** Implementers can satisfy the issue while violating the design artifact, breaking offline/fast render verification and making signature portability worse.

6. **Audit attribution lands too late.**
   ADR-0111 requires every SurfaceClient operation to log instance identity, but audit schema/plumbing is W6, after pairing, HMAC, rate limits, MCP, block rendering, nonce, and feedback all emit audit events. See [ADR-0111 §8](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/decisions/0111-surface-independent-ability-invocation.md:189), [02-issues W6-A](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1726), [02-issues W6-A deps](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1760).
   **Impact:** W6 becomes a cross-cutting rewrite wave, not an independent closeout. Earlier negative fixtures cannot honestly assert audit behavior.

7. **Clean-machine validation does not match the promised bundle.**
   The user outcome says the bundle includes Studio + plugin + theme + runtime. The W6 issue only verifies Studio is reachable, and if no WP install exists it prints "install Studio first." See [01-project-description User Outcome](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/01-project-description.md:41), [02-issues W6-B](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1797), [02-issues W6-B edge case](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1829).
   **Impact:** The ≤15 minute install claim can be gamed by excluding Studio installation time. That does not prove the free-tier onboarding claim.

8. **The release gate drops its own audit fixture bundle.**
   The issue defines bundles 19 through 24, including audit attribution, but the wave plan and release gate only require bundles 1 through 23. It also says "five bundles minimum" after naming six. See [02-issues W5-C](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/02-issues.md:1668), [03-wave-plan release gate](/Users/jamesgiroux/Documents/dailyos-repo/worktrees/dos-546-wp-studio-spike/.docs/plans/dos-546/v1.4.2-project/03-wave-plan.md:492).
   **Impact:** Audit attribution can miss the mandatory hermetic gate.

## Reasoning

The direction is plausible. This plan is not.

It claims to freeze the foundation, but key contracts are still mutable: AbilityPolicy shape, client-side execution, artifact 11, WP API hooks, tamper crypto, audit schema. It claims three-view consistency, but punts the reconciliation work that would make that true. It claims a clean-machine bundle, but assumes Studio may already exist. It claims release-gate rigor, but the fixture bundle list does not close over its own issues.

This should be refactored before Linear creation: freeze W0 contracts first, add an owned account-overview composition ability, resolve artifact 11 or cut editable behavior, move audit schema before trust-boundary waves, and split clean install/theme/MCP into separate gates unless they are truly required for v1.4.2.

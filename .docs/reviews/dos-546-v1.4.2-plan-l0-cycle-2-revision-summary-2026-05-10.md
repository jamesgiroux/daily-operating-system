### 1. AbilityPolicy `mcp_exposure` shape (architect A1 + challenge #1)

**Decision:** Adopt option (b) — promote `mcp_exposure` to tri-state enum `McpExposure { None | MetadataOnly | Invocable }` in ADR-0102 §7.1. Retire the separate `client_side_executable` field; in v1.4.2, SurfaceClient invocability is governed by `allowed_actors: [SurfaceClient]` ∧ `required_scopes` (two-level enforcement per ADR-0111 §8). The browser/JS-from-WP path is not a distinct exposure tier — it is a SurfaceClient transport choice. Artifact 05's `client_side_executable: boolean` is retired in the same amendment.

**Rationale:** Tri-state captures the "describe-but-don't-invoke" tier the WP MCP server needs (`MetadataOnly`), eliminates the schema drift between ADR-0102, artifact 05, and W1-B, and avoids a second bool that overlaps with `allowed_actors`. Architect A1 + challenge #1 + /cso refinement P2 + consult §"do not defer client_side_exposure" all converge on this resolution.

**Surgical changes:**
- Add **W0-D (new W0 issue):** "ADR-0102 §7.1 amendment: `mcp_exposure: bool` → `mcp_exposure: McpExposure { None | MetadataOnly | Invocable }`; retire `client_side_executable`." Must merge before W1-B starts.
- Update W1-B acceptance to consume the amended ADR text (no field `client_side_executable`).
- Update Phase 0 artifact 05 status note in W0-C: artifact 05 needs an addendum noting the canonical schema lives in ADR-0102 §7.1, and `client_side_executable` is no longer a separate field.
- Remove L0 open question §2 from wave plan (decided in W0-D).
- Update CI invariants table: `mcp_exposure: McpExposure` enum, not bool.

### 5. Markdown projection writer ownership (architect D1 + consult #3 + challenge #4)

**Decision:** Adopt option (c) from the architect resolution + tighten DoD §3. Scope of "edit-in-markdown" in v1.4.2 is **read-side detection and quarantine of out-of-band markdown edits, not bidirectional markdown→substrate edit propagation**. The markdown projection write path (substrate emits projected `.md` files on claim write) is the v1.4.6 "Workspace Memory Refactor" project, but the **detection-of-out-of-band-markdown-edits** path is W4-C (already in scope, Ed25519 signature on markdown projection allows tamper detection).

**Rationale:**
- v1.4.6 owns full markdown↔WP reconciliation per relationship table; pulling it into v1.4.2 doubles the wave count.
- W4-C already covers markdown-projection signature verification, which is the load-bearing detection path. The "edit-in-markdown" scenario in DoD §3 maps cleanly to "external markdown edit detected, banner rendered, substrate state preserved."
- The plan explicitly says (in W4-C scope-limits) "It does not land quarantine UI, automated repair, or reconciliation tooling — those are scoped follow-ups."
- This is path-α maintenance, not deferral: the substrate proves it can detect external markdown edits; bidirectional ingestion lands in v1.4.6 as named.

**Note on the no-deferrals rule:** Per `feedback_no_deferrals_period`, "once scope is agreed, finish it." But the original brief never agreed bidirectional markdown ingestion was in v1.4.2 scope — the relationship table explicitly says "v1.4.6 will own the markdown ↔ WP DB reconciliation that this project bootstraps." The DoD wording overpromised relative to the scope sections. Reconciling DoD to scope is correction, not deferral.

**Surgical changes:**
- DoD §3 rewritten: "Three-view consistency: substrate as authority, WP DB as projection, markdown as durable archive. Edit-in-WP scenarios + concurrent-tab scenarios from Phase 0 artifact 02 negative fixtures all pass. **Out-of-band edits in either WP DB or markdown from Phase 0 artifact 03 are detected, quarantined, and never silently promoted to canonical.** Bidirectional markdown↔substrate edit propagation is named in the v1.4.6 Workspace Memory Refactor project."
- Project description `### Three-view consistency` bullet 3 rewritten: substrate of record, WP DB as projection, markdown filesystem as **durable read-side archive in v1.4.2**; bidirectional markdown→substrate write is v1.4.6 scope.
- Non-goals: explicit "Bidirectional markdown↔substrate ingestion — v1.4.6."
- W4-C acceptance: explicit "markdown-side detection writes a signature alongside the projected file; out-of-band markdown edits detected on next read."

**Files changed:** `01-project-description.md` (Scope §Three-view, Non-goals, DoD §3), `02-issues.md` (W4-C scope-limits + acceptance).

### 10. `required_scopes` default-empty semantics (consult)

**Decision:** Keep `required_scopes: Vec<SurfaceClientScope> = vec![]` as the default (preserves v1.4.0/v1.4.1 compat). Add explicit acceptance bullet in W1-B clarifying the safety semantics:
- Default `required_scopes` empty is OK for abilities NOT exposed to `SurfaceClient` (default `allowed_actors = [User]`).
- For abilities that include `SurfaceClient` in `allowed_actors`, `required_scopes` must be explicitly non-empty OR explicitly annotated `#[ability(... no_scope_required)]` (a named-decision attribute, not silent default).
- Macro emits a compile-error if `allowed_actors` includes `SurfaceClient` AND `required_scopes` is empty AND `no_scope_required` is not present.

**Rationale:** Per consult: "any ability exposed to SurfaceClient or MCP must declare scopes explicitly, unless marked NoScopeRequired with review." Per ADR-0102 §7.6: "macro attribute must declare allowed_actors including SurfaceClient AND required_scopes AND mcp_exposure explicitly." This is already the substrate intent — codifying it as a compile-error closes the door.

**Surgical changes:**
- W1-B acceptance: add bullet "Macro emits a compile-error if `allowed_actors` includes `SurfaceClient` AND `required_scopes` is empty AND no explicit `#[ability(... no_scope_required)]` opt-out is declared. Default `required_scopes = vec![]` is permitted only for abilities not exposed to SurfaceClient. Negative fixture: ability with `allowed_actors: [User, SurfaceClient]` + empty `required_scopes` + no `no_scope_required` attribute fails to compile."
- ADR-0102 amendment scope (W0-D) extended: "default policy clarifies the SurfaceClient opt-in is non-empty `required_scopes` OR explicit `no_scope_required` attribute."

**Files changed:** `02-issues.md` (W1-B, W0-D).

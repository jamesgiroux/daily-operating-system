## Finding-by-finding resolutions

### 1. mcp_exposure collapse vs separation — REVERSE cycle-2 collapse (codex-challenge CRITICAL #1)

**Decision:** **Restore `mcp_exposure` and `client_side_executable` as two independent fields.** Cycle-2 had collapsed them under architect A1; cycle-3 reverses after re-reading Phase 0 artifact 05 lines 383-412.

**Rationale:** Artifact 05 is unambiguous: *"Keep `mcp_exposure` and `client_side_executable` separate. They govern different trust boundaries and should not be collapsed into one overloaded exposure enum… A WordPress Studio block may need to invoke an ability to hydrate a composition block even when that ability should never be listed as an MCP tool. Conversely, an MCP tool may be invocable by an agent while the WordPress client only renders returned data and should not call the ability directly."* The cycle-2 collapse pushed WP renderability toward MCP exposure, which is exactly the rigidity Phase 0 warned against. The codex-challenge finding stands; cycle-2's architect-driven collapse was wrong.

**Note on swing-past-center:** Per `feedback_dont_swing_past_center_when_correcting`, this is a course correction *within* the cycle-2 framework, not a strip. The tri-state `McpExposure` enum is **retained** (it captures the MetadataOnly tier and is the right shape for MCP enumeration); only the field collapse is reversed. `client_side_executable: bool` returns as a separate field with default `false`, gated at SurfaceClientBridge with `PolicyError::ClientInvocationDisabled`.

**Surgical changes:**
- `02-issues.md` W0-D: title + Problem + Why-now + acceptance rewritten to reflect "promote tri-state, **retain** `client_side_executable`." Cites artifact 05 lines 389-412 verbatim.
- `02-issues.md` W1-B: `AbilityPolicy` struct adds `client_side_executable: bool`; `#[ability]` macro accepts it; SurfaceClientBridge gate added.
- `02-issues.md` W3-A acceptance bullet for WP 7.0 `executeAbility()`: governed by `client_side_executable` ∧ `allowed_actors` ∧ `required_scopes`; clarified that `mcp_exposure` does NOT govern SurfaceClient invocation.

- `02-issues.md` W1-C inventory schema fields: `client_side_executable` added to canonical TypeScript interface field list.
- `01-project-description.md` Substrate-to-surface contract bullet: rewritten to describe both fields per artifact 05; ADR tense softened ("W0-D amends ADR-0102 §7.1 to…") per finding #7.

### 4. Browser-vs-PHP boundary single-source — PHP-ONLY CANONICAL (codex-consult R1)

**Decision:** PHP-only is canonical per Phase 0 artifact 15. Rewrite W2-A Origin guard so empty/absent Origin (PHP curl path) is the **primary** allow path; `site_url`-matching Origin is retained only as defense-in-depth backup for narrow PHP-side proxies, not as an invitation to browser-direct calls. This reconciles W2-A with W3-A and W4-E which both say browser→runtime is out of scope.

**Rationale:** Per Phase 0 artifact 15 §"transport" and W4-E's "transport model" bullet (browser → WP REST → PHP runtime client → loopback), the canonical caller is `class-dailyos-runtime-client.php` (W3-B) which makes server-side cURL requests with no `Origin` header. Re-introducing browser-direct calls would require rewriting W3-A, W4-E, W5-A, the W3-C custom MCP server, and the trust-boundary STRIDE analysis; PHP-only is the simpler and correct path.

**Surgical changes:**
- `02-issues.md` W2-A Origin guard acceptance bullet: rewritten — "**positive allowlist, PHP-curl-primary**" with absent-Origin as primary allow path, `site_url`-match as defense-in-depth backup. Explicitly states "browser-originated requests reach the runtime only via the PHP plugin's REST endpoint." Cross-references W4-E "transport model."
- `02-issues.md` W3-A WP 7.0 `executeAbility()` bullet: clarified to match — browser → WP REST → PHP runtime client → loopback; the W2-A guard treats absent Origin as primary.

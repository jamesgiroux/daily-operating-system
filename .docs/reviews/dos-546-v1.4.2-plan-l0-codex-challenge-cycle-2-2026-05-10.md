## Top Findings (ranked)

### 1. [CRITICAL] W0-D over-collapses MCP exposure and SurfaceClient invocation
Phase 0 says keep `mcp_exposure` and `client_side_executable` separate because a WP block may need to hydrate an ability that should never be listed as an MCP tool: `.docs/plans/dos-546/phase-0/05-ability-surface-inventory.md:389`, `:398`, `:401`.
Cycle-2 retires `client_side_executable`: `.docs/plans/dos-546/v1.4.2-project/02-issues.md:225`.
Then W3-A says WP `executeAbility()` is governed by `mcp_exposure`: `02-issues.md:1052`, and W4-A0 marks the first block producer `mcp_exposure: Invocable`: `:1252`.
That means WP renderability pushes abilities toward MCP exposure. This is the exact rigidity Phase 0 warned about.

### 2. [HIGH] W1 stage-1 parallelism is logically false
Wave plan says W1-A0, W1-A, and W1-B run in parallel with "no internal dependency": `03-wave-plan.md:127`, `:226`.
But W1-A0's helper pattern-matches `Actor::SurfaceClient`: `02-issues.md:290`, while W1-A is the issue that creates that actor: `:365`. W1-B also explicitly says SurfaceClient must merge first: `:493`.
**Fix:** W1-A first, then W1-A0/W1-B, then W1-C/D/E.

### 3. [HIGH] Markdown reconciliation is still promised in the wave plan
Project description correctly says markdown-as-input is v1.4.6, not v1.4.2: `01-project-description.md:80`, `:108`, `:140`.
But the wave plan still says "Edit in markdown → reconciliation → substrate apply → WP re-render": `03-wave-plan.md:51`. That resurrects cycle-1 blocker #4.
Also artifact 11 is simultaneously "may still be missing" and "retired": `02-issues.md:169`, `:175`; `03-wave-plan.md:59`, `:88`.

### 4. [HIGH] W4 renderer can run before the substrate rules it must consume
W4-A depends on W4-A0, W1-E, W3-B, W4-B only: `03-wave-plan.md:406`.
But its acceptance requires cached projection fallback and unknown-block projection rules: `02-issues.md:1339`, `:1340`, while W4-D owns those rules and says WP may not deviate: `:1590`.
**Fix:** Renderer should depend on W4-D, and likely W4-C for signature-aware cached projections.

### 5. [HIGH] W4-C fixed the algorithm but not Ed25519 key lifecycle
W4-C specifies private key storage and embedded public keys: `02-issues.md:1513`, `:1514`.
Phase 0 also requires unknown-key refresh, revoked-key handling, replacement key provisioning, live projection re-signing, and revoked signature IDs: `.docs/plans/dos-546/phase-0/03-tamper-detection-contract.md:381`, `:386`, `:773`, `:779`.
The plan does not own that lifecycle clearly. The phase doc still has open key-compromise questions: `:827`, `:829`.

### 6. [MEDIUM] ADR alignment is still future-tense, not locked
Current ADR-0102 still says `mcp_exposure: bool`: `.docs/decisions/0102-abilities-as-runtime-contract.md:223`.
Current ADR-0130 still has `provenance: ProvenanceEnvelope`: `.docs/decisions/0130-surface-independent-composition-contract.md:54`, `:59`.
The plan can amend these in W0, but do not describe the contract as already locked until W0-A/W0-D land.

### 7. [MEDIUM] The revision introduced stale counts and stale edge-case text
- Wave 0 still says "3 issues" despite W0-D: `02-issues.md:18`
- Wave plan still says 22 issues: `03-wave-plan.md:102`
- W2-B acceptance says 30s stale / 5s future, but edge case still says ±60s: `02-issues.md:825`, `:846`

## Required Revisions Before Approval

Keep the cycle-2 shape, but fix these before Linear creation:

1. **Reopen W0-D.** Either restore separate SurfaceClient invocation semantics or rename/split the field so WP block hydration does not imply MCP exposure.
2. **Change W1 staging** so `Actor::SurfaceClient` lands before audit/policy helpers that reference it.
3. **Remove every v1.4.2 markdown-as-input/reconciliation promise** from the wave plan.
4. **Add W4-A dependency on W4-D**, and decide whether W4-A depends on W4-C for cached projection authenticity.
5. **Expand W4-C key lifecycle acceptance:** key ring distribution, unknown key refresh, revocation, replacement key provisioning, re-signing, retired-key verification.
6. **Sweep stale counts**, artifact-11 wording, and HMAC freshness text.

After those, this probably drops to **REVISE** or **APPROVE**. Right now, **RECONSIDER** stands.

## Recommendations

- **Keep the W0→W6 ordering, but move audit earlier.** W6-A `Audit attribution: SurfaceClient instance + WP user_id round-trip` belongs in W1-A/W2-C, not release closeout. Many earlier issues already require audit fields: W1-A SurfaceClient, W2-C pairing, W2-D rate limits, W3-C MCP, W4-E nonce, W5-A feedback. Retrofitting `wp_user_id` in W6 contradicts W6-A's own problem statement.

- **W2 is not really parallel and is under-budgeted.** `03-wave-plan.md §Wave 2` calls four agents, but W2-A→B→C→D is strictly sequential. With two L issues and two M issues, "1 week" is optimistic. Either split W2 into two waves or let W3-A plugin skeleton start after W1-C while W2 continues.

- **Start the negative fixture catalog in W0/W1, not W5.** W5-C says every artifact-12 fixture lands there, but earlier issues also claim their own negative fixtures. Make W0-C create the catalog and bundle skeletons, then require each issue to land its fixtures before its wave gate. W5 should consolidate, not discover missing safety work.

- **Fix the bundle numbering contradiction.** `02-issues.md §Negative fixture catalog` lists bundles `19` through `24`, says "five bundles minimum," while `03-wave-plan.md` and `01-project-description.md §Definition of Done` require bundles `1-23`. Pick one. Keep `1-24` if audit attribution is its own bundle.

- **Add or cut the markdown projection work.** `01-project-description.md §Three-view consistency` and DoD item 3 require edit-in-WP, edit-in-markdown, WP DB, and filesystem consistency. The issue list has concurrency and tamper detection, but no clear markdown projection/write/reconciliation issue. Either add a W4/W5 issue for markdown projection ingestion, or explicitly cut edit-in-markdown to v1.4.6.

- **Resolve the `dailyos/account-overview` producer before W4.** W4-A says the ability is "assumed to exist." That is too late. Add a small W1/W4 issue for a minimal `account-overview` Composition-producing ability, or explicitly cite the existing ability and required fixture. Otherwise the first block can stall or quietly drag entity-intelligence scope back into v1.4.2.

- **Split W4-A if single-PR sizing matters.** `dailyos/account-overview Gutenberg block` includes block registration, SSR render, editor UI, account selector, trust bands, provenance panel, cached projection, fallback, and E2E. That is more than one clean PR. Split into renderer contract, editor/provenance UI, and cache/fallback integration.

- **Clarify the HMAC key threat model.** W2-B/W3-B mention malicious co-resident WP plugins, but W3-B stores the per-session key in `wp_options`. Any plugin with code execution can read options directly. Either declare malicious WP plugins out of scope, or move key custody outside plaintext WP storage.

- **Pairing bootstrapping is underspecified.** W2-A says the random port is advertised only through pairing, but WP needs the port to call `/v1/pairing/handshake`. Define whether the pairing code encodes the port, the user enters a full local URL, or Studio/bootstrap passes it securely.

- **Browser-vs-PHP transport is inconsistent.** W2-A rejects all non-empty HTTP/HTTPS `Origin` headers, but W4-E says WP block JS requests nonces. If browser JS calls the runtime directly, it will fail. Acceptance criteria should say browser JS calls WP PHP/REST, and PHP signs/forwards to loopback.

- **The save-time nonce model needs sharper acceptance.** W4-E issues 60-second click-bound nonces, while W5-A diffs block attributes on save. A user can edit, wait 90 seconds, then save. Decide: immediate explicit feedback actions, or save-button nonce covering the whole diff.

- **Do not defer `client_side_exposure` to W3-B.** `03-wave-plan.md §L0 questions` leaves WP 7.0 `executeAbility()` exposure open. That is an AbilityPolicy/schema question, so settle it in W1-B. `mcp_exposure` should not automatically imply browser/client-side invocability.

- **Tighten AbilityPolicy defaults.** W1-B defaults `required_scopes` to empty, meaning no scope required. Safer invariant: any ability exposed to `SurfaceClient` or MCP must declare scopes explicitly, unless marked `NoScopeRequired` with review.

- **Add WP admin security acceptance criteria.** W3-B pairing/revoke/settings flows need WP capability checks plus WP nonces, CSRF protection, and clear error surfaces. This is separate from DailyOS user-presence nonces.

- **Cut the multi-block "daily briefing" promise from v1.4.2.** `01-project-description.md §User Outcome` describes a composed daily briefing with entity overviews, claim summaries, callouts, prep blocks. But `§Non-goals` says briefing templates and additional blocks are later. For v1.4.2, say "a seeded page with `dailyos/account-overview` renders end-to-end."

- **If schedule slips, cut theme depth before trust depth.** Keep SurfaceClient, AbilityPolicy, pairing, HMAC, MCP allowlist, concurrency, tamper, nonce, feedback. Reduce W5-B to a minimal token-driven DailyOS block theme; move FolioBar/FloatingNavIsland/AtmosphereLayer polish to v1.4.3 if needed.

## Substrate-to-surface contract completeness

Substrate-to-surface contracts are close, but not complete yet:

- **Claims** need the markdown projection decision.
- **Signals** need an explicit WP refresh mechanism.
- **Trust rendering** needs negative fixtures per sensitivity tier.
- **Abilities** need channel-specific exposure, explicit scopes, and an earlier audit context carrying `surface_client_id`, `wp_user_id`, `wp_site_id`, and request ID.

## Findings

### F1 — AC3 violation: `AbilityPolicy::default()` is not User-only (BLOCKER)

**AC3 (this review):** "Default policy closed: `AbilityPolicy::default()` yields User-only / no scopes / `McpExposure::None` / `client_side_executable: false`."

**ADR-0102 §7.6 (canonical):** "Abilities default to `allowed_actors = [User]`, `required_scopes = vec![]`, `mcp_exposure = McpExposure::None`, `client_side_executable = false`."

**Actual implementation** (`src-tauri/abilities-runtime/src/abilities/registry.rs:372-382`):

```rust
impl Default for AbilityPolicy {
    fn default() -> Self {
        Self {
            allowed_actors: &[],         // <-- empty, not [Actor::User]
            allowed_modes: &[],
            requires_confirmation: false,
            may_publish: false,
            required_scopes: &[],
            mcp_exposure: McpExposure::None,
            client_side_executable: false,
        }
    }
}
```

`allowed_actors: &[]` is *more* restrictive than User-only on the surface (no actor can invoke), but it diverges from the canonical default specified in both AC3 and ADR-0102 §7.6. Downstream consumers using `..Default::default()` to fill in fields will end up with policies that reject *every* actor, including `Actor::User` — a silent behavior change relative to the canonical contract. The test at registry.rs locks the empty-actors default in, making this a contract-level lock rather than an oversight.

**Required:** `allowed_actors: &[Actor::User]` to match ADR-0102 §7.6 and AC3 verbatim. Update the corresponding default test to assert the User-only invariant.

### F2 — AC4 over-rejection: positive SurfaceClient declaration path is unreachable (BLOCKER)

**AC4 (this review):** "Macro compile-error gate emits `compile_error!` when `allowed_actors` includes `SurfaceClient` AND `required_scopes` empty AND `no_scope_required` absent."

The negative path is wired correctly at `src-tauri/abilities-macro/src/lib.rs:97-114` (returns `syn::Error` early). However, the *positive* path — `allowed_actors: [User, SurfaceClient], required_scopes: ["read.foo"]` (the exact shape AC line 1254 says the W1-D `dailyos/account-overview` ability will use) — is **also** broken.

`ActorArg::SurfaceClient::registry_expr()` at `src-tauri/abilities-macro/src/lib.rs:910-916` unconditionally emits:

```rust
Self::SurfaceClient => {
    quote! { compile_error!(
        "Actor::SurfaceClient cannot appear directly in AbilityPolicy::allowed_actors \
         (it is a per-invocation struct variant). Use a SurfaceClient-aware bridge \
         instead; the W1-B macro gate enforces required_scopes / no_scope_required \
         at the declaration site."
    ) }
}
```

This token stream is consumed at `lib.rs:155-159` by `actor_exprs.collect()` for the `actors_ident` static (a `&[Actor]` slice in generated code at lines 356/386/415). The gate at line 97 fires `return Err(...)` *only* when scopes are empty and `no_scope_required` is absent. When scopes ARE declared (the legitimate path), the macro falls through to codegen, the `actors_ident` static is generated, and the `compile_error!` inside `registry_expr()` fires unconditionally — meaning **no ability can ever declare `SurfaceClient` in `allowed_actors`, scoped or not**.

The comment at lib.rs:903-908 even acknowledges this: *"declaring it in `allowed_actors` at the policy level is a type error because the policy slice is `&[Actor]` and SurfaceClient cannot be const-constructed."* But the W1-B brief, ADR-0102 §7.6, and AC line 1254 all assume `allowed_actors: [..., SurfaceClient]` is the declaration syntax.

The trybuild suite confirms the gap: only `surface_client_without_scopes_fails.rs` exists (negative path). No positive-path fixture (`allowed_actors: [User, SurfaceClient], required_scopes: ["read.foo"]` → compiles) exists, so the regression slipped through the 9/9 green claim.

**Required:** Decide and implement *one* of:

1. **Generate `Actor::SurfaceClient` from `registry_expr()` properly** — this requires `Actor::SurfaceClient` to be const-constructible in `&[Actor]` slices, which conflicts with the `{ instance, scopes: ScopeSet }` struct variant landed in W1-A.1. May need a separate `ActorTag` enum for the policy-declaration slice, distinct from the runtime `Actor` enum with payloads.
2. **Generate a sentinel** (e.g., `Actor::SurfaceClientTag` or filter `SurfaceClient` out of the slice and emit a separate `exposes_to_surface_client: bool` field on `AbilityPolicy`).

Add a positive-path trybuild fixture (`surface_client_with_scopes_passes.rs`) asserting the legitimate W1-D shape compiles. Without this fixture, the gap re-drifts.

## Passed within scope

- **AC1:** `required_scopes: &'static [&'static str]` storage with `required_scopes_typed() -> Vec<SurfaceScope>` accessor is an acceptable trade-off for `inventory::submit!` static-context constructability. The conversion cost at the boundary is small and the typed accessor preserves W1-A.1's `SurfaceScope` newtype discipline.
- **AC2:** `McpExposure { None, MetadataOnly, Invocable }` with serde snake_case + `#[default] None` matches ADR-0102 §7.1 verbatim. Round-trip preserved.
- **AC5:** Allowlist init wired into `from_descriptors_checked` — `SCOPE_ALLOWLIST` OnceLock receives the union of every registered ability's scopes at registry boot, switching from lenient bootstrap to strict mode. See registry.rs:218-228.
- **AC6 (scope discipline):** No `SurfaceClientBridge` enforcement (W2-B), no MCP server `mcp_exposure` filtering (W3-C), no `emit_surface_audit` helper (W1-A0). Commit stayed in lane.

1. **W2-A Origin allowlist vs W3-A/W4-E browser boundary.** W2-A allows paired `site_url` as an Origin (i.e., browser-originated requests pass the allowlist), while W3-A and W4-E describe the browser→runtime path as rejected/out-of-scope. The transport boundary needs to be single-source: either PHP-only (browser Origin always absent or rejected) and W2-A drops `site_url` from the allowlist, or browser→runtime is explicitly in scope and W3-A/W4-E are updated to match. Strip markdown-as-input phrasing from the thesis to match the issue bodies.

3. **W2 wave shape still calls itself "4 parallel agents / 1 week" while issues read as a strict sequence** (loopback → HMAC → pairing → rate-limit). Cycle-1 flagged this same under-budget risk; cycle-2 did not adjust the wave-plan budget.

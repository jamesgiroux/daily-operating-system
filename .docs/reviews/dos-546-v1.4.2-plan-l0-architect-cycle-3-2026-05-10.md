Phase 0 artifact 05 lines 389-412 are unambiguous: the two fields govern different trust boundaries (MCP tool-surface enumeration vs. SurfaceClient invocation after WP cap + scope + actor). The cycle-2 architect A1 finding pushed for collapse based on a tri-state pattern that is right for MCP but wrong for SurfaceClient; cycle-3 correctly retains the tri-state `McpExposure` enum (the right part) and separates `client_side_executable: bool` back out (the part that should never have collapsed). Per `feedback_dont_swing_past_center_when_correcting`, this is a trim that preserves the differentiating substrate.

Coherence across the three files:
- **W0-D** (`02-issues.md:203-261`): ADR amendment retains both fields, default `mcp_exposure: None`, default `client_side_executable: false`, macro compile-error gate triggered by `allowed_actors` membership regardless of `client_side_executable` — correct.
- **W1-B** (`02-issues.md:444-456`): `AbilityPolicy` struct adds three fields; SurfaceClientBridge enforces `client_side_executable == true` after scope check, returning `PolicyError::ClientInvocationDisabled` — correct binding to the ADR amendment.
- **W4-A0** (`02-issues.md:1253`): declares `mcp_exposure: McpExposure::Invocable, client_side_executable: true` — both independently required, correct per artifact 05's "fields can disagree" worked example.
- **W3-A** (`02-issues.md:1053`): WP 7.0 `executeAbility()` gated by `client_side_executable ∧ allowed_actors: [SurfaceClient] ∧ required_scopes` with explicit "`mcp_exposure` does NOT govern SurfaceClient invocation" — correct.
- **CI invariant** (`03-wave-plan.md:153`): pins both field shapes as independent type tests — correct.

No `client_side_executable` grep-ban remains anywhere. Revert is complete and coherent. Approve.

Cycle-2 approved the producer/renderer split (W4-A0 produces, W4-A consumes) but left W4-A's dependence on W4-C (Ed25519 verification for cached projections) and W4-D (substrate-side fallback projection rules) implicit. The W4-A acceptance bullets cycle-3 now point to (`02-issues.md:1340-1341`) make these binding:

Cycle-2 was incorrect to leave these implicit — without them, W4-A could ship either (a) cached bytes rendered as trusted without signature verification (trust-boundary violation) or (b) WP-side fallback logic deviating from the substrate authority. Both are explicit cycle-2 architect concerns I should have flagged. Cycle-3 makes them load-bearing.

Wall-clock: W4 stage-3 narrows (W4-A now waits on four merges), but stage-2 fan-out is unchanged (W4-A0/C/D/E still parallel). W4 wave budget of ~1.5 weeks absorbs.

**Reviewer:** security-auditor sub-agent (per `/cso` discipline in CLAUDE.md)

**Scope:** Spec-level security review of DOS-546 spike + supporting ADR amendments (0102 §7.6/§7.1, 0111 §8, 0130 new, 0129 §4 amended)
**Verdict:** **APPROVE for formal L0 pass (security)** with 10 refinements to existing §0.8 Phase 0 deliverables
**Codex verification used:** 0 of 2 allowed calls. Findings derived from spec reading alone.

## Verdict

**APPROVE for formal L0 pass (security), with three Phase 0 design deliverables that §0.8 names but does not yet bind tightly enough.** The substrate is structurally sound after the Cycle 2 amendments. `required_scopes` and `mcp_exposure` as canonical `AbilityPolicy` fields close the two-level enforcement gap; `SurfaceClient` as a fourth actor class with per-instance identity and audit attribution is the right shape; ADR-0130 correctly cordons authorship inside the substrate. The remaining risk surface is implementation-pattern risk, not architectural risk — but three of the §0.8 commitments need sharper acceptance criteria before code starts, so they don't dissolve into "Phase 1 figures it out."

The verdict is APPROVE rather than REVISE because every finding raised is already a named §0.8 deliverable. Not blocking — refining the Phase 0 acceptance shape so the deliverables actually defend what they need to defend.

**Load-bearing STRIDE category driving the verdict:** Tampering. The substrate-side enforcement gates close the elevation-of-privilege class structurally. The remaining load-bearing problem is the three-view consistency contract — the spec genuinely defers an architectural decision to Phase 0, which is correct sequencing but means Phase 0 output is the deliverable to watch most carefully.

## STRIDE Findings Summary

### Spoofing

- **[P1] Pairing-token recovery paths are the spoofing-class attack surface.** Four recovery paths (reinstall, DB-restore, site-switch, exfiltration) need individually-defined defenses, not a single bullet covering "recovery." Each named with specific defense:
  - Reinstall → pairing token bound to hardware-resident or keychain-resident runtime-side secret; WP-side material is proof-of-pairing only
  - DB-restore → revocation list runtime-side; token expires by clock + revocation table
  - Site-switch → pairing binds to `site_url` + runtime-issued site nonce captured at first-pair
  - Exfiltration → short-lived (minutes), refreshable, paired with user-presence nonce on writes
- **[P2] SurfaceClient::WordPress identity assertion via loopback HTTP is weak by itself.** Bearer alone is not enough; loopback rules must include transport-layer signature (HMAC over request body using runtime-keychain key, retrievable WP-side only via `manage_options`-gated filter).

### Tampering

- **[P1] Three-view consistency tamper detection named but not specified.** Concurrency (write-loss, concurrent edits) and tamper (out-of-band edits to markdown or WP DB) are DISTINCT problems. Phase 0 must produce both contracts as separate sub-deliverables.
- **[P2] Same-process WP plugin attacks on read surface remain.** Defenses are correct (minimal default scopes + audit) but threat model §0.3 must explicitly name this as **accepted residual risk** rather than imply closure.

### Repudiation

- **[P2] WP-side audit gap.** Substrate logs SurfaceClient instance identity but not WP user_id. For multi-user WP, audit must round-trip the originating `wp_user_id` (or `null` for headless) into substrate logs.
- **[P3] User-presence nonce should bind to wp_user_id too.** Current binding (session+claim+field+action) doesn't defend against cross-user nonce reuse on a multi-user install within the 60s window.

### Information Disclosure

- **[P1] Custom block fallback rendering by embedding-nearest-type leaks payload structure.** Fix shape: nearest-type *projection* at JSON-pointer granularity (intersection of unknown and chosen-known schemas), explicit banner, `claim_refs` always preserved. Don't render unknown payload fields directly.
- **[P2] Ability description copy needs CI gate, not just discipline note.** ADR-0102 §7.6 names the discipline; without CI enforcement against the PII blocklist + internal-vocabulary scan, it drifts. Add to §0.7 acceptance.
- **[P3] Composition provenance must be reference-style, not envelope copy.** `Block.provenance: ProvenanceRef` (not `ProvenanceEnvelope`); renderers resolve refs against the ability's top-level envelope. Keeps ADR-0105 "lives once" invariant and avoids ADR-0108's 64KB cap.

### Denial of Service

- **[P2] Loopback HTTP rate-limit unspecified.** Phase 0.3 should commit to per-SurfaceClient-instance req/sec + req/min budgets enforced at the bridge.
- **[P3] Custom block fallback embedding lookup adds O(N) embed-model calls per unknown-block composition.** Cap unknown-block-per-composition count at substrate side.

### Elevation of Privilege

- **[P1 → already closed by Cycle 2 amendments]** Two-level enforcement (`allowed_actors` + `required_scopes` + `mcp_exposure`) is substrate-enforceable now that canonical schema fields exist.
- **[P2] One bypass path to verify.** WP 7.0 client-side `executeAbility()`: is `mcp_exposure: true` sufficient to mean "exposed to client-side JS too," or do we need separate `client_side_exposure: bool`? Phase 0.7 must resolve before client-side path is wired.
- **[P3] Maintenance abilities composing SurfaceClient-callable abilities.** Probably safe (Maintenance > SurfaceClient by construction) but worth naming in composition rules for hygiene.

## Answers to the 8 Focus Areas

1. **Pairing/token model:** Mostly adequate; recovery paths under-specified (each of four paths needs its own defense, not one bullet).
2. **Threat model defenses:** Substantially adequate; explicitly name read-side info disclosure to co-resident hostile plugins as accepted residual risk.
3. **Two-level enforcement:** Closed at substrate; verify client-side `executeAbility()` path doesn't bypass.
4. **User-presence nonce:** Adequate against the named threat; add wp_user_id to binding.
5. **Audit attribution:** Forensically complete substrate-side; gap is WP user_id round-trip.
6. **Custom block fallback:** NOT adequate as described; needs projection rules at JSON-pointer granularity.
7. **Composition provenance reconciliation:** Safe IF block-level field is a reference, not duplicate envelope; clarify in Phase 0.
8. **Three-view consistency:** Concurrency and tamper-detection are distinct problems; split into two sub-contracts.

## Phase 0 Refinements (10 items, all additions to §0.8 acceptance)

These sharpen what "done" means for the existing §0.8 deliverables. Two are genuinely new items; eight are refinements to existing items.

### Refinements to existing §0.8 deliverables

1. **Pairing/token model (§0.1):** acceptance must include per-recovery-path defenses (reinstall, DB-restore, site-switch, exfiltration) as four discrete sub-deliverables, not one bullet. Each names its defense.

2. **Three-view consistency (§0.2):** split into two sub-contracts — concurrency model AND tamper-detection model. Each ships acceptance fixtures (negative cases). Tamper detection requires projection signing on write + verification on read.

3. **Threat model (§0.3):** explicitly state which threats are *accepted residual risk* (e.g., read-side info disclosure to co-resident hostile WP plugins, mitigated by minimal-scope + audit), versus which are *closed* (e.g., write-side via user-presence nonce + audit). Forensically and politically important.

4. **Audit attribution (§0.5):** every substrate operation log entry carries SurfaceClient instance identity AND WP `user_id` (or `null` for non-WP / headless invocations).

5. **Ability-surface inventory (§0.7):** resolve `mcp_exposure` vs. `client_side_exposure` question. Reuse `mcp_exposure` for both (preferred — fewer fields, same gate) or add a parallel `client_side_exposure` field. Decide before the WP 7.0 client-side path is wired.

6. **Ability description CI gate (§0.7):** descriptions enter the same lint regime as user-facing copy (PII blocklist + internal-vocabulary scan via `.claude/hooks/pre-commit-gate.sh` equivalent). Add to acceptance.

7. **Loopback HTTP request signing (§0.3):** add HMAC-over-body using a runtime-keychain-resident key, retrievable from the WP side only via a `manage_options`-gated WP filter, so the runtime can distinguish "real DailyOS plugin code" from "other code presenting the same bearer token" with non-trivial confidence.

8. **Rate-limit budgets per SurfaceClient instance (§0.3):** per-SurfaceClient-instance req/sec + req/min budgets enforced at the bridge, with rejection on exceeded.

### New §0.8 items (not currently in DOS-546)

9. **Custom block fallback projection rules.** Field-pointer-granularity projection: intersect known and unknown block schemas at JSON-pointer level; preserve `claim_refs`; banner "rendered as nearest known type — payload may be incomplete"; never render unknown payload fields directly.

10. **Composition provenance reference shape.** Clarify `Block.provenance: ProvenanceRef` (not `ProvenanceEnvelope`). Single line in the type-clarification work, but load-bearing for ADR-0105 invariant compliance and ADR-0108 size cap.

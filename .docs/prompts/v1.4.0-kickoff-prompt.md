### Start here — Phase 0

Pick ONE of the two hard blockers and ship it. They're independent; work on one at a time unless you're spawning a parallel worktree.

**Recommended first task: DOS-209 ServiceContext**

Why first: every other substrate work depends on `&ServiceContext` existing. Without it, nothing else compiles.

Scope:

1. Create `ExecutionMode` enum (`Live` / `Simulate` / `Evaluate`) in `src-tauri/src/services/execution_mode.rs` (new).
2. Create `ServiceContext<'a>` struct in `src-tauri/src/services/context.rs` (new). Fields: `mode`, `clock: &'a dyn Clock`, `rng: &'a dyn SeededRng`, `external: &'a ExternalClients`, `tx: Option<TxHandle<'a>>`, `db: &'a ActionDb`.
3. Define `Clock` trait with `fn now(&self) -> DateTime<Utc>`. Provide `SystemClock` (Live) and `FixedClock` (Evaluate/test) implementations.
4. Define `SeededRng` trait. Provide `ThreadRng` wrapper (Live) and `SeededRng` (Evaluate/test).
5. Implement `ServiceContext::new_live(...)`, `::new_simulate(...)`, `::new_evaluate(...)`. No `Default` or zero-arg.
6. Implement `check_mutation_allowed(&self) -> Result<(), WriteBlockedByMode>` that returns `Err` outside `Live`.
7. Implement `with_transaction_async` primitive. HRTB may iterate; ship simpler sync-within-async pattern if needed.
8. Migrate every mutation function in `src-tauri/src/services/` to take `&ServiceContext` and call `ctx.check_mutation_allowed()?` as first line. ~60 call sites.
9. Replace every `Utc::now()` and `rand::thread_rng()` in `services/` and `abilities/` (if it exists) with `ctx.clock.now()` / `ctx.rng`.
10. Add CI lint against direct clock/RNG use in ability code.
11. Property test: random `ExecutionMode` + random mutation attempt → correct accept/reject.

Reference: [DOS-209](https://linear.app/a8c/issue/DOS-209) has the full rubric-upgraded spec. Check it; it's authoritative.

**Alternative first task: DOS-259 IntelligenceProvider**

Why consider it: can run parallel to DOS-209 in a second worktree. Extracts the `IntelligenceProvider` trait from two scattered implementations (PTY + Glean) into a single shape per ADR-0106 §3. Blocks ADR-0106 work (DOS-213) and ADR-0119 runtime evaluator.

Scope summarized in [DOS-259](https://linear.app/a8c/issue/DOS-259). Read it before starting.

### Hard rules (CI-enforced)

- No `Utc::now()` / `rand::thread_rng()` in ability code or new service code. Use `ctx.clock` / `ctx.rng`.
- All mutations through `services/`. No direct `INSERT`/`UPDATE`/`DELETE` in abilities.
- Every `#[ability]` function returns `AbilityResult<T>` with populated `Provenance`.
- `TrustAssessment` on Transform outputs is non-zero (not defaulted).
- `field_attributions` populated for every LLM-synthesized field.
- Prompt templates have stable `prompt_template_id` + semver `prompt_template_version`.
- Every mutation function has `ctx.check_mutation_allowed()?` as first line.
- `cargo clippy -- -D warnings && cargo test && pnpm tsc --noEmit` green before merge.
- Experimental abilities have `registered_at` date; CI fails at 90 days; no production registration, no claim writes, no publish, no MCP exposure.

### Gotchas already encoded — don't re-litigate

- Tombstone check is PRE-GATE (ADR-0113 §5). Runs BEFORE commit policy. Ghost-resurrection is structurally fixed in v1.4.0.
- Trust ledger `agent_version` is a COMPOSITE: `ability_version + prompt_template_version`. Prompt edits trigger new-ledger-row trust warming.
- Claim commit uses pessimistic row-lock (ADR-0113 R2). App-level `Mutex<HashMap<LockKey, ()>>` with 500 ms timeout.
- Provenance hard cap is 64 KB (ADR-0108 Amendment). Hitting it → `AbilityError::ProvenanceTooLarge`.
- Signal emission + invalidation enqueue share a transaction (ADR-0115 R1.4).
- `PropagateSync { await_completion: true }` bounded at 500 ms (ADR-0115 R2). Used sparingly.
- Proc-macro AST inspection + composition-graph check enforces ability categories (ADR-0102 §3 + DOS-210 spec).
- "Content stays on laptop" means DailyOS server-side components see zero. Glean is user's separate contractual relationship.
- Publish is reporting, not team intelligence (ADR-0121 tracks the latter as Open).
- Auto-gen docs (`DATA-MODEL.md`, `MODULE-MAP.md`) regenerate from code; do NOT manually add v1.4.0 tables/modules until they exist in the codebase. If you need to add content now, use a sibling file (the generator fix is Wave A lane A4 of the plan).

### Spec + issue discipline

Every work item uses `.docs/SPEC-TEMPLATE.md`. Every Linear issue gets `spec:needs-review` when drafted; another engineer (or you in a separate review session) walks the 12-surface architectural vet before `spec:ready`.

When creating or updating an issue via Linear MCP:
- Use `save_issue` tool with `team: "DailyOS"` and `project: "v1.4.0 — Abilities Runtime"`.
- Apply `spec:needs-review` label until reviewed.
- Reference the governing ADR with full path in descriptions.

### Commit style

- Type-scope prefix: `feat(DOS-XYZ):`, `fix(DOS-XYZ):`, `docs(scope):`.
- Small atomic commits. Never `git add -A`. Stage files by name.
- Commit messages explain the why, not just the what.
- Co-authored-by trailer: `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>`.
- Never skip hooks, never bypass signing.

### When to stop and ask

- Two plausible architectures for the same requirement. Stop, name them, ask.
- A destructive operation where the scope is unclear.
- Three consecutive failed attempts at a task.
- Anything that would soften an ADR boundary (especially ADR-0116 metadata-only).
- If an ADR feels ambiguous when code contact forces a precise answer — amend the ADR, do not silently decide in code.

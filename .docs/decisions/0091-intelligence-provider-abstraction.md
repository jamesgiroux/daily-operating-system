# ADR-0091: IntelligenceProvider Abstraction â€” Multi-LLM Backend Support

**Date:** 2026-02-22
**Status:** Accepted
**Target:** Post-1.0 (v1.1.0)
**Extends:** ADR-0086 (Intelligence as Shared Service)

## Context

DailyOS uses `PtyManager` to spawn Claude Code as a subprocess for AI enrichment. This is a specific, tightly-coupled integration: workspace-aware, structured output via Claude Code's prompt format, authenticated via Claude Pro/Max subscription.

The I376 enrichment audit confirmed 15 `PtyManager::for_tier` construction sites across the codebase. Two of those sites are ADR-0086 compliant (intel_queue.rs, the primary enrichment path). The remaining 13 are deliberate exceptions (file processors, email enrichment, daily workflow orchestration) that fall outside the entity intelligence pipeline. The abstraction in this ADR targets the intel_queue path â€” the two compliant sites â€” not the deliberate exceptions.

Supporting other LLM backends requires abstracting away the PTY-specific transport at `intel_queue.rs`. The rest of the pipeline (processor/enrich.rs, prepare/email_enrich.rs, executor.rs, deliver.rs, risk_briefing.rs) remains PTY-based for now. Those are workflow-level operations that are Claude Code-specific in character and are not on the post-1.0 path.

## Decision

### The IntelligenceProvider Trait

A Rust trait replaces direct `PtyManager::for_tier()` construction at the intel_queue call sites. The trait surface is minimal by design:

```rust
#[async_trait]
pub trait IntelligenceProvider: Send + Sync {
    async fn complete(
        &self,
        prompt: &str,
        tier: ModelTier,
    ) -> Result<String, ProviderError>;
}
```

`intel_queue.rs` calls `state.intelligence_provider.complete(prompt, tier)` instead of constructing a `PtyManager`. Everything downstream â€” the JSON parsing, the entity intelligence write, the meeting prep invalidation cascade â€” is unchanged.

### Three Providers

**ClaudeCodeProvider** â€” current behavior. Spawns Claude Code as a PTY subprocess. Workspace-aware. Uses the existing prompt format. This is the default and remains unchanged behavior for all existing users.

**OllamaProvider** â€” calls a locally-running Ollama instance at `http://localhost:11434/api/chat` using Ollama's OpenAI-compatible endpoint. Model is configurable. Defaults: `llama3.3:70b` for Synthesis tier, `llama3.2:3b` for Extraction and Mechanical tiers. No data leaves the device.

**OpenAIProvider** â€” calls `https://api.openai.com/v1/chat/completions` with a user-supplied API key. Model is configurable. Defaults: `gpt-4o` for Synthesis, `gpt-4o-mini` for Extraction and Mechanical. Because Ollama's API is OpenAI-compatible, `OpenAIProvider` and `OllamaProvider` share the same underlying `reqwest` HTTP client; they differ only in base URL and auth header.

### Tier-to-Model Mapping

Each provider owns its tier mapping. The `ModelTier` enum is unchanged â€” it remains `Synthesis`, `Extraction`, `Mechanical`.

| Tier | ClaudeCodeProvider | OllamaProvider (default) | OpenAIProvider (default) |
|------|--------------------|--------------------------|--------------------------|
| Synthesis | claude-opus-4-6 | llama3.3:70b | gpt-4o |
| Extraction | claude-sonnet-4-6 | llama3.2:3b | gpt-4o-mini |
| Mechanical | claude-haiku-4-5 | llama3.2:3b | gpt-4o-mini |

Model names are configurable per-provider in Settings; the defaults above are what ships.

### AppState Change

`AppState` gains one field:

```rust
pub intelligence_provider: Arc<dyn IntelligenceProvider + Send + Sync>,
```

Set to `Arc::new(ClaudeCodeProvider::new(...))` during initialization. This is the only field change. No existing fields are removed. The `pty` field remains â€” it is used by the deliberate-exception call sites (executor.rs, deliver.rs, etc.) that are not being migrated.

### Provider Selection in Settings

Settings â†’ AI Provider: "Claude Code (default)", "Ollama (local)", "OpenAI API".

Switching providers:
- Takes effect immediately for the next enrichment call.
- Does not invalidate existing `entity_intel` rows. Intelligence generated by one provider is equally valid to intelligence generated by another â€” the schema is the same, the source is not tracked per-row.
- Does not affect in-flight enrichment. The provider is read at call time, so a switch mid-queue takes effect on the next dequeue.

### Prompt Format Compatibility

Claude Code's prompt format is workspace-aware and schema-driven via Claude Code's structured output mechanism. For Ollama and OpenAI, a standard system prompt + user message format is used instead. The intelligence schema is identical â€” both prompt paths target the same `IntelligenceJson` output. Prompt templates live in `intelligence/prompts.rs` with two paths: `claude_code_prompt()` and `completion_api_prompt()`.

The completion-API prompt is a condensed version of the Claude Code prompt: it includes the entity context and the `IntelligenceJson` schema as a JSON Schema in the system message, and asks the model to respond with valid JSON matching that schema. No tool use, no file references.

### Fallback Chain

If the selected provider fails â€” Ollama not running, OpenAI key invalid, network error â€” the system falls back to `ClaudeCodeProvider`. The fallback is:

1. Logged at WARN level with provider name and error.
2. Surfaced in the system status indicator (I428) as a provider warning (not a hard failure).
3. Applied per-call, not globally â€” a transient Ollama failure falls back for that call only. If Ollama recovers, the next call uses it again.

Fallback does not apply if `ClaudeCodeProvider` itself fails (no workspace path, no Claude Code subscription). That is an existing failure mode.

## Consequences

- `PtyManager` is retained and unchanged. `ClaudeCodeProvider` is a thin wrapper around the existing `PtyManager::for_tier()` construction pattern.
- The intel_queue's two call sites are the only sites migrated. All 13 deliberate-exception sites continue to construct `PtyManager` directly. They are not in scope for this ADR.
- Prompt templates need two variants for entity enrichment. The Claude Code variant already exists. The completion-API variant is new.
- `OllamaProvider` completes the local-first story: data has been local since v0.1.0 (SQLite, on-device). With Ollama selected, AI inference is also local. Nothing leaves the device.
- `OpenAIProvider` enables users with existing OpenAI subscriptions to avoid a Claude Code dependency for enrichment, at the cost of cloud API calls.
- Provider quality is not guaranteed to be equivalent. Intelligence schema is guaranteed to be equivalent. The system does not attempt to normalize or correct for model quality differences.

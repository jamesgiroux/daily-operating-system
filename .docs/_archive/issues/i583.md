# I583 — Action Aging & Zero-Guilt Cleanup

**Version:** v1.0.2
**Area:** Backend + Frontend / Actions
**Priority:** P0

## Problem

The actions list grows indefinitely. `archive_stale_actions()` and `auto_archive_old_proposed()` exist in the DB layer but are never wired to the scheduler. Proposed actions pile up because rejection doesn't teach the system. Old pending actions never age out. The "zero-guilt" design promise is broken — the system creates guilt by accumulating an ever-growing backlog with no expiry.

## Design

### Backend — Scheduler Integration

Wire existing functions to `scheduler.rs`:
- `archive_stale_actions(30)` — auto-archive pending actions older than 30 days with no activity
- `auto_archive_old_proposed(7)` — auto-archive proposed actions older than 7 days (if not accepted, they weren't useful)
- Run daily during the preparation pipeline (after email sync, before briefing generation)

**Signal emission**: Archive events emit `action_auto_archived` signal with `source: "system_lifecycle"`, confidence 0.3. Low confidence — the system is cleaning up, not making a judgment.

**Guard rails**:
- Never auto-archive actions with `waiting_on` set (delegation still pending)
- Never auto-archive P1 actions (user explicitly marked as high priority)
- Never auto-archive actions linked to an active objective (still serving The Work)

### Backend — Gentle Aging Prompts

New query `get_aging_actions()`: returns pending actions older than 14 days with no updates. Returns `Vec<AgingAction>` with `action_id`, `title`, `entity_name`, `days_since_activity`, `is_overdue`.

New field on daily briefing data (or actions page data): `aging_actions: Vec<AgingAction>`.

### Frontend — Aging Surface

On the Actions page, aging actions get a subtle visual treatment:
- Muted text after 14 days of no activity
- Inline prompt: "Still relevant?" with two actions:
  - **Keep** — touches `updated_at` (resets the aging clock for another 14 days)
  - **Archive** — archives immediately with `source: "user_triage"`
- Proposed actions older than 3 days get a gentle countdown: "Auto-archiving in X days"

### Frontend — Completion Celebration (Micro)

When an action is completed, show a brief positive moment — not confetti, but a subtle editorial acknowledgment. If the action was overdue, show "Better late than never" or equivalent personality-aware copy. This is the zero-guilt payoff.

## Mock Data

Requires I597 (action mock data expansion). Uses:
- `mock-act-stale-1` (20d), `mock-act-stale-2` (25d), `mock-act-stale-3` (35d) for aging prompts
- `mock-act-proposed-2` (5d), `mock-act-proposed-4` (6d) for auto-archive countdown
- Existing P1/waiting/objective-linked actions verify guard rails

## Acceptance Criteria

1. Scheduler runs `archive_stale_actions(30)` and `auto_archive_old_proposed(7)` daily
2. P1 actions, waiting-on actions, and objective-linked actions are never auto-archived
3. Auto-archive emits `action_auto_archived` signal per action
4. Actions with no activity for 14+ days show "Still relevant?" prompt on the Actions page
5. "Keep" resets the aging clock; "Archive" archives immediately
6. Proposed actions show auto-archive countdown after 3 days
7. No increase in total pending action count after 30 days of inactivity (verified via `get_pending_action_counts()`)
8. **Mock data**: Aging actions (20d, 25d, 35d) show "Still relevant?" prompt. Proposed actions (5d, 6d) show countdown. P1 and waiting-on actions are exempt.

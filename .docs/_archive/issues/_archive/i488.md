# I488 — ~~Semantic gap queries sent to Glean~~ ABSORBED

**Priority:** ~~P1~~ — Absorbed into I508
**Area:** Backend / Connectors + Intelligence
**Version:** 1.1.0
**Status:** ABSORBED by I508 (Intelligence Schema Redesign)

> **Why absorbed:** I488 proposed sending gap queries to Glean based on empty fields in the current intelligence.json. But the current schema only checks 3 fields for gaps (risks, recent_wins, current_state), and those are the fields most likely already filled by meetings and emails. The fields Glean could actually fill (success_metrics, competitive_context, support_health, contract_context) aren't in the gap detector — and some don't even exist in the schema yet. The real problem isn't "send gap queries to Glean" — it's "the schema doesn't know what questions to ask." I508 redesigns the schema for multi-source enrichment, and gap queries become one mechanism within that redesign.

## Problem

`semantic_gap_query()` in `intelligence/prompts.rs` (line 1096) generates targeted search terms based on what's missing in an entity's current intelligence. It inspects the prior `IntelligenceJson` and builds a query string emphasizing gaps:

```rust
fn semantic_gap_query(prior: Option<&IntelligenceJson>) -> String {
    let mut terms = vec!["account status", "risks", "wins", "blockers", "next steps"];

    if let Some(p) = prior {
        if p.risks.is_empty() {
            terms.push("risks concerns blockers challenges");
        }
        if p.recent_wins.is_empty() {
            terms.push("recent wins outcomes delivered value");
        }
        if p.current_state.is_none() {
            terms.push("working not working unknowns");
        }
    } else {
        terms.push("executive assessment context renewal sentiment");
    }

    terms.join(" ")
}
```

This function is currently `fn` (private) and is only called at line 489 for local file ranking via `search_entity_content()` — it ranks workspace files by embedding similarity to the gap query. The gap-targeted terms never reach Glean.

When Glean is connected, it has access to the organization's full knowledge graph — CRM data, support tickets, internal docs, Slack threads. A gap query like "risks concerns blockers challenges" sent to Glean could surface a Zendesk escalation or an internal risk memo that local workspace files don't contain. This is the highest-value use case for Glean: filling specifically what's missing.

## Acceptance Criteria (original)

1. Enrich account with empty `risks`. Glean queried with risk-related terms. After enrichment, risks non-empty if Glean returned relevant content
2. Gap-targeted Glean query visible in enrichment context (debug log)
3. `semantic_gap_query()` made `pub` for cross-module access
4. Gap queries only fire when Glean is connected — no-op otherwise
5. Results deduped against already-gathered Glean context to avoid redundancy

# I431 — Cost Visibility

**Status:** Open
**Priority:** P2
**Version:** 1.0.0
**Area:** Backend + Frontend

## Summary

Users who configure multiple connectors and run DailyOS daily have no visibility into how many Claude API calls are being made on their behalf or what it costs. This issue adds a local cost tracking dashboard in Settings → System that shows weekly call counts, estimated cost broken down by enrichment type, a four-week trend chart, and an optional daily budget cap to prevent runaway costs.

## Acceptance Criteria

1. Settings → System → "AI usage" shows: total Claude API calls this week, estimated cost this week (based on approximate token counts and public Claude API pricing), breakdown by type (entity enrichment, email enrichment, report generation, meeting prep).
2. The tracking is local — the app counts PTY calls via the existing `enrichment_log` table, adds estimated token counts (based on prompt size and output size), and multiplies by the current Anthropic pricing for the active model tier.
3. A weekly usage chart (last 4 weeks) shows trend. If usage is significantly higher than the prior week, a note explains why (e.g., "New connectors configured — Clay enrichment added 34 calls this week").
4. A "Daily AI budget" setting (optional, default: unlimited) can cap the number of AI enrichment calls per day. When the budget is reached, the intel_queue pauses until midnight reset. The budget does not affect user-initiated actions (report generation, manual refresh).

## Dependencies

- The `enrichment_log` table exists (migration 016) but is missing all cost-relevant columns. Current columns: `entity_type`, `entity_id`, `source`, `event_type`, `signal_type`, `fields_updated`, `raw_payload`, `created_at`. **This issue must add via migration:** `call_type TEXT` (entity_enrichment/email_enrichment/report_generation/meeting_prep), `prompt_length INTEGER`, `output_length INTEGER`, `model_tier TEXT` (haiku/sonnet/opus). Every PTY call site in `intel_queue.rs`, `reports/generator.rs`, and `prepare/` must be updated to write these fields. This is a non-trivial wiring task — not just "add columns."
- Token estimation: use `prompt_length` (character count) × 0.25 as a tokens-per-character approximation, plus `output_length` × 0.25. Multiply by the per-token price for the active `ModelTier`. Pricing constants should be in a single `cost_estimation.rs` module so they can be updated when Anthropic changes pricing.
- The four-week chart requires grouping `enrichment_log` entries by ISO week. A SQL query with `strftime('%W', created_at)` covers this without new tables.
- The "Daily AI budget" setting should be stored in `workspace_config` as `daily_ai_budget: Option<u32>`. The intel_queue poller checks `SELECT COUNT(*) FROM enrichment_log WHERE date(created_at) = date('now')` before enqueuing a new job.
- The budget cap must be enforced in the intel_queue enqueue path, not the execution path — to avoid starting a job that will be abandoned.

## Coordination

- **Depends on I435 (Token Optimization)** — I435 audits every PTY call site and assigns correct model tiers. I431's cost estimation is only meaningful after I435 establishes the correct tier assignments. Ship I435 first, then I431 can use the audited tiers as the baseline for cost calculation. Without I435, I431's cost estimates would be based on potentially incorrect tier assignments.

## Notes / Rationale

The `enrichment_log` table already exists (confirmed in the v0.13.2 audit). Token counting is approximate — use prompt character count × 0.25 as a tokens-per-character estimate, multiplied by current Anthropic pricing. This is a local estimate, not a billable figure, and the UI should make that clear ("estimated cost" not "your bill").

The budget cap prevents runaway costs for users who configure many connectors. A user who adds Clay, Gravatar, and Linear at once might see a large spike in enrichment calls as all entities are processed for the first time. The budget cap with a clear "paused until midnight" message is preferable to either silent rate limiting or an uncapped spend.

The "why is this higher" note (criterion 3) should be generated by comparing week-over-week call counts by type. If entity enrichment calls increased by > 50%, check if new connector data arrived that week and surface that as the explanation. This is a simple heuristic, not AI-generated commentary.

Pricing constants: store as a struct in `cost_estimation.rs` with a `last_updated` field. Include a comment with the Anthropic pricing page URL so it's easy to update when pricing changes. The per-token prices for Haiku, Sonnet, and Opus tiers should each be explicit constants.

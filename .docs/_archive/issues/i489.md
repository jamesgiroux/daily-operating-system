# I489 — VP Account Review report type

**Priority:** P1
**Area:** Backend / Reports
**Version:** 1.0.0
**Depends on:** I499 (health scoring engine), I508 (intelligence schema)

## Scope

**Single-user, per-account.** The VP generates this report from their own DailyOS intelligence — their meetings, their emails, their signals for one account. The VP is also an IC; they have their own relationship with the account. This report does not require access to other users' intelligence.

Portfolio-level aggregation (seeing IC intelligence, team views, cross-user data) is a v1.2.0+ concern addressed by the hybrid architecture (Google Drive + Remote DB). See `.docs/research/2026-03-01-portfolio-intelligence-architecture.md`.

## Problem

DailyOS produces CSM-level reports (Account Health, EBR/QBR) but nothing designed for the VP of Key Accounts. A VP reviewing an account needs a strategic assessment — risk/opportunity matrix with severity and timeline, stakeholder alignment against role vocabulary, engagement health trends, and leadership-level actions (escalation, exec alignment, strategic pivots). CSM-level reports recommend "schedule a call" or "send a follow-up" — a VP needs "realign executive sponsorship" or "escalate pricing to SVP Sales."

The key difference from a CSM-level Account Health Review is not the data — it's the **lens**. The same intelligence, viewed through a leadership prompt, produces strategic observations instead of operational tasks. The system role ("chief of staff preparing for a VP") and the action-level gate ("VP-level strategic actions only") are what make this a VP report.

## Design

### Report Type

Add `ReportType::VpAccountReview` variant. DB string: `"vp_account_review"`. Entity scope: single account.

### Module Structure

New file `src-tauri/src/reports/vp_account_review.rs` following the established two-phase pattern:

1. **Phase 1 (gather):** `gather_vp_account_review_input()` — brief DB lock to collect entity intelligence, stakeholder data, meeting cadence, email signals, renewal date, health score + confidence (from I499), and user context (role, priorities).
2. **Phase 2 (generate):** Dispatched via `run_report_generation()` in `generator.rs`. No DB lock held.

### Output Schema (`VpAccountReviewContent`)

```rust
pub struct VpAccountReviewContent {
    pub strategic_assessment: String,           // 3 sentences, executive-level
    pub risk_opportunity_matrix: Vec<RiskOpportunityItem>,
    pub stakeholder_alignment: Vec<StakeholderAlignmentItem>,
    pub engagement_health: EngagementHealth,
    pub value_narrative: String,                // 2-3 sentences, cited to real events
    pub renewal_expansion_outlook: Option<RenewalOutlook>,
    pub recommended_leadership_actions: Vec<LeadershipAction>, // exactly 3
}

pub struct RiskOpportunityItem {
    pub item: String,                           // max 20 words
    pub category: String,                       // "risk" | "opportunity"
    pub severity: String,                       // "critical" | "high" | "medium" | "low"
    pub timeline: String,                       // "immediate" | "30_days" | "90_days" | "strategic"
    pub source: Option<String>,                 // meeting date or signal reference
}

pub struct StakeholderAlignmentItem {
    pub name: String,
    pub role: String,                           // uses active preset vocabulary
    pub alignment: String,                      // "aligned" | "neutral" | "misaligned" | "unknown"
    pub last_engagement: Option<String>,        // date of last meeting
    pub notes: Option<String>,                  // max 15 words
}

pub struct EngagementHealth {
    pub meeting_cadence: String,                // "X meetings in 90 days, trend Y"
    pub email_responsiveness: String,           // "X email signals, sentiment Z"
    pub trend: String,                          // "improving" | "stable" | "declining"
    pub trend_rationale: String,                // 1 sentence
}

pub struct RenewalOutlook {
    pub renewal_date: String,
    pub confidence: String,                     // "high" | "moderate" | "low"
    pub rationale: String,                      // 2 sentences, grounded in data
    pub expansion_potential: Option<String>,     // 1 sentence if signals exist
}

pub struct LeadershipAction {
    pub action: String,                         // verb phrase, VP-level
    pub rationale: String,                      // why this matters strategically
    pub timeline: String,                       // "this week" | "this month" | "this quarter"
}
```

### Prompt Design

- **System role:** "You are a chief of staff preparing a VP-level account review."
- **Preset-aware vocabulary:** Stakeholder roles use the active preset's `stakeholder_roles` labels. Entity noun adapts (account/deal/client/partner).
- **Data sources:** Entity Intelligence Assessment (primary), meeting history (90 days), email signals, stakeholder list from `entity_people`, health score + confidence from I499 health engine, I508 intelligence dimensions (Strategic Assessment, Relationship Health, Engagement Cadence, Value & Outcomes, Commercial Context), renewal date from `account_events`.
- **Action level gate:** Explicit prompt instruction: "Recommended Leadership Actions must be VP-level strategic actions (escalate, realign sponsorship, adjust investment, engage executive). Do NOT recommend CSM-level operational tasks (schedule call, send email, update CRM)."
- **Citation rule:** Risk items and value narrative must cite real signal history or meeting dates. No hallucination.
- **Internal content filter:** Same as EBR/QBR — exclude internal team debrief content from customer-facing sections.
- **No internal jargon:** No "signal", "entity", "enrichment", "intelligence" in output.

## Acceptance Criteria

1. Report contains: strategic assessment (3 sentences), risk/opportunity matrix with severity and timeline, stakeholder alignment using preset vocabulary, engagement health (meeting cadence + email responsiveness + trend), value narrative cited to real events, renewal/expansion outlook with confidence rationale, 3 recommended leadership actions
2. "Recommended Leadership Actions" are VP-level (escalation, exec alignment, strategic), NOT CSM-level (schedule call, send email)
3. Report references actual stakeholders by name from entity_people
4. Risks cited from real signal history, not hallucinated
5. Account with renewal in 90 days has non-empty Renewal Outlook section
6. Preset-aware: stakeholder roles use active preset vocabulary
7. No internal DailyOS terminology in output (no "signal", "entity", "enrichment")

## Out of Scope

- Frontend page for VP Account Review (separate issue — report renders via existing report viewer)
- Comparison across accounts (that is Portfolio Health Summary, I491)
- Batch generation of VP Account Reviews for all accounts
- Custom section ordering or template editing
- Multi-user intelligence access — this report uses the current user's intelligence only. VP seeing IC intelligence requires the shared intelligence layer (v1.2.0+, see `.docs/research/2026-03-01-portfolio-intelligence-architecture.md`)

## Implementation Notes

### Shared helper: `meeting_count_in_window()`

No time-windowed meeting count query exists in the codebase. I489 is the first report to need "meeting count in last N days" per entity — this should be implemented as a shared helper in `db/meetings.rs` (or similar) so I496 (Stakeholder Map) and I498 (Coaching Patterns) can reuse it rather than each report writing its own SQL.

### Competitive context

The `competitive_position` section of VP Account Review depends on competitive intelligence from I508 (Intelligence Schema — Strategic Assessment dimension). Until I508 ships and enrichment runs, this section will be empty/omitted for most accounts. The prompt should handle this gracefully: "No competitive intelligence available. Mention competitors in meeting notes to surface this context."

## Relationship to Portfolio Architecture

This report works in single-user mode because the VP is also an IC — they have their own meetings, emails, and signals with each account. The report renders from the VP's personal intelligence.json, not aggregated team data.

When the hybrid shared intelligence layer ships (v1.2.0+), the same report can optionally pull IC intelligence from the shared layer to enrich the VP's own view. That's an enhancement, not a prerequisite. The report is useful today without it.

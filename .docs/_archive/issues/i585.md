# I585 — Persist Value Delivered

**Version:** v1.0.2
**Area:** Backend + Frontend / Intelligence
**Priority:** P0

## Problem

`value_delivered` is computed fresh by the LLM during report generation, stored in memory, and lost when the report closes. User edits are ephemeral. This means:
- The same value statements are regenerated every time (wasting LLM calls)
- User corrections don't persist (violates correction-as-signal principle)
- Value delivered doesn't feed health scoring or meeting prep context
- No historical record of value accumulation over time

## Design

### Schema

New migration: Add `value_delivered_json TEXT` column to `entity_assessment` table.

Structure (JSON array):
```json
[
  {
    "id": "uuid",
    "statement": "Reduced ticket volume by 40% in Q1",
    "date": "2026-01-15",
    "source": "meeting",
    "source_id": "meeting-uuid",
    "impact": "cost",
    "confirmed_by_user": false,
    "added_at": "2026-01-20T10:00:00Z"
  }
]
```

### Backend — Persistence

**On enrichment**: When intelligence enrichment extracts `value_delivered`, merge with existing persisted values:
- New items: append with `confirmed_by_user: false`
- Existing items (fuzzy match on statement): update date/source if newer, don't overwrite user-confirmed items
- Never delete user-confirmed items during re-enrichment

**On user edit**: New service method `update_value_delivered(entity_id, items)`:
- Persists the full array to `entity_assessment.value_delivered_json`
- Items edited by user get `confirmed_by_user: true`
- Emits `value_delivered_updated` signal with `source: "user_correction"`, confidence 1.0

**On report generation**: Report reads from persisted `value_delivered_json` instead of regenerating. LLM only fills gaps (entities with no persisted value). User edits in report UI call `update_value_delivered` to persist changes.

### Intelligence Loop Integration

**Health scoring**: `value_delivered` count and recency feeds the **value_success dimension** (dimension 5). Accounts with recent, user-confirmed value statements score higher than accounts with only AI-inferred value.

**Meeting prep context**: Add to `gather_account_context()`:
- "Value already delivered: [list of confirmed statements]"
- CS user walks into a meeting knowing what value has been proven

**Enrichment prompt**: Add persisted value to `build_intelligence_context()` so the LLM can reference existing value when generating new assessments (avoids contradicting what the user has already confirmed).

### Frontend

Report components (`ValueDeliveredSlide.tsx`, account detail Value chapter) read from persisted data. Edits trigger `update_value_delivered` backend call. Visual indicator for user-confirmed vs AI-inferred items (subtle, e.g., checkmark on confirmed).

## Mock Data

Requires I597 (action mock data expansion). Seeded `value_delivered_json`:
- Acme: 2 items (1 user-confirmed, 1 AI-inferred) — verifies confirmed items survive re-enrichment
- Globex: 1 item (user-confirmed) — verifies sparse account renders
- Initech: empty array — verifies empty state (too new for value)

## Acceptance Criteria

1. `value_delivered_json` column exists on `entity_assessment` table
2. Intelligence enrichment writes extracted value_delivered to the column
3. User edits in report UI persist to the column via `update_value_delivered` service method
4. User-confirmed items are never overwritten by re-enrichment
5. `update_value_delivered` emits signal with `source: "user_correction"`
6. Report generation reads from persisted data, only fills gaps via LLM
7. Meeting prep context includes persisted value delivered statements
8. Value count and recency feeds value_success health dimension
9. Account detail Value chapter renders persisted (not ephemeral) data
10. **Mock data**: Acme shows 2 value items (1 confirmed badge, 1 AI-inferred), Globex shows 1, Initech shows empty state. Meeting prep for Acme includes value statements.

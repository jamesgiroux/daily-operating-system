# I588 — Recommended Actions from Intelligence

**Version:** v1.0.2
**Area:** Backend / Intelligence
**Priority:** P1

## Problem

`EntityAssessment.recommended_actions` exists as a field but is always initialized as `Vec::new()`. The intelligence enrichment pipeline never generates concrete next-step recommendations. This is the gap between "here's what's happening with Acme" (assessment) and "here's what you should do about it" (actionable intelligence). A chief of staff doesn't just report — they recommend.

## Design

### Enrichment Prompt Addition

In `intelligence/prompts.rs`, add a `recommended_actions` section to the entity enrichment prompt:

```
Based on the account's current health, recent signals, and open commitments,
recommend 2-3 specific, concrete actions the CS manager should take.

Rules:
- Each action must be specific enough to execute without further research
- Reference specific people, meetings, or signals when possible
- Prioritize: risk mitigation > relationship deepening > expansion opportunity
- Do NOT recommend actions that duplicate existing pending actions
- Format: { "title": "...", "rationale": "...", "priority": "P1|P2|P3", "suggested_due": "ISO date or null" }
```

### Context for Recommendations

Feed the prompt:
- Current health score and dimension breakdown (which dimensions are weak)
- Open actions (so recommendations don't duplicate)
- Recent signals (what's changed)
- Upcoming meetings (time-sensitive opportunities)
- Cadence data (silence = recommend outreach)
- Objective progress (stalled milestones = recommend action)

### Backend — Persistence & Lifecycle

Store recommendations in `entity_assessment.recommended_actions_json` (already exists conceptually as `recommended_actions` field).

Lifecycle:
- Generated during enrichment (intel queue)
- Persisted per entity
- When user accepts a recommendation → create proposed action with `source_type: "intelligence"`, `source_id: entity_id`
- When user dismisses → emit `recommendation_rejected` signal (feeds Bayesian learning)
- Regenerated on next enrichment cycle (stale recommendations replaced)

### Frontend — Surface

**Account Detail**: New "Recommended" section in The Work chapter. Shows 2-3 cards with title, rationale, and "Track" / "Dismiss" actions. "Track" creates a proposed action. "Dismiss" hides with correction signal.

**Daily Briefing**: If an account with a meeting today has recommendations, include top recommendation as a callout: "Consider: [recommendation title] before your Acme call."

**Actions Page**: Recommendations surface as a distinct category in the "Suggested" tab — distinguished from transcript-extracted proposals by source label "Based on account intelligence."

### Signal Emission

- `recommendation_accepted` (confidence 0.8) — user tracked the recommendation
- `recommendation_rejected` (confidence 0.3) — user dismissed
- Both feed Bayesian source weights for intelligence-generated actions

## Mock Data

Requires I597 (action mock data expansion). Pre-seeded recommendations:
- Acme: 2 recommendations (exec review + adoption metrics) — verifies account detail rendering
- Globex: 2 recommendations (Lisa Park 1:1 + renewal brief) — verifies at-risk recommendations
- Initech: none (sparse account, verifies empty state)

## Acceptance Criteria

1. Entity enrichment generates 2-3 recommended actions per entity (when sufficient context exists)
2. Recommendations are specific and concrete (not generic "follow up with customer")
3. Recommendations don't duplicate existing pending actions
4. Recommendations persist in `entity_assessment` and survive page reload
5. Account detail shows recommendations with "Track" and "Dismiss" actions
6. "Track" creates a proposed action with `source_type: "intelligence"`
7. "Dismiss" emits rejection signal for Bayesian learning
8. Daily briefing includes top recommendation for accounts with meetings today
9. Sparse accounts (insufficient context) get no recommendations (not empty/generic ones)
10. **Mock data**: Acme and Globex show recommendations on account detail. Initech shows none. "Track" action creates proposed action visible in Actions page.

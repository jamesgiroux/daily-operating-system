# I498 — Coaching Patterns report type (absorbs I461)

**Priority:** P2
**Area:** Backend / Reports
**Version:** 1.0.0
**Depends on:** I499 (health scoring engine), I508 (intelligence schema)

## Problem

CS managers have no structured way to coach CSMs based on actual behavioral patterns. Coaching conversations default to anecdotes and gut feel rather than data. DailyOS has the raw data — meeting cadence per account, stakeholder engagement breadth, signal response latency, and Monthly Wrapped reports — but nothing synthesizes this into a coaching document. A manager needs to see: "You're meeting Account X weekly but Account Y hasn't had a touchpoint in 45 days. Your executive engagement is 15% of meetings vs. a book-level norm of 25%."

This absorbs I461 (Coaching Patterns from the original v1.0.1 scope). It is a manager-to-CSM document with an observational, factual tone — never surveillance-oriented or judgmental.

## Design

### Report Type

Add `ReportType::CoachingPatterns` variant. DB string: `"coaching_patterns"`. Entity scope: **user entity** (cross-account, like Portfolio Health). Only available when 3+ Monthly Wrapped reports exist for the user.

### Availability Gate

The gather function checks: `SELECT COUNT(*) FROM reports WHERE report_type = 'monthly_wrapped' AND entity_type = 'user'`. If count < 3, return error: "Coaching Patterns requires at least 3 Monthly Wrapped reports to identify meaningful patterns. You have N."

This ensures the report has enough longitudinal data to identify patterns rather than producing noise from a single month's snapshot.

### Module Structure

New file `src-tauri/src/reports/coaching_patterns.rs` following the two-phase pattern.

### Output Schema (`CoachingPatternsContent`)

```rust
pub struct CoachingPatternsContent {
    pub meeting_cadence_analysis: MeetingCadenceAnalysis,
    pub stakeholder_engagement_patterns: StakeholderEngagement,
    pub signal_response_patterns: SignalResponsePatterns,
    pub book_level_comparison: BookLevelComparison,
    pub coaching_recommendations: Vec<CoachingRecommendation>, // exactly 3
}

pub struct MeetingCadenceAnalysis {
    pub total_meetings_90d: i64,
    pub meetings_per_account: Vec<AccountCadence>,
    pub cadence_narrative: String,                   // 2 sentences: what the pattern shows
    pub over_indexed: Vec<String>,                   // account names with disproportionate meetings
    pub under_indexed: Vec<String>,                  // account names with too few meetings
}

pub struct AccountCadence {
    pub account_name: String,
    pub meeting_count_90d: i64,
    pub meeting_count_30d: i64,
    pub trend: String,                               // "increasing" | "stable" | "decreasing"
    pub last_meeting_date: Option<String>,
    pub days_since_last_meeting: i64,
}

pub struct StakeholderEngagement {
    pub total_unique_contacts: i64,
    pub executive_meeting_ratio: f64,                // exec meetings / total meetings
    pub breadth_vs_depth: String,                    // "broad" | "balanced" | "narrow"
    pub breadth_narrative: String,                   // 1 sentence
    pub single_threaded_accounts: Vec<String>,       // account names with only 1 active contact
}

pub struct SignalResponsePatterns {
    pub avg_response_days: Option<f64>,              // avg days between signal creation and next action
    pub fastest_response: Option<ResponseExample>,
    pub slowest_response: Option<ResponseExample>,
    pub pattern_narrative: String,                   // 1 sentence on responsiveness
}

pub struct ResponseExample {
    pub account_name: String,
    pub signal_type: String,
    pub days_to_response: f64,
}

pub struct BookLevelComparison {
    pub user_meeting_cadence_avg: f64,               // meetings per account per 90d
    pub book_norm_cadence_avg: f64,                  // computed from all accounts
    pub user_exec_ratio: f64,
    pub book_norm_exec_ratio: f64,
    pub comparison_narrative: String,                // 2 sentences
}

pub struct CoachingRecommendation {
    pub recommendation: String,                      // specific, names people/accounts
    pub rationale: String,                           // what data supports this
    pub category: String,                            // "cadence" | "breadth" | "responsiveness" | "executive_access"
}
```

### Prompt Design

- **System role:** "You are a coaching advisor preparing a data-driven coaching patterns report. Your tone is observational, factual, and developmental — never judgmental or surveillance-oriented. You are helping a professional grow, not auditing them."
- **Tone gate:** Explicit prompt instruction: "Language must be coaching-oriented. Use: 'The data shows...', 'Consider...', 'An opportunity exists...'. Never use: 'Failed to...', 'Neglected...', 'Should have...', 'Problem with...'. Frame everything as an observation leading to a growth opportunity."
- **Real data only:** Meeting cadence analysis reflects actual `meetings_history` data. Stakeholder engagement uses real `meeting_attendees` data. Signal response patterns use real `signal_events` timestamps.
- **Book-level norms:** Computed mechanically in the gather function by averaging across all accounts. These are the user's own norms across their book, not team-level benchmarks (team aggregation is out of scope).
- **Specificity rule:** "Coaching recommendations must name specific accounts and/or people. 'Schedule executive touchpoint with [Name] at [Account]' not 'engage more with stakeholders'."

```rust
    //    - For each signal_event, find next meeting or action within 14 days
    //    - Compute response latency distribution
    // 5. Monthly Wrapped report summaries (last 3+ reports' content_json)
    //    - Extract key themes for longitudinal pattern
    // 6. Book-level norm computation:
    //    - avg meetings per account
    //    - avg exec meeting ratio
    // 7. Output schema + tone rules
}
```

### Monthly Wrapped Integration

The gather function reads the last 3+ Monthly Wrapped reports from the `reports` table (`report_type = 'monthly_wrapped'`) and includes a summary of each in the prompt. This gives the LLM longitudinal context to identify patterns over time rather than just a point-in-time snapshot.

## Acceptance Criteria

1. Meeting cadence analysis reflects actual calendar data, not hallucinated patterns — verify against `meetings_history` table
2. Stakeholder engagement patterns: breadth vs depth, executive vs operational ratio
3. Signal response patterns: how quickly signals are acted on (delta between signal creation and next meeting/action)
4. Comparison to book-level norms uses real data from other accounts' intelligence
5. 3 coaching recommendations that are specific ("Schedule executive touchpoint with [name]"), not generic ("engage more with stakeholders")
6. Language is coaching-oriented, not surveillance-oriented — observational, factual, never judgmental
7. Only available when 3+ Monthly Wrapped reports exist for the user
8. No internal DailyOS terminology in output

## Out of Scope

- Team-level coaching (comparing across CSMs) — requires governance design and multi-user support
- Manager dashboard or coaching session tracking
- Automated coaching cadence (e.g., generate coaching report monthly)
- Integration with 1:1 meeting prep (surfacing coaching points in manager/CSM meeting briefings)
- Historical coaching pattern trends (comparing this report to previous coaching reports)

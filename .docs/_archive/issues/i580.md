# I580 — Commitment → Action Promotion

**Version:** v1.0.1
**Area:** Backend + Frontend / Email
**Priority:** P1

## Problem

Email commitments are extracted and displayed ("Will send renewal proposal by Friday") but are dead ends. Users see them, nod, and forget. There's no path from "someone committed to X" to "I'm tracking X." This is a broken loop — intelligence is produced but never consumed as action.

## Design

### Frontend

Each commitment item in the COMMITMENTS section gets a **"Track"** action (plus icon or similar, consistent with triage action style from I579).

Clicking "Track" reveals a lightweight **inline form** (not a modal — stays in context):
- **Title**: pre-filled with commitment text (editable)
- **Entity**: pre-filled from email's linked entity (editable, entity picker)
- **Due date**: pre-filled with AI-extracted date if present, otherwise blank (date picker)
- **Owner**: pre-filled with sender name (the person who made the commitment)
- **Submit** / **Cancel** buttons

After promotion:
- Commitment item shows **"Tracked"** state — muted text, small link icon to the action
- Not dismissed — still visible as context, but clearly handled
- Tracked state persisted so it survives page reload (check if action with `source_id = email_id` and matching text exists)

### Backend

Calls existing `create_action` service method via `ServiceLayer` with:
- `title`: commitment text
- `entity_id`, `entity_type`: from email linkage
- `source`: `"email_commitment"`
- `source_id`: email_id (links action back to originating email)
- `due_date`: if provided
- `owner`: sender name

**Signal emission**: Action creation emits signal via existing ServiceLayer action creation flow. Source `"email_commitment"` allows feedback learning to distinguish email-derived actions from manual ones.

### Intelligence Loop

- Action creation from email commitment → signal with source `"email_commitment"` → feeds entity engagement dimension
- If the action is later completed, the completion signal references the email origin — closing the full loop from "email received" → "commitment extracted" → "action tracked" → "action completed"

## Acceptance Criteria

1. Each commitment in the COMMITMENTS section has a "Track" action
2. Clicking "Track" shows an inline form pre-filled with commitment text, entity, and AI-extracted due date
3. Submitting creates an Action via `ServiceLayer` with `source: "email_commitment"` and `source_id` linking to the email
4. After creation, the commitment shows a "Tracked" indicator with visual distinction from untracked items
5. "Tracked" state survives page reload (derived from existing action with matching source_id)
6. The created action appears on the Actions page with entity linkage
7. Action creation emits a signal that feeds entity engagement scoring

# I534 — Portfolio Reader: Cross-IC Intelligence from Shared Drive

**Priority:** P1
**Area:** Backend + Frontend
**Version:** v1.1.0
**Depends on:** I533 (publication engine — reports must be published before they can be read), I492 (portfolio page — consumer of aggregated data)

## Problem

In v1.0.0, the Portfolio Health report (I491) and Portfolio page (I492) only see accounts in the current user's local DailyOS instance. A VP managing 20 accounts directly gets a portfolio view of those 20. But a VP who manages 3 accounts directly and has 5 ICs who each manage 15-20 accounts sees only their 3 — not the 75-100 accounts in their org.

The publication engine (I533) solves the output side: ICs publish their reports to a Google Drive Shared folder. The portfolio reader solves the consumption side: a VP's DailyOS reads those published reports and synthesizes them into a portfolio-level view that spans their entire org.

### Why read from Drive, not from Glean?

- **Structured parsing.** Published reports have a known format (DailyOS generates them). The reader can parse structured sections reliably. Glean search returns snippets, not full documents with predictable structure.
- **Completeness.** The reader needs the full report content (all sections, all data points). Glean search optimizes for relevance, not completeness — it may omit sections it considers less relevant to the query.
- **Freshness.** Drive API returns the latest file content immediately. Glean crawls every 10 minutes — a report published 5 minutes ago might not be in Glean yet.
- **No Glean dependency.** The portfolio reader works even if Glean is not connected. It only needs Google Drive access.

## Design

### 1. Reading published reports

```rust
pub async fn read_published_reports(
    config: &PublicationConfig,
    google: &GoogleApi,
) -> Result<Vec<PublishedReport>, ReaderError> {
    // 1. List all subfolders in the configured Shared Drive folder
    // 2. For each subfolder (= one account):
    //    a. List files, sorted by most recent
    //    b. Read the most recent report of each type
    //    c. Parse report content into structured data
    // 3. Return aggregated results
}

pub struct PublishedReport {
    pub account_name: String,          // from subfolder name
    pub report_type: String,           // parsed from filename
    pub published_date: String,        // from filename or file metadata
    pub content: ParsedReportContent,  // structured content extracted from the document
    pub drive_url: String,             // link to the original document
    pub published_by: Option<String>,  // from file owner metadata
}

pub struct ParsedReportContent {
    pub health_score: Option<i64>,
    pub health_trend: Option<String>,
    pub bucket: Option<String>,         // Growth Focus / At Risk / Autopilot
    pub renewal_date: Option<String>,
    pub renewal_readiness: Option<String>,
    pub arr: Option<String>,
    pub key_risks: Vec<String>,
    pub exceptions: Vec<String>,
    pub executive_summary: Option<String>,
}
```

### 2. Report parsing strategy

Published reports are DailyOS-generated, so their structure is predictable. Two parsing approaches:

**Option A: JSON sidecar (recommended).** When publishing a report, also write a `{report-name}.meta.json` file alongside the PDF/Doc. This JSON contains the structured data (`ParsedReportContent`) pre-extracted. The reader reads the sidecar instead of parsing the document.

```json
{
  "report_type": "vp_account_review",
  "account_name": "Acme Corp",
  "published_at": "2026-03-04T09:00:00Z",
  "health_score": 72,
  "health_trend": "stable",
  "bucket": "autopilot",
  "renewal_date": "2026-10-15",
  "renewal_readiness": "ready",
  "arr": "$1.2M",
  "key_risks": ["Champion recently promoted — new POC untested"],
  "executive_summary": "Stable account with strong engagement..."
}
```

**Why sidecar over document parsing:** PDFs and Google Docs are rendering formats, not data formats. Parsing structured data from a PDF is fragile and slow. Writing a machine-readable sidecar alongside the human-readable document is clean and reliable.

**Option B: Parse from document.** Use Google Docs API to read document structure (headings, paragraphs) and extract structured data. More brittle, but works without a sidecar. Fallback if sidecar is missing.

### 3. Portfolio synthesis

The portfolio reader feeds into two consumers:

**I491 (Portfolio Health Summary report):** The report's gather function checks for published reports from other ICs. If found, the prompt receives per-account summaries from published reports in addition to locally-managed accounts. The report clearly distinguishes "my accounts" from "team accounts."

**I492 (Portfolio page):** The page shows both local accounts (with full intelligence) and published accounts (with summary-level data from sidecar). Published accounts have a visual indicator: "via {IC name}" or "published" badge. Clicking navigates to Drive URL (not account detail, since the account doesn't exist locally).

### 4. Freshness and caching

- **Read frequency:** Portfolio reader queries Drive once per app launch and once per hour thereafter
- **Cache:** Parsed results cached in memory with 1-hour TTL
- **Staleness indicator:** If the most recent report for an account is > 7 days old, show "Stale" indicator. > 30 days: "Outdated"
- **No write-back:** The reader never modifies files in Drive. It's purely consumptive.

### 5. Permission model

The reader only sees files the authenticated user has access to in the Shared Drive. Google Drive permissions are the governance layer — if a VP has access to the Shared Drive, they see all published reports. If an IC has access only to their own subfolder, they see only their own.

No DailyOS-level permission logic needed. Google Workspace governance applies.

## Acceptance Criteria

1. VP DailyOS reads published reports from Shared Drive folder. Portfolio page shows both local accounts (full detail) and published accounts (summary from sidecar).
2. JSON sidecar written alongside every published report with structured metadata (health score, trend, bucket, ARR, risks, renewal)
3. Published accounts show "via {IC name}" or "published" badge on portfolio page
4. Click on published account opens Drive URL (not a local account detail page)
5. Staleness indicator: report > 7 days shows "Stale", > 30 days shows "Outdated"
6. Portfolio Health Summary report incorporates published accounts — clearly distinguishes "my accounts" from "team accounts"
7. Cache: Drive queries happen once per hour, not on every page load
8. Works without Glean — only requires Google Drive access
9. Reader is purely consumptive — never modifies files in Drive

## Out of Scope

- Creating local account entities from published reports (the reader shows summary data, not full intelligence)
- Merging published intelligence with local intelligence for the same account (if VP also has the account locally, local data takes precedence)
- Historical trend from published reports (comparing this week's sidecar to last week's)
- Publishing health scores to a Glean custom datasource for structured queries
- Conflict resolution between ICs publishing reports for the same account

## Relationship to Other Issues

- **I533** (publication engine) is the prerequisite — reports must be published with sidecars before the reader can consume them
- **I491** (Portfolio Health report) is a consumer — the report gathers both local and published accounts
- **I492** (Portfolio page) is a consumer — the page renders both local and published accounts
- **I535** (Glean Agent integration) is complementary — Glean Agents provide org-level analysis, the reader provides IC-published intelligence. Different sources, same portfolio synthesis.

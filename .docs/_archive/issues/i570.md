# I570 — Move Keychain retry/backoff off Tokio threads

**Priority:** P2
**Area:** Backend / Performance
**Version:** v1.0.0 (Phase 6)
**Depends on:** None
**Type:** Enhancement

## Problem Statement

Three `run_security_cmd()` functions use `std::thread::sleep` for exponential backoff when the macOS Keychain returns transient errors (errno 35 / EAGAIN / "temporarily unavailable"):

1. **`google_api/token_store.rs:182`** — `run_security_cmd()` for Google OAuth token operations. Backoff: 150ms base, up to 4 retries, max single sleep = 2.4s.
2. **`glean/token_store.rs:64`** — `run_security_cmd()` for Glean OAuth token operations. Same parameters.
3. **`clay/oauth.rs:44`** — `run_security_cmd()` for Clay/Smithery API key retrieval. Same parameters.

All three use the same pattern:
```rust
let delay = std::time::Duration::from_millis(BASE_MS * 2u64.pow(attempt));
std::thread::sleep(delay);
```

`std::thread::sleep` blocks the OS thread it runs on. When these functions are called from an async context (e.g., `get_valid_access_token()` in Glean auth refresh, or Google token refresh during `refresh_emails`), they block whatever thread they happen to land on. If that's a Tokio worker thread (or even a `spawn_blocking` thread from a limited pool), the blocked thread can't serve other tasks.

The worst case is 4 retries with exponential backoff: 150 + 300 + 600 + 1200 = 2.25 seconds of thread blocking. During that time, one Tokio worker thread is completely idle.

## Solution

Wrap each `run_security_cmd()` function in `tokio::task::spawn_blocking` at the call sites where they're invoked from async code, OR convert the retry loop to use `tokio::time::sleep` in an async version.

### Approach: Async wrapper with tokio::time::sleep

Add an `async fn run_security_cmd_async()` alongside the existing sync version in each file. The async version:
1. Calls `std::process::Command` (sync, but fast — Keychain operations are <10ms when not contending).
2. On transient failure, uses `tokio::time::sleep(delay).await` instead of `std::thread::sleep(delay)`.

The sync `run_security_cmd()` remains for callers that are already on a blocking thread (e.g., `peek_account_email()` which is called from sync contexts).

### Call sites to convert

**`google_api/token_store.rs`:**
- `load_token()` / `save_token()` / `delete_token()` — called from both sync and async contexts. The macOS-specific functions (`load_token_macos`, `save_token_macos`, `delete_token_macos`) all use `run_security_cmd`. Add async variants that use `run_security_cmd_async`.
- `peek_account_email()` — sync probe, keep using sync version.

**`glean/token_store.rs`:**
- `load_token()` / `save_token()` / `delete_token()` — all called from async Glean auth flow. Add async variants.
- `peek_account_email()` / `peek_account_name()` — sync probes, keep using sync version.

**`clay/oauth.rs`:**
- `get_smithery_api_key()` — called from enrichment pipeline which is async. Add async variant.

### Alternative: spawn_blocking at call sites

Simpler but coarser — wraps the entire token operation (not just the sleep) in `spawn_blocking`. Acceptable if I569 already puts the Glean provider on a blocking thread. For Google token refresh (called during email sync), `spawn_blocking` around the refresh call is the right granularity.

## Acceptance Criteria

1. No `std::thread::sleep` executes on a Tokio worker thread. Either: (a) the sleep is replaced with `tokio::time::sleep` in an async code path, or (b) the entire operation is wrapped in `spawn_blocking` at the call site.
2. Keychain retry logic still works: transient Keychain contention (EAGAIN) is retried with exponential backoff up to 4 times.
3. Google OAuth token refresh during `refresh_emails` does not block the Tokio worker thread during retry backoff.
4. Glean token refresh does not block the Tokio worker thread during retry backoff.
5. Sync callers (`peek_account_email`, `peek_account_name`) continue to work without requiring a Tokio runtime.
6. `cargo test` — all pass.
7. `cargo clippy -- -D warnings` — clean.

## What This Does NOT Do

- Does not change the retry parameters (4 retries, 150ms base backoff) — those are appropriate for Keychain contention.
- Does not convert `std::process::Command` (the `security` CLI invocation) to an async process — the command itself is fast, only the backoff sleep is the problem.
- Does not address `block_in_place` in the Glean context provider — that's I569.

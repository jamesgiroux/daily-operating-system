# I652 — Email Enrichment Filtering & Workflow Optimization

**Area:** Backend / Workflow / Performance
**Priority:** P1
**Version:** v1.1.0 (Phase 6)
**Effort:** Large
**Status:** Spec finalized 2026-03-30 (Plan-Eng-Review complete)
**Source:** User feedback (2026-03-30). Email enrichment blocks daily briefing generation for 2-5 minutes, enriches all 20+ candidate emails when only 5 are displayed, and lacks recency filtering. Real-world observation: workflow attempted to enrich 27 emails, causing 12+ minute hang. Root cause: token optimization in v1.1.0 early work increased workflow efficiency elsewhere without proportional reduction in enrichment scope. This issue corrects the proportions.

---

## Problem

Email enrichment currently runs early in the daily workflow (Phase 1: Prepare) as a blocking step before briefing generation. This causes three critical issues:

1. **Blocking pipeline:** If email AI calls timeout or queue up, the entire briefing is delayed. Users see "preparing..." for 2-5 minutes instead of <45 seconds.

2. **Over-enrichment:** All qualifying emails are enriched regardless of display count. Current filter selects 27/56 emails when briefing only shows 5. Wastes 50-75% of email enrichment token budget.

3. **Broken filtering:** The enrichment selector (high-priority + medium-priority with known domains) lacks:
   - Display count limit (enriches 20+ when showing 5)
   - Recency filtering (re-enriches old emails every day)
   - New response detection (enriches stale threads with no recent activity)
   - Relevance decay (old resolved topics still enriched)
   - Duplicate detection (same email enriched multiple times)

Real-world impact: Workflow attempted to enrich 27 emails, hung for 12+ minutes.

---

## Design Direction

1. **DB-first architecture:** Read email list from `emails` table (I368), not from emails.json (which is write-only export per ADR-0062). All filtering and deduplication logic operates on DB state. emails.json is the briefing artifact output only.

2. **Genuine fire-and-forget async enrichment:** Enrichment spawns as truly async task immediately after schedule.json is written. Does NOT block briefing generation or narrative completion. Briefing renders with unenriched data; email enrichment completes independently in background. Next refresh picks up enriched data.

3. **Three-gate filtering with deduplication:**
   - **Gate 0 (Primary Deduplication):** Skip if already enriched in last 24h AND content unchanged (compare current snippet/subject against prior state). If content changed (new reply in thread), re-enrich.
   - **Gate 1 (Selective Filtering):** Apply ALL of: received in last 7 days, priority (high OR medium+known_domain), thread has response since yesterday.
   - **Gate 2 (Bounding):** Limit to 5-7 emails max, sorted by priority-tier-first then recency within tier.

4. **Graceful timeout & partial results:** Per-email 90s timeout (configurable). Timeout on one email → log warning, skip that email, continue enriching others. Partial result (some enriched, some not) is acceptable and correct.

5. **Smart re-enrichment:** Content-change detection prevents wasteful re-enrichment of stale emails while allowing re-enrichment when conversations evolve.

## Acceptance Criteria

### AC1: Skip Already-Enriched Emails (Gate 0 — Primary Deduplication)
**Do not re-enrich emails that are already enriched.** Gate 0 runs FIRST, before any selective filtering:

1. **Timestamp check:** If `enriched_at IS NOT NULL`, check if it's within the 24-hour window (enriched_at > NOW() - 24h).
2. **Content-change detection:** If enriched_at is recent (within 24h), compare current email `snippet` and `subject` against prior snapshot (batch-loaded from DB). If EITHER field changed → email content evolved → proceed to Gate 1 (re-enrich).
3. **Skip if no changes:** If enriched_at is recent AND both snippet and subject unchanged → skip email entirely, move to next.
4. **Proceed if expired:** If enriched_at IS NULL or older than 24h → proceed to Gate 1.

This prevents wasteful re-enrichment of stale emails while allowing re-enrichment when conversations evolve (new replies). Example: Email enriched Monday 9:00 with snippet "Can you send the contract?" is skipped Tuesday morning if snippet is identical. But if someone replies and Gmail adds "RE: Can you send the contract?" to snippet → re-enriched Tuesday.

**Data source:** Content snapshots (snippet, subject, received_at) are batch-loaded from `emails` table in a single query before Gate 0 loop begins (no N+1). Uses HashMap<email_id, snapshot> lookup inside loop.

### AC2: Selective Enrichment Logic (Gate 1 — Filtering)
For emails NOT already enriched (Gate 0), only enrich if meeting ALL three checks:

1. **Recency gate:** `received_at >= NOW() - 7 days`. Emails older than 7 days are skipped regardless of priority.
2. **Priority + domain gate:** EITHER of:
   - Priority = 'high' (no other checks needed), OR
   - Priority = 'medium' AND sender_domain (extracted from sender_email by split('@')) is in `known_domains` (customer/contact entity list from account/person entities).
3. **Newness gate:** Thread must have received a message after yesterday's workflow run. Compare `last_response_date` (field on email from directive) >= NOW() - 1 day. Prevents re-enriching stale threads where no one has replied since last briefing.

**Gates run in sequence:** Gate 0 (skip enriched) → Gate 1 (selective filtering) → Gate 2 (limit)

**Implementation:** `select_emails_for_enrichment()` function in `src-tauri/src/workflow/email_filter.rs`. Takes DB email list + yesterday DateTime + known_domains set, returns filtered list before Gate 2.

**Data source clarification:** `last_response_date` is extracted from Gmail API and added to `DirectiveEmail` struct during email fetch. Passed through directive → stored in `emails` table. Available at enrichment decision time.

### AC3: Limit to Display Count (Gate 2 — Bounding)
- Enrich maximum 5-7 emails (configurable, default 5 to match briefing display limit)
- **Sort priority-tier-first:** ALL high-priority emails (sorted by received_at descending) come before ANY medium-priority emails (sorted by received_at descending). This prevents medium-priority from pushing out high-priority.
- Example: 3 high-priority + 5 medium-priority qualifying → take [3 high, 2 medium] (limit=5). If limit=7, take [3 high, 4 medium].
- Never exceed limit even if more qualify
- Secondary sort within tier: received_at descending (most recent first)

### AC4: Pipeline Restructuring (Genuine Fire-and-Forget Async)
**Phase 1 (Prepare):**
- Generates `schedule.json` (calendar events, unenriched)
- Generates `emails.json` (email list, all unenriched, enriched_at = None on all)
- Returns to Phase 2 immediately (does NOT wait for enrichment)

**Phase 2 (Deliver & Enrich):**
1. Generates briefing narrative and focus (synchronously)
2. Writes `schedule.json` with narrative to disk
3. **Spawns async task for email enrichment** (fire-and-forget, does NOT await)
4. Returns immediately (briefing complete in <45 seconds)
5. Email enrichment runs independently in background:
   - Reads `emails` table from DB (not JSON)
   - Applies three-gate filtering (AC1-AC3)
   - Calls Claude AI per selected email (90s timeout per email)
   - Writes enriched results to DB (immediate per-email commits)
   - Updates `emails.json` export artifact (after enrichment completes or times out)
   - Invalidates briefing cache on completion (so next refresh picks up enriched data)

**Failure modes:**
- If enrichment times out or crashes: briefing already rendered and displayed (no blocking)
- Partial enrichment OK: some emails enriched, others not, both states in DB and exported to JSON
- Next workflow run respects enriched_at on already-enriched emails (Gate 0)

- Add DB migration to create column with default NULL

### AC6: New Response Detection
- Extend `DirectiveEmail` struct with `last_response_date: Option<String>` field
- During email fetch from Gmail API, extract the most recent message timestamp in each thread and populate `last_response_date`
- Pass through directive → store in `emails` table as `last_response_date` column
- Gate 1 (AC2) checks: `if last_response_date IS NOT NULL AND last_response_date >= NOW() - 1 day` → proceed. Otherwise fail gate.
- Example: Email from Monday with last reply Sunday → fails Gate 1 newness check on Tuesday (skipped). But if someone replies Monday evening, `last_response_date` updates → passes newness check on Tuesday morning.
- **Data source:** Gmail API response for thread metadata. Extracted during email sync, not computed.

### AC7: Graceful Degradation
- **Per-email timeout: 90 seconds (configurable via `Config.email_enrichment_timeout_seconds`, default 90)**
  - If Claude API call for one email exceeds 90s, immediately cancel that email's enrichment
  - Log warning: `[WARN] Email enrichment timeout: {email_id} after 90 seconds`
  - Skip that email (don't set enriched_at), continue with next email in batch
  - Return partial result: some emails enriched, others not (mixed state is correct)
  - Briefing continues rendering (already completed in Phase 2)
- **Empty enrichment response:**
  - If Claude returns empty/malformed enrichment, keep email in briefing with no summary
  - Render gracefully in UI (no summary badge, but email still shown, marked unenriched)
  - Log as warning for monitoring
- **No batch timeout:** Each email gets full 90s. If 5 emails enrich sequentially, total time can be 450s (OK, enrichment is async)
- **Partial writes to DB:** Each successful enrichment commits to DB immediately (no N+1 issue, each commit is final). If process crashes mid-batch, we keep what we have.

### Phase 2: Filtering Logic Module (email_filter.rs)

Create `src-tauri/src/workflow/email_filter.rs` with three-gate filter:

```rust
use chrono::{DateTime, Utc};
use std::collections::{HashMap, HashSet};

/// Email snapshot for content-change detection in Gate 0
pub struct EmailSnapshot {
    pub snippet: Option<String>,
    pub subject: Option<String>,
    pub received_at: DateTime<Utc>,
}

/// Gate 0: Skip already-enriched emails, unless content changed
pub fn gate_0_skip_enriched(
    email_id: &str,
    enriched_at: Option<DateTime<Utc>>,
    current_snippet: &Option<String>,
    current_subject: &Option<String>,
    prior_snapshots: &HashMap<String, EmailSnapshot>,
    now: DateTime<Utc>,
) -> bool {
    // If never enriched, pass to next gate
    let Some(enr_at) = enriched_at else { return false; };

    // If expired (>24h), pass to next gate
    if enr_at <= now - chrono::Duration::hours(24) {
        return false;
    }

    // Within 24h: check if content changed
    if let Some(prior) = prior_snapshots.get(email_id) {
        if current_snippet != &prior.snippet || current_subject != &prior.subject {
            // Content changed, allow re-enrichment
            return false;
        }
    }

    // Content unchanged, skip enrichment
    true
}

/// Gate 1: Selective filtering (priority + recency + newness)
pub fn gate_1_selective(
    priority: Option<&str>,
    received_at: DateTime<Utc>,
    last_response_date: Option<DateTime<Utc>>,
    known_domains: &HashSet<String>,
    sender_email: Option<&str>,
    now: DateTime<Utc>,
) -> bool {
    // Recency: must be within 7 days
    if received_at < now - chrono::Duration::days(7) {
        return false;
    }

    // Priority gate
    let is_high = priority.map(|p| p == "high").unwrap_or(false);
    let is_medium_known_domain = if priority.map(|p| p == "medium").unwrap_or(false) {
        let domain = sender_email
            .and_then(|e| e.split('@').nth(1))
            .unwrap_or("")
            .to_lowercase();
        !domain.is_empty() && known_domains.contains(&domain)
    } else {
        false
    };

    if !is_high && !is_medium_known_domain {
        return false;
    }

    // Newness gate: thread must have response since yesterday
    if let Some(response_date) = last_response_date {
        if response_date >= now - chrono::Duration::days(1) {
            return true;  // Pass all gates
        }
    }

    false  // Fail newness gate
}

/// Gate 2: Limit to N emails, priority-tier-first sort
pub fn gate_2_limit_and_sort(
    emails: Vec<DbEmail>,
    limit: usize,
) -> Vec<DbEmail> {
    let mut sorted = emails;

    // Sort: priority tier first (high before medium before low)
    // Then within tier: received_at descending (most recent first)
    sorted.sort_by(|a, b| {
        let priority_order = |p: Option<&str>| match p {
            Some("high") => 0,
            Some("medium") => 1,
            _ => 2,
        };

        let a_priority = priority_order(a.priority.as_deref());
        let b_priority = priority_order(b.priority.as_deref());

        if a_priority != b_priority {
            a_priority.cmp(&b_priority)
        } else {
            b.received_at.cmp(&a.received_at)  // Descending
        }
    });

    sorted.truncate(limit);
    sorted
}

/// Select emails for enrichment: apply all three gates
pub fn select_emails_for_enrichment(
    emails: Vec<DbEmail>,
    prior_snapshots: HashMap<String, EmailSnapshot>,
    known_domains: HashSet<String>,
    limit: usize,
    now: DateTime<Utc>,
) -> Vec<DbEmail> {
    let candidates: Vec<_> = emails
        .into_iter()
        .filter(|e| {
            // Gate 0
            if gate_0_skip_enriched(
                &e.email_id,
                e.enriched_at,
                &e.snippet,
                &e.subject,
                &prior_snapshots,
                now,
            ) {
                return false;
            }

            // Gate 1
            gate_1_selective(
                e.priority.as_deref(),
                e.received_at,
                e.last_response_date,
                &known_domains,
                e.sender_email.as_deref(),
                now,
            )
        })
        .collect();

    // Gate 2: limit and sort
    gate_2_limit_and_sort(candidates, limit)
}
```

pub async fn enrich_emails(
    state: &AppState,
    workspace: &Path,
    user_ctx: &UserContext,
    known_domains: &HashSet<String>,
) -> Result<(), String> {
    let data_dir = workspace.join("_today/data");
    let now = Utc::now();

    // Load emails from DB
    let emails: Vec<DbEmail> = state
        .db_read(|db| db.get_all_active_emails())
        .await
        .unwrap_or_default();

    // Batch-load prior snapshots for Gate 0 content-change check
    let prior_snapshots = get_email_snapshots_for_content_check(state).await?;

    // Apply three-gate filter
    let emails_to_enrich = select_emails_for_enrichment(
        emails,
        prior_snapshots,
        known_domains.clone(),
        5,  // limit (configurable from config)
        now,
    );

    if emails_to_enrich.is_empty() {
        log::info!("enrich_emails: no emails qualify for enrichment");
        return Ok(());
    }

    // Enrich each email with per-email timeout
    for email in emails_to_enrich {
        match tokio::time::timeout(
            Duration::from_secs(90),  // Configurable timeout
            claude_enrich_email(&email, user_ctx),
        ).await {
            Ok(Ok(enrichment)) => {
                // Success: write to DB immediately (no N+1, each is final)
                state
                    .db_write(|db| {
                        db.update_email_enrichment(
                            &email.email_id,
                            Some(&enrichment.summary),
                            Some(&enrichment.action),
                            Some(&enrichment.arc),
                            Some(&enrichment.signals),
                            now,  // enriched_at
                        )
                    })
                    .await?;
                log::info!("enriched email: {}", email.email_id);
            }
            Ok(Err(e)) => {
                // API error: skip and continue
                log::warn!("enrichment failed for {}: {}", email.email_id, e);
            }
            Err(_) => {
                // Timeout: skip and continue
                log::warn!("enrichment timeout for {}: exceeded 90 seconds", email.email_id);
            }
        }
    }

    // Update emails.json export artifact
    // ... (existing code to export enriched data)

    // Invalidate briefing cache so next refresh picks up enriched data
    invalidate_briefing_cache(&data_dir);

    Ok(())
}

## Key Architectural Decisions (from Plan-Eng-Review 2026-03-30)

**Why DB-first, not JSON?** ADR-0062 ("Briefing Artifacts vs. Live Queries") establishes that emails.json is a write-only export. Email data is read from the DB (`emails` table, I368), filtered there, enriched there. JSON is the output artifact only.

**Why priority-tier-first sorting?** Prevents medium-priority emails from pushing out high-priority emails when the limit is exceeded. All high-priority emails are enriched before any medium-priority.

**Why smart re-enrich (content-change detection)?** Allows re-enrichment when conversations evolve (new replies) while skipping stale emails. Compare current snippet/subject against prior snapshot. Load all snapshots once in a batch query (no N+1).

**Why fire-and-forget async?** Enrichment is decoupled from briefing completion. Briefing renders in <45 seconds with unenriched data. Email enrichment starts immediately after schedule.json written, runs independently, doesn't block anything. Next refresh picks up enriched data.

**Why immediate per-email DB writes?** If enrichment crashes mid-batch, we keep what we have. Partial results are correct and durable. No N+1 issue: each write is a final commit, not bulk-loaded.

**Why 90s timeout per email?** Graceful degradation: timeout on one email doesn't crash workflow. Continue with remaining emails. Partial enrichment (some enriched, some not) is acceptable and renders correctly.

**Why NOT batch timeout?** Per-email timeouts are sufficient. 5 emails × 90s = 450s total is OK because enrichment is async/non-blocking.

**Why nullable enriched_at?** Clearer semantics: None = never enriched, Some(ts) = enrichment timestamp. Gate 0 check: `if enriched_at.is_none()` proceed to selective filtering.

---

## Out of Scope

- **Email content re-ranking or relevance scoring:** Gate 1's priority/recency/newness is heuristic. ML-driven relevance scoring deferred to I643 (broader efficiency controls).
- **Archive/trash filtering:** I652 assumes all emails are in inbox. Filtering by folder/label status is orthogonal work.
- **Sender reputation or domain ranking:** Known domains come from customer/contact entity lists. Building a sender reputation model (frequency, engagement, reply rate) deferred.
- **Real-time email subscription:** I652 is workflow-triggered. Real-time push notifications on incoming email deferred to I544.
- **Expansion opportunity enrichment:** I652 extracts current subscriptions. Expansion signals (open opportunities, upsell candidates) handled by separate enrichment (Outlook chapter).
- **Product classification from emails:** I624 (product discovery from transcripts) is separate. Email content reading for product signals deferred.
- **Batch timeout:** No maximum total time for all enrichment. Each email gets 90s, enrichment runs until complete (OK because async, non-blocking).

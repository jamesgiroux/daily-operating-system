# I497 — Success Plan report type (absorbs I460)

**Priority:** P1
**Area:** Backend / Reports
**Version:** 1.0.0
**Depends on:** I499 (health scoring engine), I508 (intelligence schema)

## Problem

There is no structured success plan document that can be shared with customers. CSMs track objectives informally through meeting notes and intelligence fields (`success_metrics`, `open_commitments`), but there is no synthesized document that presents: "Here is what we agreed to deliver, here is where we stand, here is what is next." A success plan must be customer-shareable — no internal DailyOS language, no "signals", no "entities."

This absorbs I460 (Success Plan from the original v1.0.1 scope) with richer acceptance criteria grounded in real intelligence data sources.

## Design

### Report Type

Add `ReportType::SuccessPlan` variant. DB string: `"success_plan"`. Entity scope: single account.

### Module Structure

New file `src-tauri/src/reports/success_plan.rs` following the two-phase pattern.

### Output Schema (`SuccessPlanContent`)

```rust
pub struct SuccessPlanContent {
    pub account_context: AccountContext,
    pub what_we_deliver: String,                     // from user value_proposition — or prompt if empty
    pub customer_objectives: Vec<CustomerObjective>,
    pub agreed_outcomes: Vec<AgreedOutcome>,
    pub current_progress: Vec<ProgressItem>,
    pub blockers_and_dependencies: Vec<BlockerItem>,
    pub next_milestones: Vec<Milestone>,
}

pub struct AccountContext {
    pub account_name: String,
    pub lifecycle_stage: Option<String>,             // "onboarding" | "adoption" | "renewal" | "expansion"
    pub renewal_date: Option<String>,
    pub relationship_start: Option<String>,           // earliest meeting date
    pub primary_contact: Option<String>,
    pub health_summary: String,                      // 1 sentence, no score number
}

pub struct CustomerObjective {
    pub objective: String,                           // what the customer wants to achieve
    pub source: String,                              // "success_metrics" | "meeting_YYYY-MM-DD" | "open_commitment"
    pub status: String,                              // "on_track" | "at_risk" | "achieved" | "not_started"
    pub priority: String,                            // "primary" | "secondary"
}

pub struct AgreedOutcome {
    pub outcome: String,                             // specific deliverable or result
    pub owner: String,                               // "us" | "customer" | "joint"
    pub timeline: String,                            // "Q1 2026" | "by March 15" | "ongoing"
    pub status: String,                              // "in_progress" | "complete" | "blocked" | "not_started"
}

pub struct ProgressItem {
    pub objective: String,                           // references a CustomerObjective
    pub progress_narrative: String,                  // 1-2 sentences on current state
    pub evidence: Option<String>,                    // meeting date or metric that proves progress
    pub completion_estimate: Option<String>,          // "70%" or "on track for Q2"
}

pub struct BlockerItem {
    pub blocker: String,
    pub impact: String,                              // which objective this blocks
    pub owner: String,                               // who can resolve
    pub status: String,                              // "open" | "in_progress" | "resolved"
}

pub struct Milestone {
    pub milestone: String,                           // next thing to achieve
    pub target_date: Option<String>,
    pub owner: String,
    pub depends_on: Option<String>,                  // what must happen first
}
```

### Prompt Design

- **System role:** "You are a customer success professional preparing a shared success plan document. This document will be shared with the customer — use professional, partnership-oriented language. No internal jargon."
- **Objective sourcing:** Explicit prompt instruction: "Customer objectives must be sourced from intelligence `success_metrics` and `open_commitments` fields. If these fields contain objectives, use them. Do not invent objectives not grounded in the data."
- **Progress assessment:** "Assess progress based on signal history: metric movements (improving signals), commitment resolutions (completed actions), meeting discussion topics. Cite evidence where available."
- **"What we deliver" section:** Pulls from user entity's `value_proposition` and `success_definition`. If both are empty, the section renders a prompt: "Set your value proposition on the /me page to populate this section."
- **Customer-shareable language gate:** Explicit blocklist in prompt: "Never use: signal, entity, enrichment, intelligence, health score, DailyOS, pipeline, enriched, ingested. Use instead: update, progress, health, context, insight."
- **Internal content filter:** Same as EBR/QBR — exclude internal team debrief content.

### Intelligence Field Access

The gather function needs to parse specific fields from the `entity_intelligence` JSON:
- `success_metrics` — array of customer success criteria
- `open_commitments` — array of outstanding commitments with status
- These are extracted from the `executive_assessment` or dedicated JSON fields in the intelligence row.

## Acceptance Criteria

1. Objectives sourced from intelligence `success_metrics` and `open_commitments` — not hallucinated
2. Progress assessment based on signal history (metric movement, commitment resolution)
3. Contains: customer objectives, agreed outcomes with timelines and owners, current progress per objective, blockers and dependencies, next milestones, account context (lifecycle stage, renewal date)
4. "What we deliver" section pulls from user entity's `value_proposition` and `success_definition` — renders prompt if empty
5. Shareable with customer — no internal-only language (no "signal", "entity", "enrichment")
6. No internal DailyOS terminology in output — customer-shareable language throughout

## Out of Scope

- Customer-facing portal or shared link for the success plan (report is exported/shared manually)
- Success plan versioning or change tracking (showing what changed since last version)
- Integration with external success plan tools (Gainsight, Totango)
- Automatic objective detection from meeting transcripts (objectives come from existing intelligence fields)

# I566 — User-facing semaphore timeout + cancellation

**Version:** v1.0.0 Phase 6
**Depends on:** I565 (typed resource permits)
**Type:** Enhancement — Backend + Frontend / Performance
**Scope:** Backend: IPC command handlers for manual refresh. Frontend: toast/status feedback.

---

## Problem

When a user clicks "Refresh Intelligence" on an account detail page, the IPC handler in `services/intelligence.rs` (line 60–64) awaits the semaphore with no timeout:

```rust
let _permit = state
    .heavy_work_semaphore  // After I565: state.permits.user_initiated
    .acquire()
    .await
    .map_err(|_| "Heavy work semaphore closed".to_string())?;
```

If background work is running, this `await` blocks indefinitely. The user sees:
- No progress indicator (the "Refreshing..." text appears immediately, but nothing else happens)
- No way to cancel
- A potential beach ball if the blocked future holds other resources
- The PTY subprocess has its own 180-second timeout (`PtyManager::with_timeout(180)` at `intel_queue.rs` line 1037), but the semaphore wait before the PTY even starts has no bound

The same problem affects `commands/projects_data.rs` (line 165) for project enrichment.

After I565 lands, `permits.user_initiated` should rarely contend with background work, but it can still contend with another user-initiated refresh (clicking Refresh on two accounts in quick succession), and the pipeline permit may still cause waits.

---

## Solution

### 1. Add timeout to user-initiated semaphore acquisition

In `services/intelligence.rs` and `commands/projects_data.rs`, replace unbounded `acquire().await` with a timeout:

```rust
use tokio::time::{timeout, Duration};

let permit = match timeout(
    Duration::from_secs(10),
    state.permits.user_initiated.acquire(),
).await {
    Ok(Ok(permit)) => permit,
    Ok(Err(_)) => return Err("Resource permits closed".to_string()),
    Err(_) => {
        // Timeout — return a structured error the frontend can handle
        return Err("REFRESH_QUEUED:Background work in progress. Your refresh will start when it completes.".to_string());
    }
};
```

### 2. Frontend: Handle queued/timeout response

In `useAccountDetail` (or wherever `invoke("enrich_entity")` is called), detect the `REFRESH_QUEUED:` prefix in the error response and show a toast instead of an error:

```typescript
try {
  await invoke("enrich_entity", { ... });
} catch (err) {
  const msg = String(err);
  if (msg.startsWith("REFRESH_QUEUED:")) {
    toast.info(msg.replace("REFRESH_QUEUED:", ""));
  } else {
    toast.error("Refresh failed");
  }
}
```

### 3. Add cancellation token for PTY operations

Add an `AbortHandle`-style cancellation mechanism so the user can cancel a running enrichment:

- Add a `CancellationToken` (from `tokio_util::sync`) to `AppState` for user-initiated work.
- When the user clicks "Refresh" while enrichment is already running for that entity, cancel the existing operation and start a new one.
- The PTY subprocess is killed via `PtyManager::kill()` on cancellation.

This is the stretch goal — the timeout alone resolves the UX issue. Cancellation can be a follow-up if needed.

### 4. IPC response type

Consider returning a structured response instead of using error-string prefixes. A `RefreshResult` enum:

```rust
#[derive(Serialize)]
#[serde(tag = "status")]
enum RefreshResult {
    #[serde(rename = "completed")]
    Completed { enriched_at: String },
    #[serde(rename = "queued")]
    Queued { message: String },
    #[serde(rename = "failed")]
    Failed { error: String },
}
```

This is cleaner than error-string parsing but requires changing the IPC return type. Evaluate whether the current `Result<String, String>` contract allows this or if it needs a new command.

## Acceptance Criteria

1. User clicks "Refresh Intelligence" while background enrichment is running. Within 10 seconds, the UI shows either: (a) enrichment starts, or (b) a toast message explaining the refresh is queued.
2. The toast message uses product vocabulary — no system terms like "semaphore" or "permit". Example: "Background work in progress — your refresh will start when it completes."
3. The "Refreshing..." button text shows "(queued)" when the timeout fires, so the user knows their request was received.
4. If no contention exists, manual refresh starts immediately — no 10-second delay. The timeout is a ceiling, not a floor.
5. Two rapid clicks of "Refresh" on the same account do not queue two enrichments — the second click is debounced or deduped.
6. Project enrichment via "Refresh" on project detail has the same timeout + feedback behavior.
7. `cargo test` passes. `cargo clippy -- -D warnings` clean. `pnpm build` succeeds.

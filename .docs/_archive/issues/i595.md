# I595 — Rejection Learning for Proposed Actions

**Version:** v1.0.2
**Area:** Backend / Intelligence
**Priority:** P2

## Problem

When a user rejects a proposed action, the system captures `rejection_source` and `rejected_at` in the database and emits an `action_rejected` signal — but nothing uses this data to improve future suggestions. The system keeps proposing the same patterns the user has already rejected. This erodes trust: if the AI suggests 10 actions and 8 get rejected, the user stops looking at suggestions.

## Design

### Rejection Pattern Learning

New module: `intelligence/action_learning.rs`

**Pattern extraction**: When an action is rejected, extract features:
- `source_type` (which pipeline generated it)
- Account/entity context
- Title keywords (normalized, stop-words removed)
- `rejection_source` (where the user rejected it — briefing, actions page, meeting detail)

**Suppression rules** (heuristic, no ML needed):

1. **Exact title suppression**: If an action with the same normalized title for the same account was rejected, never propose it again. Store in `rejected_action_patterns` table.

2. **Source fatigue**: If a user rejects > 70% of proposed actions from a given `source_type` over 30 days, reduce that source's proposal rate:
   - Briefing-generated: skip low-confidence proposals (only emit P1-worthy)
   - Email-extracted: raise the commitment extraction confidence threshold
   - Emit `proposal_source_fatigued` signal for observability

3. **Keyword suppression**: If a user rejects 3+ actions containing the same keyword cluster for the same account (e.g., "follow up," "schedule," "check in"), suppress future proposals matching that pattern for that account.

### Schema

New table `rejected_action_patterns`:
- `id` INTEGER PRIMARY KEY
- `account_id` TEXT (nullable — null means global pattern)
- `pattern_type` TEXT ("exact_title" | "keyword" | "source_fatigue")
- `pattern_value` TEXT (normalized title, keyword, or source_type)
- `rejection_count` INTEGER
- `first_rejected_at` DATETIME
- `last_rejected_at` DATETIME
- `suppressed` BOOLEAN (true when threshold crossed)

### Integration Point

In `prepare/orchestrate.rs` and `processor/email_actions.rs`, before creating a proposed action:
1. Check `rejected_action_patterns` for matching suppression
2. If suppressed: skip creation, log to debug
3. If not suppressed: create as normal

### Feedback Loop Closure

- Rejection data feeds Bayesian source weights (existing mechanism in `signals/feedback.rs`)
- `source_type` weight adjusts: sources producing mostly-rejected actions lose confidence
- Weight recovery: if user starts accepting from a suppressed source, gradually restore

## Mock Data

Requires I597 (action mock data expansion). Pre-seeded rejection patterns:
- Acme: exact_title "Schedule weekly check-in" (rejected 3x, suppressed)
- Globex: keyword "follow up" (rejected 4x, suppressed)
- Verify proposed action pipeline skips these patterns when generating mock proposals

## Acceptance Criteria

1. Rejecting an action stores a pattern in `rejected_action_patterns`
2. Same-title actions for the same account are never re-proposed after rejection
3. Sources with > 70% rejection rate over 30 days get reduced proposal frequency
4. Keyword patterns (3+ rejections) are suppressed for the account
5. Suppressed patterns checked before creating proposed actions (both briefing and email pipelines)
6. Rejection data feeds Bayesian source weight adjustment
7. Suppression is account-scoped (rejecting "schedule follow-up" for Acme doesn't suppress it for Beta Corp)
8. Debug logging shows when proposals are suppressed and why
9. **Mock data**: 2 suppression patterns seeded. Verify pipeline debug log shows "suppressed" when matching mock patterns.

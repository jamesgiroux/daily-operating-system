# I645 - Timeliness, staleness, and feedback-aware learning

**Area:** Backend / Intelligence Loop + Feedback + Ranking
**Priority:** P0
**Version:** v1.1.0 Phase 5
**Effort:** Large
**Depends on:** I644, existing Bayesian feedback weights, existing user correction flows
**Source:** Architecture review follow-up. The user explicitly asked how we time-box account content, avoid surfacing stale issues as if they are current, and make thumbs up / thumbs down / dismiss / live edits materially change the next refresh.

---

## Problem

Account intelligence has a time problem and a learning problem.

### The time problem

Not all facts age the same way:

- ARR can stay current until superseded
- an open ticket can become resolved and irrelevant quickly
- a compliance blocker can stay live for months
- a product theme mentioned on a call can matter for 90 days and then go cold

If DailyOS does not model timeliness explicitly, account detail and meeting detail drift toward "interesting history" instead of "what matters now."

### The learning problem

Current feedback mechanisms do not yet guarantee material behavioral change on the next refresh.

If a user:

- clicks thumbs down on an item
- dismisses it with `x`
- edits a field inline
- accepts or rejects a conflicting source suggestion

the system should not simply store a generic signal. It should alter relevance, weighting, suppression, and prompt assembly in the next refresh cycle.

Without that, correction feels cosmetic instead of instructional.

---

## Solution

Add an explicit freshness and feedback model for every source-backed field or item.

### 1. Timeliness metadata

Every source-backed fact, signal, or inference should carry:

- `observed_at`
- `refreshed_at`
- `stale_after`
- `state`: `current | historical | stale | resolved | dismissed | superseded`
- optional `resolved_at`

This applies both to promoted facts and structured assessment items.

### 2. Relevance windows by evidence class

DailyOS should not use one global staleness rule. Use class-specific defaults.

Initial policy:

| Evidence class | Active window | Historical handling |
|---|---|---|
| CRM contract facts | current until superseded | remain current unless explicitly replaced |
| Renewal opportunity stage | 14 days unless re-confirmed | move to historical if not refreshed |
| Support incidents | 45 days after resolution | collapse into record/history |
| Active blockers | current until resolved or stale_after | historical after resolution |
| Technical themes from calls | 90 days | historical unless reaffirmed |
| Stakeholder engagement / coverage | 120 days | historical if not re-seen |
| Compliance concerns | 180 days or until resolved | keep visible longer due to renewal impact |
| Value delivered | durable | stays visible but dates matter |

These windows should be config-driven in code, not hardcoded in UI components.

### 3. Surface behavior

By default:

- account detail shows `current` and `recent` items
- meeting detail shows only `current` and high-relevance recent items
- stale or resolved items move into "The Record" / history sections

This makes timeliness a first-class rendering rule.

### 4. Feedback model

Add a feedback event table keyed to entity + field/item + source.

Minimum shape:

```sql
CREATE TABLE entity_feedback_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_id TEXT NOT NULL,
    field_key TEXT NOT NULL,
    item_key TEXT,
    feedback_type TEXT NOT NULL,      -- thumbs_up | thumbs_down | dismiss | accept | reject | live_edit
    source_system TEXT,
    source_kind TEXT,
    previous_value TEXT,
    corrected_value TEXT,
    reason TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 5. Next-refresh behavior

Feedback must change what happens next.

Rules:

- `thumbs_up`: modestly increase Bayesian weight for the source + prompt path that produced the correct item
- `thumbs_down`: immediately remove the item from the current UI, create a suppression record, and decrease Bayesian weight for that source + prompt path
- `dismiss`: same immediate removal behavior as thumbs down, but without the stronger negative learning penalty
- `live_edit`: create a user override / correction record that is injected into the next prompt context as ground truth
- `reject source suggestion`: lower trust in that source for that field on that entity
- `accept source suggestion`: promote the source-backed value and reinforce the source for that field/entity pair

### 6. Suppression and tombstones

Dismissed items should not bounce back on the next refresh just because the prompt sees roughly the same source again.

Add a tombstone / suppression model:

- `field_key`
- `item_key` or hash
- `dismissed_at`
- `source_scope`
- `expires_at`

An item may reappear only if:

- there is newer evidence with `observed_at > dismissed_at`, or
- a stronger source class supports it, or
- the user explicitly requests a refresh that ignores suppression

### 7. Override safety

User edits should be sticky.

If the user corrects ARR, lifecycle, renewal date, or another vital:

- the corrected value remains current
- the system still stores incoming conflicting evidence
- DailyOS offers accept/dismiss instead of silently overwriting the user

This is the mechanism that lets Glean reinforce account truth without steamrolling user context.

---

## Acceptance Criteria

1. Every source-backed field or item rendered on account detail or meeting detail can be classified as `current`, `historical`, `stale`, `resolved`, `dismissed`, or `superseded`.
2. A resolved Zendesk incident older than the support-health active window no longer appears in the active watch list or pre-meeting briefing by default, but remains available in history.
3. A CRM-backed ARR remains current until a newer factual source supersedes it.
4. Clicking thumbs down on a surfaced item removes it from the UI immediately, writes a suppression record, and prevents that same item from reappearing on the next refresh unless newer or stronger evidence is present.
5. Clicking `x` also removes the item immediately and suppresses it for the next refresh, but is modeled as a lighter-weight signal than thumbs down.
6. A live edit creates a user override that is included in future prompt context and is not silently overwritten by later AI output.
7. Accepting or rejecting a source-backed suggestion updates Bayesian or per-field weights in a way that is visible in logs or diagnostics.
8. Meeting detail defaults to current and relevant account intelligence. Historical but no-longer-relevant issues are relegated to history or omitted by default.

---

## Implementation Notes

- Timeliness should be enforced in the backend query / assembly layer, not as ad hoc frontend filtering.
- Keep the initial relevance windows opinionated and simple. We can tune them later with usage data.
- The learning model should operate at both:
  - source level (`glean_crm` vs `glean_chat` vs `glean_gong`)
  - field or prompt-slot level (`renewal_likelihood`, `support incident summary`, `stakeholder sentiment`)
- Feedback should be visible enough in diagnostics that we can prove the loop is working.

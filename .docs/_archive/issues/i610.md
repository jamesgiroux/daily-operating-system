# I610 — Consolidate AppLockState into single struct

**Area:** Code Quality / Backend
**Priority:** P2
**Version:** v1.0.1
**Source:** v1.0.0 GA engineering review (2026-03-16)
**Related:** I404 (AppState decomposition Phase 1), I465 (app lock feature)

---

## Problem

App lock state (I465) is spread across 4 separate fields in `AppState`, each with its own synchronization primitive:

```rust
pub is_locked: AtomicBool,
pub last_activity: Mutex<Instant>,
pub last_failed_unlock: Mutex<Option<Instant>>,
pub failed_unlock_count: AtomicU32,
```

The Atomic + Mutex mix creates a window for inconsistent reads. Example: a thread can read `is_locked = true` but `failed_unlock_count = 0` during a concurrent unlock attempt. No bugs have been reported, but the invariant is not enforced by the type system.

## Scope

1. Create `AppLockState` struct:
   ```rust
   pub struct AppLockState {
       pub is_locked: bool,
       pub last_activity: Instant,
       pub last_failed_unlock: Option<Instant>,
       pub failed_unlock_count: u32,
   }
   ```
2. Replace the 4 fields with `pub lock_state: Mutex<AppLockState>` (or `RwLock` if read-heavy)
3. Update all lock/unlock/check callsites to acquire the single lock
4. Verify no deadlock potential (lock_state should never be held while acquiring other locks)

## Acceptance Criteria

1. `AppState` has a single `lock_state` field instead of 4 separate lock-related fields
2. All lock operations (lock, unlock, check idle, record failed attempt) go through one acquisition
3. No new `unwrap()` calls on the lock — use `lock().expect()` with context
4. All tests pass, clippy clean

## Out of Scope

- Full AppState decomposition (I404)
- Changing lock behavior or adding new lock features

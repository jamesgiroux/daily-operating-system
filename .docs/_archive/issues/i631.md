# I631 — Transcript filing for person + project entities

**Area:** Backend / Transcripts
**Priority:** P2
**Version:** v1.0.2
**Effort:** Low (1-2 hours)
**Source:** Planning session (2026-03-21). Transcript routing already works for accounts but not for person or project entities.

---

## Problem

When a meeting transcript is processed (`processor/transcript.rs`), the system routes the markdown file to the entity directory:

- **Account-linked meeting** → `~/Accounts/{AccountName}/Call-Transcripts/{date}-{slug}-transcript.md` — **works**
- **1:1 meeting (person-linked)** → should go to person directory — **doesn't route**
- **Project-linked meeting** → should go to project directory — **doesn't route**
- **Unlinked meeting** → `~/_archive/{date}/` — **works (correct fallback)**

The routing logic in `transcript.rs` checks for a linked account via `meeting.linkedEntities` but doesn't check for person or project entities. 1:1 meetings and project meetings fall through to the archive fallback.

This matters because:
1. MCP `search_content` uses entity directories for semantic search — transcripts in `_archive/` aren't discoverable by entity
2. Claude Code users working on an account, person, or project benefit from having transcripts in the entity's directory
3. The Intelligence Loop integration check question #3 asks: "Should it appear in `build_intelligence_context()`?" — entity-filed transcripts feed enrichment context

## Solution

Extend the transcript routing logic to handle all entity types in priority order:

1. If meeting has a linked **account** → route to `~/Accounts/{Name}/Call-Transcripts/`
2. Else if meeting has a linked **project** → route to `~/Projects/{Name}/Call-Transcripts/`
3. Else if meeting is a 1:1 (type `one_on_one`) and has a linked **person** → route to `~/People/{Name}/Call-Transcripts/`
4. Else → route to `~/_archive/{date}/`

Account takes priority because a customer meeting with 2 attendees might match both "account" and "person" — the account context is more valuable for CS workflows.

## Implementation

In `src-tauri/src/processor/transcript.rs`, the routing function that determines the destination path:

1. Find the existing account routing logic (likely a match on `linkedEntities` with `entityType == "account"`)
2. Add a second branch: if no account match, check for `entityType == "project"` → route to project directory
3. Add a third branch: if meeting type is `one_on_one` and `linkedEntities` contains a person → route to person directory
4. Ensure directory creation (`create_dir_all`) handles person and project paths

The destination path pattern should be consistent:
- `{workspace}/{EntityTypePlural}/{EntityName}/Call-Transcripts/{date}-{slug}-transcript.md`

## Acceptance Criteria

1. Process a transcript for a 1:1 meeting linked to person "Jane Smith." Markdown file appears at `~/People/Jane Smith/Call-Transcripts/{date}-{slug}-transcript.md`.
2. Process a transcript for a project meeting linked to project "Platform Migration." Markdown file appears at `~/Projects/Platform Migration/Call-Transcripts/{date}-{slug}-transcript.md`.
3. Process a transcript for a customer meeting linked to account "Acme Corp." Markdown file still appears at `~/Accounts/Acme Corp/Call-Transcripts/` (existing behavior preserved).
4. Process a transcript for an unlinked meeting. Markdown file goes to `~/_archive/` (existing behavior preserved).
5. Meeting linked to both an account and a person → account routing wins (priority order).
6. `meeting_transcripts.transcript_path` in DB reflects the correct entity-routed path.
7. MCP `search_content` for a person entity finds transcripts filed in their directory.

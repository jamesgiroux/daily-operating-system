# I614 — DB growth monitoring + age-based purge scheduler

**Area:** Backend / Data Lifecycle
**Priority:** P2
**Version:** v1.0.1
**Source:** v1.0.0 GA engineering review (2026-03-16)
**Related:** ADR-0098 (data governance), I528 (data lifecycle infrastructure)

---

## Problem

The live DB is 152MB with no automatic cleanup for aged data. Three growth vectors have no purge triggers:

1. **Signals**: `signal_events` rows retained indefinitely. Decay rules exist in `signals/propagation.rs` (half-life scoring) but no deletion scheduler — decayed signals still occupy storage.
2. **Emails**: 90-day search scope in FTS5, but raw email data has no age-based purge.
3. **Embeddings**: `content_embeddings` stores 768 floats per document with no cleanup for deleted/archived entities.

A power user running for 6+ months could reach 500MB+ with no automatic relief. Desktop apps shouldn't silently consume unbounded disk.

## Scope

### Phase 1: Monitoring (ship first)

1. On startup, check DB file size via `std::fs::metadata()`
2. Log info at any size, warn at 300MB, error at 500MB
3. Emit `db-size-warning` Tauri event at 500MB+ so frontend can show a toast
4. Add DB size to DiagnosticsSection in Settings

### Phase 2: Purge scheduler

5. Add `purge_aged_signals(days: i64)` to `db/data_lifecycle.rs`:
   - Delete `signal_events` where `created_at < now - days` AND `source != 'user_correction'`
   - User corrections (confidence 1.0) are never purged
   - Default: 180 days
6. Add `purge_aged_emails(days: i64)`:
   - Delete `emails` and `email_signals` where `received_at < now - days`
   - Default: 365 days
7. Add `purge_orphaned_embeddings()`:
   - Delete `content_embeddings` where `entity_id` not in any active entity table
8. Run all three in `run_hygiene_loop()` once per day (not every scan cycle)
9. Log purge counts to audit log

## Acceptance Criteria

1. App logs DB file size on startup
2. Settings → Diagnostics shows current DB size in human-readable format
3. Frontend shows toast warning when DB exceeds 500MB
4. After purge: `SELECT COUNT(*) FROM signal_events WHERE created_at < date('now', '-180 days') AND source != 'user_correction'` returns 0
5. User corrections (source = 'user_correction') are never deleted regardless of age
6. Purge runs once daily, not on every hygiene scan
7. Audit log has entries for each purge operation with row counts
8. `cargo test` passes — add test for `purge_aged_signals` with user_correction preservation

## Out of Scope

- Embedding quantization (int8 instead of float) — separate optimization
- User-configurable retention periods — ship sensible defaults first
- WAL size management — SQLite handles this with existing checkpoint pragmas

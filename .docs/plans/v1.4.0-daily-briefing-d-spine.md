# v1.4.0 Daily Briefing D-spine Redesign Track

**Date:** 2026-05-06
**Status:** Parallel design/reference track
**Reference surface:** `.docs/design/reference/surfaces/briefing-d-spine.html`

**Parallel plan:** `.docs/plans/v1.4.0-waves.md` is on W6. This track must not disturb W6 release-gate ownership.

## Goal

Decide whether D-spine becomes the v1.4.0 DailyBriefing redesign, using the
existing Daily Briefing reference as the foundation and extracting only the
missing reusable component code needed to make the reference real.

## Scope

In scope for this design track:

- Reference candidate in `.docs/design/reference/surfaces/briefing-d-spine.html`.
- `DailyBriefingDSpine` surface spec.
- `MeetingSpineItem` source component and CSS module.
- Pattern metadata updates for the D-spine component set.
- QA against reference rendering, inspector tags, and mobile/desktop layout.

Not in scope without a follow-up implementation plan:

- Switching the routed `/` DailyBriefing page to D-spine.
- Replacing `BriefingMeetingCard` in production.
- Shipping DayStrip without a route-level navigation decision.
- Claim/trust UI coupling beyond consuming W6 outputs when they exist.

## Reuse-first component contract

Use these existing design-system pieces first:

- Chrome/layout: `FolioBar`, `DayStrip`, `FloatingNavIsland`, `AtmosphereLayer`, `MarginGrid`, `FinisMarker`.
- Current briefing substrate: `editorial-briefing.module.css`, `DailyBriefing.module.css`, `DailyBriefingAttentionSection`.
- D-spine patterns: `DayChart`, `MeetingSpineItem`, `InferredActionSelector`.
- Primitives: `Pill`.

New extraction allowed in this track:

- `MeetingSpineItem` only. It composes `Pill` and keeps state labels in the time rail.

## Rollout plan

1. **Reference candidate**
   - Build the static reference from existing DailyBriefing chrome and margin grid.
   - Keep `data-ds-state="proposed"` on the surface.
   - Do not add the candidate to `surface-manifest.json` until a routed source page exists.

2. **L0 implementation plan before route changes**
   - If the candidate is approved, write a routed implementation plan before code.
   - Review the plan with design, accessibility, and engineering reviewers because this is a user-facing surface.
   - Decide whether `BriefingMeetingCard` remains as an expanded variant or is folded into `MeetingSpineItem` plus an expansion pattern.

3. **Production integration**
   - Add a feature flag or dev scenario for the D-spine DailyBriefing.
   - Wire real dashboard data into `DayChart`, `MeetingSpineItem`, Moving, and Watch.
   - Preserve current DailyBriefing empty/loading/error/cached states.
   - Keep FolioBar and FloatingNavIsland; decide whether DayStrip replaces Weekly Forecast access.

4. **Review loops**
   - Run reference audit and browser QA at desktop and mobile widths after each iteration.
   - Compare `briefing.html` and `briefing-d-spine.html` section by section so no current shipped capability disappears silently.
   - Re-audit inspector coverage: every major visual block should resolve to a primitive, pattern, or surface.

5. **Release decision**
   - Ship in v1.4.0 only if W6 release gates remain green and D-spine passes reference QA, accessibility, and route-level data parity.
   - Otherwise keep the reference and source extraction in the design system and defer route cutover.

## Data parity checklist

- Lead sentence without the removed focus block.
- Day shape and meeting count.
- Past, in-progress, upcoming, needs-briefing, building, and cancelled meetings.
- Expanded/full briefing path for meetings.
- Account/person movement rows.
- Watch items with inferred action options.
- Priority emails/actions/lifecycle changes if those remain outside Watch.
- Empty, loading, error, cached/stale states.

## Risks

- **Production parity drift:** D-spine can look complete while missing current DailyBriefing rows. Mitigation: section-by-section parity review against `briefing.html` and `DailyBriefing.tsx`.
- **Pattern taxonomy drift:** Proposed surface work can be mistaken for production. Mitigation: `data-ds-state="proposed"`, separate proposed-reference section, and no strict manifest entry yet.
- **W6 interference:** The redesign can collide with trust/claim cutovers. Mitigation: keep this track reference-first until W6 gates are stable.

# v1.4.0 Wave Protocol Amendments

**Status:** Lean version. Three small rules added to `v1.4.0-waves.md`. Pending L0.
**Date:** 2026-05-07
**Scope:** Operational adjustments to the wave protocol surfaced during orchestration v1-lite design.

These amendments are operational config, not safety machinery. The orchestration v1-lite doc at `.docs/plans/orchestration/v1-lite.md` is the consumer.

---

## Amendment 1 — Cycle cap stays at 2 for now

`v1.4.0-waves.md` line 96 stays as written: 2 revision cycles ⇒ L6 escalation.

If the cap bites in practice and observed averages diverge meaningfully, James loosens via the existing tuning protocol (line 365) as a normal L6 ruling between waves. No phased activation, no convergence-trigger taxonomy, no cycle budget caps separate from line 96. Live with it; tune when it bites.

## Amendment 2 — Codex outage handling

If `/codex challenge`, `/codex review`, or `/codex consult` returns empty, errors, or off-topic output:

1. Retry up to 3 times with 1 / 3 / 5 minute backoff
2. After 3 failed retries → escalate L6 with reason `reviewer_infrastructure_failure`
3. Never substitute another reviewer (e.g. `code-reviewer`) into the codex slot. The L0/L2/L3 independence model relies on codex being a separate context.
4. L2 codex outage → status check stays red; PR sits.

Reference: `.docs/plans/wave-W2/escalations/DOS-209-DOS-259-l6-decision.md` (codex companion reliability bug; Linear issue tracking it should be open).

## Amendment 4 — Plan canonical location: Linear ticket, not git

For orchestration-driven work (cloud routines per `.docs/plans/orchestration/v1-lite.md`), the plan artifact lives on the Linear ticket itself, not in `.docs/plans/wave-WN/DOS-NNN-plan.md`. Linear is canonical per CLAUDE.md; co-locating the plan with the ticket eliminates git-vs-Linear drift, removes the need for `plan_sha` pinning ceremony, and uses Linear's `updatedAt` as the optimistic-concurrency token.

L0 reviewers read the plan from Linear and post verdicts as ticket comments. Iterations happen by editing the plan section.

Wave-scoped artifacts (`proof-bundle.md`, `retro.md`, ADRs) continue to live in git — they're wave-scoped, not ticket-scoped, and don't have a Linear analogue.

For existing in-flight work that already has a plan at `.docs/plans/wave-WN/DOS-NNN-plan.md`, no migration is required. New tickets follow the new convention.

## Amendment 3 — `security-auditor` on L0 panel for security-relevant plans

Added to the existing L0 trio (codex challenge + domain reviewer + codex consult): `security-auditor` runs as an additional reviewer when the changeset touches:

- `src-tauri/src/services/**`, `src-tauri/src/abilities/**`, `src-tauri/src/bridges/**`, `src-tauri/src/commands/**`, `src-tauri/src/intelligence/**`
- Schema / migrations: `.sql` files, `src-tauri/src/migrations.rs`, `src-tauri/migrations/**`, claim-substrate tables
- Prompt templates under `.claude/`
- Sensitivity / render-policy code (backend or frontend)
- MCP configs, tool registry, skill definitions
- Wave-protocol or orchestration docs (`.docs/plans/v1.4.0-waves*.md`, `.docs/plans/orchestration/**`)
- Adds new trust boundary or privileged action

When triggered, the L0 panel becomes a quartet (codex challenge + architect-reviewer + security-auditor + codex consult), unanimous required. For non-security plans the standard trio runs unchanged.

Same trigger criteria apply at L2 (code PRs).

## What this replaces

Previous versions of this doc (v1–v6, in git history) attempted a much larger amendment with phased activation, cycle taxonomies, convergence triggers, transcript-token verifiers, magic-comment detection, etc. That was over-engineered for the scope. The `feedback_l0_partial_convergence_when_class_recurs` memory captures why.

The lean version above is what's actually needed. The rest of what those drafts tried to specify is either already covered by existing wave protocol gates (L2 domain reviewers, L3 Suite S/P/E) or doesn't need to be specified in the protocol doc at all.

# v0.15.2 Version Brief

**Thesis:** v0.15.1 closes the encryption and injection gaps. v0.15.2 adds two capabilities: (1) an observability layer via a tamper-evident audit log that records what the app did, when, and whether anything anomalous occurred; and (2) a dual-mode context architecture that lets DailyOS intelligence draw from local data alone or from Glean's organizational knowledge graph. By the end of v0.15.2, DailyOS has a complete security posture (encrypted, injection-hardened, observable) and the architectural seam for enterprise knowledge integration.

**Status:** Implemented. Audit log (I471-I473) and context provider (I479-I481) work committed on `dev`.

**ADRs:** 0094 (Audit Log and Enterprise Observability), 0095 (Dual-Mode Context Architecture), 0096 (Glean Context Provider)

**Prerequisite:** v0.15.1 shipped. `AuditLogger` stubs from I462 and I465 can be upgraded to real calls once this version lands.

### Explicitly out

- Remote audit log shipping (SIEM integration) -- local-first; no external destinations
- Real-time anomaly push notifications -- the log record is the baseline; macOS notifications for injection anomalies are a future affordance
- Log signing with a user certificate -- hash chain is the tamper detection mechanism; full cryptographic signing is disproportionate
- Multi-user audit model -- single macOS user is the scope; team deployments revisit this
- Direct Glean OAuth -- users provide endpoint + token manually; automated OAuth discovery is out of scope

### I471 — AuditLogger core

Full criteria in `i471.md`. Summary:

1. `~/.dailyos/audit.log` is created on first launch. Permissions: `0o600`.
2. Each line is a valid JSON object with `ts`, `v`, `category`, `event`, `detail`, `prev_hash`.
3. After two launches: `record[N].prev_hash == sha256(record[N-1] line)`. Chain is continuous across restarts.
4. `app_started` event appears in the log after every launch. `audit_log_rotated` appears after rotation.
5. 90-day rotation: records older than 90 days are pruned on startup. Recent records are retained.
6. `verify_audit_log()` returns `Ok(N)` for an intact log. Deleting a line manually, then calling verify, returns `Err`.
7. `cargo test` passes including all `audit_log.rs` unit tests.

### I472 — Instrument all pipeline events

Events that must be verified in the running app (check `audit.log` after triggering each):

**Security events (from I462/I465 upgrades):**
1. `db_key_generated` appears in log on first launch (new installation or after key deletion).
2. `db_key_accessed` appears in log on every subsequent launch.
3. `db_migration_started` and `db_migration_completed` appear during the plaintext → encrypted migration (one-time, already complete for most users -- verify on a test installation or by reverting to a plaintext DB).
4. `oauth_connected` appears after connecting Google in Settings. `oauth_revoked` appears after disconnecting.
5. `app_unlock_succeeded` and `app_unlock_failed` appear after Touch ID interactions (I465 upgrade from stub).

**Data access events (new instrumentation):**
6. `google_calendar_sync` appears in log after a calendar fetch. `detail.events_fetched` is a non-zero integer (if calendar has events). No event names or attendees in the detail.
7. `gmail_sync` appears after an email sync. `detail.emails_fetched` is the count. No subjects or senders in the detail.
8. `clay_enrichment` appears after a Clay API call fires. `detail.entity_type` is `"person"` or `"account"`.
9. `gravatar_lookup` appears after a Gravatar fetch. `detail.count` is the number of lookups.

**AI operation events (new instrumentation):**
10. `entity_enrichment_started` and `entity_enrichment_completed` appear after an entity enrichment cycle. `detail.duration_ms` is present and reasonable (> 0, < 300000).
11. `entity_enrichment_failed` appears if the PTY fails or schema validation fails. `detail.error_category` is one of `"pty_error"`, `"schema_validation"`, `"timeout"`.
12. `email_enrichment_batch` appears after an email enrichment run. `detail.emails_processed` and `detail.failed` are present.

**Anomaly events (from I470 upgrade):**
13. `injection_tag_escape_detected` appears when `</user_data>` is found in external content (I466 catches and escapes it -- this event fires when the escape fires). Detail: `{"source": "email_subject", "escaped": true}`. Verify by sending an email with `</user_data>` in the subject and waiting for sync.
14. `injection_instruction_in_output` appears when the output validator detects system prompt terms in model output. Detail includes the detected terms.
15. `schema_validation_failed` appears when model output fails schema parsing.

**Config change events:**
16. `workspace_path_changed` appears after a workspace path change in Settings. Detail uses path categories (`"documents"`, `"home"`, `"custom"`), not the literal path. No PII.
17. `ai_provider_changed` appears after switching AI provider in Settings. Detail: `{"from": "claude_code", "to": "ollama"}`.

**PII rule verification:**
18. `grep -E '"name"|"email"|"subject"|"title"|"company"' ~/.dailyos/audit.log` -- returns no matches. Audit records contain only IDs, counts, categories, and classifications. Never content.

### I473 — Settings → Data → Activity Log UI

1. Settings → Data contains an "Activity Log" section (or subsection). It shows the last 100 audit records grouped by calendar day. Each record shows: timestamp (HH:MM:SS), category chip, event name in plain English (see translation table below), and a collapsed detail view on click.
2. Records with `category: "anomaly"` are highlighted with an amber left border or icon. They are not alarming -- just visually distinct.
3. A category filter: All / Security / Data Access / AI / Anomalies / Config. Selecting a filter shows only matching records. Filter state resets on page navigation (not persisted).
4. An "Export log" button writes the full `audit.log` file to a user-selected path via the system file save dialog. The exported file is the raw JSON-lines format. Verify: export, open in a text editor, valid JSON on each line.
5. A "Verify integrity" button calls `verify_audit_log()`. On success: "Log integrity verified -- N records checked." On failure: "Warning: log integrity check failed at record N. Some records may have been modified." Neither blocks the UI; the result is shown inline.
6. Plain-English event name translations (used in the UI, not stored in the log):
   - `db_key_generated` → "Encryption key created"
   - `db_key_accessed` → "Database opened"
   - `oauth_connected` → "Google account connected"
   - `oauth_revoked` → "Google account disconnected"
   - `app_unlock_succeeded` → "App unlocked"
   - `google_calendar_sync` → "Calendar synced (N events)"
   - `gmail_sync` → "Email synced (N emails)"
   - `entity_enrichment_completed` → "Intelligence updated"
   - `injection_tag_escape_detected` → "Injection attempt detected and blocked"
   - `schema_validation_failed` → "AI output rejected (unexpected format)"
   - `audit_log_rotated` → "Log maintenance"
   - Any unknown event → show the raw event name

Full criteria in `i479-done.md`. Summary:

1. `ContextProvider` trait in `context_provider/mod.rs` with `gather_entity_context()` and `mode()`.
2. `LocalContextProvider` in `context_provider/local.rs` wrapping existing `build_intelligence_context()`.
3. `state.rs` has `context_provider: Arc<dyn ContextProvider>` initialized as `LocalContextProvider` by default.
4. `intel_queue.rs` and all 4 report generators call through `ContextProvider` trait.
5. Zero behavior change — all intelligence and report output identical before and after.
6. ADR-0095 written.

### I480 — GleanContextProvider + cache + migration

Full criteria in `i480-done.md`. Summary:

1. `GleanMcpClient` in `context_provider/glean.rs` with `search`, `search_people`, `read_document` MCP calls.
2. DashMap + SQLite cache with TTL (docs 1h, profiles 24h, org graph 4h).
3. Migration 052 adds `glean_document_cache` and `context_mode_config` tables.
4. Two-phase gather: Phase A (local DB, ms) + Phase B (Glean network, 200-2000ms).
5. Additive vs Governed strategy in context merging.
6. Glean outage falls back to local context gracefully.
7. ADR-0096 written.

---

### I481 — Connector gating + mode switching + Settings UI

Full criteria in `i481-done.md`. Summary:

1. Gmail and Drive pollers disabled in Governed mode. Clay/Gravatar disabled in any Glean mode. Linear stays active.
2. Glean signals weighted at 0.7 confidence, 60-day half-life.
3. `set_context_mode` command triggers full re-enrichment sweep.
4. `ContextSourceSection.tsx` in Settings: Local/Glean mode selector, endpoint, token, Additive/Governed strategy.
5. Configuration persists to DB and takes effect immediately.

## Ship Criteria

`~/.dailyos/audit.log` exists after launch, contains valid JSON-lines, hash chain is intact. All 18 event types in I472's acceptance criteria appear in the log after triggering the corresponding actions. Settings → Data → Activity Log renders. Export produces a valid JSON-lines file. Glean mode gathers from both local DB and Glean API. Connector gating disables the correct pollers per mode. Settings UI allows mode switching.

### End-to-end smoke test (mandatory before tagging)

1. Launch app. Open `~/.dailyos/audit.log`. Verify `app_started` and `audit_log_rotated` records are present. Parse the JSON on each line -- all valid.

2. Trigger an enrichment cycle (enrich any account). Verify `entity_enrichment_started` and `entity_enrichment_completed` appear in the log with `detail.duration_ms > 0`.

3. Disconnect and reconnect Google in Settings. Verify `oauth_revoked` followed by `oauth_connected` appear in the log.

4. Send an email with subject containing `</user_data>`. Wait for email sync. Verify `injection_tag_escape_detected` appears in the log with `"source": "email_subject"`.

5. Settings → Data → Activity Log. The four events from steps 1-4 are visible. Click an anomaly record (from step 4) -- it is highlighted. Export log -- file is written. "Verify integrity" -- returns "N records checked."

6. `grep -E '"name"|"email"|"subject"|"title"' ~/.dailyos/audit.log` -- no matches. No PII in the log.

7. Manually delete a line from `audit.log` (in a text editor). Settings → Data → Activity Log → "Verify integrity" reports a failure.

8. `cargo test` passes. `cargo clippy -- -D warnings` passes.

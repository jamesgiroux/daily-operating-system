# Concurrency Consistency Contract

## Context

This is Sub-contract A for DOS-546, the DailyOS WordPress Studio surface viability spike.

It covers only concurrency for projected claims and compositions.

Tamper detection, signature verification, and audit-chain integrity belong in the companion artifact.

The problem space is a multi-writer surface over one DailyOS substrate.

The runtime, MCP client, WP editor, and future `SurfaceClient`s can all observe or attempt to mutate a projected claim.

The WP Studio surface is not a detached document export.

It is a rendered surface over ability outputs, claim projections, and composition blocks.

Therefore a WordPress block must carry enough substrate state to reject stale writes and refresh itself after corrections.

DailyOS remains the concurrency authority.

WordPress persists projected blocks, but it does not decide the current version of a claim.

Surface clients may cache and render, but they must write through the DailyOS mutation boundary.

The concurrency contract extends the existing ability/provenance model:

- ADR-0102 requires all capability outputs to flow through typed abilities and `AbilityOutput<T>`.
- ADR-0105 makes provenance a first-class output envelope.
- ADR-0129 establishes WordPress-side rendering as the primary surface for this spike.
- ADR-0130 establishes a surface-independent composition contract and block shape.

The requested ADR-0129 and ADR-0130 files were not present in this worktree at draft time.

This document still uses their stated contract names as binding inputs from DOS-546.

Where this artifact defines concrete paths or schemas, Phase 1 should treat them as the proposed compatibility layer for those ADRs.

Non-goals:

- No conflict-free replicated data type for rich text editing.
- No last-writer-wins behavior for claims.
- No polling fallback as a correctness mechanism.
- No tamper detection schema beyond fields needed to carry concurrency watermarks.

Terms:

- `Claim`: an assertion in the DailyOS claim substrate.
- `ProjectedClaim`: a claim rendered into a WordPress block or composition block.
- `Composition`: a surface-independent block tree containing claim projections and local presentation data.
- `SurfaceClient`: any live client rendering or editing a DailyOS projection.
- `Watermark`: the concurrency token a writer must present when mutating a projected claim or composition.
- `Correction`: a runtime-issued replacement, supersession, withdrawal, or tombstone that changes the current claim state.

## Watermark Model

### Decision

The conflict authority is a server-assigned monotonic integer per claim.

Each claim has a `claim_version: u64`.

The version increments exactly once per accepted mutation to that claim's authoritative state.

The version is scoped to a stable `claim_id`.

The version is never generated by WordPress, MCP clients, browser clients, or agents.

A hybrid logical clock is also recorded for ordering, replay, and diagnostics.

The hybrid logical clock is not the conflict authority.

The concrete watermark is:

```ts
type ClaimWatermark = {
  claim_id: string;
  claim_version: number;      // u64, server-assigned, monotonic per claim
  source_hlc: string;         // "physicalMs-counter-nodeId"
  event_seq: number;          // global event bus sequence for replay ordering
  produced_by: ActorRef;      // runtime | user | mcp | surface_client | system
  produced_at: string;        // ISO-8601 from ServiceContext clock
};

type ActorRef =
  | { kind: "runtime"; ability_name: string; ability_version: string }
  | { kind: "user"; user_id: string }
  | { kind: "mcp"; client_id: string; tool_name: string }
  | { kind: "surface_client"; surface_client_id: string }
  | { kind: "system"; component: string };
```

### Why This Shape

A monotonic per-claim integer is simple to compare.

It gives the write service one check: `expected_version == current_version`.

It is stable across clients with unreliable clocks.

It avoids accidental last-writer-wins behavior from HLC ordering.

The hybrid logical clock is still necessary.

It lets the event bus preserve causality across claim updates, composition updates, and correction events.

It gives SurfaceClients useful diagnostics when a conflict occurred within the same wall-clock window.

It supports deterministic replay after reconnect.

The `event_seq` is a global bus cursor.

It is assigned by the runtime event bus after the mutation transaction commits.

It gives subscribers a total order for delivery even when multiple claims mutate in one burst.

### WordPress Block Attribute Location

The canonical path inside a WordPress block is:

```ts
block.attributes.dailyos.projection.watermarks.claims[claim_id]
```

For a block whose data is a composition rather than a single claim, the composition-level watermark lives at:

```ts
block.attributes.dailyos.projection.watermarks.composition
```

The recommended block attribute shape is:

```json
{
  "dailyos": {
    "projection": {
      "projection_id": "proj_01HX...",
      "surface": "wordpress-studio",
      "composition_id": "comp_01HX...",
      "schema_version": 1,
      "watermarks": {
        "composition": {
          "composition_id": "comp_01HX...",
          "composition_version": 12,
          "source_hlc": "1778377440123-0002-runtime-a",
          "event_seq": 88214,
          "produced_at": "2026-05-10T22:24:00.123Z"
        },
        "claims": {
          "claim_01HXAA": {
            "claim_id": "claim_01HXAA",
            "claim_version": 7,
            "source_hlc": "1778377440123-0001-runtime-a",
            "event_seq": 88213,
            "produced_by": {
              "kind": "runtime",
              "ability_name": "prepare_meeting",
              "ability_version": "1.4"
            },
            "produced_at": "2026-05-10T22:24:00.123Z"
          }
        }
      }
    }
  }
}
```

The path is deliberately namespaced under `dailyos.projection`.

The block may contain ordinary WordPress attributes for typography, layout, labels, and content.

Concurrency metadata stays separate from presentation attributes.

### AbilityOutput Carrying Watermarks

Watermarks ride on the same envelope as provenance.

The domain output must not embed concurrency fields.

For Phase 1, extend the runtime envelope as:

```ts
type AbilityOutput<T> = {
  data: T;
  provenance: Provenance;
  ability_version: string;
  diagnostics: Diagnostics;
  projection?: ProjectionEnvelope;
};

type ProjectionEnvelope = {
  projection_schema_version: 1;
  composition?: CompositionWatermark;
  claims: Record<string, ClaimWatermark>;
  cursor: EventCursor;
};

type EventCursor = {
  event_seq: number;
  stream_id: string;
};
```

The `projection` field is optional because not every ability output is projected into a surface.

When an ability output is renderable as a WP Studio block, the runtime populates `projection.claims`.

When the output is a composed block tree, the runtime also populates `projection.composition`.

The provenance envelope remains the source of attribution, trust, and source freshness.

The projection envelope is the source of concurrency truth.

The two are siblings because they answer different questions:

- Provenance: why this output exists and what sources shaped it.
- Projection: which claim/composition versions this rendered output is based on.

### Composition Watermark

Composition edits have their own watermark because a composition can change without mutating a claim.

Examples include block order, presentation mode, local grouping, or replacing one projected claim block with another.

```ts
type CompositionWatermark = {
  composition_id: string;
  composition_version: number; // u64, server-assigned, monotonic per composition
  source_hlc: string;
  event_seq: number;
  produced_at: string;
};
```

A claim mutation can cause a composition update if the composition stores materialized claim text.

A pure layout edit increments only `composition_version`.

A correction to a projected claim increments `claim_version` and emits an event that may also produce a derived `composition.updated` if the rendered block payload changes.

## Conflict Resolution Matrix

All writes use compare-and-set.

The writer must supply `expected_claim_version` or `expected_composition_version`.

Missing expected versions are invalid for projected mutations.

The runtime checks the expected version inside the same transaction that applies the mutation.

The result is one of:

- `accepted`: mutation committed and new watermark returned.
- `rejected`: mutation not committed.
- `merged`: mutation committed after deterministic merge.
- `correction_event`: mutation not applied as written; client receives the authoritative current state and the correction reason.

| Conflict shape | Resolution | Rule | Event emitted | Client response |
|---|---|---|---|---|
| Stale-watermark write | Reject | If `expected_claim_version < current_claim_version`, reject the write. | `claim.write_rejected` then `claim.corrected` payload if current value differs from attempted value. | `409 Conflict` with current watermark and correction payload. |
| Concurrent writes to same claim from two SurfaceClients | Reject one | First transaction to commit wins. Second writer sees stale version and is rejected. | Winner emits `claim.updated`; loser receives `claim.write_rejected`. | Loser gets explicit `ConflictSignal` and current projection. |
| WP editor edits projected block while runtime has issued correction | Correction-event | If editor submits against the old version after correction, editor text is preserved as a local draft but not committed. | `claim.corrected` already emitted; stale submit emits `claim.write_rejected`. | UI shows conflict panel with "Runtime corrected this claim" and current value. |
| MCP client invokes mutating ability for claim already mid-flight | Reject | If the claim has an active mutation lock or pending commit transaction, reject with `MidFlightMutation`. MCP must retry after event. | `claim.write_rejected` with reason `mid_flight_mutation`; no claim update. | MCP receives structured tool error including current or pending watermark. |

### Stale-Watermark Write

Stale writes are rejected.

They are never merged automatically.

The stale writer may submit a new write after it refreshes against the current watermark.

The service response shape is:

```ts
type StaleWatermarkConflict = {
  ok: false;
  error: "stale_watermark";
  claim_id: string;
  expected: ClaimWatermark;
  current: ClaimWatermark;
  correction: CorrectionPayload;
};

type CorrectionPayload = {
  correction_id: string;
  reason:
    | "runtime_correction"
    | "user_superseded"
    | "source_refresh"
    | "tombstone"
    | "contradiction";
  current_claim: ProjectedClaimSnapshot;
  attempted_patch?: JsonPatch;
};
```

The correction payload is required even if the writer is MCP.

Agents should not infer the current claim by issuing a separate read after a rejected write.

The conflict response must carry the version it lost to.

### Concurrent SurfaceClient Writes

Two SurfaceClients can write the same claim within milliseconds.

The runtime does not compare wall-clock timestamps to decide a winner.

The database transaction order decides.

The first accepted compare-and-set increments the claim version.

The second compare-and-set observes a changed current version and fails.

The losing client receives:

```ts
type ConflictSignal = {
  type: "claim.conflict";
  claim_id: string;
  attempted_by: ActorRef;
  winning_watermark: ClaimWatermark;
  losing_expected_watermark: ClaimWatermark;
  current_projection: ProjectedClaimSnapshot;
  retry_policy: "refresh_then_resubmit";
};
```

No automatic text merge is attempted for claim values.

Claims are semantic assertions, not collaborative rich text documents.

If two edits are both valuable, the user or agent can produce a new claim that combines them after reading the current version.

### WP Editor Versus Runtime Correction

Runtime corrections have priority over stale editor drafts.

The correction represents new substrate truth.

The WP editor's local content is not discarded, but it becomes a draft attached to the block session.

The rejected mutation records a `surface_draft_preserved` diagnostic.

```ts
type SurfaceDraftPreserved = {
  draft_id: string;
  claim_id: string;
  based_on_version: number;
  current_version: number;
  stored_until: string;
};
```

The draft is not part of the authoritative claim graph.

If the editor chooses to apply it anyway, the editor must submit a fresh mutation against the current watermark.

That second submission is a new user assertion, not a replay of the stale write.

### MCP Mid-Flight Mutation

MCP is an ability surface, not a direct data writer.

An MCP mutating ability must route through the ability registry and claim service.

If the target claim is mid-flight, the service rejects the invocation before invoking a side-effectful maintenance write.

The rejected tool result shape is:

```ts
type McpMidFlightConflict = {
  ok: false;
  error: "mid_flight_mutation";
  claim_id: string;
  mutation_id: string;
  current_watermark?: ClaimWatermark;
  pending_since: string;
  retry_after_event: {
    stream: "claims";
    cursor: EventCursor;
  };
};
```

The MCP client must wait for the push event stream.

It must not poll claim state in a loop.

## Stale-Edit UX

The WordPress editor must show stale state as an editing condition, not as a generic save failure.

When a block is behind the current claim version, the block enters `outdated` state.

The user-visible behavior is:

1. The block remains visible in place.
2. The block gets a non-blocking stale indicator in the block toolbar.
3. Editing controls remain available, but save/apply actions show a conflict confirmation path.
4. The current DailyOS value is available in an inline comparison panel.
5. The user's unsaved edit is preserved locally until the user resolves or discards it.

The block should not silently rewrite while the cursor is active inside editable text.

During active editing, incoming `claim.updated` or `claim.corrected` events mark the block as stale.

After focus leaves the block, the editor may refresh passive render content if there is no local draft.

Required UI states:

```ts
type BlockConcurrencyState =
  | { state: "current"; watermark: ClaimWatermark }
  | { state: "outdated"; local: ClaimWatermark; current: ClaimWatermark }
  | { state: "conflict"; localDraftId: string; current: ClaimWatermark; reason: string }
  | { state: "refreshing"; cursor: EventCursor }
  | { state: "offline"; lastKnown: ClaimWatermark; reconnecting: boolean };
```

Toolbar behavior:

- `current`: no concurrency badge.
- `outdated`: badge says "Updated in DailyOS".
- `conflict`: badge says "Resolve update".
- `refreshing`: spinner in the DailyOS block control.
- `offline`: badge says "Offline - edits saved locally".

Conflict panel behavior:

- Show "Your draft" and "Current DailyOS value".
- Show the current version number and timestamp.
- Offer "Replace with current" and "Apply my draft to current version".
- "Apply my draft to current version" submits a new mutation using the current watermark.
- If that resubmission conflicts again, the panel remains open with the newer current value.

Runtime correction behavior:

- A `claim.corrected` event changes the label from "Updated in DailyOS" to "Corrected in DailyOS".
- The correction reason is shown in short form if safe for the surface.
- The provenance control links to the current sources and trust band.

MCP-originated update behavior:

- If an agent correction is committed, the badge says "Updated by DailyOS agent".
- If the update is only proposed and not committed, the projected block does not change.
- Proposed agent claims can appear in a separate review surface, not as silent block edits.

Offline behavior:

- Edits made offline remain local drafts.
- The editor may not mark them as saved to DailyOS.
- On reconnect, the client replays missed events first.
- Only after replay catches up may the editor offer to apply the offline draft.

Accessibility requirements:

- The stale/conflict state must be available in text, not color only.
- Conflict controls must be keyboard reachable.
- The current version and draft version must be announced as separate regions.

## Event Bus Contract

Mutation propagation is push-based.

Every SurfaceClient subscribes to the runtime event bus.

Polling is not part of the correctness contract.

### Event Names

Claim mutation events:

```ts
type ClaimEvent =
  | ClaimUpdatedEvent
  | ClaimCorrectedEvent
  | ClaimSupersededEvent
  | ClaimTombstonedEvent
  | ClaimWriteRejectedEvent
  | ClaimConflictDetectedEvent;

type ClaimUpdatedEvent = {
  type: "claim.updated";
  cursor: EventCursor;
  claim_id: string;
  previous: ClaimWatermark;
  current: ClaimWatermark;
  patch: JsonPatch;

};

type ClaimCorrectedEvent = {
  type: "claim.corrected";
  cursor: EventCursor;
  claim_id: string;
  previous: ClaimWatermark;

type ClaimSupersededEvent = {
```

### Subscription

SurfaceClients subscribe by stream and cursor.

The subscription is push, not poll.

For WP Studio, use a websocket or server-sent-events transport depending on deployment constraints.

The semantics are transport-independent:

```ts
type SubscribeRequest = {
  surface_client_id: string;
  surface: "wordpress-studio" | "mcp" | "tauri" | "other";
  streams: Array<"claims" | "compositions">;
  subjects: Array<
    | { kind: "claim"; claim_id: string }
    | { kind: "composition"; composition_id: string }
    | { kind: "subject"; subject_ref: string }
  >;
  from_cursor?: EventCursor;
  max_batch_size?: number;
};

type SubscribeAck = {
  ok: true;
  subscription_id: string;
  heartbeat_ms: number;
  replay_from?: EventCursor;
  current_cursor: EventCursor;
};
```

A SurfaceClient must subscribe to every claim and composition it has mounted.

For block-heavy pages, the client may subscribe by `composition_id` or `subject_ref` instead of individual claims.

The server expands those subjects to the claim set and sends affected events.

### Backpressure

The event bus must handle many subscribers and bursty mutations without dropping correctness events.

Backpressure rules:

- Each subscriber has a bounded outbound queue.
- Events are coalesced only by claim/composition when safe.
- `claim.write_rejected` is never coalesced.
- `claim.corrected` is never coalesced away.
- If multiple `claim.updated` events for the same claim are queued, the server may coalesce them into the latest snapshot plus an `events_coalesced` count.
- Coalesced delivery must preserve the latest `claim_version`, `source_hlc`, and `event_seq`.
- If the queue exceeds the bound, the server sends `subscription.backpressure` and requires replay from cursor.

Backpressure event:

```ts
type BackpressureEvent = {
  type: "subscription.backpressure";
  cursor: EventCursor;
  subscription_id: string;
  reason: "client_slow" | "burst_coalesced" | "queue_limit";
  replay_required: boolean;
  replay_from: EventCursor;
  dropped_event_count?: number;
};
```

When `replay_required` is true, the client must pause local "current" indicators.

It marks affected blocks as `refreshing`.

It then reconnects or issues a replay request from the provided cursor.

### Reconnection and Replay

The event bus stores an ordered event log for at least the replay window.

Phase 1 acceptance should assume a minimum replay retention of 24 hours or 100,000 events, whichever is larger.

On reconnect, the client sends its last acknowledged cursor.

The server replies with all missed events in event-sequence order.

```ts
type ReplayRequest = {
  surface_client_id: string;
  subscription_id?: string;
  from_cursor: EventCursor;
  streams: Array<"claims" | "compositions">;
  max_batch_size: number;
};

type ReplayResponse = {
  ok: true;
  events: RuntimeEvent[];
  next_cursor: EventCursor;
  has_more: boolean;
};
```

If the cursor is too old:

```ts
type ReplayExpired = {
  ok: false;
  error: "cursor_expired";
  oldest_available_cursor: EventCursor;
  required_action: "resync_snapshot";
};
```

`resync_snapshot` means the client must fetch the current projected composition snapshot once.

After snapshot load, it subscribes from the snapshot cursor.

Snapshot fetch is allowed only as recovery from an expired cursor or initial page load.

It is not a polling loop.

### Ordering Guarantees

The push contract guarantees:

- Per-claim events are delivered in increasing `claim_version`.
- Per-composition events are delivered in increasing `composition_version`.
- Cross-claim delivery follows increasing `event_seq`.
- Replay returns events in the same order subscribers would have received them live.
- A `claim.write_rejected` response includes the current watermark known at rejection time.
- A client that catches up to `current_cursor` can render "current" without issuing a poll.

### Idempotency

Clients must treat events as idempotent.

If a client receives an event with `event_seq <= last_applied_event_seq`, it ignores it.

If a client receives a claim event with `claim_version <= local_claim_version`, it ignores it unless the event is `claim.write_rejected` addressed to that client.

The runtime may redeliver events after reconnect.

Redelivery must not duplicate visible conflict panels.

## Polling Rejection

Polling is unacceptable as a concurrency mechanism for DOS-546.

It fails the product and system requirements:

- Battery: WordPress editor tabs can remain open for hours; polling every block or claim wastes CPU and network on idle pages.
- Latency: a 5-second poll interval means stale edits are accepted by the UI for up to 5 seconds after correction.
- Scale: N open editors multiplied by M projected blocks creates avoidable read load on the DailyOS runtime.
- Correctness: polling can miss intermediate correction context and only observe the latest snapshot.
- User trust: stale save failures feel random when the UI was not told immediately that the substrate changed.

Push is the contract because it gives explicit guarantees:

- The runtime tells clients about mutations immediately after commit.
- Clients receive correction payloads, not just changed values.
- Reconnect uses ordered replay from cursor.
- Backpressure is explicit and recoverable.
- The UI can move blocks into `outdated`, `conflict`, or `refreshing` state at the right time.

Snapshot reads are still allowed in two cases:

1. Initial render of a WordPress Studio page.
2. Recovery after replay cursor expiry.

Snapshot reads are not polling.

They do not replace live subscription.

## Phase 1 Acceptance Fixtures

### Fixture 1: Two SurfaceClients Write Same Claim Within 50ms

Setup:

```json
{
  "claim_id": "claim_fixture_001",
  "initial_claim_version": 3,
  "surface_clients": ["surface_a", "surface_b"],
  "write_window_ms": 50
}
```

Steps:

1. Both clients render the claim at `claim_version = 3`.
2. `surface_a` submits patch A with `expected_claim_version = 3`.
3. `surface_b` submits patch B with `expected_claim_version = 3` within 50ms.
4. Runtime applies compare-and-set in transaction order.

Expected:

- Exactly one write returns `accepted`.
- The accepted write returns `claim_version = 4`.
- The loser returns `409 Conflict`.
- The loser receives `claim.write_rejected` with reason `stale_watermark`.
- Both clients receive `claim.updated` for version 4.
- No last-writer-wins overwrite occurs.

### Fixture 2: SurfaceClient Reconnect Receives Missed Events In Order

Setup:

```json
{
  "surface_client_id": "surface_replay_001",
  "last_cursor": { "stream_id": "claims", "event_seq": 100 },
  "mutations_while_offline": [
    { "claim_id": "claim_a", "from_version": 1, "to_version": 2 },
    { "claim_id": "claim_b", "from_version": 4, "to_version": 5 },
    { "claim_id": "claim_a", "from_version": 2, "to_version": 3 }
  ]
}
```

Steps:

1. Client disconnects after acknowledging `event_seq = 100`.
2. Runtime commits three mutations at `event_seq = 101`, `102`, and `103`.
3. Client reconnects with `from_cursor.event_seq = 100`.

Expected:

- Server replays events 101, 102, and 103 in order.
- `claim_a` applies version 2 before version 3.
- Client receives `has_more = false` after catching up.
- Client marks affected blocks `current` without polling.
- Duplicate replay after reconnect does not duplicate visible UI alerts.

### Fixture 3: Stale-Watermark Write From MCP Is Rejected With Correction Payload

Setup:

```json
{
  "claim_id": "claim_mcp_001",
  "mcp_expected_version": 8,
  "current_claim_version": 10,
  "current_reason": "runtime_correction"
}
```

Steps:

1. MCP client invokes an Agent-accessible ability that attempts to mutate `claim_mcp_001`.
2. The ability input carries `expected_claim_version = 8`.
3. Runtime claim service observes current version 10.

Expected:

- Ability returns structured tool error, not a generic failure.
- Error reason is `stale_watermark`.
- Payload includes `current.claim_version = 10`.
- Payload includes `correction.reason = "runtime_correction"`.
- No claim mutation occurs.
- Event bus emits `claim.write_rejected` addressed to the MCP client.

### Fixture 4: Runtime Correction Arrives While WP Editor Has Active Draft

Setup:

```json
{
  "claim_id": "claim_wp_001",
  "editor_version": 5,
  "runtime_correction_version": 6,
  "editor_state": "active_text_focus"
}
```

Steps:

1. WP editor focuses projected claim text at version 5.
2. User types a local draft.
3. Runtime commits correction version 6.
4. Event bus pushes `claim.corrected`.
5. User attempts to save draft with `expected_claim_version = 5`.

Expected:

- Block enters `outdated` state when correction arrives.
- Text under active cursor is not overwritten.
- Save attempt is rejected as stale.
- Conflict panel opens with "Your draft" and "Current DailyOS value".
- Draft is preserved with `based_on_version = 5`.
- Applying draft requires a new write against version 6.

### Fixture 5: Mid-Flight MCP Mutation Is Rejected

Setup:

```json
{
  "claim_id": "claim_midflight_001",
  "active_mutation_id": "mut_001",
  "mcp_client_id": "mcp_fixture_001"
}
```

Steps:

1. Runtime starts a claim mutation transaction for `claim_midflight_001`.
2. MCP client invokes a mutating ability targeting the same claim.
3. Claim service checks mid-flight state before applying mutation.

Expected:

- MCP receives `mid_flight_mutation`.
- Payload includes `mutation_id`.

- Event bus emits `claim.write_rejected`.

## Open Questions

1. Should Phase 1 use websocket or SSE for WP Studio transport?

   The contract allows either.

   Websocket is better for bidirectional editor sessions.

   SSE is simpler if writes already go through HTTP ability invocation.

2. What is the final replay retention window?

   This draft proposes 24 hours or 100,000 events.

   Phase 1 should validate memory/storage cost under bursty composition updates.

3. Should composition snapshots be materialized in WordPress post meta?

   The block attribute path is enough for per-block concurrency.

   Large compositions may need post-meta indexing to avoid loading oversized block JSON.

4. How does ADR-0130 name the canonical block type?

   This draft uses `dailyos.projection` as the namespace and is compatible with a block type like `dailyos/composition`.

   Phase 1 should align the final attribute path with ADR-0130's exact block schema.

5. Are MCP-originated corrections always agent-authored?

   ADR-0111 treats MCP as `Actor::Agent`.

   If a future MCP session carries a verified user delegation token, the actor model may need a `delegated_user` variant.

6. How much of the conflict panel is WordPress-native versus DailyOS React?

   The behavior is required either way.

   The rendering implementation can be native Gutenberg components, embedded React, or a hybrid.

7. Should rejected writes be durable audit events?

   This artifact requires event-bus emission.

   The companion tamper contract should decide which rejected writes enter the tamper-evident audit chain.

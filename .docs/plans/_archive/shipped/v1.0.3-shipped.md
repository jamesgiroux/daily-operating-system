# v1.0.3 — The Meeting Record

**Thesis:** The meeting briefing page is DailyOS's crown jewel — "walk in prepared" happens here. But after the meeting, the page becomes two documents fighting for space: outcomes stacked on faded prep. v1.0.3 transforms the meeting page into a temporal journey. Each stage of the meeting lifecycle — upcoming, in-progress, just-ended, processed — is a purpose-built editorial document. After processing, the page becomes a Meeting Record: executive summary, prep prediction scorecard, structured outcomes, and meeting-to-meeting continuity. A consolidated markdown record is generated for MCP consumption. The CSM opens it two hours after a tough QBR and it reads like a polished executive brief they'd share with their VP unprompted.

**Status:** Not started. Depends on v1.0.2 (I621 editorial treatment of transcript outcomes is the foundation this builds on, I622 past-meeting opacity is resolved).

**Branch model:** Short-lived per-issue branches from `dev`.

**Origin:** CEO plan review (2026-03-21). SCOPE EXPANSION mode — the crown jewel page deserves the cathedral treatment. Health scoring accuracy (I633) discovered as a blocking prerequisite.

---

## Principles

1. **The page tells a temporal story.** Not four modes of the same page — four chapters of a meeting's life. Each feels intentional.
2. **AI produces; users curate.** The meeting record writes itself. The CSM reviews it, corrects what's wrong, and moves on.
3. **Zero guilt.** If the CSM never looks at the meeting record, nothing breaks. Intelligence flows into the dossier and briefing automatically.
4. **The prediction scorecard builds trust.** Showing "what we predicted vs what happened" makes the prep credible — and feeds Bayesian learning when predictions are right or wrong.
5. **The meeting record is MCP-ready.** A markdown file in the entity directory means Claude Code can answer "what happened in the Acme QBR?" from a single document.

**Acceptance criteria:**

## Phase 2: Temporal Transformation

**Goal:** The meeting page transforms through 4 stages, each with purpose-built editorial content.

| ID | Title | Notes |
|----|-------|-------|
| **I634** | Meeting page temporal transformation | The page has 4 states: upcoming (prep briefing, editable), in-progress (prep locked, recording indicator), just-ended (processing progress UI), processed (meeting record — transformed layout with executive summary, outcomes, prediction scorecard, continuity thread). Pre-meeting content becomes a collapsed appendix in the processed state. |

**The 4 Stages:**

```
1. UPCOMING (existing, enhanced)
   ─────────────────────────────
   Full prep briefing. Editable.
   "Brief me before this meeting."

2. IN-PROGRESS (enhanced)
   ────────────────────────
   Prep locked (read-only). Live status.
   "Meeting in progress — recording active."
   Granola auto-check continues.

3. JUST-ENDED (new)
   ──────────────────
   "Meeting ended 15 min ago."
   Waiting for transcript → processing progress:
   ● Phase 1: Summary & actions... done (12 actions)
   ● Phase 2: Intelligence... done (3 wins, 2 risks)
   ○ Phase 3: Deep analysis...
   Uses existing transcript-progress events.

4. PROCESSED (transformed)
   ──────────────────────────
   The Meeting Record — executive document:
   ├── Headline (AI-generated one-liner)
   ├── Executive Summary
   ├── The Thread (since last meeting)
   ├── What We Predicted vs What Happened (I635)
   ├── Structured Outcomes (I621 editorial)
   │   ├── Key Findings (wins/risks/decisions)
   │   ├── Engagement Dynamics
   │   ├── Champion Health
   │   └── Commitments & Actions
   ├── The Room (attendee summary)
   └── Pre-Meeting Context (collapsed appendix)
```

**Acceptance criteria:**

1. Navigate to an upcoming meeting. Full prep briefing shows, inline editable.
2. During the meeting (time-based detection), prep content becomes read-only. "Meeting in progress" status indicator visible.
3. After meeting ends but before transcript arrives (0-30 min window), page shows "Meeting ended X minutes ago. Waiting for transcript..." No stale prep at full opacity.
4. When transcript processing starts, phase-by-phase progress shows with counts (actions, wins, risks).
5. When processing completes, the page transforms into the Meeting Record layout. Pre-meeting content is in a collapsed "Pre-Meeting Context" appendix at the bottom.
6. The Meeting Record reads top-to-bottom as a cohesive executive document, not two documents stacked.
7. Folio bar updates to reflect the current stage: "Briefing" (upcoming), "In Progress" (during), "Processing..." (just-ended), "Meeting Record" (processed).
8. Transition between stages is smooth — no full page reload, no layout shift.
9. Direct URL to a processed meeting renders the Meeting Record, not the prep-first layout.
10. Back navigation from Meeting Record returns to the correct prior surface when navigated from briefing or dossier.

## Phase 3: Intelligence Layers

**Goal:** The meeting record includes prediction comparison and meeting continuity.

| ID | Title | Notes |
|----|-------|-------|
| **I635** | Prep prediction scorecard | Compare pre-meeting prep (from `prep_frozen_json`) against transcript outcomes. Show: ✓ predicted risks that materialized, ✗ predicted risks that didn't come up, ⚡ surprises (outcomes with no prior signal). Feeds Bayesian learning — correct predictions increase signal source confidence. |
| **I637** | Meeting-to-meeting continuity | "The Thread" section in the meeting record: what changed between this meeting and the previous one with the same entity. Actions completed since last meeting, risks that resolved or escalated, new stakeholders. Data already computed for prep's "Since Last Meeting" — extend into post-meeting context. |

**Acceptance criteria (I635):**

1. Processed meeting record shows "What We Predicted vs What Happened" section with three categories: ✓ confirmed, ✗ not raised, ⚡ surprise.
2. Predicted risks from prep compared against transcript risks using semantic similarity (not exact string match).
3. Predicted wins from prep compared against transcript wins.
4. Items with no prep match classified as ⚡ surprises.
5. Each prediction result shows provenance: where the prediction came from and whether it was confirmed, not raised, or a surprise.
6. Clicking a confirmed or surprise prediction result links to the relevant outcome below when there is a matching transcript item.
7. Correct predictions emit positive feedback signal for the source that generated the prep risk. `✗ not raised` does not emit negative feedback; `⚡ surprise` logs a coverage gap.
8. If there is no frozen prep or no transcript outcomes, the scorecard section is hidden rather than rendered as an empty state.

**Acceptance criteria (I637):**

1. Meeting record shows "The Thread" section: "Since your last meeting with {entity} on {date}..."
2. Lists: actions completed, actions still open with overdue indication, health score change, new attendees.
3. If no previous meeting exists with this entity, section shows "First meeting with {entity}."
4. If the meeting has no linked entity, the thread section is omitted entirely.
5. Thread data appears in the meeting record markdown.
6. Thread data comes from existing `sinceLast` prep computation — reused, not recomputed.

## Phase 4: Meeting Record Markdown

**Goal:** A consolidated meeting record markdown file for MCP consumption.

| ID | Title | Notes |
|----|-------|-------|
| **I636** | Meeting record markdown | After processing completes, generate a structured markdown file at `~/Accounts/{Name}/Meeting-Records/{date}-{slug}-record.md` (or person/project directory per I631). Contains: meeting metadata, executive summary, prediction scorecard, structured outcomes, actions, The Thread, attendee summary. NOT the raw transcript (that's separate). This is the intelligence output document. |

**Acceptance criteria:**

1. After transcript processing completes, a meeting record markdown file is generated in the entity directory under `Meeting-Records/`.
2. The markdown file contains: YAML frontmatter (meeting metadata), executive summary, prediction scorecard, outcomes (wins/risks/decisions), engagement dynamics, champion health, commitments, actions, The Thread, attendees.
3. MCP `search_content` for the entity finds the meeting record.
4. File is generated after Phase 3 completes — includes all intelligence layers.
5. If entity is a person (1:1 meeting), record goes to `~/People/{Name}/Meeting-Records/`.
6. If entity is a project, record goes to `~/Projects/{Name}/Meeting-Records/`.
7. If prediction scorecard or continuity data is unavailable, those sections are omitted rather than rendered empty.
8. Reprocessing a transcript regenerates and overwrites the meeting record.
9. `meeting_transcripts.record_path` stores the generated record path.
10. Meeting record file is separate from the transcript file (which contains raw text).

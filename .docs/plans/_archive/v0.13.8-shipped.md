# v0.13.8 Version Brief

**Thesis:** AppState is a 30-field god struct. Every subsystem — scheduler, capture loop, hygiene scanner, calendar poller, signal engine — reaches into the same monolith for its 2-3 dependencies. This makes it impossible to reason about what a subsystem needs, impossible to test subsystems in isolation, and creates implicit coupling between unrelated domains. Meanwhile, ~47 signal emission call sites across 15 files all manually import `crate::signals::bus::emit_signal_and_propagate` and pass `&state.signal_engine` — the last major module boundary without a service wrapper. This version does two things: (1) splits AppState into domain containers so the struct becomes a facade, and (2) formalizes the signal system behind a SignalService so callers stop reaching into bus internals. Both are mechanical refactors — no logic changes, no behavior changes. This is the last structural prerequisite before v0.14.0 feature work.

**Status:** Planning.

When v0.13.8 creates `HygieneState`, that container covers both `hygiene.rs` fields AND the new `self_healing/` module fields. The boundary is already clean — `hygiene.rs` owns structural/mechanical fix state; `self_healing/` owns quality scoring state. `HygieneState` becomes the parent container for both. No logic changes, same mechanical extraction pattern as the rest of this version.

### I403 — SignalService formalization

1. A `services/signals.rs` file exists with a `SignalService` struct (or a set of public free functions following the same pattern as other services) exposing these methods:
   - `emit(db, engine, entity_type, entity_id, signal_type, source, value, confidence) → Result<(String, bool)>` — wraps `bus::emit_signal` (no propagation)
   - `emit_and_propagate(db, engine, entity_type, entity_id, signal_type, source, value, confidence) → Result<(String, bool)>` — wraps `bus::emit_signal_and_propagate`
   - `get_for_entity(db, entity_id) → Vec<SignalEvent>` — wraps existing query
   - `get_callouts(db, entity_id) → Vec<Callout>` — wraps callout generation
   - `run_propagation(db, engine, entity_id) → Result<()>` — fire cross-entity rules for a single entity
   - `invalidate_preps(db, entity_id, prep_queue) → Result<()>` — queue affected meeting preps for re-generation
2. All **service-layer** call sites (`services/accounts.rs`, `services/people.rs`, `services/actions.rs`, `services/meetings.rs`, `services/projects.rs`, `services/emails.rs`, `services/intelligence.rs`) call `SignalService::emit` or `SignalService::emit_and_propagate` instead of `crate::signals::bus::emit_signal*` directly. Verify: `grep -r "signals::bus::emit" services/` returns zero matches.
3. **Internal signal callers** within `signals/` itself (`cadence.rs`, `post_meeting.rs`, `email_bridge.rs`, `feedback.rs`, `propagation.rs`, `callouts.rs`) continue to call `bus::emit_signal*` directly — they are internal to the module boundary, not external consumers. The service wraps the public API; internal wiring stays internal.
4. **Infrastructure callers** outside `services/` (`commands.rs`, `google.rs`, `clay/enricher.rs`, `hygiene.rs`, `executor.rs`, `prepare/`, `processor/`, `gravatar/`) are migrated to SignalService where they have access to AppState. Call sites that only have a raw `db` handle (no AppState) may keep the direct call if wrapping would require threading AppState through — document these exceptions.
5. `cargo test` passes. `cargo clippy -- -D warnings` passes.
6. Signal behavior is unchanged — same events emitted, same propagation rules fire, same callouts generated. Verify: run the app, trigger a field edit on an account, confirm `signal_events` table gets the same `field_updated` signal as before.

## Risk Assessment

**Blast radius:** Wide but shallow. Every file that accesses an AppState field needs a path update, and every file that calls `bus::emit_signal*` needs a SignalService import. But each change is mechanical — no judgment calls, no logic changes. The risk is typos and missed call sites, both caught immediately by the compiler.

**Recommended approach:**

For I403 (SignalService): Create the service file, migrate one consumer file at a time, `cargo check` after each. Internal `signals/` callers stay on direct `bus::` calls.

For I404/I405 (AppState): Extract one container at a time. After each extraction:
1. `cargo check` — compiler finds every missed call site
2. Fix all errors
3. `cargo test` — verify no logic regression
4. Move to next container

**Estimated call sites:**
| Work item | Est. call sites | Key files |
|-----------|----------------|-----------|
| SignalService (I403) | ~30 external | services/*.rs (~20), commands.rs (~4), google.rs (~2), clay/ (~2), hygiene.rs (~3) |
| HygieneState | ~20 | hygiene.rs, self_healing/, commands.rs, scheduler.rs |
| CaptureState | ~20 | capture.rs, commands.rs, post_meeting.rs |
| CalendarState | ~25 | google.rs, commands.rs, proactive/ |
| WorkflowState | ~30 | executor.rs, scheduler.rs, commands.rs |
| SignalState | ~15 | signals/, services/, commands.rs |
| IntegrationState | ~10 | clay/, quill/, linear/, google.rs |

Total: ~150 call site updates across ~15 files.

**I403-specific risk:** The `signals/` internal callers (`email_bridge.rs`, `cadence.rs`, `post_meeting.rs`, `feedback.rs`, `propagation.rs`, `callouts.rs`) have ~17 `emit_signal*` calls that stay as direct `bus::` calls. The boundary is: anything inside `signals/` is internal; everything else goes through SignalService. This matches the pattern where `signals/bus.rs` defines the functions and `signals/` submodules are co-located consumers.

## Ship criteria

All acceptance criteria for I403, I404, and I405 met. SignalService exists and all external consumers use it — `grep -r "signals::bus::emit" services/` returns zero. AppState has ≤15 top-level fields. All 30 original fields are accessible through domain containers. `cargo test` passes. `cargo clippy -- -D warnings` passes. The app starts normally and every surface renders correctly. No logic changes — `git diff` shows only path renames, struct definitions, constructor updates, and service wrapper additions.

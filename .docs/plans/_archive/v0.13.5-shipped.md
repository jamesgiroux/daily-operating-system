# v0.13.5 Version Brief

**Thesis:** People are the most under-represented entities in DailyOS. A user managing a buying committee knows five individuals by name but sees them as independent contacts with no structural relationship to each other. They don't see that the champion is locked out of decisions because of a blocker, or that the executive sponsor will only approve what the technical evaluator signs off on. This version adds typed relationship networks for people — a qualitatively different intelligence model because people don't form trees, they form relationship graphs. It builds a `person_relationships` table with directional, context-scoped edges (champion, blocker, peer, ally, etc.), extends person enrichment to include relationship context, adds person→person signal propagation so relationship changes ripple through the network, and surfaces the network as a new chapter on person detail pages with AI-synthesized cluster summaries, risks, and opportunities. (ADR-0088)

**Status:** Planning.

## Design Decisions

### Person page chapter naming: "Their" pattern

Person detail pages shift from "The ___" to "Their ___" for person-specific chapters. This subtly signals that person pages are *about someone*, not about a system concept. The existing Chapter 3 "The Network" (linked accounts/projects) becomes **"Their Orbit"**. The new person-to-person relationship chapter is **"Their Network"**. The "Their" prefix becomes a design pattern exclusive to person entity pages.

Updated chapter order for `PersonDetailEditorial`:

| # | Current Name | New Name | Content |
|---|-------------|----------|---------|
| 1 | The Profile | The Profile | Hero, vitals, metadata (unchanged) |
| 2 | The Dynamic / The Rhythm | The Dynamic / The Rhythm | Relationship health (unchanged) |
| 3 | The Network | **Their Orbit** | Linked accounts + projects (renamed) |
| 4 | *(new)* | **Their Network** | Person-to-person relationships (I392) |
| 5 | The Landscape | The Landscape | Risks, wins, unknowns (unchanged) |
| 6 | The Record | The Record | Meetings, captures, email signals (unchanged) |
| 7 | The Work | The Work | Open actions + upcoming meetings (unchanged) |

### Confidence decay model: computed-on-read

Edge confidence in `person_relationships` decays using the same exponential half-life formula as signals (`decayed = base * 2^(-age_days / half_life_days)`), but computed on read — not by a background job. The `confidence` column stores the original confidence at creation or last reinforcement. The `last_reinforced_at` column tracks when evidence last supported the edge. Decay is calculated at query time from `last_reinforced_at` (or `created_at` if never reinforced). Default half-life: 90 days for inferred edges, no decay for user-confirmed edges (`source = 'user_confirmed'`).

### Edge creation in v0.13.5: enrichment pipeline + Tauri CRUD command

Edges are created by two paths:
1. **Enrichment pipeline** (I391): When person enrichment detects relationship patterns, it writes edges with `source = 'inferred'` and low initial confidence (~0.4). No confirmation UI in this version — inferred edges contribute at reduced confidence.
2. **Tauri command** (`upsert_person_relationship` / `delete_person_relationship`): Ships in I390 for programmatic and testing use. Frontend calls this when implementing the edge suggestion confirmation UI in v0.13.6.

### Preset vocabulary expansion: `relationship_vocabulary` field

`RolePreset` gains a `relationship_vocabulary: Record<string, string>` field mapping internal relationship types to display labels. Resolved at render time. Backend preset definitions ship with vocabulary for all 9 presets. The field is optional — presets without it fall back to the internal type name (title-cased).

## Implementation Context (from codebase research)

### I390 — What exists today

- **Migration 038** is the next available (current max: 037).
- `entity_people` table already links persons to accounts/projects with `relationship_type DEFAULT 'associated'`. The new `person_relationships` table is person-to-person only — no collision.
- `account_team` has a `role` concept (csm, champion, tam) for account-team assignments. Different concern from person-to-person relationships.
- `DbPerson` struct in `db/types.rs:363` has 24 fields. No changes needed — relationships live in a separate table.

### I391 — What exists today

- **Propagation engine** in `signals/propagation.rs` uses `default_engine()` with `engine.register()`. Adding `rule_person_network` follows the same pattern as the 7 existing rules (6 active + 1 dead).
- **Loop prevention** is source-tag-based: check `signal.source.contains("propagation:network")`. Existing rules use `"propagation:hierarchy"` — no overlap.
- **`IntelligenceJson`** in `intelligence/io.rs:72` has no `network` field today. Add `network: Option<NetworkIntelligence>` following the same pattern as `portfolio: Option<PortfolioIntelligence>`.
- **Person enrichment prompt** (`intelligence/prompts.rs`) currently includes: facts, meetings, email signals, entity connections. Adding relationship edges + neighbor signal summaries is a prompt expansion, not a restructure.
- **Edge-type-sensitive multipliers** (ADR-0088): The rule function needs a `match` on `relationship_type` to apply modifiers (1.0x for champion/sponsor/decision_maker, 1.2x for blocker/detractor, 0.7x for peer/ally, 0.0x for collaborator below threshold, context-gated for dependency). More nuanced than the existing hierarchy rules (flat 0.6/0.5).

### I392 — What exists today

- **`PersonDetailEditorial.tsx`** renders 6 chapters via `buildChapters()` at line 98. Chapter 3 "The Network" uses `PersonNetwork.tsx` for entity links.
- **`PresetVocabulary`** in `types/preset.ts:35` has 7 fields. Needs the new `relationshipVocabulary` field.
- **Callout pattern** (`BriefingCallouts.tsx`): Severity-coded left border + headline + detail. Reusable for risks/opportunities — no new component needed.
- **`StateBlock`** component: Colored label + prose paragraphs. Reusable for cluster summary.
- **Entity navigation**: `PersonNetwork.tsx:160` uses React Router `<Link>` to navigate to entity detail pages. Same pattern applies for person-to-person links (route: `/people/$personId`).
- **`usePersonDetail.ts`** hook orchestrates data loading. Listens for `intelligence-updated` events to refresh. No changes needed — the hook already loads `EntityIntelligence` which will include the new `network` field.

### I390 — Person relationship graph

1. `person_relationships` table exists in the DB (migration 038) with columns: `id`, `from_person_id`, `to_person_id`, `relationship_type`, `direction`, `confidence`, `context_entity_id`, `context_entity_type`, `source`, `created_at`, `updated_at`, `last_reinforced_at`. Creating an edge via the `upsert_person_relationship` Tauri command succeeds without error.

2. All 10 relationship types are valid enum values: `champion`, `executive_sponsor`, `decision_maker`, `technical_evaluator`, `blocker`, `peer`, `ally`, `detractor`, `collaborator`, `dependency`. Attempting to insert an invalid type is rejected by the Rust type system (enum validation before DB insert).

3. Context scoping works: insert two edges for the same person pair — one scoped to Account X (`context_entity_id` and `context_entity_type` both set), one scoped to Project Y. Query returns both edges. Filtering by `context_entity_id` returns only the relevant edge(s).

4. Confidence decay is computed on read: query an edge created 30 days ago with initial confidence 0.8 and `source = 'inferred'` (90-day half-life). The returned effective confidence is `0.8 * 2^(-30/90) ≈ 0.635`. A `user_confirmed` edge does not decay regardless of age.

5. `upsert_person_relationship` and `delete_person_relationship` Tauri commands exist. Upserting an existing edge (same `from_person_id`, `to_person_id`, `relationship_type`, `context_entity_id`) updates its fields and advances `updated_at`. Deleting an edge removes it from the DB.

6. A DB migration handles the `person_relationships` table creation. `cargo test` passes. All existing person data is unaffected — no person records are deleted or corrupted.

### I391 — People network intelligence

1. Person enrichment prompt includes: the person's own signals (Layer 1), their relationship edges from `person_relationships` (with decayed confidence), and brief signal summaries for each connected person. Verify by adding a debug log that prints the assembled prompt for one person with ≥2 known relationships — confirm the logged prompt contains real relationship type data and connected person names.

2. Person `intelligence.json` includes a `network` top-level field. A person with ≥2 relationship edges has non-null values in `key_relationships` (array with real entries), `cluster_summary` (non-empty string), and `influence_radius` (integer > 0).

3. A person with zero relationship edges in the DB has `network.health = "unknown"` and empty arrays (`key_relationships: []`, `risks: []`, `opportunities: []`). The `network` field exists but is sparse — not missing entirely.

4. Person→person signal propagation: Emit a high-confidence signal (≥0.65) for Person A who has a `champion` edge to Person B with edge confidence 0.8. Within one `intel_queue` propagation cycle, `signal_events` contains a row for Person B with `source = 'propagation:network'` and `confidence = (original_signal_confidence * 0.8 * 0.4)`. Verify with real data by querying `signal_events`.

5. Edge-type-sensitive propagation: A person has a `blocker` edge (confidence 0.8) to another person. Emit a 0.65 confidence signal for the first person. The derived signal confidence should be `0.65 * 0.8 * 0.4 * 1.2 ≈ 0.250` (0.4 base × 1.2 blocker multiplier = 0.48 effective multiplier). Verify in `signal_events` that the derived signal has confidence ≈ 0.25.

6. No propagation cascade: Person A emits a signal that propagates to Person B (A's neighbor). Person B does NOT generate a propagated signal to Person C (B's neighbor) in the same cycle. Query `signal_events` — confirm no `propagation:network` signal has itself spawned another `propagation:network` signal in the same cycle.

7. Network enrichment is triggered when a person's relationship graph changes. Add a new relationship edge for a person; within one `intel_queue` cycle, that person's `entity_intel.updated_at` timestamp advances. Verify by querying `entity_intel` for the person before and after adding an edge.

8. Inferred edges from enrichment: After enriching a person who attends meetings with another known person, if the enrichment output identifies a relationship pattern, a row appears in `person_relationships` with `source = 'inferred'` and confidence ~0.4. Verify by checking `person_relationships` before and after enrichment for a person with shared meeting attendance.

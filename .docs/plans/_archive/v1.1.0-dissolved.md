# v1.1.0 — Intelligence Foundation + CS Report Suite

**Thesis:** DailyOS becomes the operating system for the VP of Key Accounts. Health scores are null for ~80% of accounts, inferred relationships are extracted then discarded, and Glean data flows into prompts but never into the signal bus or people table. Reports built on hollow intelligence are hollow reports. v1.1.0 fixes the intelligence foundation so every account has a health score and relationship data, connects Glean structurally into the people table and signal bus, and builds the complete CS report suite — including VP-specific types that encode operational rigor.

**Status:** Planning.

**Prerequisite:** v1.0.0 shipped. Glean dual-mode context (v0.15.2) production-ready. v0.14.x user entity + role presets shipped. v0.15.0 report infrastructure shipped.

**Context:** Gap analysis (`.docs/research/2026-02-28-hook-gap-analysis.md`) concluded DailyOS + Glean covers ~80% of what a VP of Key Accounts needs. The remaining gaps are intelligence depth and VP-facing surfaces. Three-layer model: Glean retrieves facts, DailyOS interprets them, humans decide.

**Dissolves:** v1.0.1 (I458–I461). All CS report types absorbed here with richer acceptance criteria because they now build on a fixed intelligence foundation rather than hollow data.

### I499 — Health scoring engine

**Problem:** No algorithmic health scoring. LLM invents health_score meaning each enrichment. ADR-0097 defines 6 relationship dimensions computed deterministically.

**Files:** `src-tauri/src/intelligence/health.rs` (NEW), `src-tauri/src/intel_queue.rs`, `src-tauri/src/intelligence/prompts.rs`

1. Every enriched account has non-null `AccountHealth` with all 6 dimension sub-scores
2. Sparse account (1 meeting, no email) gets health score with low confidence band (0.2-0.4), not null
3. Dimension scores are deterministic — same data produces same scores (no LLM randomness)
4. Lifecycle weighting changes composite: at-risk account with declining momentum scores lower than mature account with same momentum
5. When org baseline exists (from I500), divergence detection fires when relationship composite differs by 15+ points
6. User's manual RAG influences composite by ~15% per industry consensus
7. Unit tests for each dimension, weight normalization, sparse handling, divergence detection

### I507 — Source-attributed correction feedback

**Problem:** The Bayesian feedback engine (Thompson Sampling in `signals/bus.rs`) works but is narrowly wired. Only entity-resolution replacements close the full feedback loop (user action → identify wrong source → penalize). Intelligence field corrections blame `"intel_queue"` generically — never the actual source (Glean, email, transcript). Person profile corrections blame `"clay"` even when Glean wrote the field. Email disposition corrections are dead ends (written to `email_signals`, never read). The system never learns which sources are reliable for which intelligence.

**Why it matters:** Day-to-day corrections at the account level compound upward. When a CSM corrects a risk, that should penalize the source that produced the bad risk. After enough corrections, reports for the VP are built on better-calibrated intelligence. Without source attribution, every source stays at its default weight forever.

**Depends on:** I487, I504, I505 (signals must exist before feedback can operate on them)

**Files:** `src-tauri/src/intelligence/prompts.rs` (add source_attribution to schema), `src-tauri/src/intelligence/io.rs` (parse/store attribution), `src-tauri/src/services/intelligence.rs` (attributed correction routing), `src-tauri/src/services/people.rs` (profile correction routing), `src-tauri/src/commands.rs` (email disposition feedback)

1. After enrichment, `intelligence.json` includes `source_attribution` mapping fields to source names
2. User corrects a risk → source that informed that risk (e.g. `"glean_search"`) gets beta += 1 in `signal_weights`
3. After 5+ corrections penalizing a source, `get_learned_reliability()` returns < 0.5 for that source
4. User corrects a Glean-sourced person title → `"glean"` penalized for `("person", "profile_enrichment")`
5. User corrects email disposition → source that scored the email penalized
6. VP Account Review for account with corrected risks produces different risk section on re-enrichment

### I508 — Intelligence schema redesign for multi-source enrichment

**Problem:** `IntelligenceJson` was designed for calendar + email + transcript inputs. With Glean, entirely new intelligence dimensions (competitive context, support health, product adoption, org changes, commercial context) have no home in the schema. The enrichment prompt asks questions calibrated for meeting transcripts — it doesn't ask the questions Glean can answer. Local mode benefits equally: the richer schema defines what good intelligence looks like regardless of source. The system fills what it can, makes gaps visible.

**Files:** `src-tauri/src/intelligence/io.rs` (new types, fields with `#[serde(default)]`), `src-tauri/src/intelligence/prompts.rs` (expand prompt schema, evolve gap detection), `src/types/index.ts` (matching TypeScript types), `src-tauri/src/context_provider/glean.rs` (wire gap queries)

1. New intelligence dimensions added: competitive_context, opportunities, blockers, support_health, product_adoption, organizational_changes, contract_context, expansion_signals, renewal_outlook, coverage_assessment, source_attribution — all with `#[serde(default)]`
2. Existing intelligence.json files deserialize without error (backward compatible, additive only)
3. Enrichment prompt asks for all new fields with evidence-based guidance — LLM fills what it can from available context, omits what it can't
4. **Local mode**: enrich with only meetings/emails. Existing fields populated as today. New fields empty — correct behavior
5. **Remote mode**: enrich with Glean connected. At least 2 new dimension fields populated from Glean context
6. `semantic_gap_queries()` evolved from 3-field checker to dimension-aware gap detector, sent to Glean when connected
7. Frontend types match — `pnpm tsc --noEmit` passes
8. No DB migration needed — intelligence.json is a JSON blob column

### I509 — Transcript interaction dynamics extraction

**Problem:** DailyOS processes the same call transcripts Gong analyzes but extracts only meeting outcomes (summary, actions, wins, risks). Gong extracts an additional layer: interaction dynamics — talk ratios, per-speaker sentiment, competitor mentions, escalation language, decision-maker engagement. Via Glean, we get the raw Gong transcript but NOT Gong's extracted insights. The transcript extraction prompt can be expanded to extract Gong-caliber interaction dynamics from transcripts we already process, feeding directly into I508's intelligence dimensions.

**Files:** `src-tauri/src/processor/transcript.rs` (expand prompt + parsing), `src-tauri/src/types.rs` (InteractionDynamics struct), `src-tauri/src/db/meetings.rs` (store dynamics), `src-tauri/src/intelligence/prompts.rs` (include dynamics in enrichment context), `src-tauri/src/signals/bus.rs` (new signal types)

**Blocked by:** I508

1. Transcript with clear speaker labels → `InteractionDynamics` populated with talk balance, per-speaker sentiment, engagement signals
2. Competitor mentioned in transcript → `competitor_mentions` populated, `competitor_mentioned` signal emitted, feeds `competitive_context` on next enrichment
3. Escalation language detected → `escalation_signals` populated, `escalation_detected` signal emitted, feeds `risks` with cited evidence
4. Decision-maker silent → `decision_maker_active: "no"` in engagement signals, `stakeholder_disengagement` signal emitted
5. Transcript without speaker labels → dynamics that require attribution return "unclear" (no hallucination)
6. Existing transcript extraction (summary, discussion, actions, wins, risks, decisions) unchanged — no regression
7. No new AI call — dynamics extracted in the same `process_transcript()` prompt
8. Structured dynamics data included in enrichment context (replacing or augmenting raw transcript text)

### End-to-end smoke test (mandatory before tagging)

1. Enrich a sparse account (1 meeting, no email). Health score is non-null with low confidence band (0.2-0.4). All 6 dimensions show scores (populated ones with evidence, empty ones at neutral 50).

2. Enrich account with 3+ stakeholders. `SELECT * FROM person_relationships WHERE source = 'ai_enrichment'` has rows. Person detail shows "suggested" relationships with source label.

3. With Glean connected, enrich account. New contacts discovered from Glean appear in people table. entity_people rows with `data_source = 'glean'` exist. People show Glean-sourced title/department. Manager relationships created with source `"glean"`. `signal_events` has `source = 'glean'` rows.

# v0.15.1 Version Brief

**Thesis:** The reports in v0.15.0 become meaningfully better when entity intelligence is richer. v0.15.1 closes two gaps that v0.15.0 exposes: the first is Glean — VIP's organisational knowledge graph is now accessible and it's exactly the enrichment source DailyOS is missing for enterprise accounts. The second is the relationship map: a CSM's success plan requires understanding the stakeholder network, not just individual contacts. Renewal Readiness, Stakeholder Map, and Success Plan close out the CS report suite. Coaching Patterns emerges from the first three months of Monthly Wrapped data. By end of v0.15.1, the intelligence layer has both personal depth (everything the user knows) and organisational breadth (everything VIP knows), and the report suite covers the full CS workflow.

**Status:** Planning.

**Prerequisite:** v0.15.0 shipped. Glean API access confirmed (VIP rollout March 2026). 3+ months of Monthly Wrapped history for Coaching Patterns.

### I340 — Glean integration

Full criteria in `i340.md`. Summary:

1. Glean connector card in Settings → Connectors. API token stored in keychain. "Test connection" passes against VIP's Glean instance.
2. `search_glean_for_entity(entity_name, entity_domain, limit)` returns Zendesk tickets, Gong summaries, or Salesforce notes from VIP's Glean index for a known account. Permission-filtered (user sees only what they're allowed to see).
3. Entity enrichment prompts include an "Organisational context from Glean" section when results exist. Verify via DEBUG prompt log — real Glean snippets appear. The resulting `entity_intel.executive_assessment` references context that only exists in VIP's Glean index.
4. **The combined intelligence quality gate:** Enrich the same account with Glean OFF, then with Glean ON. The Glean-ON version must include at least one concrete detail (ticket number, Gong call topic, or Salesforce note) that is not derivable from the user's personal signals alone. Generic restatements of account name don't count.
5. `signal_events` has rows with `source = 'glean'` after enrichment cycles that included Glean context.
6. Glean results cached for 4 hours per entity. Graceful failure (timeout, rate limit, unavailable) returns empty list without breaking enrichment.
7. Connector card shows: connected status, last queried timestamp, entity count enriched with Glean context in last 7 days.

### I458 — Renewal Readiness report

1. `generate_report(entity_id, 'renewal_readiness')` produces a report covering: renewal profile (date, ARR, risk rating), health assessment and trend (from `entity_intel.health_trend`), champion/executive alignment (from `entity_intel.relationship_depth` and stakeholder insights), competitive context if available, open items before renewal, and recommended actions with owners.
2. Report is accessible from an account detail page via Reports → "Renewal Readiness." The button only appears for accounts that have a renewal date set (either in account metadata or as a lifecycle event).
3. Renewal risk rating is one of: `low`, `medium`, `high`, `critical` — derived from `health_score`, `health_trend.direction`, and signal patterns. Not hallucinated. Verify: a healthy account with no risk signals gets `low` or `medium`; an account with multiple negative signals and a declining trend gets `high` or `critical`.
4. **User context shapes the "Recommended actions":** With the user's value proposition and current priorities set in `/me`, the recommended actions reference the user's delivery lens. Verify: the recommended actions for an at-risk account differ observably before/after setting a value proposition in the user entity.
5. Export to PDF: clean, under 2MB, no internal DailyOS terminology (no "signal", "entity_intel", "confidence score"). Could be shared with a manager before a renewal QBR.
6. `is_stale` invalidation works when entity intel updates.

### I459 — Stakeholder Map report

1. `generate_report(entity_id, 'stakeholder_map')` produces a report covering: full contact roster for the account (name, title, last interaction date), relationship type for each (champion/sponsor/blocker/technical/economic buyer — using preset vocabulary), engagement level (active/warm/cold/unknown), coverage gaps (buying committee roles with no known contact), and recommended next actions per gap.
2. Relationship types use the active role preset's vocabulary (I442). CS preset: champion, economic buyer, technical evaluator. Sales preset: relevant sales vocabulary. Verify: with CS preset active, relationship types use CS labels; with Sales preset, they shift.
3. **The coverage gap section is non-negotiable:** If the account has no known contact in the "Economic Buyer" role (or equivalent for the active preset), the gap appears explicitly with a suggested action: "No economic buyer identified — consider asking [champion name] for an introduction." Not generic filler. Verify: an account with only technical contacts shows coverage gaps for executive and economic buyer roles.
4. People relationship data (from I390–I392, v0.13.5) is incorporated: if `person_relationships` has edges for contacts at this account, the relationship map reflects them.
5. Export to PDF: visual enough to be usable in a team review or planning session. Under 2MB.

### I460 — Success Plan report

1. `generate_report(entity_id, 'success_plan')` produces a report covering: shared objectives (what the customer is trying to achieve), success metrics (agreed KPIs with targets and current status from `entity_intel.success_metrics`), milestones and timeline, responsibilities (what DailyOS/the vendor delivers, what the customer commits to), and current progress.
2. **This report requires user context to be meaningful.** The "what we deliver" section pulls from the user's `value_proposition` and `success_definition` in the user entity (`/me` page). If these are empty, the section renders a prompt: "Add your value proposition on your profile to personalise this section." The report does not hallucinate generic vendor value propositions.
3. Both sides of the intersection are populated: account goals/metrics (from entity intelligence) AND user delivery context (from user entity). Verify: generate a Success Plan before and after setting value proposition and success definition — the "what we deliver" section is observably different and specific after.
4. Success metrics render as a table: metric name / target / current status / owner. If `entity_intel.success_metrics` is empty, the section shows a prompt to define KPIs — not hidden.
5. Export to PDF: customer-presentable, under 2MB. Could be shared with the account's decision-maker as a mutual success framework.

### I461 — Coaching Patterns (conditional on 3+ months of Wrapped history)

1. `generate_report(user_entity_id, 'coaching_patterns')` is only available when `SELECT COUNT(*) FROM reports WHERE report_type = 'monthly_wrapped' AND entity_type = 'user'` returns ≥ 3. If fewer than 3 Monthly Wrapped reports exist, the report type is not offered in the UI.
2. The report surfaces: patterns in what moved priorities forward vs. what stalled, meeting-to-outcome correlation (meeting types / cadences that correlate with wins), relationship patterns (accounts where engagement increased vs. declined, and what preceded the change), time allocation patterns (where the user's attention has been vs. their declared priorities).
3. **The honest miss is a pattern, not just a single instance.** Unlike the Monthly Wrapped "honest miss" (one item), Coaching Patterns identifies recurring misses: "For the third month in a row, accounts in your quarterly priorities received fewer than 2 meetings per month." Verify by reading `content_json.recurring_patterns` — each item cites multiple Monthly Wrapped reports as evidence.
4. The coaching language is factual and specific, never judgemental: "Accounts you met with 3+ times per month had a 60% higher positive signal rate than accounts you met with once or less." Not: "You should meet with accounts more often." Verify: the report reads as data-backed observation, not unsolicited advice.
5. Export to PDF: personal, reflective in tone. Under 2MB.

### End-to-end smoke test (mandatory before tagging)

1. Connect Glean in Settings → Connectors with VIP API token. "Test connection" passes. Trigger enrichment for a VIP customer account. Open the account detail — the intelligence references something from Glean (a Zendesk ticket topic, a Gong call outcome, or a Salesforce note) that was not in the previous enrichment. `SELECT source FROM signal_events WHERE source = 'glean' ORDER BY created_at DESC LIMIT 1` — returns a row.

2. Open an account with a renewal date set in the next 90 days. Reports → Renewal Readiness. Generation completes. Report shows a risk rating that matches the account's actual health state. Recommended actions reference the user's value proposition (if set on `/me`). Export PDF — under 2MB, professional quality, no internal terminology.

3. Open an account with 3+ contacts. Reports → Stakeholder Map. Coverage gaps section appears for any missing buying committee roles. At least one contact shows a relationship type derived from the people relationship network (I390–I392). Export PDF — structured, readable.

4. Open an account where you've had 2+ meetings and some signals. Navigate to `/me` and confirm value_proposition is set. Reports → Success Plan. "What we deliver" section references the user's value proposition. Success metrics section shows either real KPIs or a clear prompt to define them. Export PDF — customer-presentable.

5. If 3+ Monthly Wrapped reports exist: Reports → Coaching Patterns (from `/me` page). At least one "recurring pattern" cites multiple Monthly Wrapped reports as evidence. Language is observational, not prescriptive.

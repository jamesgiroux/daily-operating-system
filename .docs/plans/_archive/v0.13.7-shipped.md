# v0.13.7 Version Brief

**Thesis:** The current hygiene system finds gaps and fills them. That's necessary but not sufficient. It doesn't know whether the intelligence it produced is *right* — only whether it exists and is recent. The Jefferson Advisors meeting briefing mentioning Adobe Fonts is the canonical example: the intelligence was generated, enriched, and served without anyone or anything noticing that the content had nothing to do with the entity's actual meeting history. The intelligence was confident. It was also wrong. v0.13.7 makes the system aware of the quality of what it produces, not just the presence of it. Four changes: quality scores per entity (continuous, not binary), semantic coherence validation (embedding distance catches topic drift before it reaches the user), an enrichment trigger function (continuous score replacing the hardcoded 14-day threshold), and feedback closure (user corrections propagate back to source reliability). Together, these close the loop from "we enriched it" to "we enriched it and the output is coherent with what the entity is actually about."

**Status:** Planning.

**Research:** See `.docs/research/self-healing-intelligence.md`

**Prerequisite:** v0.13.6 must ship first. IntelligenceService (I402) shipped in v0.13.6 with 6 methods — the ones v0.13.7 needs (`enrich_entity`, `update_intelligence_field`, `update_stakeholders`) are all present. SignalService (I403) was deferred from v0.13.6 — it does not exist. self_healing/ follows the same direct-call pattern as all other services: `bus::emit_signal_and_propagate()` with `&state.signal_engine`. When SignalService eventually ships, self_healing/ migrates along with everything else.

|----|-------|------|
| I406 | Entity quality scoring — Beta distribution quality model per entity; replaces binary has/missing classification | Backend / Intelligence |

- New user-visible features
- Person deduplication via embedding (Jaro-Winkler + embedding cascade) — valuable, but scoped to its own issue post-v0.13.7 since it requires careful calibration
- Jaccard attendee group matching (patterns.rs replacement) — follow-on to I406
- Linfa / ML framework dependency — the four issues above are implementable using existing infrastructure (nomic-embed-text, Thompson Sampling, signal bus); no new ML library needed
- Web search enrichment — Clay already covers this; building in-house duplicates it
- New LLM enrichment sources — signal bus and intel_queue already handle enrichment; this version improves the decision logic, not the sources

## v0.13.6 Service Integration Contract

v0.13.6 shipped IntelligenceService (`services/intelligence.rs`, 242 lines, 6 methods). SignalService (I403) was deferred — no `services/signals.rs` exists. All services call `bus::emit_signal_and_propagate()` directly.

**IntelligenceService** — self_healing/ uses:
- `enrich_entity(entity_id, entity_type, state)` — to trigger re-enrichment when coherence fails or trigger score exceeds threshold (unified method, not separate per-entity-type)
- `update_intelligence_field(entity_id, entity_type, field_path, value, state)` — feedback closure hooks into this (I409)

**Signal emission** — self_healing/ uses the same pattern as accounts.rs, people.rs, etc.:
- `crate::signals::bus::emit_signal_and_propagate(db, &state.signal_engine, ...)` — for `entity_coherence_flagged` signal emission
- I410 hooks into signal arrival by adding a call at the end of `emit_signal_and_propagate` in `bus.rs`, or by having the caller (service layer) invoke `self_healing::scheduler::evaluate_on_signal()` after signal emission

**Rule:** self_healing/ uses IntelligenceService for enrichment operations. Signal emission follows the existing direct-call pattern until SignalService (I403) ships in a future version.

### I406 — Entity quality scoring

1. An `entity_quality` table exists in the DB with columns: `entity_id` (TEXT PK), `entity_type` (TEXT), `quality_alpha` (REAL DEFAULT 1.0), `quality_beta` (REAL DEFAULT 1.0), `quality_score` (REAL, computed: alpha/(alpha+beta)), `last_enrichment_at` (TEXT), `correction_count` (INTEGER DEFAULT 0), `coherence_retry_count` (INTEGER DEFAULT 0), `coherence_window_start` (TEXT), `coherence_blocked` (INTEGER DEFAULT 0), `created_at` (TEXT), `updated_at` (TEXT). Migration exists.
2. Quality score starts at 0.5 (`Beta(1,1)`) for newly-created entities. Verify: create a new account; `SELECT quality_score FROM entity_quality WHERE entity_id = '<id>'` returns 0.5.
3. After a successful enrichment completes (IntelligenceService writes new `entity_intel`): `quality_alpha` increments by 1.0. `quality_score` increases. Verify with a known entity that has not been corrected.
4. After a user edits an AI-generated intelligence field (via IntelligenceService.update_field): `quality_beta` increments by 1.0. `quality_score` decreases. Verify by editing a field on a real account and checking the table.
5. `get_entities_without_intelligence()` and `get_stale_entity_intelligence(14)` are replaced (or augmented) by `get_entities_below_quality_threshold(threshold: f64)`. The hygiene scan uses the quality score to prioritize enrichment — entities with quality < 0.45 are enqueued before entities that are merely stale.
6. The hygiene report (`HygieneReport`) includes a `low_quality_entities` count: entities with `quality_score < 0.45`. The UI hygiene panel surfaces this as a new gap type.
7. Entities with no `entity_quality` row (pre-existing entities not yet scored) default to 0.5 in all quality-aware code paths. No null pointer / missing-row errors. `cargo test` passes.
8. The `entity_quality` table is entity-type-agnostic: accounts, projects, and people all use it. The schema accommodates the user entity coming in v0.14.0 without migration changes.

### I407 — Semantic coherence validation

1. A `coherence_check` function exists in `self_healing/detector.rs` that takes an entity's `executive_assessment` text and the entity's linked meeting corpus (last 90 days of meeting titles + summaries), embeds both using the existing `EmbeddingModel`, and returns a cosine similarity score in [0, 1].
2. The coherence check runs as a **post-enrichment validation step** — triggered when intel_queue completes writing new entity intelligence (wired via I410), NOT in the meeting prep assembly hot path. If cosine similarity < 0.30 (configurable threshold), the intelligence is marked `coherence_flagged = true` in `entity_intel` and the entity is re-enqueued via IntelligenceService at `ContentChange` priority.
3. **The Jefferson Advisors/Adobe Fonts test:** Open the running app. For the Jefferson Advisors meeting (entity: Agentforce project), the coherence check should detect that the intelligence text mentioning "Adobe Fonts" is semantically distant from meeting titles like "Agentforce", "Salesforce", "VIP". Verify: `SELECT coherence_score FROM entity_intel WHERE entity_id = '<agentforce_id>'` returns a score below threshold. After re-enrichment runs, the intelligence no longer mentions Adobe Fonts.
4. Coherence check is skipped for entities with < 2 linked meetings (insufficient reference corpus — not a coherence failure, just sparse data). Verify: an entity with 1 linked meeting does not get `coherence_flagged`.
5. Coherence scores are stored in `entity_intel.coherence_score` (new column). Verify: after running the check for accounts with real meeting data, `SELECT entity_id, coherence_score FROM entity_intel WHERE coherence_score IS NOT NULL` returns rows with scores between 0 and 1.
6. A coherence failure emits a signal (`entity_coherence_flagged` signal type) via `bus::emit_signal_and_propagate()` so downstream consumers (callouts, invalidation) can react. Verify: `SELECT signal_type, entity_id FROM signal_events WHERE signal_type = 'entity_coherence_flagged' ORDER BY created_at DESC LIMIT 5` returns rows after the check runs.
7. The embedding model being unavailable or not yet loaded causes the coherence check to skip gracefully (log a warning, proceed without flagging). No crashes if the model is loading.

### I408 — Enrichment trigger function

1. A `compute_enrichment_trigger_score(entity_id, entity_type, db) -> f64` function exists in `self_healing/remediation.rs`. It returns a continuous score in [0, 1] computed from:
   - `meeting_imminence`: 1.0 if linked meeting within 24h, 0.5 if within 7 days, 0.1 otherwise
   - `staleness`: days_since_enrichment / 14.0 (capped at 1.0 — the 14-day threshold is now the "normal" staleness ceiling, not an absolute gate)
   - `entity_importance`: derived from account tier, meeting frequency, or signal count (normalized to [0,1])
   - `signal_delta`: number of new signals since last enrichment / 10.0 (capped at 1.0)
   - Weighted sum: `(imminence × 0.4) + (staleness × 0.3) + (importance × 0.2) + (signal_delta × 0.1)`
2. `self_healing::evaluate_portfolio()` (called from hygiene Phase 3) uses `compute_enrichment_trigger_score` to prioritize the queue: entities are sorted by trigger score descending before being enqueued via IntelligenceService. Entities with trigger score < 0.25 are not enqueued even if they exist in `get_entities_without_intelligence()` (they're not important enough to spend AI budget on right now).
3. An entity with a meeting in 12 hours and intelligence last enriched 20 days ago with 5 new signals has a higher trigger score than an entity with no upcoming meetings and intelligence 8 days old. Verify by computing the function on known entities in the DB and confirming the ordering makes sense.
4. The `check_upcoming_meeting_readiness()` function is refactored to use `compute_enrichment_trigger_score` instead of its own `PRE_MEETING_STALE_DAYS` constant. The pre-meeting window becomes a special case of the trigger score: when `meeting_imminence = 1.0`, even modestly stale entities get enqueued.
5. The trigger score is logged at DEBUG level for each entity evaluated, allowing manual inspection of why an entity was or was not enqueued. Verify via logs during a hygiene scan run.

### I409 — Feedback closure

1. When IntelligenceService.update_intelligence_field is called (user editing an AI-generated field), in addition to emitting a `user_correction` signal via `bus::emit_signal_and_propagate()`, it calls `self_healing::feedback::record_enrichment_correction(entity_id, entity_type, "intel_queue")`. This: (a) increments `quality_beta` in `entity_quality`, (b) calls `signals::sampling::update_source_reliability("intel_queue", entity_type, false)` to decrement the source's `learned_beta`.
2. When `update_account_field`, `update_person_field`, or `update_project_field` is called (user editing a non-intelligence field that may have been populated by Clay enrichment), and the field's known source is `clay`, call `self_healing::feedback::record_enrichment_correction(entity_id, entity_type, "clay")`. The Clay source's reliability for that entity type decrements.
3. Verify the feedback loop closes end-to-end: edit an intelligence field on a known account. Check: (a) `signal_events` has a new `user_correction` signal; (b) `entity_quality` has `quality_beta` incremented for this entity; (c) the source reliability store has an updated `learned_beta` for `intel_queue`. All three should change in one edit action.
4. A successful enrichment that is NOT subsequently corrected (no user edits within 7 days) results in a `quality_alpha` increment. Verify: identify an entity whose last enrichment was 10+ days ago with no corrections; check `quality_alpha` increased after enrichment vs. before.
5. Verify the system gets better over time: run the app for a week with normal usage. `SELECT entity_id, quality_alpha, quality_beta, quality_score FROM entity_quality ORDER BY quality_score ASC LIMIT 10` — entities the user has corrected most should have lower quality scores, not equal quality scores. The scores must be differentiated, not uniformly reset.

### I410 — Hygiene event-driven triggers

1. When a signal arrives for entity X (any signal via `bus::emit_signal_and_propagate`), the signal handler calls `self_healing::scheduler::evaluate_on_signal(entity_id, entity_type, db)`. This function: (a) updates the entity's trigger score via `compute_enrichment_trigger_score`, (b) if the trigger score exceeds 0.7, enqueues the entity via IntelligenceService at `ContentChange` priority without waiting for the next 4-hour scan.
2. The 4-hour hygiene scan (`run_hygiene_loop`) is unchanged and serves as the catch-all sweep for entities that haven't received signals recently. The event-driven path handles freshly-active entities.
3. The coherence check (I407) fires on intel_queue completion events — when entity intelligence is freshly written, `self_healing::scheduler::on_enrichment_complete(entity_id)` runs the coherence check on the new content. If it fails, re-enqueue via IntelligenceService at `ContentChange`. This is the ONLY path coherence checks run (not in prep assembly, not in the 4-hour scan).
4. **Circuit breaker** prevents thrashing. State tracked on `entity_quality`:
   - `coherence_retry_count` — incremented on each coherence-triggered re-enrichment
   - `coherence_window_start` — ISO8601 timestamp of first retry in current 24h window
   - `coherence_blocked` — set to 1 when count >= 3 within the window
   - **Window logic:** On coherence failure: if `coherence_window_start` is within last 24h, increment `coherence_retry_count`; if not, reset count to 1 and set new window start. If count >= 3, set `coherence_blocked = 1` and do NOT re-enqueue.
   - **Auto-expiry:** When `coherence_window_start` is >72h ago (24h window + 48h cooldown), reset `coherence_blocked = 0` and `coherence_retry_count = 0` on next evaluation. The entity gets one more try with whatever new context has accumulated.
   - **User override:** If the user manually triggers "Refresh Intelligence" (Manual priority), the circuit breaker is bypassed. User intent overrides the safety valve. The coherence retry count resets on manual trigger.
   - Blocked entities surface in the hygiene report as needing review. Verify: `SELECT entity_id, coherence_blocked, coherence_retry_count FROM entity_quality WHERE coherence_blocked = 1`.
5. The event-driven path does not hold the DB Mutex longer than the 4-hour scan path. Signal emission is already O(1) and the quality evaluation is a single DB read + write. Verify via latency logs: signal emission latency does not measurably increase after this change.

## Module Structure

The new logic ships as a dedicated `self_healing/` module, **not** added to `hygiene.rs`. `hygiene.rs` stays as-is for its Phase 1/2 mechanical fixes. The self-healing layer is new code that sits alongside it.

```
src-tauri/src/
  hygiene.rs ............... unchanged — Phase 1/2 mechanical fixes (relationships,
                              summaries, dedup, renewals, etc.) + orchestration loop
  self_healing/
    mod.rs ................. Public API + orchestration (detect → score → remediate)
    quality.rs ............. I406 — Beta distribution quality model, entity_quality table
    detector.rs ............ I407 — Coherence check, gap detection, staleness scoring
    remediation.rs ......... I408 — Enrichment trigger function, queue prioritization
    feedback.rs ............ I409 — Correction → quality score + source reliability
    scheduler.rs ........... I410 — Event-driven triggers, circuit breaker, auto-expiry
```

**Integration surfaces (post-v0.13.6):**
- `self_healing::evaluate_portfolio(db, state)` — called from hygiene Phase 3, replaces `enqueue_ai_enrichments()`. Uses IntelligenceService for enrichment enqueuing.
- `self_healing::scheduler::evaluate_on_signal(entity_id, entity_type, db, state)` — called after `bus::emit_signal_and_propagate` in service call sites (same pattern as prep invalidation)
- `self_healing::scheduler::on_enrichment_complete(entity_id, db, embedding_model, state)` — called from intel_queue after enrichment write
- `self_healing::feedback::record_enrichment_correction(entity_id, entity_type, source, db)` — called from IntelligenceService.update_intelligence_field

self_healing/ uses IntelligenceService for enrichment operations. Signal emission uses `bus::emit_signal_and_propagate()` directly (same pattern as all other services until SignalService ships).

`hygiene.rs:run_hygiene_scan()` calls into `self_healing::evaluate_portfolio()` as its Phase 3 step. The separation is clean: `hygiene.rs` detects structural and mechanical gaps; `self_healing/` makes quality-aware enrichment decisions.

This structure is intentional for v0.13.8: the AppState decomposition will create a `HygieneState` container. That container naturally covers both `hygiene.rs` fields AND the `self_healing/` module fields — the boundary is already drawn.

**Ship criterion addition:** `wc -l src-tauri/src/hygiene.rs` must not have grown from its v0.13.5 value. All new logic is in `self_healing/`. If the line count increased significantly, the module boundary was not respected.

## Design Principle: Intelligence Quality is First-Class

This version establishes a behavioral contract:

**The system should know when its intelligence is wrong, not just when it's missing.**

| State | Current system | v0.13.7 system |
|-------|---------------|----------------|
| No intelligence | Detected (missing_intelligence count) | Detected + quality = 0.5 (uncertain) |
| Stale intelligence (> 14 days) | Detected | Detected + trigger score incorporates staleness with other factors |
| Wrong intelligence (topic drift) | **Not detected** | Detected via coherence check; flagged + re-enriched |
| Improving over corrections | **Not tracked** | Quality score decreases on correction; source reliability updates |
| Getting worse over time | Not tracked | Quality beta accumulates; surfaces in hygiene report |
| Thrashing on bad entity | Not possible (no re-enrichment) | Circuit breaker: 3 failures in 24h → blocked 72h → auto-retry |

## Ship criteria

All acceptance criteria above are met. The `entity_quality` table exists and has rows for all entities (accounts, projects, people). The Jefferson Advisors/Adobe Fonts coherence failure is detectable and auto-remediated: after one enrichment cycle, the intelligence for the Agentforce entity no longer contains topics unrelated to Salesforce/AI. User corrections demonstrably change quality scores — entities that have been corrected have lower quality scores than entities that have not. The trigger function produces differentiated enrichment priority across entities in the portfolio. Source reliability updates are written after corrections. The circuit breaker prevents thrashing on persistently incoherent entities and auto-expires after 72h. Manual "Refresh Intelligence" bypasses the circuit breaker. Enrichment operations route through IntelligenceService. Signal emission follows the same direct `bus::emit_signal_and_propagate()` pattern as all other services. `cargo test` passes. `cargo clippy -- -D warnings` passes. No user-visible regressions.

### End-to-end smoke test (mandatory before tagging)

1. Clean app restart (`pnpm dev`). `SELECT COUNT(*) FROM entity_quality` returns a count matching the number of accounts + projects + people in DB.

2. Open the Jefferson Advisors account (or any account whose intelligence contains content that doesn't match its meeting history). Check: `SELECT coherence_score, coherence_flagged FROM entity_intel WHERE entity_id = '<id>'`. If coherence_flagged = 1, wait one intel_queue cycle. Reopen the account — intelligence should no longer contain the anomalous content.

3. Edit an intelligence field on a known account (e.g., change a stakeholder's role). Check: `SELECT quality_beta, quality_score FROM entity_quality WHERE entity_id = '<id>'` — `quality_beta` should be higher than before the edit, `quality_score` lower.

4. Find an account with a meeting in the next 24 hours. Check trigger scores — this entity should rank near the top of the enrichment queue, not behind entities with only stale-intelligence triggers.

5. Check `signal_events` for `entity_coherence_flagged` signals: `SELECT entity_id, created_at FROM signal_events WHERE signal_type = 'entity_coherence_flagged' ORDER BY created_at DESC LIMIT 5`. At least one row should exist if any entity failed the coherence check during startup.

6. `wc -l src-tauri/src/hygiene.rs` — hygiene.rs should not have grown significantly. The new quality/coherence logic is in `self_healing/`, not crammed into hygiene.rs.

7. Trigger "Refresh Intelligence" on a `coherence_blocked` entity. Verify: enrichment runs despite the block, `coherence_retry_count` resets, and the circuit breaker state clears.

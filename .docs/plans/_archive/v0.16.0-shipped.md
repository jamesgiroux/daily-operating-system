# v0.16.0 Version Brief

**Thesis:** The first minute determines whether someone becomes a user. Before v0.16.0, a new user opens DailyOS and sees an empty shell — no guidance, no demonstration of what's possible, no clear path to getting value. This version makes the first-run experience the product's best argument for itself. Demo data shows what "ready" looks like before the user has connected anything. A guided first-run wizard covers seven decisions: verify Claude Code (required — without it the app cannot build context for anything and is a calendar viewer), connect Google, set your role, set your domain, add your first account, tell DailyOS about your work, and prime the system with its first piece of context. That last step teaches the most important habit: feed it, and it feeds back. Drop a transcript now, connect Quill, link a Google Doc — any of these starts the loop. All user-facing copy follows ADR-0083 product vocabulary throughout (context not intelligence, updates not signals, briefings not prep).

**Status:** Planning.

## Acceptance Criteria

Each criterion must be verified with real data in the running app after a full `pnpm dev` restart. Compilation is not acceptance.

---

### I56 — Educational Redesign (demo data + guided tour)

1. A "Try with demo data" option appears prominently on the empty dashboard state (before Google is connected). Activating it loads a pre-populated workspace with: 3 customer accounts with real-looking intelligence, 4 upcoming meetings with prep content, 5 active actions, 2 recent email signals. Demo data is clearly marked as demo — a persistent "Demo mode" badge in the folio bar.

2. The guided tour activates on first real launch (not demo mode). It consists of 4–6 contextual callouts, one per surface, that explain what each section does in plain language. Callouts are dismissible individually or all-at-once. They do not block interaction — the user can skip ahead at any time.

3. Demo mode is fully exitable. "Connect real data" clears demo data and starts the normal first-run flow. `SELECT count(*) FROM accounts WHERE is_demo = 1` returns 0 after exiting demo mode.

4. The guided tour state persists per session, not per launch. After completing the tour (or dismissing all callouts), it does not reappear on subsequent launches. `SELECT has_completed_tour FROM app_state` returns 1.

---

### I57 — First-Run Wizard (connect, configure, and launch)

1. On first launch with no Google auth and no workspace data, a full-screen first-run wizard appears rather than the empty dashboard. It does not appear on subsequent launches.

2. The wizard has 6 steps, each completable in under 60 seconds:
   - **Step 1 — Claude Code** (cannot be skipped): Verify Claude Code is installed and authenticated via `PtyManager::is_claude_available()` and `PtyManager::is_claude_authenticated()`. If not installed, shows install instructions (claude.ai/download) with a "Check again" button. If not authenticated, shows instructions to run `claude` in terminal. The step is explicit: "DailyOS uses Claude Code to build briefings and insights. Without it, the app cannot find new information about your meetings, accounts, or emails." This is the only step without a Skip option.
   - **Step 2 — Connect Google**: Connect Calendar + Gmail. Skip option available.
   - **Step 3 — Your role**: Role preset selection. Default: Customer Success.
   - **Step 4 — Your domain**: Enter your work email domain(s).
   - **Step 5 — Your first account**: Add one customer account. Optional but encouraged.
   - **Step 6 — About you** (if v0.14.0 is shipped): Prefills the user entity with value proposition and one priority. Skipped if v0.14.0 hasn't shipped.

3. Completing all steps lands the user on the daily briefing with a "Your first briefing is generating" state — not an empty state. If Google is connected, the poller runs immediately. If not, the demo briefing template is shown.

4. Skipping the entire wizard (clicking "Skip setup") lands on the empty dashboard with a persistent "Complete setup" prompt in the Settings area.

5. Each step's data persists immediately on completion — if the user closes the app mid-wizard, progress is retained on next launch.

### I437 — Empty State Redesign

1. Every surface that currently renders a generic empty state has a purpose-specific version. Minimum: Accounts page, People page, Actions page, Email page, Daily briefing.

2. Every empty state includes: a one-sentence explanation of what the surface does, a direct action button that begins filling it ("Add account", "Connect Google", "Process your inbox"), and a brief benefit statement (not just "Nothing here yet").

3. The personality-driven empty state copy uses the active role preset's vocabulary. A CS user sees CS-appropriate language; a Sales user sees sales language.

4. No surface renders `null` or a blank white area when it has no data. Verify by clearing the test DB and navigating every page — each shows a purposeful empty state.

---

### I478 — Remove Feature Toggle Section from Advanced Settings

1. The Advanced Settings panel no longer shows any feature enable/disable toggles. Only AI Models, Hygiene, Capture, Data Management, and Lock Timeout remain.

2. `get_features` and `set_feature_enabled` Tauri commands are removed. `is_feature_enabled()` still works — features gate based on entity_mode + role preset defaults, not user overrides.

3. Existing config files with `features: {}` do not cause errors on load.

## Ship criteria

A brand new user who has never used DailyOS can: see what the app does before connecting anything (demo mode), complete the wizard in under 5 minutes, and see a live briefing within 10 minutes of opening the app for the first time. The app never shows a blank white page.

### End-to-end smoke test (mandatory before tagging)

1. Fresh install, first launch: wizard appears. Skip it — "Complete setup" prompt appears in Settings.

2. Fresh install, complete wizard in full: all 6 steps, Claude Code verified, Google connected, one account added. Lands on daily briefing with "generating" state. Calendar events appear within 5 min.

3. Demo mode: click "Try with demo data." Pre-populated briefing renders with 3 accounts, 4 meetings, 5 actions. "Demo mode" badge visible. Click "Connect real data" — demo data clears, wizard appears.

4. Guided tour: on first real launch (no demo), 4–6 contextual callouts appear. Dismiss them. Relaunch — they do not reappear.

5. All empty states: disconnect Google, clear all entities. Navigate each page — every page shows a purposeful empty state with an action button, never a blank page.

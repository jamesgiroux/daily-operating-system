# v0.15.0 Version Brief

**Thesis:** Reports become meaningful when the system knows both sides of the equation. v0.14.0 gave DailyOS professional context about the user — their value proposition, their definition of success, what they're trying to accomplish. v0.15.0 uses that context to generate two categories of reports that couldn't exist without it. The first category faces outward: EBR/QBR "Value Delivered" now references the user's actual narrative, not generic outcome summaries any vendor could have written. The Account Health Review frames risks and opportunities through the user's product lens. The success plan — the hardest document for a CSM to write — is now possible because both sides of the intersection (what the account wants and what the user delivers) are machine-readable. The second category faces inward: for the first time, DailyOS answers the question no productivity tool answers — "what did I actually accomplish?" The Weekly Impact Report looks back at the last seven days through the user's declared priorities, surfacing concrete evidence of movement (or the honest absence of it). The Monthly Wrapped synthesises the prior month into a celebratory narrative: top wins cited to real events, priority progress stated honestly, and one coaching observation a good manager would make. Together, these two report directions — outward-facing CS intelligence and inward-facing personal impact — make v0.15.0 the version where DailyOS becomes a system that not only helps the user do their work but helps them see what their work amounted to. Reports are entity intelligence and personal signal history, reformatted for an audience, framed by professional context. One AI call per report type per entity or user, cached in DB, invalidated when the underlying intelligence updates.

**Status:** Planning.

## Acceptance Criteria

Each criterion must be verified with real data in the running app after a full `pnpm dev` restart. Compilation is not acceptance. A criterion is met when the described behavior is observable with real accounts in the running app.

---

### I397 — Report infrastructure

1. `reports` table exists in DB: `entity_id`, `entity_type`, `report_type`, `content_json`, `generated_at`, `intel_hash`, `is_stale`. A migration creates it.
2. `generate_report(entity_id, report_type)` Tauri command exists. Calling it for a known account with `report_type = 'swot'` produces a non-empty row in the `reports` table within 2 minutes.
3. `get_report(entity_id, report_type)` returns the cached report without a PTY call. Verify: generate once, call `get_report` twice — the second call is instant (< 100ms).
4. **Intel hash invalidation:** Update an account's `entity_intel`. Within one invalidation cycle, `SELECT is_stale FROM reports WHERE entity_id = '<id>'` returns 1. The "Intelligence updated since this report was generated — regenerate?" banner appears on next report view.
5. `ReportShell` renders correctly for SWOT, Account Health Review, and EBR/QBR. All text fields are editable inline. Edits are NOT persisted to DB — close and reopen: original AI-generated content is shown.
6. **PDF export:** "Export PDF" from any report produces a file with Newsreader serif headers, DM Sans body, design token colors, under 2MB, readable in Preview. No HTML artifacts.
7. **SWOT bundled format:** Four quadrants (Strengths, Weaknesses, Opportunities, Threats), each pre-populated from entity_intel signals. At minimum 2 items per quadrant for an account with 3+ signals. Items reference real signal types or meeting titles — not generic placeholder text.
8. The existing `RiskBriefingPage` is unaffected — risk briefing migration (I397) is separate.

### I398 — Risk report migration

1. `generate_risk_briefing` internally routes through `generate_report(entity_id, 'risk')`. Output is stored in the `reports` table, not in `risk-briefing.json` on disk.
2. After generating: `SELECT content_json FROM reports WHERE entity_id = '<id>' AND report_type = 'risk'` returns a non-empty row. No `risk-briefing.json` is written to disk.
3. `get_risk_briefing` reads from the `reports` table. Verify: delete any existing `risk-briefing.json`; call `get_risk_briefing`; report loads correctly from DB.
4. When entity intel updates for the account, `is_stale = 1` is set for the risk report row.
5. `RiskBriefingPage` renders all 6 slides correctly with the migrated data source.
6. Fallback: if `reports` table has no row and a `risk-briefing.json` exists on disk, read from disk (migration grace period for existing briefings).
7. `save_risk_briefing` (user slide edits) persists to `reports.content_json`, not to disk.

### I398 — Account Health Review

1. `generate_report(entity_id, 'account_health')` produces a report with the following sections: Health Summary, Health Score & Trend, Key Risks, Stakeholder Coverage, Engagement Cadence, Open Commitments. Renewal Outlook section appears only if renewal date is set.
2. The report is accessible from the account detail page via a "Reports" button or tab — not buried in the folio bar.
3. All section content references real account data: actual risk descriptions, real stakeholder names from `stakeholder_insights`, real commitment descriptions from `open_commitments`. Verify by reading `content_json` and checking that content is account-specific.
4. For sparse accounts (1 signal, 1 meeting): report renders with reduced content and a "Limited intelligence — some sections may be incomplete" notice. No hallucinated content.
5. Export to PDF: all sections visible, under 2MB, no internal DailyOS terminology visible to a reader (no "signal", "entity_intel", "confidence score" in the PDF text).
6. `is_stale` invalidation works: update entity intel, open the report, see the "Intelligence updated" banner.

### I399 — EBR/QBR

1. `generate_report(entity_id, 'ebr_qbr')` produces a report with all 8 sections: Partnership Overview, Goals Recap, Value Delivered, Success Metrics, Challenges & Resolutions, Strategic Roadmap, Customer Asks, Next Period Priorities.
2. **Value Delivered quality gate (non-negotiable):** The Value Delivered section references at least one specific, real event from the account's history — a completed commitment with its source meeting ID or signal type. Generic filler text ("the team has worked closely together") is a failure criterion. Verify by reading `content_json.value_delivered` — each item must have a `source` field pointing to a real meeting ID or signal ID.
3. **Success Metrics:** Renders as a table (metric / target / current / status). If `entity_intel.success_metrics` is empty, the section shows a placeholder row prompting the CSM to add KPIs — it does NOT hide the section.
4. The report is flagged as customer-facing in `ReportShell`. The PDF export is customer-presentable: no internal confidence scores, no signal debug data, no DailyOS system terminology in body text.
5. All 8 sections are individually editable inline. The CSM workflow is review → edit → export.
6. Export to PDF: clean, customer-presentable, under 2MB. Could be sent to a customer executive without embarrassment.
7. A progress indicator shows during the PTY generation (estimated 60–120s).
8. `is_stale` invalidation works — "Intelligence updated — regenerate to include latest signals" banner appears when entity intel has updated since last generation.

### I418 — Weekly Impact Report

1. `generate_report(user_entity_id, 'weekly_impact')` produces a report covering the last 7 calendar days. The `reports` table stores it under `entity_type = 'user'` and `entity_type_id = 1`. `report_type = 'weekly_impact'`.
2. The report auto-generates every Monday. A scheduler entry (or hygiene scan trigger) checks: does a `weekly_impact` report exist for the current week? If not, enqueues generation. Verify: on Monday, `SELECT generated_at FROM reports WHERE report_type = 'weekly_impact' ORDER BY generated_at DESC LIMIT 1` — timestamp is from this week.
3. **Priorities moved is the quality gate:** At least one item in the Priorities section must reference a real signal, meeting ID, or commitment that occurred in the reporting period AND is linked to a declared priority. Generic statements ("you worked on Cox this week") are a failure criterion. Verify by reading `content_json.priorities_moved[0]` — it must contain a `source` field pointing to a real event ID.
4. The report renders on the `/me` page under a "My Impact" section — a new chapter below the declared context sections. It shows the most recent weekly report with a "Last week" label and a "View previous" affordance for older reports.
5. All text sections are editable inline (draft state only — not persisted). A "Share as PDF" export produces a clean single-page personal summary, under 1MB. No external DailyOS terminology in the export (no "signal_events", "entity_intel").
6. If the user has no declared priorities (both `annual_priorities` and `quarterly_priorities` are empty), the Priorities moved section shows a prompt: "Add priorities on your profile to see how your week connected to what matters." It does not hallucinate priority alignment.
7. Invalidation: the weekly report is invalidated (`is_stale = 1`) if new signals arrive for entities linked to the user's priorities after generation. This is rare but ensures the report reflects the full week if late-arriving data lands before Monday's next generation.

### I419 — Monthly Wrapped

1. `generate_report(user_entity_id, 'monthly_wrapped')` produces a report covering the calendar month immediately prior to generation. `report_type = 'monthly_wrapped'`. Stored in `reports` table under `entity_type = 'user'`.
2. The report auto-generates on the 1st of each month covering the prior month. Verify: on February 1st, `SELECT report_type, json_extract(content_json, '$.period') FROM reports WHERE report_type = 'monthly_wrapped' ORDER BY generated_at DESC LIMIT 1` — period is "January 2026" (or the prior month).
3. **Top wins quality gate:** Each item in `top_wins` must have a `source` field referencing a real meeting ID or signal ID from the reporting month AND must relate to a declared annual or quarterly priority where possible. At least one win must be priority-linked if any priorities were declared. Generic wins without source citations are a failure criterion.
4. **Priority progress is honest:** If a quarterly priority had zero meeting or signal activity during the month, it appears in the "no activity" state — it is NOT omitted to make the report look better. Verify: add a quarterly priority, generate a report for a month with no activity on that priority — the priority appears in the "no activity" group.
5. **The honest miss is present and specific:** The report always includes exactly one honest miss (if any priority had no activity — otherwise the section is omitted). The miss names the specific priority/account/commitment. It is never generic ("you could have done more"). Verify by reading `content_json.honest_miss` — it contains a priority text and account name where relevant.
6. The report renders on the `/me` page in the "My Impact" section, below the most recent Weekly report. Labelled by month ("January 2026"). Up to 12 months of reports are listed; older ones are accessible via archive.
7. **PDF export is polished and personal.** The Monthly Wrapped PDF is designed for personal reflection and potential sharing — not the clinical look of the CS reports. It uses the editorial design system but with a warmer tone: the headline stat is prominent, wins are featured, the honest miss is visually distinct but not harsh. File under 2MB. Verify: export a Wrapped PDF and confirm it reads like a personal achievement summary, not a business intelligence report.
8. The report includes a comparison where history exists: "This month vs. last month: +3 meetings, -2 open commitments, +1 account health improvement." Only shown if a prior month's report exists.
9. `cargo test` passes. Generation handles gracefully when user entity has no declared priorities — still produces a factual summary from activity data with a note that priorities would personalise the framing.

**Recommended build order:**
1. I397 (report infrastructure + SWOT) — platform that all report types use; I396 fields already exist from v0.14.0
2. I398 (risk report migration) — validate infrastructure with existing known-good report type
3. I399 (Account Health Review) — first new report type, internal audience (lower stakes)
4. I418 (Weekly Impact Report) — shift to personal reports while I400 design settles; simpler (user data only, no customer-facing quality bar); validates personal report pattern
5. I400 (EBR/QBR) — flagship, customer-facing; benefits from lessons learned in I399 and I418; hardest deliverable gets the most runway
6. I419 (Monthly Wrapped) — larger narrative format, builds on I418's personal report framing
7. I413 (document attachment) — P2, enriches report generation context

## Ship criteria

All acceptance criteria above are met. A real account with 3+ signals renders an Account Health Review that references the user's actual product lens — not generic analysis. The same account's EBR/QBR "Value Delivered" section cites specific events from the account's history, framed through the user's value proposition set in v0.14.0. The EBR exports to PDF that could be sent to a customer executive without editing the AI framing of what success looks like. The Risk Report reads from `entity_intel` DB, not disk. A Monthly Wrapped for the prior calendar month renders with at least one top win citing a real event ID and a priority progress section that honestly shows priorities with no activity in their "no activity" state rather than omitting them. `cargo test` passes. `cargo clippy -- -D warnings` passes.

### End-to-end smoke test (mandatory before tagging)

1. Prerequisite: v0.14.0 user entity is populated. Navigate to `/me` — value proposition, success definition, and at least one strategic priority are set. These were set in v0.14.0 and should still be present.

2. Open an account with 3+ signals and 2+ meetings that is linked to one of the user's strategic priorities. Trigger entity intelligence enrichment. Read the resulting `executive_assessment` — it should reference the user's product context, not only the account's internal situation. Verify the user context block was included in the prompt (DEBUG log).

3. From that account's detail page, click "Reports" → "SWOT". Generation completes (< 2 min). Strengths and Opportunities quadrants should reference the user's value proposition where relevant. Items reference the account's actual signal history — not placeholder text.

4. Click "Reports" → "Account Health Review". Generation completes. Every section has real content. Edit the Health Summary field. Export PDF. Open in Preview — all sections readable, no HTML artifacts, under 2MB. Close and reopen — edit is gone (draft only).

5. Click "Reports" → "EBR/QBR". Generation completes. `SELECT json_extract(content_json, '$.value_delivered[0].source') FROM reports WHERE report_type = 'ebr_qbr' AND entity_id = '<id>'` — returns a real meeting ID or signal ID. Export PDF — customer-presentable, references the user's success definition in framing what was delivered.

6. Generate a risk briefing on a known at-risk account. `SELECT id FROM reports WHERE report_type = 'risk' AND entity_id = '<id>'` returns a row. No `risk-briefing.json` written to disk. `RiskBriefingPage` loads all 6 slides.

7. Update entity intelligence. `SELECT is_stale FROM reports WHERE entity_id = '<id>'` returns 1 for all report types. Open any report — "Intelligence updated" banner visible.

8. Drop a product deck PDF onto the user entity page attachments section. Wait for file processor. `SELECT * FROM content_files WHERE source = 'user_context'` — returns a row for the PDF. Enrich an account related to the product deck topic — the assembled enrichment prompt (DEBUG) includes a relevant passage from the document.

9. Navigate to `/me` → "My Impact" section — the most recent Weekly Impact Report and Monthly Wrapped are both visible. The Weekly report's "Priorities moved" section references a real signal or meeting ID (read `content_json.priorities_moved[0].source` — it is a non-null, non-empty string corresponding to a real event in the DB). The Monthly Wrapped's "Priority progress" section shows at least one priority in the "no activity" state if any declared priority had no signals or meetings in the prior month.

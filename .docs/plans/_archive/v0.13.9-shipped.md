# v0.13.9 Version Brief

**Thesis:** Connectors are the raw material for intelligence. Every external data source — transcripts, enrichment, project management — should produce signals that flow through the signal bus into entity intelligence. Most of DailyOS's connectors don't. Gravatar stores avatars in files and never writes back to the people table. Linear syncs issues to the DB and then stops — nothing reads them, nothing surfaces them. Granola holds the DB mutex across an AI pipeline call (the exact bug Quill fixed months ago). Clay has never run in production. This version hardens every existing connector, wires Linear into the signal system, and renames the surface from "Connections" to "Connectors" — because that's what they are: the inbound connections that make intelligence possible.

**Status:** In progress. I421, I424, I425 done. I422 (Clay) done — Smithery transport, unified enrichment pipeline, background processor draining queue end-to-end. I423 (Gravatar) wired through unified pipeline, untested with real data (requires Gravatar API key).

### I422 — Clay connector — SSE transport + OAuth flow

**Context:** Clay API keys are beta-locked (not publicly available). The stdio path (`npx @clayhq/clay-mcp` + `CLAY_API_KEY`) is a dead end. The correct path is SSE to `https://mcp.clay.earth/mcp` with a Bearer token via Clay OAuth — the same auth path used when Clay MCP tools work in Claude Code. `rmcp 0.1.5` lacks `transport-sse`; implement via raw `reqwest` + JSON-RPC-over-SSE (already documented as the intended approach in `client.rs` lines 171–174). Full criteria in `i422.md`.

1. `reqwest` in `Cargo.toml` gains `stream` feature. `cargo build` passes clean.
2. `ClayClient::connect_sse(token)` exists and connects to `https://mcp.clay.earth/mcp` via reqwest streaming + JSON-RPC. `ClayClient::connect(token)` calls SSE first; stdio is retained as an unreachable fallback.
3. Clay Settings card replaces the API key input with a "Connect Clay" OAuth flow (web browser → Clay login → token stored in keychain). Clay OAuth is confirmed available. Follows the same PKCE pattern as the existing Google OAuth flow.
4. After authenticating: `SELECT count(*) FROM clay_sync_state WHERE state = 'completed'` > 0; `SELECT count(*) FROM signal_events WHERE source = 'clay'` > 0.
5. Legacy `enrich_person_from_clay` (plain `emit_signal`) removed. All enrichment uses `enrich_person_from_clay_with_client` (`emit_signal_and_propagate`). Clay signals propagate to linked accounts.

The fix mirrors Quill's approach: acquire the mutex to read required data, drop it, run the pipeline without the lock. 3. `state.granola_poller_wake` (an `Arc<Notify>` or equivalent) is added to `AppState` and wired into `lib.rs` alongside the existing `quill_poller_wake`. Verify: `grep -n "granola_poller_wake" src-tauri/src/state.rs src-tauri/src/lib.rs src-tauri/src/google.rs` — appears in all three files.
4. `cargo test` passes. No regressions on existing Quill functionality. The fix is isolated to the Granola code path.

### I425 — Linear signal wiring

1. **Signal emission on sync:** When Linear issues are synced via `upsert_issues`, signals are emitted for state changes. Signal types:
   - `linear_issue_blocked` — when an issue's `state_type` changes to `started` or `inProgress` with `state_name` containing "blocked"
   - `linear_issue_completed` — when `state_type` changes to `completed`
   - `linear_issue_overdue` — when `due_date` is in the past and `state_type` is not `completed`
   Verify: `SELECT signal_type, source FROM signal_events WHERE source = 'linear' ORDER BY created_at DESC LIMIT 10` — returns rows after a Linear sync with active issues.
2. **Entity linking:** When Linear issues or projects are linked to DailyOS entities (accounts or projects), the signal carries the entity context. A new `linear_entity_links` table (or a `entity_id`/`entity_type` column on `linear_issues`) maps Linear issues to DailyOS entities by project name matching and user-confirmed links.
3. **Meeting prep enrichment:** When assembling meeting prep for a meeting linked to a project entity that has associated Linear issues, the prep includes a "Related work" section showing: blocked issues, overdue issues, issues completed since the last meeting. Verify by opening a meeting detail page for a meeting linked to a project that has Linear issues — "Related work" appears with real issue data.
4. **Briefing surface:** In the daily briefing attention section, a meeting with project-linked Linear issues that are blocked or overdue shows a signal callout. Verify: `SELECT count(*) FROM signal_events WHERE source = 'linear'` is non-zero after configuring Linear with a real API key.
5. Linear Settings card (`LinearConnection.tsx`) shows: connected status, issue count, project count, "last synced" timestamp, and a list of the most recently synced issues. Currently it shows counts but no list — add the issue list (last 5 issues, truncated with a "View all" action if applicable).

Old `clay/oauth.rs` stores Smithery API key in keychain.

2. **Unified enrichment pipeline** — Replaced separate Clay poller + Gravatar fetcher with:
   - `db/people.rs: update_person_profile()` — single write function for all sources, handles source priority, provenance tracking, enrichment_log audit

3. **Frontend** — Avatar accepts `photoUrl` prop, PersonHero renders bio/phone/social tooltips, searchable entity link picker for Linear.

### Clay enrichment — the blocking issue

**Symptom:** The unified enrichment processor spawns (confirmed by running binary directly — `eprintln` output verified) but `npx tauri dev` swallows stderr so no logs are visible. The DB queue (`clay_sync_state`) drains extremely slowly — most contacts fail.

**Root cause chain (investigated in order):**

1. **`map_person_row` hardcoded enrichment fields to None** — Fixed. All 11 person SELECT queries now include enrichment columns.

2. **Smithery requires `Accept: application/json, text/event-stream` header** — Fixed. Without it, returns 406.

3. **Smithery `searchContacts` returns `{"total":N,"results":[...]}` wrapper** — Fixed. Parser now handles the wrapper format.

4. **Smithery `getContact` returns completely different schema** — Fixed. `displayName` not `name`, `work_history[]` not `titleHistory`, `social_links[]` not `linkedinURL`, `emails[]` not `email`. Manual JSON parsing in `get_contact_detail`.

5. **Smithery `searchContacts` uses different tool name** — Fixed. `searchContacts` not `search_contacts`, `getContact` not `get_contact`.

6. **Contact ID type mismatch** — Smithery returns `id` as a JSON number (215631727), but `ClayContact.id` was `String`. Serde's `#[serde(default)]` + `#[serde(deserialize_with)]` interaction causes silent fallback to `""` when deserialization fails. **Attempted fix:** changed `id` from `String` to `serde_json::Value` with `id_str()` helper. Status: fix is in source but unverified in running app.

7. **Contact matching returns wrong person** — keyword search for `andrea.grassi@automattic.com` returns YOUR profile first (matches `automattic`). Fixed: use `name` filter for non-email queries, match on `displayName` against email.

8. **`npx tauri dev` swallows stderr** — `eprintln!` confirmed working when binary is run directly but invisible via `npx tauri dev`. No logger backend (`env_logger`, `tauri-plugin-log`, etc.) is configured — all `log::info!` calls are no-ops. This made debugging extremely difficult throughout the session.

9. **Stale binary problem** — `cargo clippy` compiles the binary, then `git commit` happens after, leaving the binary older than the commit. `npx tauri dev` runs `cargo run --no-default-features` which may or may not trigger recompilation depending on cargo's freshness check. Multiple rounds of "restart and check" were wasted because the running binary didn't have the latest fixes.

### What needs to happen next

1. **Add a logger backend** — without this, debugging background tasks is blind. `env_logger` or `tauri-plugin-log` so `log::info!` actually outputs somewhere visible.

2. **Verify the `serde_json::Value` id fix** — the latest build has `ClayContact.id` as `Value` instead of `String`. Need to confirm this fixes the empty-id deserialization issue by running the binary and checking the DB.

3. **Build workflow fix** — always build with `--no-default-features` to match what `npx tauri dev` uses. Or add a `just` / `Makefile` command that does the right thing.

4. **Gravatar end-to-end test** — once the processor is draining Clay, verify Gravatar also processes through the unified path.

5. **PersonHero bio rendering** — fixed (shows below lede), needs visual review.

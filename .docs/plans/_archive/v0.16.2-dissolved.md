# v0.16.2 Version Brief

**Thesis:** The design system says: type does the structural work. Borders, containers, and dividers are secondary. When that principle is violated — when structure comes from hardcoded inline styles instead of the type scale and the margin grid — pages feel like database output rather than curated editorial documents. This version closes that gap across every surface. The app already has the editorial vocabulary: ChapterHeading, FinisMarker, margin grid, section rules, editorial-reveal, EditorialEmpty, EditorialLoading, AtmosphereLayer. The work is making sure every page actually uses them. No new patterns. No new components. Just consistent application of the system we already built.

**Status:** Planning.

**The editorial test (four questions for every page):**
1. Does it open with a conclusion or a chrome shell?
2. Does content use the margin grid and section rules, or flex columns of divs?
3. Do sections breathe at 48–64px, or collapse at 24–32px?
4. Does it end with a FinisMarker, or just stop?

### I448 — ActionsPage editorial rebuild

**Context:** 42 inline `style={{}}` occurrences, no CSS module, max-width 900px (not 1100px), group headers are raw divs. The most-visited secondary page reads entirely as database output.

1. A CSS module `ActionsPage.module.css` is created. All structural styling moves from `style={{}}` to module classes. `grep -c "style={{" src/pages/ActionsPage.tsx` — returns ≤ 5 (only genuinely dynamic values like computed widths or animation values, each justified with a comment).

2. Page container uses `var(--page-max-width)` and `var(--page-padding-horizontal)`. The Actions page renders at 1100px maximum width — consistent with Account and Project detail pages.

3. Group headers in `PendingGroupedView` ("Acme QBR · Thursday") use `<ChapterHeading>` component, not inline-styled divs. Each meeting group is introduced by an editorial header, not a raw `<div>`.

4. Time-band headers ("Overdue," "This Week," "Later") are rendered as editorial section labels using the mono label pattern (`s.marginLabel` class, 11px mono, uppercase, letter-spacing). Not inline-styled divs.

5. The page uses the margin grid (`s.marginGrid`, `s.marginLabel`, `s.marginContent`) for the main pending view. Action items appear as styled text rows in the content column, not as a flat list.

6. `<EditorialLoading>` replaces the inline-styled loading skeleton. `<EditorialError>` replaces the inline-styled error div.

7. `<FinisMarker />` appears unconditionally at the bottom of every tab view — including when there are no actions. Verify: clear all actions, navigate to Actions page — FinisMarker renders.

8. The hero heading "Actions" uses the correct type scale: 76px Newsreader for the page title (matching the Daily Briefing's hero heading scale), not `fontSize: 36` (which is between scale levels and creates visual ambiguity).

**Context:** Both AccountDetailEditorial.tsx and ProjectDetailEditorial.tsx have ~280-line inline-styled Portfolio chapters. The hotspot lists read as JSON output. They violate "conclusions before evidence."

3. Hotspot items use an existing signal prose component pattern, not raw divs. Each hotspot shows account name (Newsreader 19px, turmeric tint), one-line reason (DM Sans 14px, secondary), and a link to the child account detail page.

5. The "Business Units" child account list is preceded by a mono label "PORTFOLIO" or "BUSINESS UNITS" in the margin label position — not an inline-styled section title.

6. The `var(--color-garden-eucalyptus)` reference in the folio bar (from I447 criterion 2) is audited and either resolved (now that the token is defined) or replaced with the appropriate token for that specific button's intent.

### I483 — Theme infrastructure + shipped presets

**Context:** Every style value flows through CSS custom properties after I447. That token discipline is the prerequisite for theming — swapping a palette means overriding the same custom properties under a different selector. This issue builds the layering, ships three themes, adds typography controls, and documents the path for custom themes.

**1. Token layering architecture:**

The current `:root` block in `design-tokens.css` is restructured into two layers:
- **Semantic layer** (`:root`): Contains all semantic tokens (`--color-text-primary`, `--color-entity-account`, `--font-serif`, `--space-md`, etc.) that map to palette tokens. Components reference only semantic tokens. This layer does NOT change between themes.
- **Palette layer** (`[data-theme="warm"]`, `[data-theme="dark"]`, `[data-theme="cool"]`): Contains the raw color values, font stacks, and base sizing that semantic tokens resolve to. The `warm` palette matches the current production values exactly — zero visual change on upgrade.

`:root` sets `data-theme="warm"` as the default. If no `data-theme` attribute is present, the warm palette applies. Verify: remove the `data-theme` attribute from `<html>` — the app looks identical to the current production build.

**2. Three shipped themes:**

Each theme provides a complete override of the palette layer. Every palette token must be defined in every theme — no falling through to `:root` defaults.

- **Warm** (default): Current production palette. Cream/linen backgrounds, charcoal text, turmeric/sage/larkspur accents. Newsreader serif, DM Sans body. This is the editorial magazine aesthetic.
- **Dark**: Low-light palette. Dark charcoal/slate backgrounds (`#1a1d23` ground, `#242830` surface), light cream/warm-white text, desaturated accent colors (turmeric dims to `#b8922a`, sage to `#6b9468`, larkspur to `#7b91b0`). Same font stacks. Rule colors invert to light-on-dark (`rgba(245,242,239,0.08)` and `rgba(245,242,239,0.12)`). Frosted glass and backdrop effects adapt.
- **Cool**: Blue-grey palette. Cool white backgrounds (`#f0f2f5` ground, `#f7f8fa` surface), slate-blue text (`#2c3345`), cool-shifted accents (turmeric shifts to steel blue `#5a7ba8`, sage stays, terracotta shifts to dusty rose `#a06b7a`). Slightly lower contrast for a calmer reading experience.

Verify each theme: navigate every page listed in the end-to-end visual check table. Text is readable (WCAG AA contrast minimum on all text-to-background pairs). Accent colors are distinguishable from each other. Entity colors remain semantically distinct (accounts, projects, people each have a visually different color in every theme).

**3. Typography controls:**

Two user-adjustable settings stored in `user_preferences`:

- **Font size scale**: A base multiplier (`--font-size-scale`) applied to all typographic tokens. Three presets: Compact (0.875×), Default (1×), Large (1.125×). The multiplier is applied via `calc()` on the type scale — e.g., a 14px body becomes `calc(14px * var(--font-size-scale))`. Heading hierarchy is preserved at all scales.
- **Font family override**: Users can swap the serif and sans-serif font stacks independently. Two dropdowns in Settings: "Headline font" (default: Newsreader) with options for system serif, Georgia, or "Custom…" (user provides font name). "Body font" (default: DM Sans) with options for system sans-serif, Inter, or "Custom…". The mono font is not user-overridable (it's a data display font, not a preference). Custom font entry: a text input where the user types a font name — the app sets `--font-serif-override` or `--font-sans-override` and the font stack resolves via `var(--font-serif-override, 'Newsreader'), Georgia, serif`.

Verify: set font size to Large, navigate to Daily Briefing — headlines and body scale proportionally. Set headline font to Georgia — all `--font-serif` usage renders in Georgia. Set body font to Inter — all `--font-sans` usage renders in Inter.

**4. Settings UI — Appearance section:**

A new "Appearance" section in Settings, positioned before the existing Connectors sections. Contains:

- **Theme picker**: Three cards showing a thumbnail preview of each theme (a small rendered sample of the color palette — not a screenshot, just colored rectangles showing background/text/accent). The active theme has a subtle border highlight. Clicking a card applies the theme instantly (no save button).
- **Text size**: Three radio buttons (Compact / Default / Large) with a live preview line of sample text that scales as the user selects.
- **Font overrides**: Two dropdowns (Headline font, Body font) with the options listed above. "Custom…" reveals a text input.
- **Reset to defaults**: A single text button that restores Warm theme, Default text size, and default fonts.

The section uses the editorial treatment from I452 (section rules, no inline styles, CSS module). All labels use product vocabulary — "Appearance", "Theme", "Text size", "Fonts" — no system terms.

Verify: open Settings, scroll to Appearance. Select Dark theme — the entire app repaints immediately. Select Cool — repaints again. Select Compact text size — text shrinks across all pages. Reset to defaults — returns to Warm, Default, Newsreader/DM Sans.

**5. Preference persistence:**

Theme choice, font size scale, and font overrides are stored in the SQLite `user_preferences` table (existing table, new keys: `theme`, `font_size_scale`, `font_serif_override`, `font_sans_override`). On app launch, the preferences are loaded and applied to the `<html>` element before first paint (via the Tauri `setup` hook or an early React effect) to prevent a flash of wrong theme.

Verify: set Dark theme, quit and relaunch the app — Dark theme loads immediately without a flash of Warm.

**6. Custom theme documentation:**

A user-facing doc (in-app or linked from Settings) explains:
- What a theme is (a set of CSS custom property overrides)
- How to create a custom theme CSS file (copy the Warm palette block, change values)
- Where to place the file (`~/.dailyos/themes/my-theme.css`)
- That the app will detect and list custom theme files in the theme picker

The custom theme loader reads `.css` files from `~/.dailyos/themes/`, parses them for `[data-theme="..."]` blocks, and adds them to the theme picker dynamically. Invalid files (missing required tokens) are ignored with a console warning.

Verify: create a file `~/.dailyos/themes/midnight.css` with a `[data-theme="midnight"]` block containing all palette tokens. Relaunch the app. "Midnight" appears as a fourth option in the theme picker. Select it — the app renders with the custom palette.

# v0.14.1 Version Brief

**Thesis:** DailyOS knows everything about the world the user operates in — accounts, people, projects, signals, meetings. It knows almost nothing about the user themselves. This version changes that. The user becomes a first-class entity with a dedicated professional context surface: their own page, outside Settings, with the same editorial treatment as account and person detail pages. They can declare their value proposition, define what success looks like for their customers, set annual priorities (year-level bets) and quarterly priorities (current quarter focus) linked to accounts and projects, write professional knowledge as embedded context entries, and attach documents the system uses as enrichment context. With this in place, entity intelligence can do something it couldn't before: frame analysis through the user's actual professional lens. A cost-pressure signal at an account becomes an opportunity when the system knows the user has a cost-reduction story. An at-risk stakeholder is prioritised differently when the system knows that stakeholder is blocking a quarterly priority the user declared. Neither priority layer expires automatically — set and forget until ready to update. CS-first implementation in v0.14.0. All role presets expanded holistically in v0.14.1. (ADR-0089, ADR-0090)

**Status:** Planning.

### Explicitly out

- Role preset expansion beyond CS-first — v0.14.1. The data model and page structure are preset-agnostic. In v0.14.0, only the CS vocabulary, placeholders, and playbook labels are fully implemented. Other presets see the generic structure.
- User entity intelligence generation (the AI enriching the user entity itself) — the user entity is authored, not generated. The system consumes it, it does not produce it.
- Sharing user entity context externally (MCP, integrations) — user entity is local and private. Never exposed.
- Success Plan report type — requires user context (this version) + report infrastructure (v0.15.0). Candidate for v0.15.1.

### I350 — In-app notifications

The auto-updater (`tauri-plugin-updater`) is already fully wired: config in `tauri.conf.json`, capabilities granted, `UpdateSection` component in Settings, GitHub Releases endpoint configured. This issue adds proactive notification and system status surfaces.

1. **Background update check on launch:** App silently calls `check()` via `tauri-plugin-updater` during startup. Result stored in React context. No user action required. Verify: start app with a newer version published to GitHub Releases → update state is available without navigating to Settings.
2. **Update-available banner:** When a newer version is detected, a persistent but dismissible banner appears (editorial toast or top-of-page strip): "DailyOS vX.Y.Z available — [What's New] [Install & Restart]." "What's New" opens a modal with release notes (from `update.body`). "Install & Restart" triggers `downloadAndInstall()` + `relaunch()`. Verify: banner appears on next launch after a new release is published. Clicking "Install & Restart" installs and relaunches.
3. **"What's New" modal:** Renders release notes as editorial markdown. Dismisses on close. After install+restart at new version, the modal auto-shows once with the notes for the version just installed. Tracks `last_seen_version` in SQLite — appears exactly once per version. Verify: dismiss, restart, confirm it does not reappear for the same version.
4. **System status toasts:** If email enrichment pipeline fails, a toast fires: "Email enrichment paused: [reason]" with Retry action. Not silent. Verify: simulate a pipeline failure → toast appears within 10 seconds.
5. **OAuth expiry notification:** If Google OAuth expires (401 response), a macOS Notification Center alert fires: "DailyOS — Action Required: Reconnect your Google account in Settings." Requires `tauri-plugin-notification` with proper entitlements. Verify: revoke OAuth token → native notification appears.
6. **No third-party notification SaaS:** `grep -r "beamer\|intercom\|pendo\|appcues\|segment" src-tauri/ src/` returns 0.
7. **Existing UpdateSection in Settings preserved:** The manual "Check for Updates" flow in Settings continues to work alongside the new proactive check.

### I411 — User entity data model

1. `user_entity` table exists with all columns per the issue spec. Single row enforced. Migration creates it.
2. **Identity field migration:** Migration copies `user_name`, `user_company`, `user_title`, `user_focus` from `workspace_config` into corresponding columns on `user_entity` (`name`, `company`, `title`, `focus`). After migration, these four columns on `workspace_config` are dropped. All reads/writes for identity fields go through `user_entity`. The `set_user_profile` command is updated to write to `user_entity`. No dual storage. Verify: existing name/company/title/focus values appear in `SELECT name, company, title, focus FROM user_entity` after migration.
3. `user_context_entries` table exists: `id`, `title`, `content`, `embedding_id`, `created_at`, `updated_at`.
4. Tauri commands: `get_user_entity`, `update_user_entity_field(field, value)`, `get_user_context_entries`, `create_user_context_entry`, `update_user_context_entry`, `delete_user_context_entry` — all registered in `commands.rs`.
5. `_user/context.json` written to workspace on every field update. DB is the source of truth for all reads — the JSON file exists for local access by Claude Desktop and other tools that read workspace files. Verify: update `value_proposition` → file exists at path and contains updated value. Then read the value back via `get_user_entity` → it reads from DB, not from the JSON file.
6. Annual priorities support entity linking to accounts and projects (via `linked_entity_id` + `linked_entity_type`). Quarterly priorities support the same linking pattern. Verify via `SELECT annual_priorities FROM user_entity` — the linked ID is present. People/meeting/action linking deferred to v0.14.1.
7. Neither annual nor quarterly priorities expire automatically. `get_user_entity` returns both arrays in full regardless of age. Verify: insert a quarterly priority, update `created_at` to 60 days ago in DB, call `get_user_entity` — the item is still present in `quarterly_priorities`.
8. **Objections as flat strings:** The `objections` field is a JSON array of strings (not Q&A objects). Simplifies editing and vector DB ingestion. Verify: `update_user_entity_field("objections", '["They say we're too expensive", "Integration concerns with legacy systems"]')` — parses and stores correctly.
9. `cargo test` passes. No regressions.

### I412 — User context in enrichment prompts

1. `build_user_context_block(db) -> Option<String>` function exists. When user entity has at least one non-null field, returns a formatted block. When all fields are NULL, returns None. Reads from DB (not `_user/context.json`).
2. Every entity enrichment prompt via intel_queue includes the user context block when non-null. Verify by logging one assembled prompt at DEBUG — it contains the user's `value_proposition` text if set.
3. The reframe is observable with real data: set `value_proposition` to describe a cost-reduction story. Find an account with cost-pressure signals. Trigger enrichment. Read the resulting `entity_intel.executive_assessment` — it references the cost-reduction angle alongside the risk framing, not only risk. This is a quality judgement, not binary pass/fail, but the difference must be visible. Log before/after prompts for comparison.
4. Updating `current_priorities` and triggering re-enrichment on a high-trigger-score entity produces intelligence that references the updated priority within one intel_queue cycle.
5. When all user entity fields are NULL, the enrichment prompt is identical to its pre-v0.14.0 shape. No empty section headers or placeholder text injected.

### I413 — User context document attachment

1. A `_user/attachments/` directory in the workspace is created on first use (when a file is dropped on the attachments dropzone). Files dropped there trigger the existing file processor pipeline.
2. Processed attachments are indexed in `content_files` and `content_embeddings` tables under a `user_context` collection label. Verify: drop a PDF → after processing, `SELECT * FROM content_files WHERE source = 'user_context'` returns a row with summary and classification.
3. User context attachments do NOT appear in the main inbox or any account's file list. They are isolated to the `/me` page and the enrichment pipeline. Verify: drop a file on `/me` → navigate to inbox → file is not listed.
4. The enrichment prompt builder (I412) optionally includes top-K semantic matches from user documents when relevant (cosine similarity > 0.70 threshold). Verify: attach a product deck about "cost reduction" → enrich an account with cost-pressure signals → assembled prompt (DEBUG log) includes a passage from the attached document.
5. If the file processor is unavailable, the file is saved to disk and queued for processing on next cycle. No crashes or silent failures.

### I414 — User-context-weighted signal scoring

1. `compute_user_relevance_weight(signal, user_entity) -> f64` returns 1.0 when user entity is empty or no match (neutral — no behavior change from today).
2. When `annual_priorities` or `quarterly_priorities` names an account or project explicitly (via `linked_entity_id` or text match), signals from that entity receive a weight > 1.0. Verify: add a strategic priority linked to a known account. Check that account's signals rank higher in the attention section than equivalent-confidence signals from unlisted accounts.
3. Signal scoring changes are observable in the daily briefing: the top items in the Attention section reflect both signal confidence AND alignment with declared priorities.
4. A `user_relevance_weight` column exists on `entity_intel` (added in the I414 migration or the I411 migration). `SELECT COUNT(*) FROM entity_intel WHERE user_relevance_weight > 1.0` — returns a non-zero count after at least one strategic priority is set and a scoring cycle has run.
5. `cargo test` passes. Scoring without user context produces identical results to pre-v0.14.0.

### I396 — intelligence.json report fields

1. Six new fields in `entity_intel`: `health_score` (REAL), `health_trend` (TEXT JSON), `value_delivered` (TEXT JSON), `success_metrics` (TEXT JSON), `open_commitments` (TEXT JSON), `relationship_depth` (TEXT JSON). Migration creates them. TypeScript types include all six. `pnpm tsc` passes.
2. AI enrichment prompt for accounts includes instructions to populate all six fields. Every field name appears in the output schema definition in the prompt.
3. After enrichment for an account with 3+ signals and 2+ meetings: `health_score` is 0–100; `health_trend.direction` is one of `improving`, `stable`, `declining`, `volatile`; `value_delivered` has at least one entry if a completed commitment or win signal exists.
4. Fields are NULL or empty arrays for sparse accounts (< 2 signals). The AI is instructed to return null for fields it cannot populate — the code defaults to null when the AI omits a field. Both paths handled.
5. With user context active (I412 complete), the `value_delivered` items are framed through the user's value proposition where relevant. Verify by comparing `value_delivered` output for the same account before and after setting a value proposition — the framing should differ.
6. Existing fields unaffected. `cargo test` passes.

### I415 — User entity page

1. `/me` route exists and renders. The page has six sections: About Me, What I Deliver, My Priorities (two-layer), My Playbooks, Context Entries, Attachments.
2. All declared context fields are editable inline (EditableText / EditableList). Save on blur via `update_user_entity_field`. No explicit save button anywhere on the page.
3. **Annual priorities:** Render as a list with optional entity chip per item. "+ Add priority" row. Entity linking via existing entity picker component (accounts and projects only). Each item deletable.
4. **Quarterly Priorities:** Separate list labelled "This Quarter." Same entity linking as annual (accounts and projects). No reset action, no expiration. Visually distinct from annual priorities. Users edit, add, and remove items freely — when the quarter shifts, they update the list themselves.
5. **Context Entries:** List of existing entries (title + truncated content + date). "+ Add context entry" opens inline form. Entries can be edited and deleted. Entries appear immediately on create — embedding is background, does not block UI.
6. **Attachments dropzone:** Drag-and-drop zone targeting `_user/attachments/`. Dropped files appear in file list immediately. After processor runs (up to 2 min): `SELECT * FROM content_files WHERE source = 'user_context'` returns a row. Verify a PDF dropped here does NOT appear in the main inbox or any account's file list.
7. **CS preset shaping:** With CS preset active, What I Deliver section is expanded. Three named playbook sections appear: "At-Risk Accounts," "Renewal Approach," "EBR/QBR Preparation." Placeholders are CS-specific. With a non-CS preset, generic structure shown, single "My Methodology" playbook field.
8. **Activity indicator:** A static line near the top: "Your details are enhancing context." Visible when at least one user entity field is non-null. Hidden when user entity is empty. No dynamic count — simple presence check.
9. Design system compliance: no hardcoded hex or rgba values, no new typography variants, no inline style props with static values. Section headers use existing editorial section rule treatment.

### I417 — Context entries

1. New context entry → saved to `user_context_entries` → enqueued for embedding. After embedding processor runs: `SELECT embedding_id FROM user_context_entries WHERE id = '<id>'` returns non-null.
2. `search_user_context(query, limit)` function exists. Returns semantically relevant context entries by cosine similarity against `content_embeddings` where `collection = 'user_context'`.
3. The enrichment prompt builder (I412) calls `search_user_context` with the entity name + domain as query. Top-2 results above threshold are included in the user context block. Verify: write a context entry about "how I handle technical evaluators." Enrich an account with technical evaluation signals. Assembled prompt (DEBUG log) includes the context entry text.
4. Context entries are not visible on account detail pages or anywhere in the entity surfaces — they only appear on the `/me` page and are injected by the enrichment pipeline.
5. If embedding processor is not loaded, entry saves to DB without embedding_id and is retried when the model is available. No crashes or silent failures.

## Ship criteria

All acceptance criteria met. Navigate to `/me` — the user entity page loads with six sections, all editable inline. Set a value proposition and a strategic priority linked to a real account. Wait one enrichment cycle. Open the account detail — the intelligence references the user's value narrative. Open the Attention section of the daily briefing — signals from the linked account appear near the top. The Settings page contains no identity fields. Update-available banner appears when a new version exists. System failures surface as toasts or native notifications.

### End-to-end smoke test (mandatory before tagging)

1. Clean app restart. Navigate to `/me` (new nav item). The page loads. All six sections render. About Me shows existing name/company/title migrated from workspace_config into user_entity.

2. Fill in Value Proposition: "We help media companies publish at scale without infrastructure headaches." Fill in Current Priorities: "Expand the Crestview Media account." Add an annual priority linked to the Crestview Media account entity. Save on blur — `SELECT value_proposition, annual_priorities FROM user_entity` confirms both are persisted. `_user/context.json` exists with updated values.

3. Navigate to Settings. Confirm: no name, company, title, or focus fields visible. Settings contains only technical config, connections, role preset, domains, personality.

4. Return to `/me`. Add a context entry titled "How I think about infrastructure conversations." Wait 2 minutes. `SELECT embedding_id FROM user_context_entries WHERE title LIKE '%infrastructure%'` — returns non-null.

5. Navigate to `/me` → Attachments. Drop a product deck PDF. The file appears in the list. After 2 minutes, `SELECT id FROM content_files WHERE source = 'user_context'` returns a row. The file does NOT appear in inbox.

6. Navigate to the Crestview Media account detail. Trigger entity intelligence enrichment. Read the `executive_assessment` — it references infrastructure or the scale narrative, shaped by the user's value proposition. Compare against a pre-v0.14.0 enrichment for the same account — the framing is observably different.

7. Open the daily briefing Attention section. Crestview Media signals appear at or near the top, above equivalent-confidence signals from accounts not in the user's strategic priorities.

8. Start app with a newer version available on GitHub Releases. Update-available banner appears without navigating to Settings. Click "Install & Restart" — app updates and relaunches. "What's New" modal shows release notes. Dismiss it. Restart — modal does not reappear.

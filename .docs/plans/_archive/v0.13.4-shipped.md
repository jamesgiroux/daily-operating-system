# v0.13.4 Version Brief

**Thesis:** v0.13.3 proved that parent accounts become portfolio surfaces when two-layer intelligence and bidirectional signal propagation are wired correctly. A Marketing manager needs the same view for their project portfolio — which campaigns are on track, which workstreams need resources, what patterns are emerging across the program that no individual campaign page reveals. This version applies the identical architecture from I384/I385 to project entities: parent projects become portfolio surfaces for users in project-mode roles (Marketing, Product, Agency), child project signals propagate upward to feed portfolio intelligence, and parent-level signals cascade down to direct children without explicit specification. This also wires entity-mode-aware navigation: users whose role preset defaults to projects see Projects before Accounts in navigation; account-mode users see the inverse. (ADR-0087 generalization)

**Status:** Planning.

## Acceptance Criteria

Each criterion must be verified with real data in the running app after a full `pnpm dev` restart. Compilation is not acceptance. Types aligning is not acceptance. A criterion is met when the described behavior is observable with real entities in the running app.

---

### I388 — Project hierarchy intelligence

1. The AI enrichment prompt for any project that has child projects includes: the `intelligence.json` of every direct child (or a summarized excerpt if the child count is large), key active signals for each child, and the parent project's own signals (from meetings tagged to the parent and user edits on the parent entity).
2. The resulting `intelligence.json` for a parent project contains a `portfolio` field that is absent from leaf-node intelligence. The `portfolio` field contains: `health_summary` (portfolio-level health assessment), `hotspots` (array of child projects needing attention with reason), `cross_project_patterns` (signal types or topics appearing in 2+ children), and `portfolio_narrative` (AI-synthesized executive view of program health).
3. Verify with a real parent project that has 3+ children with existing intelligence: open the project detail page. The portfolio chapter renders with real content — not placeholder text, not null values. `hotspots` contains at least one entry if any child has a risk signal.
4. When a child project's `intelligence.json` is updated (by any means — enrichment, user edit, signal cascade), the parent project is enqueued in `intel_queue` for re-enrichment. Verify: update a child's intelligence; within one intel_queue cycle, the parent's `intelligence.json` `updated_at` timestamp changes.
5. Leaf-node projects (no children) are unaffected. Their enrichment prompt and `intelligence.json` shape are unchanged. Verify: `SELECT id FROM projects WHERE id NOT IN (SELECT DISTINCT parent_id FROM projects WHERE parent_id IS NOT NULL)` returns leaf projects; check one — no `portfolio` field in its `intelligence.json`.
6. The prompt vocabulary is project-appropriate: "campaign," "workstream," "milestone," "program health," "deliverable" — not "account health," "renewal risk," "spend." Verify by checking the prompt template or reading the resulting `intelligence.json` for a parent project: no `renewal_risk`, `spend`, or account-centric fields should appear; `program_health`, `campaign_status`, or project-centric fields should.
7. The bidirectional propagation rules from v0.13.3 (upward at 60%, downward at 50% with 0.7 threshold, no loops, direct children only) apply to the project entity type. Verify: a child project signal appears in `signal_events` with `source = 'propagation:hierarchy_up'` for the parent project; a parent project signal ≥ 0.7 produces `propagation:hierarchy_down` signals for direct children only; no `propagation:hierarchy` signal is re-propagated.

---

### I388 — Parent project detail page (bundled with I388)

8. Opening a parent project detail page (any project with children) renders a distinct **Portfolio** chapter as the primary content section. This chapter does not appear on leaf-node project pages.
9. The Portfolio chapter contains:
   - **Health summary** — portfolio-level narrative from `intelligence.json.portfolio.portfolio_narrative`
   - **Hotspots** — list of child projects with active risk or opportunity signals, with a one-line reason per child. Sorted by signal recency/urgency. Each hotspot links to that child's detail page.
   - **Cross-project patterns** — signal types or topics appearing in 2+ children, surfaced as a distinct callout. If no cross-project patterns exist, this section is hidden (not an empty state).
10. Below the Portfolio chapter, the existing project intelligence sections render (the parent's own signals, meetings, actions) — the same content a leaf-node project would show for its own entity.
11. The child project list renders as a condensed view within the Portfolio chapter — each child shown with its name and health indicator. Clicking a child navigates to that child's detail page.
12. Meetings tagged directly to the parent project appear in the parent's meeting section. Meetings tagged to child projects do NOT appear in the parent's meeting list. Verify: the parent's meeting list contains only meetings where the parent's `id` is in `meeting_entities`, not meetings linked to any child.
13. The page uses the existing design system and editorial layout patterns. The Portfolio chapter uses the existing chapter/section rule treatment. Hotspots use the existing callout/signal prose component. No new design patterns are invented.

---

### I389 — Entity-mode-aware surface ordering

1. A user with a role preset where `entityModeDefault = "project"` (e.g., Marketing, Product) sees Projects before Accounts in the main navigation.
2. A user with a role preset where `entityModeDefault = "account"` (e.g., CS, Sales) sees Accounts before Projects in the main navigation.
3. A user with a role preset where `entityModeDefault = "both"` sees the current ordering unchanged (Accounts before Projects, or the existing order).
4. Switching role presets in Settings (without restarting the app) immediately updates the nav order to reflect the new preset's `entityModeDefault`. Verify by switching from a CS preset to a Product preset: the nav reorders to reflect the change without a full app restart.
5. The change is a nav ordering change only — no page layouts, no content changes, no routing changes. Verify: `git diff src/` shows changes only to navigation-related component(s).
6. The nav order change persists across app restarts. Verify: set a project-mode preset, close the app, reopen, confirm Projects still appears first.

## Issue Dependencies

```
I388 (project portfolio intelligence + detail page)
  └── independent — extends existing project entity enrichment,
      same pattern as I384/I385 from v0.13.3; no other issues block it

I389 (entity-mode-aware navigation)
  └── blockedBy I388 — project-mode users need the portfolio surface
      to exist before nav ordering change is meaningful; reordering nav
      before projects have portfolio intelligence creates a confusing UX
```

**Recommended build order:**
1. I388 (project portfolio intelligence + detail page) — the substance, independent
2. I389 (entity-mode-aware navigation) — the surface adaptation, meaningful only once I388 exists

## Ship criteria

All acceptance criteria above are met across both issues. A real parent project with 3+ children renders a Portfolio chapter with a synthesized health summary, at least one hotspot if any child has a risk signal, and cross-project patterns if applicable. Child project signals propagate to the parent without user action. A high-confidence parent project signal fans out to all direct children within one propagation cycle. Navigation ordering adapts to role preset `entityModeDefault` without app restart. Project-mode users (Marketing, Product) see Projects before Accounts; account-mode users see the inverse. No regressions on leaf-node project, account, or person pages.

### End-to-end smoke test (mandatory before tagging)

1. Clean app restart (`pnpm dev`). Create or identify a parent project with 3+ child projects. Open the parent project detail page. Confirm: Portfolio chapter renders with real data — health summary, at least one child in the hotspots list, portfolio narrative is not null or placeholder.
2. In the running app, edit a field on a child project. Wait one intel_queue cycle (≤2 min). Open the parent project detail page. Confirm: the parent's portfolio narrative reflects the updated child context (or at minimum, `entity_intel.updated_at` for the parent has changed since before the edit).
3. Check `signal_events` directly: `SELECT source, entity_id, entity_type, created_at FROM signal_events WHERE source LIKE '%hierarchy%' AND entity_type = 'project' ORDER BY created_at DESC LIMIT 20`. Confirm both `hierarchy_up` and `hierarchy_down` rows exist from recent propagation cycles.
4. Switch to a project-mode role preset (Marketing or Product). Confirm nav shows Projects before Accounts.
5. Switch to an account-mode role preset (CS or Sales). Confirm nav shows Accounts before Projects.
6. Switch back to a different project-mode preset without restarting. Confirm nav reorders immediately to reflect the change.
7. Open a leaf-node project detail page. Confirm: no Portfolio chapter renders (it has no children).

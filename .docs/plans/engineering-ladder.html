DailyOS · Engineering Ladder

From plan to merge, in seven gates.

L0–L6 with Plan, Implement, Review, and Capture roles per rung.

The Engineering Ladder is how work moves from intake to merge. Each rung names the skills that apply at that phase — not just review. A continuous Knowledge Channel feeds findings into L0 and captures findings out of L3 / L5 so the same drift doesn't surface twice.

Adopted 2026-05-18 L0–L6 numbering preserved Plan / Implement / Review / Capture per rung K-channel: docs/solutions/ + .docs/decisions/

Each rung, four roles.

Plan, Implement, Review, and Capture skills per rung. K-in (substrate-grep at L0) and K-out (`/ce-compound` capture at L3 retro) are mandatory.

Knowledge Channel

Drift can't be prevented at the gate — it has to be prevented at authoring.

The Knowledge Channel K is a continuous feedback loop parallel to L0–L6 — not a rung. Findings flow out of L3 / L5 into the knowledge store, and back into L0 / L1 / L2 as substrate-grep obligations.

Autonomous

Claude runs /ce-compound mode:headless for each qualifying finding as part of L3 retro work. Same set-and-forget authorization as impl→L1→commit→L2→fix→retro→tag.

Reminder hook

.claude/hooks/k-out-reminder.sh (Stop hook) scans the last assistant turn for trigger phrases — class-pattern finding, same-shape twice, substrate already existed, L3 retro complete — and prints a one-line nudge.

L6 escalation policy

When to pull James in, and when not to.

L6 is expensive — every escalation is a context-switch. The orchestration handles as much as possible downstream so L6 is reserved for genuine human-judgment calls.

MUST escalate

Objective triggers — agents do not get to skip these.

  1. Cycle cap exceeded (Amendment 1)
  2. Reviewer flags "needs human judgment" with a specific structural question
  3. L5 drift detected without remediation path
  4. Suite gate failure requiring regression acceptance or budget relaxation
  5. Reviewer infrastructure failure (codex outage after 3 retries)
  6. Net new scope expansion (NOT scope discovery within the agreed outcome)
  7. Contract amendment — plan needs fundamental change
MUST NOT escalate

Handle downstream. These are the orchestrator's remit.

  1. Routine review iteration within cycle cap — read findings, fix, resubmit
  2. Single flaky codex output — retry per Amendment 2
  3. Lint, type-check, test failures — fix them
  4. Reviewer dissent (1/3 or 2/3 against) — investigate the dissenter, don't break the tie
  5. Same finding class repeats twice — class-level sweep, file new tickets on the wave
  6. Scope discovery within DoD — create tickets, attach to wave, continue
  7. Choice between equivalent implementations — pick the more complete option
  8. "I don't know what to do" without a specific structural question
  9. Deferral requests — "park for v1.4.x" is not a valid escape valve
  10. Half-finished implementations — finish or escalate as net-new-scope
  11. Asking for context already in codebase, memory, or plan
  12. Tool choice when codebase has a convention

Pacing

No wave starts until the prior wave clears L3 and L5 (where applicable). No agent codes before L0 clears unanimously. Two revision cycles on the same plan or PR without convergence → L6 escalation.

Bounding

L2 reviews are bounded by acceptance criteria. Path-α findings (theoretical hardening beyond AC) file to the maintenance project, not cycle-N+1. Substrate PRs unblock once AC violations are closed.

Wave

Wave is the project-management primitive for big version programs. Parallel agents fan out within a wave; hard ordering across waves. Smaller versions collapse to L0 → code → L2 → ship.

# v1.3.0 — Report Engine Rebuild: Intelligence-First, Display-Only Reports

**Thesis:** Every report in DailyOS re-derives intelligence from raw tables via expensive PTY calls, ignoring the pre-computed intelligence that the Intelligence Loop already stores in `entity_assessment`. v1.3.0 inverts this: reports become *assembly surfaces* that render pre-computed intelligence, not independent AI targets. With 3 new columns on `entity_assessment` and 3 new tables (`portfolio_assessment`, `user_weekly_summary`, `user_monthly_summary`), most reports become display-only — no generation step, no loading spinner, no "Generating..." wait. The few genuinely editorial AI fields are pre-computed during enrichment or at week/month rollover, not on demand. The Book of Business is rebuilt on this foundation and re-enabled. Weekly Impact and Monthly Wrapped get the delight treatment (animations, archetypes, achievements) on top of the new instant-render foundation.

**Branch:** `dev`

**Absorbs:** Former I601-I603 (report animation and delight).

---

## Principles

1. **Reports assemble, they don't generate.** `entity_assessment` is the authoritative source. Reports format, aggregate, and render — they don't re-analyze. This mirrors ADR-0086: meeting prep is a consumer of entity intelligence, not an independent AI target. Reports follow the same pattern.
2. **PTY calls happen in the background, not on demand.** The few genuinely editorial fields (portfolio synthesis, weekly headline, monthly personality) are pre-computed by scheduler tasks or enrichment triggers — never blocking the user.
3. **Mechanical where possible, AI where necessary.** Stats, scores, trends, and structured data render mechanically from DB. AI generates only what requires cross-entity synthesis or editorial voice.
4. **Display-only is the target.** Opening a report should be like opening a page — instant render from stored data. "Generate Report" buttons become "Refresh Report" for the rare case where the user wants to force a re-computation.
5. **No dead code.** The rebuild cleans up all legacy paths, schema mismatches, unreachable functions, and `#[allow(dead_code)]` annotations.

## Schema Additions

### New columns on `entity_assessment`

```sql
ALTER TABLE entity_assessment ADD COLUMN best_customer_quote TEXT;
-- Verbatim quote selected during enrichment from captures.evidence_quote.
-- Used by: Account Health, EBR/QBR.

ALTER TABLE entity_assessment ADD COLUMN recommended_next_steps_json TEXT;
-- [{action, owner_hint: "csm|customer|joint", timeline_hint: "this_week|this_month|this_quarter"}]
-- Richer than health_json.recommended_actions (plain strings).
-- Used by: EBR/QBR next_steps, Account Health (upgraded).

ALTER TABLE entity_assessment ADD COLUMN ebr_cache_json TEXT;
-- Pre-generated EBR/QBR content (executive_summary, story_bullets, strategic_roadmap).
-- Generated during enrichment cycle, not on-demand. Customer-facing register.
-- Used by: EBR/QBR report (display-only with this cache).
```

The existing `risks_json` schema is extended in the enrichment prompt to include `status: "open|mitigated|resolved"` per risk item. No new column needed — the field is widened in place.

### New table: `portfolio_assessment`

Cross-entity intelligence for Book of Business. Generated by a background task when any entity's `enriched_at` changes.

```sql
CREATE TABLE portfolio_assessment (
    id INTEGER PRIMARY KEY CHECK (id = 1),  -- singleton like user_entity
    generated_at TEXT NOT NULL,
    intel_hash TEXT NOT NULL,
    period_label TEXT NOT NULL,
    executive_summary TEXT,
    leadership_asks_json TEXT,
    quarterly_focus_json TEXT,
    key_themes_json TEXT,
    projected_churn_narrative TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### New table: `user_weekly_summary`

Pre-computed weekly report content. Mechanical fields computed at week rollover; editorial fields (headline, what_you_did) generated by one background PTY call.

```sql
CREATE TABLE user_weekly_summary (
    week_start TEXT PRIMARY KEY,
    week_label TEXT NOT NULL,
    total_meetings INTEGER NOT NULL DEFAULT 0,
    total_actions_closed INTEGER NOT NULL DEFAULT 0,
    headline TEXT,
    what_you_did TEXT,
    priorities_moved_json TEXT,
    wins_json TEXT,
    watch_json TEXT,
    into_next_week_json TEXT,
    meetings_by_day_json TEXT,
    prior_week_headline TEXT,
    generated_at TEXT,
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### New table: `user_monthly_summary`

Pre-computed monthly report content. Stats computed mechanically; editorial fields generated by one background PTY call at month rollover.

```sql
CREATE TABLE user_monthly_summary (
    month_start TEXT PRIMARY KEY,
    month_label TEXT NOT NULL,
    total_conversations INTEGER NOT NULL DEFAULT 0,
    total_entities_touched INTEGER NOT NULL DEFAULT 0,
    total_people_met INTEGER NOT NULL DEFAULT 0,
    signals_captured INTEGER NOT NULL DEFAULT 0,
    top_entity_name TEXT,
    top_entity_touches INTEGER DEFAULT 0,
    priority_alignment_pct INTEGER,
    top_win TEXT,
    moments_json TEXT,
    hidden_pattern TEXT,
    personality_json TEXT,
    carry_forward TEXT,
    word_one TEXT,
    word_two TEXT,
    word_three TEXT,
    achievements_json TEXT,
    month_over_month_json TEXT,
    top_entity_sparkline_json TEXT,
    generated_at TEXT,
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## Report-by-Report: From Generated to Display-Only

| Report | Today | After v1.0.3 | PTY Calls at Report Time |
|--------|-------|-------------|--------------------------|
| **Account Health** | 1 monolithic PTY, re-queries raw tables | Display-only assembly from `entity_assessment` | **0** |
| **SWOT** | 5 parallel PTY calls | Mechanical assembly from `entity_assessment` fields | **0** |
| **EBR/QBR** | 1 monolithic PTY | Display-only from `entity_assessment` + `ebr_cache_json` | **0** (pre-generated during enrichment) |
| **Book of Business** | Mechanical + 1 synthesis PTY | Mechanical + display from `portfolio_assessment` | **0** (pre-generated by background task) |
| **Weekly Impact** | 1 monolithic PTY | Display from `user_weekly_summary` | **0** (pre-generated at week rollover) |
| **Monthly Wrapped** | 1 monolithic PTY | Display from `user_monthly_summary` | **0** (pre-generated at month rollover) |

Total PTY calls at report-open time: **0 across all report types.**

AI work moves to: enrichment pipeline (entity-scoped), portfolio synthesis task (triggered by intel_hash change), weekly rollover (Sunday/Monday), monthly rollover (1st of month).

## Version-Level Acceptance Criteria

1. **Zero PTY calls at report-open time** across all 6 report types
2. **Consistent narratives:** Account Health says the same thing as the account detail page — same source data
3. **Instant render:** Reports open as fast as any other page — no "Generating..." state for cached content
4. **Graceful degradation:** If pre-computed data is missing (first run, stale), reports show what's available mechanically and offer a "Refresh" action to trigger background generation
5. **BoB re-enabled** behind feature flag, validated with real data
6. **Delight delivered:** Weekly Impact animated and crisp. Monthly Wrapped celebratory with archetypes, achievements, and confetti
7. **No dead code:** Zero `#[allow(dead_code)]`, zero unreachable functions, zero legacy schema paths
8. **Zero new npm dependencies** for animation work
9. **All reports support in-place editing** — edits persist in the summary tables, survive regeneration via user_edits tracking

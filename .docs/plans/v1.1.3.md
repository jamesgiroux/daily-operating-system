## D1: Navigation Rethink

### Problem

The navisland component switches between a global page menu and a local context menu depending on where you are in the app. This is confusing — users don't know which mode they're in, and the transition between global and local navigation is disorienting.

### What needs to happen

1. Run `/design-consultation` to explore navigation patterns for a CS desktop app
2. Decide: persistent global nav + contextual breadcrumbs? Sidebar + detail? Tab-based?
3. The navigation model should make it obvious where you are and how to get back
4. Consider: the app has ~5 top-level views (dashboard, accounts, people, meetings, settings) and deep entity detail pages

### Acceptance criteria

- [ ] Design session completed with navigation pattern chosen
- [ ] Wireframe or mockup approved
- [ ] Navigation component refactored to chosen pattern
- [ ] User can always tell where they are in the app hierarchy
- [ ] Back/forward browser-style navigation works intuitively

## D2: Account Detail Page Information Architecture

### Problem

The account detail page is filling up. v1.1.0 added lifecycle intelligence, provenance, technical footprint, products, and stakeholder gallery. v1.1.1 decomposed the code (1245→241 lines) but the visual layout still has too many sections competing for attention. The page needs an information architecture review, not more sections.

### What needs to happen

1. Run `/office-hours` or `/design-consultation` to rethink the page layout
2. Decide: which sections are above the fold? Which are expandable? Which move to tabs or sub-pages?
3. Consider information hierarchy: what does a CSM need to see FIRST when opening an account?
4. The v1.1.1 component decomposition makes layout changes much easier (each section is now its own component)

### Acceptance criteria

- [ ] Design session completed with layout pattern chosen
- [ ] Information hierarchy defined (primary, secondary, tertiary sections)
- [ ] Layout refactored — above-the-fold shows only the most critical account state
- [ ] Secondary information accessible via expand/tab/scroll without cluttering the initial view
- [ ] Page feels clean and focused, not overwhelming

## Context from v1.1.1

v1.1.1 laid the groundwork for both items:

- **AccountDetailEditorial.tsx** was decomposed into 12 focused components. Layout changes are now surgical — move a `<AccountProductsSection />` call, not 200 lines of inline JSX.
- **`MarginSection`** was extracted as a reusable layout wrapper, making it easy to experiment with section visibility and ordering.
- **`useAccountFieldSave`** hook centralizes all field editing logic, so layout changes don't risk breaking edit flows.

## Product thesis alignment

v1.4.1 is the **substrate completion** release — not a surfaces release. The project description is explicit:

> **Non-goals:** Claim receipt UI / Activity Log / Lint Mode → v1.4.4. Recommendation contract / salience scoring → v1.4.5. Proactive trigger policy / notifications → v1.4.6. MCP exposure → v1.5.0.

Every wave in v1.4.1 should preserve the v1.4.0 product invariants and add three more:

6. **Signals propagate durably**, not in-process best-effort. Claim invalidation survives crashes and restarts.
7. **Trust scoring biases real surfaces** through shadow-mode comparison data, not just unit tests.
8. **Capability migrations are validated end-to-end** against bundles that prove parity vs the legacy code path being replaced.

Same Mission Gate as v1.4.1 project: *"Does this make DailyOS better at maintaining and updating the user's working understanding of their professional world?"* — for v1.4.1 the answer should usually be: yes, because it makes the underlying working model more durable, trustworthy, revisable, or measurable.

## What "parallel" means here

Same shape as v1.4.0:
1. **A frozen contract.** The Linear ticket text is the contract. Agents do not invent shapes.
2. **An exclusive file/table allowlist.** Listed below per agent.
3. **A deny list.** Files no other agent in the wave will touch.
4. **A merge gate.** Concrete artifact (test, lint, command output) proving done before the next wave starts.

## Product thesis alignment

v1.4.0 is the first proof of DailyOS as a personal intelligence system.

The goal is not to ship an abilities runtime for its own sake. The goal is to
prove that DailyOS can maintain a user's working understanding of their world as
structured, revisable, trustworthy intelligence.

Every wave should preserve five product invariants:

1. Corrections stick.
2. Claims carry evidence, time, and provenance.
3. Trust is computed and visible, not implied.
4. Stale, contradicted, or uncertain intelligence is treated differently from current fact.
5. Core primitives remain domain-neutral; CS/account behavior belongs in domain interpretation and surfaces.

The Golden Daily Loop gate passes only if the user-facing result is more
trustworthy, not merely if the runtime compiles.

## What "parallel" means here

Each agent in a wave gets:

1. **A frozen contract.** The Linear ticket text — including the 2026-04-24 amendments — is the contract. Agents do not invent shapes.
2. **An exclusive file/table allowlist.** Listed below per agent.
3. **A deny list.** Files no other agent in the wave will touch.
4. **A merge gate.** Concrete artifact (test, lint, command output) proving done before the next wave starts.

Within a wave the agents fan out. **Across waves the ordering is hard.** Skipping a wave gate is what produced the Codex round 1 + 2 findings; the cost of ignoring it is weeks.

# Review system — the integrity layer

This section is the binding execution contract for how the architecture is protected during the spine push. The bar is **methodical, not fast**. Human review (L6) is the last layer, never the first. Every layer below L6 must be exhausted before a human is pulled in.

## Review ladder L0–L6

| Layer | When | Reviewers (independent) | Pass rule |
|---|---|---|---|
| **L0 Plan review** | Per agent, **pre-code** | (1) `/codex` challenge against plan (2) domain reviewer per matrix (3) `/codex` consult, independent read — no shared notes | Unanimous approval |
| **L1 Self-validation** | Per agent, pre-PR | Implementing agent | Evidence artifacts in PR body |
| **L2 Diff review** | Per PR, pre-merge | (1) `/codex review` (2) code-reviewer subagent (3) domain reviewer per matrix | All three approve |
| **L3 Wave adversarial** | After all wave PRs merged | (1) `/codex` challenge mode against integrated wave + ADRs (2) architect-reviewer on integrated diff (3) Suite S/P/E reports per wave gate | All approve, all required suites green |
| **L4 Surface QA** | W4 onward | `/qa-only` first, `/qa` in W6, accessibility-tester for user-facing | Zero blockers |
| **L5 Drift check** | After W3 and W5 | `/plan-eng-review` + architect-reviewer comparing integrated state to v1.4.0 end-state | No drift |
| **L6 Human review** | Only when L0–L5 escalate | You | Tiebreaker / direction call |

**Pacing rule.** No wave starts until prior wave clears L3 *and* L5 (where applicable). No agent codes before L0 clears unanimously. **2 revision cycles** on the same plan or PR without convergence ⇒ L6 escalation. Do not attempt a third revision unilaterally.

## 5. Performance

## Reviewer matrix (L0 plan + L2 diff domain slot)

The third reviewer slot in L0 and L2 is a domain specialist keyed to the agent's risk profile. Slots stack: a single agent can pull multiple domain reviewers if it touches multiple risk dimensions.

| Agent profile | Domain reviewer |

## Escalation to human (L6)

Triggered only when one of these occurs:

1. **2 revision cycles** on the same plan or PR fail to converge.
2. Reviewers **disagree on severity** and a tiebreaker pass on a different model still doesn't resolve.
3. A reviewer explicitly flags **"needs human judgment"** (scope cut, contract amendment, end-state question).
4. **L5 drift** detected that can't be remediated without direction.
5. **Suite gate failure** that requires accepting a regression or relaxing a budget.

## Pilot wave, retro, and tuning protocol

**W0 is the explicit pilot for the L0–L6 review system.** The two bug-fix agents are the lowest-risk validation point we get: real correctness work, but no new substrate, clean dependency state, hours-scale wall-clock. If a layer is over-engineered or under-engineered, we find out cheaply.

W0 will validate: the L0 three-reviewer triangle, the plan template, the 2-revision-cycle rule, L1 evidence requirements, L2 diff review, L3 wave adversarial, Suite E mechanics, the proof-bundle format.

W0 will *not* validate: Suite P baseline (first runs at W1), Suite S coverage on new write paths (first runs at W3), L5 drift check (W3+), reviewer-matrix stacking (W3-C carries 3 domain reviewers), 8-agent fan-out coordination (W3).

### Tuning protocol

Changes to the review system are allowed but bounded:

| Change type | Allowed | Approval |
|---|---|---|
| Tighten a layer (more reviewers, stricter pass rule, additional invariant, additional suite coverage) | Any time | Document in retro; effective next wave |
| Loosen a layer (fewer reviewers, relaxed pass rule) | Between waves only | **L6 approval required**; logged in retro |
| Drop a layer entirely | Never within v1.4.0 | Defer to post-v1.4.0 retrospective |
| Add a new layer or suite | Between waves | Document in retro; effective next wave |
| Tune reviewer prompts (e.g., `/codex challenge` phrasing) | Mid-wave allowed if false-positive rate is high | Document in retro |
| Adjust per-suite gate timing | Between waves | L6 approval; logged in retro |
| Tune CI invariants (add new ones) | Any time | Document in retro |
| Remove a CI invariant | Never within v1.4.0 | v1.4.1 retrospective |

**Hard rule: the L6 escalation triggers themselves do not relax mid-spine.** The 2-revision-cycle rule, severity-disagreement, needs-human-judgment, drift, and suite-failure triggers stay binding through v1.4.0. Their tuning is a v1.4.1 retrospective conversation.

Principle: the system is binding but not sacred. Friction with no signal gets tuned. Catching mechanisms stay. If retros consistently show a layer producing only ceremony, that's a v1.4.1 decision to remove it — not a mid-spine cut.

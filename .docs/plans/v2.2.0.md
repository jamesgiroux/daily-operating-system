## Acceptance Criteria

Each criterion must be verified with real data in the running app after a full `pnpm dev` restart.

### I456 — In-App Markdown Reader (Core)

1. **Read-only rendering** — Open any .md file from entity Documents/. Headings, lists, code blocks, tables, links, images render correctly. Verify: Open a markdown file in account detail; all formatting displays properly.

2. **Syntax highlighting** — Code blocks use language-aware highlighting (JavaScript, Python, SQL, etc.). Colors adapt to light/dark theme. Verify: Code block with `js` fence displays JavaScript syntax colors.

3. **Entity breadcrumb** — Reader header shows entity name + type (e.g., "Acme Account › Documents › GTM Strategy"). Clicking entity name navigates to entity detail. Verify: Click entity name in breadcrumb → navigate to entity page.

4. **Entity-aware links** — Markdown links to other entities (e.g., `[Acme Renewal Playbook](Acme)`) auto-link to those entities. Verify: Document mentions "Acme", click link → navigate to Acme account.

5. **Lazy loading** — Large documents (> 5MB) load content on open; doesn't block list render. Verify: Open large document; content loads within 1 second.

6. **Error handling** — Missing files show friendly message; broken images show alt text or placeholder. Verify: Delete a file, try to open → error message displays.

7. **Consistent with editorial design** — Uses design tokens, spacing grid, magazine aesthetic (no inline styles, proper margins). Verify: Reader matches typography and spacing of other DailyOS pages.

### Document Intelligence Integration

8. **Transcripts trigger enrichment** — Add propagation rule: `transcript_outcomes` signal → enqueue intel_queue for linked entity. Entity intelligence now includes raw transcript context (quotes, discussion notes) beyond AI extractions. Verify: Upload a meeting transcript with wins/risks to an account. When account intelligence is refreshed, transcript context appears in enriched intelligence.

9. **Drive docs in enrichment context** — Verify existing integration from v0.14.3: Drive documents stored in entity Documents/ are automatically retrieved and injected as context during entity enrichment. Verify: Import a Drive doc to an account. When enrichment runs: enrichment prompt (DEBUG log) includes excerpt from Drive doc.

10. **End-to-end doc → intelligence flow** — Import a Google Drive document to an account, open it in the markdown reader, verify it appears in account intelligence context. Verify: Reader shows Drive doc → enrich account → enrichment prompt includes doc excerpt → intelligence.json mentions doc insights.

### Accessibility & Polish

11. **Keyboard navigation** — Close modal with Escape. Navigate headings with arrow keys. Links are tab-focusable. Verify: Press Escape to close → modal closes. Tab through content → links are reachable.

12. **Metadata display** — File name, source (e.g., "Google Drive", "Manually added"), last modified date shown in header. Verify: Reader header displays all metadata correctly.

13. **`cargo test` passes** — No regressions. Markdown parser handles edge cases (nested lists, code in quotes, malformed links).

## Ship Criteria

Documents are readable and beautiful in-app. Markdown reader matches editorial aesthetic. Drive documents and entity Documents/ are visible and usable. Transcript context flows into entity intelligence via new propagation rule. No regressions on existing features. `cargo test` passes.

### End-to-end smoke test (mandatory before tagging)

1. Create an account. Add a markdown document to its Documents/ folder manually. Verify: File appears in account detail, opens in reader, renders correctly.

2. Import a Google Drive document to an account (requires v0.14.3 completed). Verify: File appears in Documents/, opens in reader, images/tables render.

3. Upload a meeting transcript to an account (Quill/Granola sync or manual). Verify: Transcript is processed, wins/risks are captured, linked entity intelligence includes transcript context.

4. Open markdown reader. Verify: Headings, code blocks, images, tables all render correctly. No console errors.

5. Click entity link in breadcrumb. Verify: Navigate to entity detail without closing reader.

6. `cargo test` passes. No regressions.

## Open Questions

1. **Search within document (I456 Phase 2)?** Should v1.2.0 include Cmd+F to search within a document, or defer to v1.3.0?
   - **Recommendation:** Defer. MVP is read-only, visible context. Search can come later.

2. **Document editing?** Should v1.2.0 include inline editing, or is read-only sufficient for MVP?
   - **Recommendation:** Read-only for v1.2.0. Prove the UX, add editing in v1.3.0.

3. **Entity mention auto-linking?** Should documents auto-detect mentions like "Acme" and link to the account?
   - **Recommendation:** Yes, but simple (exact name match, case-insensitive). Fuzzy matching deferred.

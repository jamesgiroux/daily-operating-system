## Release thesis

The workspace is DailyOS's primary input channel: documents, notes, Drive files, inbox drops, and entity markdown. Today these files write facts directly into SQLite and the filesystem without provenance tracking, trust scoring, or signal propagation. The result is intelligence that the system treats as implicit and immune to correction — no lifecycle state, no `source_asof`, no trust band, no invalidation path when the file changes.

v1.4.6 changes this at the substrate level. A new `services::workspace_ingestion` service becomes the single mutation boundary for all file-derived facts. The existing `_inbox/` classification-and-route pipeline is refactored to produce claim proposals rather than direct entity writes. Drive-synced files, entity documents, and user attachments enter a staged pipeline with explicit lifecycle states (`pending`, `ingesting`, `ingested`, `superseded`, `rejected`, `quarantined`). File-derived claims carry full provenance through the v1.4.1 Provenance Envelope substrate. The v1.4.2 Gutenberg block foundation hosts the source management UI, entity-seeded intake, and safe markdown preview — no new Tauri React surfaces.

This release closes the gap between "workspace memory" and "trust-scored intelligence substrate." v1.4.7 (MCP Server v2) and the recommendations layer beyond it depend on this being a real claim graph, not a filesystem mirror.

### Agent W1-B — DOS-464: Registry migration for workspace source types

- **Spec:** [DOS-464](https://linear.app/a8c/issue/DOS-464).
- **Goal:** Migrate the registry of known workspace source types (inbox, Drive, entity-doc, user-attachment, Granola transcript, Quill transcript) to the v1.4.1 `data_source` taxonomy. Establishes the allowlist of file origins the ingestion pipeline accepts and validates paths against.
- **Files owned (exclusive):** `src-tauri/src/services/workspace_ingestion/registry.rs` (W1-B is the only owner of this submodule); new migration(s) in `src-tauri/src/migrations.rs` (slot v202); no other files.
- **Don't touch:** `signals/policy_registry.rs` (v1.4.1 owns; v1.4.6 signal additions belong to W3-B); `processor/classifier.rs` (W2 owns); `google_drive/sync.rs` (W2-B refactor lane owns the Drive mutation path); `services/workspace_ingestion/lifecycle.rs` (W1-A owns); `services/workspace_ingestion/mod.rs` (W1-A owns; W1-B contributes a re-export only via the agreed convention, no edits to this file).
- **Depends on:** DOS-463 (W1-A) merged — needs the `workspace_file_lifecycle.source_type` enum and module skeleton to be frozen before this registry can enumerate and validate against them.
- **Contracts used:** ADR-0107 + v1.4.6 amendment (workspace `DataSource` variant); Shared Primitive and Service Ownership doc.
- **Security gate:** DOS-464 is security-annotated. The registry validates that file paths are within the workspace root before any ingestion proceeds. The path validation logic is a trust boundary: a malformed or traversal path must be rejected before any file content is read, AND the validation result must be unforgeable across a TOCTOU window between validate and read. `/cso` at L0 plan AND L2 diff.
  - Path validation API: `WorkspaceSourceRegistry::open_validated(path) -> Result<(File, FileIdentity), RegistryRejection>`. The function canonicalizes the path; confirms the canonical path is a strict child of the configured workspace root; rejects any path containing `..` components even if the canonical form resolves within the workspace; opens the file with `O_NOFOLLOW` (or platform equivalent — `std::os::unix::fs::OpenOptionsExt::custom_flags(libc::O_NOFOLLOW)` on Unix; corresponding `FILE_FLAG_OPEN_REPARSE_POINT` semantics on Windows); captures device + inode after open; and returns the open `File` handle plus a `FileIdentity { device, inode, canonical_path }`. **Ingestion code reads from the returned handle — it never re-opens by path.** This closes the validate-then-read TOCTOU race.
  - On rejection, emit a `WorkspaceFileRejected` signal (registered in W3-B) with reason `PathTraversalAttempt` | `SymlinkRefused` | `OutsideWorkspace` before returning the error.
  - CI gate: explicit lint script `src-tauri/scripts/check_workspace_path_validation.sh` (modeled on `check_claim_writer_allowlist.sh`) — forbids any `std::fs::File::open`, `std::fs::read*`, or `tokio::fs::*` call on a workspace-rooted path outside `services/workspace_ingestion/registry.rs::open_validated`. Allowlist exceptions carry per-line `dos-146-allowed:` rationale comments.
  - Negative fixture suite (per Phase 0 artifact-12 style): TOCTOU race (validate → swap symlink target → attempt read) must produce `RegistryRejection::SymlinkRaced`; URL-encoded `%2e%2e`; NFC/NFD Unicode normalization variant of `..`; absolute-path attempts; symlinks pointing outside workspace.
- **Intelligence Loop gate:**
  1. *Claim model:* Registry is metadata infrastructure, not a claim. No claim written by this agent.
  2. *Provenance + trust:* Each source type entry carries the `DataSource` variant it maps to (for trust scoring in W3) and a default `sensitivity` tier.
  3. *Signals + invalidation:* No signals emitted in W1-B itself. Rejection signal type is `WorkspaceFileRejected` (registered in W3-B); W1-B's `open_validated` returns a typed error, and the consumer (W2-A pipeline) decides whether to emit the signal at the boundary based on the error reason. Rejection-reason variants are documented in the security gate.
  4. *Runtime + surfaces:* Registry consumed by W2 ingestion service.
  5. *Feedback loop:* User override path is via lifecycle record, not registry. Confirm registry is read-only at runtime (no user mutation path on the registry itself).
- **Tests required:** All negative fixtures from the security gate (TOCTOU symlink race, `%2e%2e`, NFC/NFD `..`, absolute-path, outside-workspace symlink); valid path acceptance test returning `(File, FileIdentity)`; `DataSource` variant mapping round-trip across all `WorkspaceFileKind` values; CI lint script green (no path-based file reads outside `registry::open_validated`); `cargo clippy -D warnings` clean.
- **Done when:** `WorkspaceSourceRegistry::open_validated` exists and returns `(File, FileIdentity)` from a single atomic open; all negative fixtures rejected with the typed reasons listed in the security gate; CI lint script in place; ingestion call sites in subsequent lanes consume only the returned handle (verified via grep at W2-A merge); migration slot v202 used; all tests green; `/cso` L0 plan AND L2 diff approval recorded.

### Agent W1-C — DOS-465: Document/entity links and ingestion run tracking

- **Spec:** [DOS-465](https://linear.app/a8c/issue/DOS-465).
- **Goal:** Introduce the `document_ingestion_runs` table — each run of the ingestion pipeline against a file produces a record with start time, file hash, file size, ingestion mode (initial/incremental/forced), status, claim count produced, and error log. Doc/entity links (`document_entity_links`) table associates workspace files with the entities they're relevant to, carrying the confidence and attribution for the association.
- **Files owned (exclusive):** New migrations in `src-tauri/src/migrations.rs` (slots v203–v204); new tables `document_ingestion_runs` and `document_entity_links`; new submodule files `src-tauri/src/services/workspace_ingestion/runs.rs` and `src-tauri/src/services/workspace_ingestion/link.rs` (W1-C is the only owner of both submodule files).
- **Don't touch:** `services/workspace_ingestion/lifecycle.rs` (W1-A); `services/workspace_ingestion/registry.rs` (W1-B); `services/workspace_ingestion/mod.rs` (W1-A); claim substrate; signal bus.
- **Depends on:** DOS-463 (W1-A) merged — needs `workspace_file_lifecycle` PK for the FK on `document_ingestion_runs` and the module skeleton.
- **Contracts used:** ADR-0098 (source lifecycle); Shared Primitive and Service Ownership doc.
- **L0 question #10 resolution (per cycle 2 amendment):** `document_entity_links` ships as a relational table for v1.4.6, NOT as a `commit_claim` write. Justification: an entity-doc link is metadata about ingestion provenance, not an assertable claim about the world; promoting it to a claim adds substrate complexity without enabling new v1.4.6 behavior. The relational form is bounded by:
  - User correction routes through `services::workspace_ingestion::link::override_link(emitter: &dyn SignalEmitter, file_id, entity_id, actor) -> Result<()>` (cycle 3 amendment: takes `SignalEmitter` as a parameter via the W2-A trait so W3-B doesn't need to edit `link.rs`) — writes the row AND emits a `WorkspaceFileEntityLinkChanged` signal (the real emitter impl ships in W3-B's `signals.rs` and is constructed in `wiring.rs`) so downstream surfaces invalidate.
  - Graph projection (DOS-489 / W3-C) reads the table and surfaces `attribution_source` + `confidence` + `user_override` as part of the projection response, so consumers can distinguish system-attributed from user-corrected links.
  - The v1.5.x decision to migrate links to typed claims remains open and is filed separately as a maintenance item; it does not block v1.4.6.
- **Intelligence Loop gate:**
  1. *Claim model:* Ingestion run records are not claims. `document_entity_links` is a relational metadata record per the cycle 2 resolution above, not a claim.
  2. *Provenance + trust:* Ingestion run records carry `file_hash` (SHA-256 of content at ingestion time) for tamper detection and re-run idempotency. `document_entity_links` carries `attribution_source` (the signal or rule that established the link), `confidence` (0.0–1.0), and `user_override` (nullable timestamp + actor_id).
  3. *Signals + invalidation:* `link::override_link` emits `WorkspaceFileEntityLinkChanged`; no other W1-C surface emits signals (ingestion run completion signals belong to W3-B in the pipeline).
  4. *Runtime + surfaces:* `document_ingestion_runs` consumed by W5 backfill (idempotency key), DOS-472 source management UI (history display), and DOS-489 graph projection. `link::override_link` consumed by DOS-472 (user re-link action) and DOS-468 (entity-seeded intake first-link).
  5. *Feedback loop:* User correction via `link::override_link` writes `user_override` AND emits the link-changed signal. Downstream prep invalidation handled by W3-B's allowlist additions.
- **Tests required:** Migration applies cleanly against W1-A baseline; FK integrity between `document_ingestion_runs` and `workspace_file_lifecycle`; idempotency test (inserting the same run twice with the same file hash produces one record, not two); `link::override_link` writes row + emits signal (signal bus mock); `user_override` field round-trips with actor attribution.
- **Done when:** Both tables exist with all specified columns; FK constraints enforced; `link::override_link` API present with signal emission; idempotency test green; all migrations apply in sequence; `cargo test` + `cargo clippy -D warnings` clean.

  - **`PlaceDocumentInput`** (cycle 3 amendment: explicit MCP wire encoding for binary content; closes DX OPEN F#1):

    ```
    {
      "entity_id": String,                      // required; subject the file should attach to
      "filename_hint": String,                  // required; sanitized to safe identifier server-side; original recorded in run log
      "content_b64": String,                    // required; standard base64 of file bytes; max decoded size 10MB; binary detection rejects non-text after decode
      "client_source_asof": Option<ISO-8601>,   // OPTIONAL hint only; NOT used as authoritative source_asof (CSO #5 fix)

    }
    ```
    The Rust struct uses `content: Bytes` internally; MCP wire serialization decodes base64 in/out. The `client_dedup_key` is the third component of the idempotency triple — there is one, and only one, key (matches the cycle 2 changelog after cycle 3 reconciliation).

    ```
      "ingestion_run_id": String,
      "file_id": String,

    }
    ```

    ```
      "would_create_run": bool,
      "would_create_file_id": String,
      "would_lifecycle_state": LifecycleState,
      "would_emit_claim_proposals_estimate": u64,
      "duplicate_of_existing_receipt": Option<PlaceDocumentReceipt>
    }
    ```

Rationale: a malicious MCP caller cannot inflate trust band by submitting stale content with a fresh-looking timestamp.

Subject revocation between auth check and write is impossible because the read of the entity row is `SELECT ... FOR UPDATE`-equivalent (or, on SQLite, an implicit `BEGIN IMMEDIATE` covering the auth read).

- **Idempotency:** if a `(entity_id, content_sha256, client_dedup_key.unwrap_or_default())` matches an existing run within the last 1 hour, return `IdempotencyConflict { existing_receipt }` instead of creating a new run. Caller can opt out via `client_dedup_key = Some("nonce-...")` for legitimate re-uploads.
- **Security gate:** DOS-474 is security-annotated.

  - `/cso` at L0 plan AND L2 diff.

  3. *Signals + invalidation:* Placement triggers `WorkspaceFileIngested` → `EntityIntelligenceUpdated` through the W3-B signal chain.

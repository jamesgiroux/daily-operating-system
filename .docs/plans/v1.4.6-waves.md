## Release thesis

The workspace is DailyOS's primary input channel: documents, notes, Drive files, inbox drops, and entity markdown. Today these files write facts directly into SQLite and the filesystem without provenance tracking, trust scoring, or signal propagation. The result is intelligence that the system treats as implicit and immune to correction — no lifecycle state, no `source_asof`, no trust band, no invalidation path when the file changes.

v1.4.6 changes this at the substrate level. A new `services::workspace_ingestion` service becomes the single mutation boundary for all file-derived facts. The existing `_inbox/` classification-and-route pipeline is refactored to produce claim proposals rather than direct entity writes. Drive-synced files, entity documents, and user attachments enter a staged pipeline with explicit lifecycle states (`pending`, `ingesting`, `ingested`, `superseded`, `rejected`, `quarantined`). File-derived claims carry full provenance through the v1.4.1 Provenance Envelope substrate. The v1.4.2 Gutenberg block foundation hosts the source management UI, entity-seeded intake, and safe markdown preview — no new Tauri React surfaces.

This release closes the gap between "workspace memory" and "trust-scored intelligence substrate." v1.4.7 (MCP Server v2) and the recommendations layer beyond it depend on this being a real claim graph, not a filesystem mirror.

### Agent W1-A — DOS-463: Workspace file lifecycle and ownership model

- **Spec:** [DOS-463](https://linear.app/a8c/issue/DOS-463).
- **Goal:** Define and ship the authoritative lifecycle/ownership model for workspace files. Every file that DailyOS monitors has a lifecycle state (`pending`, `ingesting`, `ingested`, `superseded`, `rejected`, `quarantined`) and an ownership record linking it to a source type, entity (if known), and ingestion run. This is the schema + type contract that all of W1 and W2 build on.
- **Files owned (exclusive):** New migration(s) in `src-tauri/src/migrations.rs` (slots v173–v174); new table `workspace_file_lifecycle` (or the name from the accepted Linear issue contract); new Rust types in `src-tauri/src/services/workspace_ingestion.rs` (new module, file-exclusive at this stage — only the type definitions, not the mutation logic); doc amendment in `src-tauri/src/services/mod.rs` (mod entry only).
- **Don't touch:** `services/claims.rs`, `signals/`, `processor/`, `watcher.rs`, `google_drive/`, any existing migration files.
- **Depends on:** v1.4.1 Signal Policy Registry and `data_source` taxonomy confirmed merged (W0 close gate).
- **Contracts used:** Shared Primitive and Service Ownership doc (`c729429f-79c9-4131-a243-5500d6e2a605`); ADR-0098 (source-aware lifecycle); ADR-0107 (source taxonomy — `data_source` values for workspace file types).
- **Surfaces affected:** Tauri runtime (new table); no user-visible surface in this wave.
- **Migration slots:** v173–v174.
- **Intelligence Loop gate (per-issue, CLAUDE.md §Critical Rules):**
  1. *Claim model:* The `workspace_file_lifecycle` record is not a claim — it is the provenance carrier for claims that will be derived from the file in W2/W3. Confirm in done-when that no claim is written by this agent.
  2. *Provenance + trust:* The lifecycle record carries `source_type`, `source_asof` (file mtime or Drive `modifiedTime`), and `data_source` (ADR-0107 variant). Confirm these fields are present and typed correctly.
  3. *Signals + invalidation:* This wave emits no signals. Confirm the lifecycle state machine specifies which state transitions in W2/W3 will trigger signals (document in type comments).
  4. *Runtime + surfaces:* No ability or surface consumes the lifecycle record in W1. Confirm the type is defined for consumption by `services::workspace_ingestion` in W2.
  5. *Feedback loop:* Lifecycle record carries a `user_override` field (nullable) so a future correction can escalate a `rejected` file to `pending` re-ingestion. Confirm the field exists even if unused in W1.
- **Tests required:** Schema migration round-trip (apply + verify table shape); type shape property test (all required fields present and typed); trybuild test asserting `WorkspaceFileLifecycle` cannot be constructed without `source_type`, `source_asof`, `data_source`, and `lifecycle_state`.
- **QA/proof:** Migration applies cleanly against a v1.4.1 baseline DB; `cargo test` green; `cargo clippy -D warnings` clean.
- **Done when:** `workspace_file_lifecycle` table exists with lifecycle state enum, `source_type`, `source_asof`, `data_source` (ADR-0107 variant), `entity_id` (nullable), `ingestion_run_id` (FK to `document_ingestion_runs` from DOS-465, nullable at W1 time), `sensitivity` (ADR-0125 variant), `user_override`; all required fields non-nullable; Rust types compile inside `services::workspace_ingestion`; all tests green; Intelligence Loop gate above answered affirmatively.
- **Handoff notes:** DOS-464 and DOS-465 consume the `workspace_file_lifecycle` table shape. Freeze it before those agents start implementation.

### Agent W1-B — DOS-464: Registry migration for workspace source types

- **Spec:** [DOS-464](https://linear.app/a8c/issue/DOS-464).
- **Goal:** Migrate the registry of known workspace source types (inbox, Drive, entity-doc, user-attachment, Granola transcript, Quill transcript) to the v1.4.1 `data_source` taxonomy. Establishes the allowlist of file origins the ingestion pipeline accepts and validates paths against.
- **Files owned (exclusive):** `src-tauri/src/services/workspace_ingestion.rs` (source-type registry portion — function-level exclusive on the `WorkspaceSourceRegistry` sub-section); new migration(s) in `src-tauri/src/migrations.rs` (slot v175); no other files.
- **Don't touch:** `signals/policy_registry.rs` (v1.4.1 owns; v1.4.6 extends the signal bus, not the policy registry); `processor/classifier.rs` (not touched in W1; W2 owns); `google_drive/sync.rs` (DOS-466 in W2 owns the Drive integration mutation path).
- **Depends on:** DOS-463 (W1-A) merged — needs the `workspace_file_lifecycle.source_type` enum to be frozen before this registry can enumerate and validate against it.
- **Contracts used:** ADR-0107 (source taxonomy); Shared Primitive and Service Ownership doc.
- **Security gate:** DOS-464 is security-annotated. The registry validates that file paths are within the workspace root before any ingestion proceeds. The path validation logic is a trust boundary: a malformed or traversal path must be rejected before any file content is read. `/cso` at L0 plan AND L2 diff.
  - Path validation must: canonicalize the path (`std::path::Path::canonicalize` or equivalent), confirm the canonical path is a strict child of the configured workspace root, reject any path with `..` components even if the canonical form resolves within the workspace, and emit a `WorkspaceIngestionRejected` signal with reason `PathTraversalAttempt` before returning an error.
  - CI gate: grep test asserting `..` in any path passed to `WorkspaceSourceRegistry::validate_path` produces the `PathTraversalAttempt` rejection, not a success.
- **Intelligence Loop gate:**
  1. *Claim model:* Registry is metadata infrastructure, not a claim. No claim written by this agent.
  2. *Provenance + trust:* Each source type entry carries the `DataSource` variant it maps to (for trust scoring in W3) and a default `sensitivity` tier.
  3. *Signals + invalidation:* No signals emitted in W1-B. The `WorkspaceIngestionRejected` signal type is declared here but not wired to the bus until W2.
  4. *Runtime + surfaces:* Registry consumed by W2 ingestion service.
  5. *Feedback loop:* User override path is via lifecycle record, not registry. Confirm registry is read-only at runtime (no user mutation path on the registry itself).
- **Tests required:** Path traversal rejection test (multiple attack vectors per `/cso` review: `../../../etc/passwd`, URL-encoded `%2e%2e`, symlink traversal if workspace contains symlinks); valid path acceptance test; `DataSource` variant mapping round-trip; `cargo clippy -D warnings` clean.
- **Done when:** `WorkspaceSourceRegistry` enumerates all v1.4.6 source types with `DataSource` mapping, default sensitivity, and path-validation guard; path traversal returns `PathTraversalAttempt` error for all named attack vectors; migration slot v175 used; all tests green; `/cso` L0 plan approval recorded.

### Agent W1-C — DOS-465: Document/entity links and ingestion run tracking

- **Spec:** [DOS-465](https://linear.app/a8c/issue/DOS-465).
- **Goal:** Introduce the `document_ingestion_runs` table — each run of the ingestion pipeline against a file produces a record with start time, file hash, file size, ingestion mode (initial/incremental/forced), status, claim count produced, and error log. Doc/entity links (`document_entity_links`) table associates workspace files with the entities they're relevant to, carrying the confidence and attribution for the association.
- **Files owned (exclusive):** New migrations in `src-tauri/src/migrations.rs` (slots v176–v177); new tables `document_ingestion_runs` and `document_entity_links`; new Rust types in `src-tauri/src/services/workspace_ingestion.rs` (ingestion-run and entity-link sub-sections — function-level exclusive).
- **Don't touch:** lifecycle table (W1-A owns); source registry (W1-B owns); claim substrate; signal bus.
- **Depends on:** DOS-463 (W1-A) merged — needs `workspace_file_lifecycle` PK for the FK on `document_ingestion_runs`.
- **Contracts used:** ADR-0098 (source lifecycle); Shared Primitive and Service Ownership doc.
- **Intelligence Loop gate:**
  1. *Claim model:* Ingestion run records are not claims. `document_entity_links` IS a claim-adjacent record (it asserts a relationship) but ships as a relational table, not through `commit_claim`. Confirm in done-when that any future conversion to a proper claim goes through `services::claims`. (See L0 question #10.)

### Agent W5-A — DOS-475: Backfill existing workspace files

- **Spec:** [DOS-475](https://linear.app/a8c/issue/DOS-475).
- **Goal:** Backfill all pre-existing workspace files (entity JSON/markdown files, Drive-synced documents, inbox residues) into the `workspace_file_lifecycle` table and run the ingestion pipeline in backfill mode to produce claim proposals for files that never went through the v1.4.6 pipeline. Backfill is idempotent, resumable, dry-runnable, and never mass-promotes historical files into `likely_current` band.
- **Files owned (exclusive):** `src-tauri/src/services/workspace_backfill.rs` (new module — backfill orchestration); `src-tauri/src/bin/workspace_backfill.rs` (new binary — CLI invocation with `--dry-run` flag); migration slot v191.
- **Don't touch:** `IngestPipeline` (W2-A owns — backfill calls it); `services/claims.rs` (backfill routes through `commit_claim` via the pipeline); existing claim records (backfill never updates committed claims' `canonical_status` without going through `commit_claim` with proper tombstone/dedup logic).
- **Backfill invariants:**
  - Idempotent: running backfill twice against the same file produces the same result (use `ingestion_run_id` + `file_hash` as idempotency key).
  - Resumable: backfill records a cursor (`last_processed_file_path`) in a `backfill_state` row updated atomically with each file processed.
  - Dry-runnable: `--dry-run` mode produces the `BackfillSummary` report without writing any `workspace_file_lifecycle` records or committing any claims.
  - Trust band discipline: historical files with `source_asof` older than the freshness decay threshold for their `DataSource` variant receive `use_with_caution` or `needs_verification` band — never `likely_current` without a re-ingestion that confirms the content is still current.
- **Intelligence Loop gate:**
  1. *Claim model:* Backfill commits proposals through `commit_claim`. No direct `intelligence_claims` insert.
  2. *Provenance + trust:* `source_asof` from file mtime at backfill time. Historical files with mtime older than decay threshold receive degraded trust band per the freshness decay table (v1.4.1). Confirmed in done-when by a fixture test with an old-mtime file producing `use_with_caution` band.
  3. *Signals + invalidation:* Backfill-committed claims emit `ClaimAsserted` signals through the v1.4.1 signal path. `EntityIntelligenceUpdated` fires per entity after backfill completes for that entity's files.
  4. *Runtime + surfaces:* No UI change in W5-A. `build_intelligence_context()` picks up backfill-committed claims automatically.
  5. *Feedback loop:* Backfill-committed claims are subject to the existing feedback path. User corrections work identically.
- **Tests required:** Idempotency test (run twice, same result); dry-run produces `BackfillSummary` with no writes; resumability test (simulate interrupt + resume, final state correct); old-mtime file gets `use_with_caution` band; `cargo test` + `cargo clippy -D warnings` clean.
- **Done when:** Backfill binary runs against a fixture workspace; idempotency, resumability, dry-run all green; trust band discipline enforced on old-mtime files; `BackfillSummary` matches expected counts; all tests green.

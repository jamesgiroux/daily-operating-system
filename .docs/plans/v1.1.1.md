# v1.1.1 — Entity Linking as a Service + Security Hardening

**Thesis:** v1.1.0 shipped lifecycle intelligence, but meetings identified by title or attendee matching silently failed to generate briefings. The entity linking pipeline had 6 independent gaps that compounded into silent failures. v1.1.1 makes entity linking a proper service layer function, fixes the gaps, hardens the security posture, and adds the first real frontend test coverage.

**Status:** Implementation complete. AC7 pending real-data verification.

**Origin:** Production observation (2026-03-31). A CloudCo meeting with the account name in the title and known attendees linked to the CloudCo account did not generate a briefing. Classification correctly identified the account (slug match at 0.50 confidence), but the entity link was never persisted to `meeting_entities`, so the prep queue skipped it entirely. CSO audit on the same day found 2 HIGH and 3 MEDIUM security findings.

**Cadence:** Tuesday/Thursday release train for v1.1.x hardening series (temporary upgrade from weekly Tuesday). Reverts to weekly after hardening completes.

## What shipped

### Entity linking overhaul (I653 + beyond)

Two centralized service functions replace all inline entity linking:
- `persist_classification_entities()` — creation-time linking, no prep invalidation
- `persist_and_invalidate_entity_links_sync()` — background auto-link with stale prep invalidation

**9 fixes:**
1. `CalendarEvent.classified_entities` carries entity IDs through `to_calendar_event()` (was `None`)
2. Calendar poller persists entity links via service function
3. Name-vs-ID comparison bug fixed (was comparing account names against UUIDs)
4. `prepare_today` persists entity links (was missing, unlike `prepare_week`)
5. `prepare_week` refactored to use centralized service (DRY)
6. Background auto-link invalidates stale prep and enqueues re-generation
7. Mechanical prep passes attendees in classified JSON for signal resolution
8. Person→account chaining at classification time (stakeholder attendees boost existing account resolutions)
9. Resolution window widened from 30 minutes to 7 days (catches meetings after app restart)

**2 critical bugs found during AC validation:**
- Stakeholder chaining was creating entity resolutions for ALL accounts any attendee was linked to (fixed: only boosts existing resolutions, never introduces new ones)
- User domains (CSM's own company) were stored in `account_domains` for every customer account, causing every meeting to domain-match every account (fixed: user domain exclusion + v2 backfill purge)

### Security hardening (CSO audit findings)

- All 12 GitHub Actions pinned to commit SHA (was tag refs)
- `.github/CODEOWNERS` protects workflow files
- Gravatar API key migrated from plaintext `config.json` to macOS Keychain
- DOMPurify sanitization on release notes HTML rendering

### Frontend test coverage

47 new tests across 5 components (was 0):
- StakeholderGallery (10 tests)
- EditableVitalsStrip (10 tests)
- AccountOutlook (13 tests)
- AccountDetailEditorial (4 tests)
- DailyBriefing (10 tests)

### ADR-0101: DB is the read model

Identified 5 problematic READ paths where frozen JSON is used as source of truth instead of DB. Migration sequence defined for v1.1.2. Written to `.docs/decisions/0101-db-is-read-model.md`.

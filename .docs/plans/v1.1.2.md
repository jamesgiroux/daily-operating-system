# v1.1.2 — Transcript Routing Fix

**Thesis:** Since mid-February, every transcript and meeting record has been silently dumped into `_archive/{date}/` instead of routing to account directories. The Intelligence Loop is broken for transcript data: 57+ files are invisible to MCP search, meeting prep, and intelligence context. Three linked root causes: empty account_domains table kills entity resolution, transcript processor has no fallback routing, and stranded files need recovery. v1.1.2 fixes the pipeline, adds defensive fallbacks, and recovers the backlog.

## Root Cause Summary

**Failure chain discovered 2026-04-03:**

1. `account_domains` table is empty for all accounts (domains extracted during orchestration but never persisted)
2. `classify_meeting_multi()` loads EntityHints with empty domain lists, can't match attendee emails to accounts
3. `CalendarEvent.account` is `None` for most meetings
4. `process_transcript_with_kind()` checks `meeting.account`, gets None, falls through to `_archive`
5. No fallback: transcript processor doesn't parse YAML frontmatter or try attendee domain matching
6. Result: 57+ customer meeting transcripts stranded in `_archive` since mid-February

## Acceptance Criteria

### Pipeline Fix (I660 + I661)
1. **Entity resolution works.** A meeting with `contact@heroku.com` resolves to the Heroku account.
2. **Transcripts route to accounts.** New transcripts from customer meetings land in `Accounts/{name}/Call-Transcripts/`.
3. **Meeting records route to accounts.** Records land in `Accounts/{name}/Meeting-Records/`.
4. **Fallback chain is defensive.** When classification fails, attendee domain matching and frontmatter parsing provide secondary routing.
5. **Routing is auditable.** Logs show which method resolved each transcript (classification, domain fallback, frontmatter, archive).

### Recovery (I662)
6. **Stranded files recovered.** Customer meeting transcripts moved from `_archive` to correct account directories.
7. **Internal meetings untouched.** Team syncs, all-hands, etc. stay in `_archive` (correct behavior).
8. **DB paths updated.** `meeting_transcripts.transcript_path` reflects new locations.
9. **MCP discovers recovered files.** `search_content` for an account name finds its recovered transcripts.
10. **Idempotent.** Running recovery twice produces no changes on second run.

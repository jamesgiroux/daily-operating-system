## v1.4.3 mission

v1.4.3 ships the WordPress foundation that v1.4.4 Surface Migration assembles on top of. v1.4.2 W4-F proved WP can render DailyOS substrate content end-to-end (`account-overview` block, PR #298). v1.4.3 ships the kit, primitives, theme, stabilization, and feedback infrastructure that make any subsequent block author productive and any subsequent surface usable.

**No new composite surfaces.** `account-overview` from v1.4.2 stays the working reference. v1.4.4 builds composites.

## Release thesis

v1.4.3 closes when:
1. WordPress block authors can lift a Tauri React primitive into a Gutenberg block without reading substrate code.
2. The stabilization gaps surfaced by v1.4.2 W4-F L4 hands-on (DOS-671..675) are fixed.
3. Feedback writes (corrections, dismissals, edits) flow through the substrate claim path, not direct DB writes from WP.
4. A clean macOS install can pair DailyOS + WP and render account-overview within 5 minutes.

## What v1.4.0–v1.4.2 leave behind (consumed contracts)

- **v1.4.0–v1.4.1:** abilities-runtime substrate (W3 abilities, claims, lifecycle, trust bands). All v1.4.3 blocks consume these unchanged.
- **v1.4.2 W4-A/B/D:** composition substrate (`Composition`, `ProjectedComposition`, `ProjectedBlock`, field-binding paths, trust-band hoist). v1.4.3 blocks consume this verbatim.
- **v1.4.2 W4-F (PR #298):** local-to-local read path. `surface_runtime/project_composition` route, signed-loopback transport, surface session lifecycle. v1.4.3 W0 hardens the lifecycle holes this surfaced; the read contract is otherwise consumed unchanged.
- **v1.4.2 magazine theme spike + `dailyos/account-overview` block:** the reference implementation v1.4.3 generalizes into the C1 starter kit (W1) and translates into the Wave 1 primitive blocks (W2).

## Architecture invariants

All v1.4.0–v1.4.2 invariants remain active. v1.4.3 adds (or re-asserts):

| Invariant | Mechanism | Activated by |
|---|---|---|
| **Local-to-local threat model holds.** Reads do not write to surface_client_sessions, audit tables, rate-limit buckets, or any other DB state outside the deliberate render-path producer commit on cache miss. | Runtime test that 100 consecutive `project_composition` reads against a stable cache key fire zero `db_write` calls | W0 (Packet B §5.3 cache key correction merge) |
| **Producer commit on cache miss is the materialization channel + de-facto invalidation.** No signal-propagation invalidation bus required at v1.4.3 scope. | Rust test: external composition-version advance invalidates old cache key on next read | W0 (Packet B §5.3 merge) |
| **Render-path authorization mandatory before cache lookup.** `authorize_local_render` skips rate-budget consumption (ability/scope buckets only); descriptor/actor/mode/scope checks remain. | grep gate on `surface_runtime/mod.rs` — `authorize_local_render` precedes `cache_lookup` | W0 (Packet B §5.5 merge) |
| **Keychain cleanup outside the SQLite transaction.** Three `revoke_pairing_row` call sites (explicit revoke, re-pair, suspicious-replay) return `KeychainCleanupTarget`; cleanup runs after commit; failure logs + audits but does not rollback. | Rust test: mocked-slow keychain CLI; SQLite writer lock released within 10ms of DB commit | W0 (Packet A §5.3 merge) |
| **Three-variant `SessionKeyLookup` classification.** Only documented `security` item-not-found signatures classify as `NotFound` → revoke. Everything else → `Unavailable` → leave DB row active. | grep CI on `load_session_master_key` return type | W0 (Packet A §5.1 merge) |
| **Explicit sentinel cleanup before listener abort.** `explicit_sentinel_cleanup()` called from `stop` + `Drop` before `abort()`. | grep CI: `explicit_sentinel_cleanup()` precedes `abort.abort()` in both `stop` and `Drop::drop` bodies | W0 (Packet A §5.4 merge) |
| **C1 starter kit harness gates every new block.** No new block (in v1.4.3 W2 or any subsequent version) lands without a passing integration test against the kit's producer→projection→renderer fixture. | CI workflow `block-kit-integration.yml` runs against every PR touching `wp/dailyos/blocks/` | W1 merge |
| **Block primitives consume tokens from a single source.** Token-to-`theme.json` generator runs from the three canonical token sources (`.docs/design/tokens/`, `src/styles/tokens.css`, `wp/dailyos/theme/theme.json`); no per-block hardcoded values. | grep CI on `wp/dailyos/blocks/*/style.css` for raw color/spacing values | W2 merge |
| **DailyOS magazine theme owns no trust/provenance styling.** Plugin owns the essential trust/provenance CSS; theme can override but is not a security/runtime dependency. Stock WP theme is a supported fallback. | Suite S test: switch to TwentyTwentyFive; account-overview block still renders trust band + provenance label legibly | W3 merge |
| **Feedback writes flow through the substrate claim path.** No direct DB writes from WP; no MCP route bypassing `services::claims::record_claim_feedback` or its sibling for the W4-E nonce path. | grep CI on `wp/dailyos/` for DB-write attempts (rusqlite handles, raw SQL); allowlist on `commands/` for write entry points | W4 merge |
| **Clean-machine install path is documented + tested.** Pair from-scratch reaches account-overview render in ≤5 minutes from a fresh macOS user account. | DOS-577 clean-machine fixture run | W6 merge |

## Wave 4 — Feedback Write Infrastructure

Per architecture invariant: feedback writes flow through the substrate claim path, not direct DB writes from WP. Per execution constraint: W4-E user-presence nonce + W5-A click-bound router (from v1.4.2 lineage).

- **Goal:** Inline edit affordances in WP blocks (corrections, dismissals, corroborations, contradictions) write through the user-presence-bound nonce path to the substrate's existing claim/feedback service.
- **Stage 4a — Substrate (nonce lifecycle):**
  - `surface_feedback_nonces` table (migration v181) — nonce_id, surface_client_id, session_id, claim_id, action_kind, user_intent_text (optional, sensitivity-flagged), issued_at, consumed_at, expires_at (24h default).
  - Service: `services::surface_feedback::issue_nonce`, `consume_nonce`, `expire_nonces_sweep`.
  - Audit events: `pairing.feedback.nonce_issued`, `.consumed`, `.expired`, `.replay_rejected`.
- **Stage 4b — Click-bound router (WP):**
  - PHP REST endpoint `wp/dailyos/v1/feedback` that mints a nonce via signed runtime call, then waits for user click confirmation before consuming.
  - JS: feedback affordance UI (button + optional textarea for `note` per cycle 2 W4-F class E security limit: 500 chars, sensitivity=User, redaction in audit log copy).
  - Wire to existing `record_claim_feedback` substrate service.
- **Stage 4c — Proof:** end-to-end feedback fixture (Tauri runtime + WP editor session) — user clicks "Mark outdated" on an account-overview claim, nonce issues, user confirms, substrate records feedback, next render reflects the dismissal.
- **L2 reviewer matrix:** codex challenge + code-reviewer + codex consult + CSO (mandatory — nonce lifecycle + write path).
- **Migration slots:** v181 (nonce table), v182 (index), v183 (rollback path if needed).

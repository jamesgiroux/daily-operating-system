## Things to be aware of

- Resolution of DOS-217 vs DOS-264 is already locked: **abilities are operations; MCP tools are registry-derived**. Do not reopen this design question.
- Four bridge types: `TauriAbilityBridge`, `McpAbilityBridge`, `WorkerAbilityBridge`, `EvalAbilityBridge`. Each wraps the same `invoke_ability` core differently depending on caller context.
- The generic `invoke_ability` Tauri command is new — it replaces the per-ability `get_entity_context`, `prepare_meeting` etc. Tauri commands. New command must be in `src-tauri/src/commands/`.
- MCP auto-registration: abilities registered in the W3-A registry (DOS-210) are auto-exposed as MCP tools. The bridge reads the registry at startup — no hardcoded tool list.
- Actor-filtered discovery: MCP tool discovery must filter by actor (human / agent / eval / worker). An eval actor must NOT see tools that require human context and vice versa. This is the primary security surface — get it right.
- Hard depends on W3-A registry (DOS-210) for ability metadata and W3-B Provenance (DOS-211) for input envelope shape. Read those W3 L0 plans.
- **Security is the lead concern for this agent.** The Suite S evidence at end of W4 covers: auth/authz on `invoke_ability`, MCP actor-filter discovery, schema-validation on input JSON, capability-handle leakage.

## The two hard blockers (Phase 0)

**Nothing else compiles without these.** Do not skip to Phase 1.

### DOS-209 — ServiceContext + ExecutionMode

**File:** [`src-tauri/src/services/context.rs`](../../src-tauri/src/services/context.rs) (new)

Create `ServiceContext` struct with:

- `mode: ExecutionMode` (`Live` / `Simulate` / `Evaluate`)
- `clock: &'a dyn Clock` — trait with `fn now(&self) -> DateTime<Utc>`
- `rng: &'a dyn SeededRng`
- `external: &'a ExternalClients` — mode-aware wrappers for Glean, Gmail, Salesforce, Slack
- `tx: Option<TxHandle<'a>>`
- `db: &'a ActionDb`

Add `check_mutation_allowed(&self) -> Result<(), WriteBlockedByMode>` that returns `Err` outside `Live`.

Add `with_transaction_async(ctx, |tx_ctx| async { ... })` primitive (HRTB/lifetime design may iterate).

Migrate every mutation function in `src-tauri/src/services/` (~60 call sites) to take `&ServiceContext` and call `ctx.check_mutation_allowed()?` as first line.

Replace every direct `Utc::now()` and `rand::thread_rng()` in services/ with `ctx.clock.now()` / `ctx.rng.*`. CI lint this going forward.

Brownfield-as-greenfield: no backward-compat shim. `ActionDb::open()` zero-arg wrapper remains for existing callers (per ADR-0116 R1.3), but new code uses `ServiceContext` explicitly.

**Gotchas:**

- DB-side `CURRENT_TIMESTAMP` defaults bypass injected clock — follow-on issue to convert to explicit `NOW()`. Document as known limitation.
- Existing process-wide singletons (`LAST_TRANSCRIPT_NOTIFICATION`, `NODE_BINARY`/`NPX_BINARY`/`NPM_BINARY`, `DISCOVERY_CACHE`, `CACHED_KEY`) need to be wrapped through `ServiceContext` or per-impl state (see MODULE-MAP refreshed section).
- Async transaction signature may need HRTB iteration — spike signature early; acceptable to ship with a simpler sync-within-async pattern if HRTB design slips.

**Acceptance:** DOS-209 spec (amended 2026-04-20) fully listed. Property test: random `ExecutionMode` + random mutation attempt → correct accept/reject.

**Parallel with:** DOS-259 (touches `intelligence/` and `services/intelligence.rs`, not `services/context.rs`), DOS-212, DOS-234, docs work.

### DOS-259 — IntelligenceProvider trait extraction

**File:** [`src-tauri/src/intelligence/provider.rs`](../../src-tauri/src/intelligence/provider.rs) (new)

Define trait per [ADR-0106 §3](../decisions/0106-prompt-fingerprinting-and-provider-interface.md):

```rust
#[async_trait]
pub trait IntelligenceProvider: Send + Sync {
    async fn complete(&self, prompt: PromptInput, tier: ModelTier)
        -> Result<Completion, Error>;
    fn provider_kind(&self) -> ProviderKind;
    fn current_model(&self, tier: ModelTier) -> ModelName;
}

pub struct Completion {
    pub text: String,
    pub fingerprint_metadata: FingerprintMetadata,
}
```

Implement for:

- `PtyClaudeCode` — migrates current PTY orchestration from `services/intelligence.rs` behind the trait.
- `GleanProvider` — refactors `intelligence/glean_provider.rs` to implement the trait.

Add a `ReplayProvider` under `#[cfg(test)]` or Evaluate-mode gating that returns canned `Completion` from fixtures.

Callers use `provider.complete(…).await?.text` for text-only needs; fingerprint-aware callers consume the full `Completion`.

**Brownfield-as-greenfield:** no `DeprecatedIntelligenceProvider` compat shim. Break all callers, migrate, ship.

**Acceptance:** DOS-259 spec fully listed. Enrichment, Trust Compiler shadow (when it exists), meeting prep all run byte-identically pre/post refactor on dev data.

**Parallel with:** DOS-209 (they touch different files).

## Phase 2 — The end-to-end slice (DOS-218)

**The proof point for v1.4.0.** One ability through every substrate surface. If this works, the substrate is real. If it doesn't, v1.4.0 shipped intentions.

### Target ability: `get_entity_context`

Chosen for:
- Highest read volume in the product.
- Read category (no mutation) — simplest contract to exercise.
- Composes most of the substrate (Reads from services, reads claims, reads trust, produces Provenance, renders through surfaces).

### Flow the slice must exercise end-to-end

1. **Services layer:** reads through `&ServiceContext` — no direct `ActionDb` access.
2. **Claim reads:** from `intelligence_claims` with default filter (`committed` + `tombstoned`; not superseded).
3. **Provenance:** populated envelope with `invocation_id`, `sources[]`, `field_attributions` for each returned field, `TrustAssessment` computed (even if Trust Compiler returns 0.5 placeholder first).
4. **Observability:** `#[ability]` macro expansion opens root tracing span; invocation_id threads through log lines, `signal_events.caused_by_invocation_id`, and any evaluation trace (when evaluator runs in Phase 3).
5. **Eval fixture:** one hand-authored fixture at `src-tauri/tests/abilities/get_entity_context/fixtures/fixture_001/` — full `state.sql` + `inputs.json` + `provider_replay.json` + `expected_output.json` + `expected_provenance.json`. CI runs it.
6. **Surface rendering:** Tauri `invoke_ability("get_entity_context", ...)` returns the output; frontend renders `Provenance` per ADR-0108 rules.
7. **Verification:** output parity between pre-migration path (direct query) and post-migration path (through ability) on ≥20 real entities in dev DB.

### Success criteria

- Single `cargo test --test abilities::read::get_entity_context` passes against the fixture.
- Manual smoke test: open 5 accounts in the dev app; briefing renders; provenance panel visible; trust band visible.
- `clippy -D warnings && cargo test && pnpm tsc --noEmit` green.
- Parallel-run divergence ≤1% across 20 entities.

**When DOS-218 lands, v1.4.0 substrate is proven.** Everything after is replication.

## Guardrails and pitfalls

### Hard rules enforced in CI

- No `Utc::now()` or `rand::thread_rng()` in ability code or new service code — use `ctx.clock.now()` / `ctx.rng`.
- All mutations through `services/` — no direct `INSERT`/`UPDATE`/`DELETE` in abilities.
- Every `#[ability]` function returns `AbilityResult<T>` with populated `Provenance`.
- `TrustAssessment` on every Transform output is non-zero (not defaulted to 0).
- `field_attributions` populated for every LLM-synthesized field.
- Prompt templates have stable ID + semver.
- Experimental abilities expire at 90 days via CI check.
- Every mutation function has `ctx.check_mutation_allowed()?` as first line.
- `clippy -D warnings && cargo test && pnpm tsc --noEmit` green before merge.

### Gotchas to avoid (all already encoded in ADRs/specs)

- **Tombstone check is PRE-GATE.** Not inside the commit policy. Ghost-resurrection fix depends on this order.
- **Trust ledger `agent_version` is composite** (`ability_version + prompt_template_version`). A prompt edit must trigger new-ledger-row trust warming.
- **Proc-macro AST inspection for category classification** catches ~95% of violations; composition-graph check + fixture-trace tests catch the rest.
- **Provenance hard cap 64 KB** per ADR-0108 amendment. Hitting it → `AbilityError::ProvenanceTooLarge`, not silent truncation.
- **PropagateSync { await_completion: true }** bounded at 500 ms. Used only for user-correction → next-briefing paths.
- **Claim commit uses pessimistic row-lock** (ADR-0113 R2). App-level `Mutex<HashMap<LockKey, ()>>`; 500 ms timeout.
- **Experimental flag waives provenance + category + fixtures** for one cycle (90 days CI-enforced). Gated behind `#[cfg(feature = "experimental")]`.
- **Signal emission + invalidation enqueue share a transaction** (ADR-0115 R1.4). Either both commit or both roll back.
- **"Content stays on laptop" is precise** — DailyOS server-side components see zero; Glean processes prompts + completions on their infrastructure (per the user's contract with Glean).
- **Publish is reporting, not team intelligence** — ADR-0121 tracks the latter as Open.
- **Auto-gen docs regenerate periodically.** `DATA-MODEL.md` and `MODULE-MAP.md` are auto-generated from the live codebase. Do NOT manually add v1.4.0 content to them until the tables/modules actually exist in code — or, extend the generators (Wave A, lane A4) to merge v1.4.0 content from a sibling file.

### When to stop and ask

Per CLAUDE.md confusion protocol:

- Two plausible architectures for the same requirement.
- A destructive operation where the scope is unclear.
- Missing context that would change approach significantly.
- Three consecutive failed attempts at a task.
- Anything that would soften an ADR boundary (ADR-0116 metadata-only especially).

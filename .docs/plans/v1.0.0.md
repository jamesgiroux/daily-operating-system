# v1.0.0 — GA: Rearchitecture + Intelligence + Launch Quality

**Thesis:** v1.0.0 is the GA release. It combines structural cleanup of the local architecture, a complete intelligence foundation, the CS report suite, and everything needed for a trustworthy, polished launch: search, offline mode, data export, privacy controls, and editorial consistency across every surface. v0.16.1 (beta hardening) and v0.16.2 (UI finesse) were dissolved and absorbed — one version, one launch, no half-measures. DailyOS stays local-first and personal. Governance and team views are solved in v1.1.0 via output-layer publication + Glean Agents, not server sync. ADR-0099 (remote-first) was proposed and withdrawn after first-principles review.

**Status:** Phases 1–3 complete. Phase 4 complete: 4a (I548-I553, I547 stashed, I550 pass 1 done/pass 2 suspended), 4b (I554-I558 done). Phase 5a: I559 done, I535 Steps 1-8 implemented but blocked on testing (4 blockers documented). I431 deferred post-1.0. Execution tracker: `.docs/plans/phase-3-execution-tracker.md`.

**Architecture audit:** `.docs/architecture/` (14 documents)

## Principles

1. **DailyOS stays local-first.** SQLite is the canonical data store. No server dependency for core operation.
2. **Intelligence is personal.** Each user's intelligence is the synthesis of Glean's org knowledge + their personal context. Not shared.
3. **Workspace files are eliminated.** DB as sole local data layer. No `_today/data/`, no `intelligence.json` on disk.
4. **ServiceLayer is mandatory.** Every mutation goes through ServiceLayer. No direct DB writes from command handlers.
5. **Sharing is at the output layer.** Reports and curated summaries are publishable. Raw signals and intelligence are private.
6. **Intelligence computation is pluggable.** DailyOS owns the personal state machine, consumption surface, and orchestration. Whether intelligence is computed locally or provided by Glean Agents, the data model, rendering, and orchestration remain the same. See `.docs/research/2026-03-04-dual-mode-intelligence-architecture.md`.

**Blocker 1: Context mode requires manual Settings toggle + app restart.** The `ContextProvider` is resolved once at `AppState::new()` and held as `Arc<dyn ContextProvider>` for the app lifetime. Connecting Glean auth (OAuth token in Keychain) does NOT automatically switch context mode to Glean. ## Phase 6: GA Readiness

**Goal:** The app is fast, stable, and trustworthy. No beach balls. No silent failures. No security gaps. This is the final pass before tagging v1.0.0 — everything before this phase built features; this phase makes them production-grade.

**Principle:** Background work must NEVER degrade UI responsiveness. A user clicking a button gets a response in < 100ms, even if the PTY is mid-enrichment, the embedding model is running, and the email pipeline is syncing. If work is in progress, tell the user — don't freeze.

### 6a: Performance — Async Runtime Health

The #1 user-reported issue: macOS beach ball during background enrichment. Root cause: PTY subprocess calls (`run_enrichment()`, `run_batch_enrichment()`) execute on Tokio async worker threads instead of blocking threads. A single enrichment blocks an async thread for 30-180 seconds, starving IPC command handlers. Combined with a single-permit `heavy_work_semaphore` shared across 6 subsystems, user-initiated actions queue behind background work indefinitely.

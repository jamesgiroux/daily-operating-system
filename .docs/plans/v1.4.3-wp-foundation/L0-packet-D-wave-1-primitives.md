## 5. What this packet authors

### 5.1 Per-primitive translation (default path)

For each primitive: run `pnpm dailyos:translate-tauri --primitive <PrimitiveName>`. The translator handles Supported-category primitives end-to-end (Pill + HealthBadge are AC §5.7 parity targets with hand-coded bodies; others get scaffold defaults that the per-primitive PR fills in).

### 5.2 Per-primitive hand-translation (where translator scaffold is insufficient)

Per W1 translator scope matrix (V1.3.1):
- **Translator parity targets (full output):** Pill, HealthBadge
- **Translator scaffolds (body TODO):** StatusDot, Avatar, IntelligenceQualityBadge, FreshnessIndicator, ProvenanceTag (Supported category, scaffold-only)
- **TrustBandBadge (SupportedWithSourcePromotion):** translator scaffolds + W2 promotes design-system status proposed→integrated
- **EntityChip + TypeBadge:** translator-NotSupported by default (editable variants); for W2, restrict to **display-only** variants and use `pnpm dailyos:new-block --template typed-display` manually. Document the editable-variant deferral.

Per-primitive deliverable: `wp/dailyos/blocks/<slug>/{block.json,render.php,render-functions.php,style.css,edit.js,editor.css}` + matching `src-tauri/abilities-runtime/tests/fixtures/<slug>_integration_fixture.rs`.

### 5.3 Fold v1.4.10-dissolved UX patterns

Per wave plan §183, three previously-parked v1.4.10 tickets dissolve into W2 as block-level enhancements:
- **DOS-9** (cite-chip age + freshness tooltip): folded into `ProvenanceTag` block as the `data-age` + tooltip attributes.
- **DOS-11** (trust-band UI): folded into `TrustBandBadge` block as the primary visual semantic. Already named-in-matrix.
- **DOS-325** (score bands + evidence drill-down): may produce a new `ScoreBand` primitive — TBD per design-reviewer L0 verdict. If yes, add as 11th primitive (the matrix expands by one); if a refinement of existing primitives, fold into `HealthBadge` or `IntelligenceQualityBadge`.

### 5.4 Implementation fan-out shape

Per wave plan §185: 2-up parallel codex per primitive (worktree-isolated). 5 sequential rounds × 2 primitives = 10 primitive PRs (or grouped into 2-3 commit-batches if PR overhead is excessive).

Per primitive, agent reads:
1. Existing TSX (`src/components/<sub>/<Name>.tsx`)
2. Existing CSS Module (`.module.css`)
3. Reference HTML (`.docs/design/reference/_shared/primitives.css` for the primitive's classes)
4. Design spec (`.docs/design/primitives/<Name>.md`)

Then: run translator (or new-block CLI if NotSupported variant), customize per spec, drop matching Rust fixture, run `cargo test -p abilities-runtime --test block_kit_integration_harness` to confirm parity. Commit.

## 6. Directional decisions resolved at L0

### 6.1 Single batched packet vs per-primitive packets

Single batched L0 packet (this one). Per-primitive packets would 10x the L0 review burden with diminishing returns — the translation pattern is identical across primitives. Per-primitive divergence handled in this packet's §5.2.

### 6.2 EntityChip + TypeBadge: display-only in W2

Both have editable variants per .docs/design/primitives/. Editable variants need the W4 feedback-write infrastructure (click-bound router + nonce). W2 ships display-only; editable variants defer to W4 follow-up tickets.

### 6.3 TrustBandBadge proposed→integrated promotion

Per primitives README, TrustBandBadge is currently `proposed`. The W2 WP block is the first integrated consumer; promote to `integrated` in the same PR (markdown-only doc change to README + create `.docs/design/primitives/TrustBandBadge.md` proper if missing). Per primitives README "Adding a primitive that already exists in src/", promotion is markdown-PR only — no code consolidation required.

### 6.4 ScoreBand: defer to design-reviewer L0 verdict

If design-reviewer says ScoreBand is genuinely distinct from HealthBadge + IntelligenceQualityBadge, add to matrix. If not, fold into one of them. Resolved before implementation starts.

### 6.5 Visual parity standard

Visual parity vs Tauri React: hero-state side-by-side screenshots at L4. CSS values may differ where W1 token generator + theme.json mapping forces them to (palette slug naming changed `--color-spice-turmeric` → `var(--wp--preset--color--spice-turmeric)`); design-reviewer judges acceptable drift per primitive.

## 7. Acceptance criteria

1. **All 10 (or 11 if ScoreBand) primitives ship as `wp/dailyos/blocks/<primitive>/` directories** with block.json, render.php, render-functions.php, style.css, edit.js, editor.css.
2. **Each block has a matching Rust integration fixture** in `src-tauri/abilities-runtime/tests/fixtures/<primitive>_integration_fixture.rs`. The W1 CI gate enforces this.
3. **All 10 (or 11) fixtures pass** the `block_kit_integration_harness` test in a single `cargo test` run.
4. **Visual parity to Tauri React** validated at L4 hands-on: side-by-side screenshots (Tauri rendering vs WP block rendering at default + 1 hero variant per primitive). Design-reviewer signs off.
5. **TrustBandBadge promoted** in `.docs/design/primitives/README.md` from `proposed` → `integrated`. `.docs/design/primitives/TrustBandBadge.md` exists with full spec.
6. **DOS-9 + DOS-11 + DOS-325 closed** as work folded into W2 (or, for DOS-325, either folded or producing a new primitive per §6.4).
7. **CI green** on the combined W2 PR(s): `cargo clippy -p abilities-runtime --all-targets -- -D warnings` + `cargo test -p abilities-runtime --test block_kit_integration_harness` + `block-kit-integration.yml` workflow.
8. **No edits to W1 substrate** (CLI / templates / translator / harness / CI workflow). Anything that *should* change there is filed as a W1-amendment ticket and resolved before W2 lands.

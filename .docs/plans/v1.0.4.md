# v1.0.4 — Test & Trust

**Thesis:** v1.0.3 shipped the Meeting Record — the crown jewel page. But the retro exposed two structural gaps: PostMeetingIntelligence was rewritten 12 times with 6 sequential bug fixes and zero frontend tests, and mock data used wrong table names and missing fields because nothing validates it against the schema. This is a hardening release: lock down what shipped with tests, and prevent mock data from drifting silently.

**Status:** Not started. No dependencies — can run in parallel with v1.1.0.

**Branch model:** Short-lived per-issue branches from `dev`.

**Origin:** Engineering retro (2026-03-22). Recommendations 1 and 2.

---

## Principles

1. **Tests are the cheapest insurance.** Every bug fixed without a regression test will come back.
2. **Mock data is documentation.** If it doesn't match the real schema, it's lying about what the product does.
3. **Small release, high leverage.** No features. Just structural quality that makes every future release safer.

## Phase 1: Frontend Component Tests

**Goal:** PostMeetingIntelligence never ships with a badge/filter/lifecycle bug again.

| ID | Title | Notes |
|----|-------|-------|
| **I638** | PostMeetingIntelligence test suite | Vitest + @testing-library/react. Test the component renders correctly for each action state (suggested/pending/completed/archived), that commitments are hidden when actions exist, that Lucide icons render (not unicode), that role changes show old→new transition, and that the prediction scorecard categorizes correctly. |

**Acceptance criteria:**

1. Test file: `src/components/meeting/PostMeetingIntelligence.test.tsx`
2. Tests cover:
   - Suggested actions render with SUGGESTED pill + Accept/Dismiss buttons
   - Pending actions render with PENDING pill + Done button
   - Completed actions render with check icon + strikethrough + Reopen button
   - Archived actions are NOT rendered
   - Commitments section hidden when actions array is non-empty
   - Commitments section shown when actions array is empty
   - Role changes render name + old status → new status + evidence
   - Prediction scorecard renders confirmed (Check icon), surprise (Zap icon), not-raised (X icon)
   - Speaker sentiments render name + sentiment badge + evidence
   - Talk balance bar renders customer/internal percentages
3. All tests pass: `pnpm test`
4. No snapshot tests — behavioral assertions only

## Phase 2: Mock Data Validation

**Goal:** Mock data never silently drifts from the real schema.

| ID | Title | Notes |
|----|-------|-------|
| **I639** | Mock data schema validation | A Rust integration test (or shell script) that parses INSERT statements from `demo.rs` and `devtools/mod.rs`, extracts table names and column names, and validates them against the current SQLite schema. Fails if a table or column doesn't exist. |

**Acceptance criteria:**

1. Validation runs as part of `cargo test` (preferred) or as a standalone script in `scripts/`
2. Catches the exact bugs from v1.0.3:
   - Table name mismatch: `enriched_captures` vs `captures` (the table was renamed in migration 070)
   - Missing column: INSERT with columns that don't exist in the current schema
   - Extra columns: INSERT with more values than the table has columns
3. If implemented as a Rust test: lives in `src-tauri/src/db/mod_tests.rs` or `src-tauri/src/demo.rs`
4. If implemented as a script: `scripts/validate_mock_data.sh` runs in CI via `test.yml`
5. Validates both `demo.rs` (onboarding demo) and `devtools/mod.rs` (full mock scenario)
6. Clear error message: "Mock data references table 'enriched_captures' which does not exist. Did you mean 'captures'?"

## Phase 3: Backend Integration Tests

**Goal:** The Meeting Record intelligence pipeline is tested end-to-end.

| ID | Title | Notes |
|----|-------|-------|
| **I640** | Meeting Record integration tests | Rust integration tests using in-memory SQLite. Seed a meeting with captures, dynamics, champion health, and actions. Verify the temporal stage detection, prediction scorecard computation, continuity thread queries, and markdown generation produce correct output. |

**Acceptance criteria:**

1. Test the prediction scorecard:
   - Jaccard similarity correctly classifies confirmed vs not-raised vs surprise
   - Empty prep or empty outcomes returns None (not empty scorecard)
   - Bayesian feedback emitted for confirmed predictions
2. Test the continuity thread:
   - Previous meeting found for same entity
   - First meeting returns `is_first_meeting: true`
   - Actions completed between meetings correctly counted
   - Health delta computed from `health_score_history`
   - New attendees detected via meeting_attendees diff
3. Test the meeting record markdown:
   - YAML frontmatter contains required fields
   - Sections present when data exists, absent when data is empty
   - File routing follows entity priority (account > project > person > archive)
4. All tests in `src-tauri/src/` — no new test infrastructure needed
5. Tests use the existing `test_db()` helper for in-memory SQLite
